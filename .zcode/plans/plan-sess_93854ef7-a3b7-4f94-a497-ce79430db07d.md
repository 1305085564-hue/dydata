二、已确认的根因
1. `src/lib/content-attribution.ts` 里 `avg_play_duration` 现在按绝对秒数 `<=8 / <=20 / >20` 硬分段，和规格冲突。
2. `src/app/api/admin/content-attribution/[videoId]/route.ts` 目前没有查询 `avg_play_ratio`，导致归因链路根本拿不到正确分段依据。
3. `content-attribution.test.ts` 现有 3 条测试把错误口径固化了。
4. `content-attribution` 与 `content-comparison` 两个接口把 self/team/top/user 参照系查询重复写了两遍。
5. team/top 的“今日”边界都在用 `new Date().setHours(0,0,0,0)`，线上会跟服务器时区走，不是上海日历日。
6. `content-comparison` 的 legacy `previous/recent3` 现在无论 `ref` 是什么都会查询，存在无谓数据库往返。

三、实现方案

A. 修正归因定位口径（P0-1）
改动文件：
- `src/lib/content-attribution.ts`
- `src/lib/content-attribution.test.ts`
- `src/app/api/admin/content-attribution/[videoId]/route.ts`
- 共享参照系模块（见 C）

具体做法：
1. 扩展归因内部 `SnapshotRow/MetricRow`，新增 `avg_play_ratio: number | null`，但只作为“定位辅助字段”。
   - 它不进入 `METRIC_MAP`、不进入 findings 指标项、不改变现有 10 指标展示面。
2. 将 `computeLocate()` 的 `avg_play_duration` 分支改为：
   - `avg_play_ratio < 1/3` → `opening`
   - `1/3 <= avg_play_ratio <= 2/3` → `middle`
   - `avg_play_ratio > 2/3` → `ending`
   - `seconds` 继续展示 `avg_play_duration`
3. 当 `avg_play_ratio === null` 时：
   - 返回 `kind: "segment"`
   - `segment_hint: null`
   - `seconds: avg_play_duration ?? null`
   也就是老实承认“只能定位到整条视频整体，不能再细分段”。
4. 保持“归因结论不用 AI”不变；这一块只改确定性规则的口径，不接任何 AI 调用。

测试计划：
- 改掉现有错误测试：
  - “avg_play_duration <= 8s → opening”
  - “avg_play_duration > 20s → ending”
  - “缺总时长用 avg_play_duration 估算段位”
- 新增/替换为：
  - `ratio=0.2 -> opening`
  - `ratio=0.5 -> middle`
  - `ratio=0.8 -> ending`
  - `ratio=null -> segment_hint=null，seconds 保留展示值`
- 保留并复测其他 tone / missing / 排序测试，确保只改定位逻辑，不误伤归因排序与好坏判定。

B. 修正“今日团队/今日高手”上海时区边界（P0-2）
改动文件：
- 共享参照系模块（见 C）
- `src/app/api/admin/content-comparison/[videoId]/route.ts`
- `src/app/api/admin/content-attribution/[videoId]/route.ts`

具体做法：
1. 不再在两个接口里写 `new Date().setHours(...)`。
2. 统一在共享参照系查询里，用项目现成 `getShanghaiDate()` 生成当日上海日期字符串。
3. 今日起点按你指定的方式构造：
   - `new Date(`${getShanghaiDate()}T00:00:00+08:00`).toISOString()`
4. team/top 查询一律使用这个 ISO 起点做 `.gte("published_at", ...)`。

测试计划：
- 为共享参照系逻辑补一个“日期边界”测试：
  - 用模拟的 UTC 凌晨时刻（对应北京时间上午）断言生成的“今日起点”仍是上海日历日的 00:00，而不是服务器本地午夜。
- 目标不是测浏览器/系统时区，而是测“我们拼出来的查询起点字符串是否始终按上海日历日”。

C. 抽共享参照系查询，消除两接口重复（P1-4）
拟新增文件：
- `src/lib/content-comparison-reference.ts`

具体做法：
1. 把现在散落在两个 route 里的共用部分收口到新模块：
   - `SNAPSHOT_SELECT`（补 `avg_play_ratio`）
   - `MetricRow` / `SnapshotData`
   - `toMetricRow()`
   - `averageMetricRows()`
   - `buildSnapshotMap()`（如果 comparison 仍需要）
   - `getReferenceMetrics(...)` 作为主入口
2. `getReferenceMetrics(...)` 的输入预计为：
   - `supabase`
   - `videoId`
   - `video`（至少含 `account_id / user_id / published_at`）
   - `ref`
   - `refUserId`
3. 输出预计为：
   - `referenceRows: MetricRow[]`
   - `reference: MetricRow | null`
   - `refLabel: string`
   - `refCount: number`
4. 四种 `ref` 行为保持现状语义不变：
   - `self`：当前账号、早于当前视频、近 3 条
   - `team`：团队今日视频均值
   - `top`：团队今日最高播放那条
   - `user`：指定人近 3 条（保持现状，不额外改成“早于当前视频”）
5. `content-attribution` 改为只负责：
   - 取当前 24h snapshot
   - 调共享参照系函数
   - 把 current/reference 送进 `computeAttribution`
6. `content-comparison` 改为只负责：
   - 取当前 24h snapshot
   - 调共享参照系函数
   - 组装 `{ current, reference, ref, ref_label, ref_count }`
7. legacy `previous/recent3` 兼容字段保留，但只在 `ref === "self"` 时查询；其余 ref 直接返回 `null`，避免多余数据库往返，同时不炸现有前端类型。

测试计划：
- 为新共享模块补单测，覆盖：
  - self / team / top / user 四种 ref 的分支行为
  - `ref=user` 缺 `refUserId` 仍由 route 层返回 400（route 层原校验保留）
  - `top` 返回单条参照且 `refCount=1`
  - `comparison` 仅在 `self` 时走 legacy previous/recent3
- 如果现有仓库没有成熟 route 单测，我会优先把“最容易回归、最关键的行为”放进共享模块测试里，route 只做轻量组装。
