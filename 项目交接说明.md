# DYData 项目交接说明

这是一份给“接手 DYData 的人”看的总文件。

目标只有两个：

1. 人能看懂这个项目现在在云端哪里、怎么维护。
2. AI 打开这份文件后，知道应该先看什么、按什么规则改，尤其是上传链路不要乱动。

---

## 一句话说明

DYData 是一个“抖音数据日报平台”。

员工每天在网页里提交数据，系统把数据写进 Supabase，再通过 Vercel 对外提供网页和接口。

## 这个项目怎么理解

可以把它理解成 3 层：

### 第 1 层：前台页面

员工和管理员打开网页，填写、查看、分析数据。

最重要页面：

- `/dashboard`：员工提交日报
- `/growth`：成长分析
- `/analytics`：数据分析
- `/admin`：管理后台
- `/admin/analytics`：经营分析

### 第 2 层：接口

前台表单提交后，会调用接口写库。

最关键接口：

- `src/app/api/video-submit/route.ts`
  这是“上传截图并提交数据”的主接口。
- `src/app/api/remind/route.ts`
  这是“日报提醒”的接口。

### 第 3 层：数据库

数据最终写到 Supabase。

数据库改动统一放这里：

- `supabase/migrations/`

规则：

- 只允许新增 migration
- 不允许修改旧 migration

## 现在维护时最重要的规则

### 1. 部署只走 Vercel

这是硬规则。

不要因为仓库里看到 `wrangler`、`open-next`、Cloudflare 相关残留文件，就误以为项目该走 Cloudflare Workers。

正确做法：

- 网站部署看 Vercel
- 域名解析看 Cloudflare

### 2. 数据库变更只能新增 migration

不要改旧 migration。

正确做法：

1. 在 `supabase/migrations/` 新增一个新文件
2. 推到远端 Supabase
3. 再验证页面

### 3. 不要直接改线上环境变量

环境变量要在 Vercel / Supabase 对应后台里维护，不是在代码里硬改。

### 4. 上传链路不要凭感觉改

上传出问题时，先看：

- 页面：`src/app/(app)/dashboard/`
- 接口：`src/app/api/video-submit/route.ts`
- 数据库 migration 是否已经执行

不要只改前端提示文字，不查数据库规则。

## 上传功能现在的真实规则

这是最容易出错的地方，单独写清楚。

### 上传入口

- 页面提交后，最终走 `src/app/api/video-submit/route.ts`

### 上传时会做什么

接口会同时处理几类东西：

1. 视频基础信息
2. 24h 数据快照
3. 日报数据
4. 视频标签

### 内容标签规则

上传界面里“内容标签”最多 3 个。

规则是：

- 每个词都是 1 条独立标签
- 每个词建议不超过 4 个字
- 最多 3 个

例如：

- `删掉`
- `但是`
- `说的`

这是 3 条独立标签，不是 1 条长字符串。

### 关键词相关数据库规则

现在数据库已经按这个逻辑修过：

- `关键词`：同一视频可以保存多条
- `话题`：同一视频只保留 1 条
- `题材 / 表达形式 / CTA类型`：同一视频各保留 1 条

对应迁移：

- `supabase/migrations/043_video_tags_keyword_multi_value.sql`

如果以后上传又报 `ON CONFLICT` 一类错误，先检查：

1. 远端 Supabase 有没有执行最新 migration
2. 接口是不是又把多个关键词错误合并或错误 upsert 了

## 本地启动最少需要什么

先安装依赖：

```bash
npm install
```

启动：

```bash
npm run dev
```

至少要有这些环境变量：

```bash
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
FEISHU_WEBHOOK_URL=
CRON_SECRET=
REMIND_SECRET=
AI_BASE_URL=
AI_API_KEY=
AI_MODEL=
```

当前约定：

- `AI_BASE_URL=https://www.aiapikey.net`
- `AI_MODEL=claude-sonnet-4-6`

## 接手的人第一天该做什么

### 如果是人来维护

先做这几步：

1. 看这份文件
2. 看 `README.md`
3. 看 `CLAUDE.md`
4. 确认自己有 GitHub / Vercel / Supabase / Cloudflare 权限
5. 本地跑起来
6. 打开线上站点，确认首页、登录、提交页、后台都能打开

### 如果是 AI 来接手

AI 先读这些文件：

1. `项目交接说明.md`
2. `CLAUDE.md`
3. `README.md`
4. `docs/数据口径.md`

AI 必须遵守这些规则：

- 部署平台按文档认定为 Vercel，不能自作主张改平台
- 改数据库只能新增 migration
- 上传问题优先检查接口 + migration，不要只猜前端问题
- 看到历史残留配置，不能自行推断这是当前正式方案
