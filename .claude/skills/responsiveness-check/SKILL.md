# 响应式检查

测试网站布局如何响应视口宽度变化。在单个浏览器会话中依次调整断点宽度、截图、对比相邻尺寸，并报告布局断裂的位置。

**测试范围**：布局响应式 —— 溢出、堆叠、导航切换、内容重排。

**不测试的内容**：通用无障碍（ARIA、语义化 HTML、标题层级、色彩对比度）。这些不随视口宽度变化 —— 请改用 ux-audit skill。

## 浏览器工具优先级

开始前，检测可用的浏览器工具：

1. **playwright-cli**（首选）—— 支持 resize、命名会话和子代理并行。如已安装，先运行 `/playwright-cli` 加载完整命令参考。
2. **Playwright MCP** (`mcp__plugin_playwright_playwright__*`) —— `browser_resize` 用于视口调整。
3. **Chrome MCP** (`mcp__claude-in-chrome__*`) —— `resize_window` 用于视口调整。使用用户已登录的 Chrome 会话。

如果以上都不可用，告知用户并建议安装 playwright-cli 或 Playwright MCP。

## 运行模式

### 模式 1：标准检查

**何时使用**："check responsive"、"responsiveness check"、"test breakpoints"

测试覆盖设备谱系的 8 个关键断点：

| 宽度 | 设备场景 |
|-------|---------------|
| 320px | 小屏手机 (iPhone SE) |
| 375px | 标准手机 (iPhone 14) |
| 768px | 平板竖屏 (iPad) |
| 1024px | 平板横屏 / 小笔记本 |
| 1280px | 笔记本 |
| 1440px | 桌面显示器 |
| 1920px | 全高清 |
| 2560px | 超宽 / 4K |

**流程**：

1. 在单个浏览器会话中打开 URL（高度：900px）
2. 从 320px 开始。对每个断点宽度：
   a. 调整视口大小
   b. 短暂等待 CSS 重排（布局过渡）
   c. 截取首屏区域
   d. 如页面有显著的下拉内容，滚动后截图
   e. 执行 8 项布局检查（见下方矩阵）
   f. 记录问题及严重程度
3. 对比相邻宽度 —— 识别布局切换发生的位置
4. 撰写报告

### 模式 2：全面扫描

**何时使用**："responsive sweep"、"sweep all breakpoints"、"find where it breaks"

从 320 到 2560 每 160px 测试一次（共 15 个宽度）。与标准模式相同的单会话方式 —— 只是数据点更多。此模式用于找到布局断裂的**精确宽度**。

宽度：320, 480, 640, 800, 960, 1120, 1280, 1440, 1600, 1760, 1920, 2080, 2240, 2400, 2560

开始扫描模式前**简要确认**（15 张截图是一个有意义的工作量）。

### 模式 3：指定范围

**何时使用**："check between 768 and 1024"、"test tablet breakpoints"、"focus on mobile widths"

以 80px 为增量测试用户指定范围。用于已知问题区域需要详细排查时。

示例："check between 768 and 1024" 测试：768, 848, 928, 1008（加上 1024 作为终点）。

### 多 URL

测试多个 URL 时（如 "check the homepage, about page, and contact page"）：

- 启动**并行子代理**，每个 URL 一个（不是每个断点一个）
- 每个子代理在其命名会话中对各自 URL 运行标准检查
- 合并结果到单个报告

```
# 子代理模式 (playwright-cli)
playwright-cli -s=page1 open https://example.com/ &
playwright-cli -s=page2 open https://example.com/about &
```

## 布局检查矩阵

这 8 项检查针对**实际随视口宽度变化**的问题：

| # | 检查项 | 关注点 |
|---|-------|-----------------|
| 1 | **水平溢出** | 内容超出视口宽度 —— 出现水平滚动条、元素被裁切 |
| 2 | **文本溢出** | 文本在单词中间截断、与相邻元素重叠、字号不可读 (< 12px) |
| 3 | **导航切换** | 汉堡菜单在正确宽度出现/消失，模式之间没有"损坏"状态 |
| 4 | **内容堆叠** | 多列布局在窄屏按逻辑阅读顺序堆叠为单列 |
| 5 | **图片/媒体缩放** | 图片溢出容器、宽高比失真、缺少响应式尺寸 |
| 6 | **触控目标** | 移动端 (< 768px) 交互元素 < 44px —— 按钮、链接、表单输入框 |
| 7 | **留白平衡** | 移动端过于拥挤（没有呼吸空间），宽屏过于稀疏（内容迷失在空白中） |
| 8 | **CTA 可见性** | 主要行动号召在每个宽度下无需滚动即可在首屏可见 |

## 过渡检测

此 skill 的独特价值在于**找到布局切换发生的位置**以及是否干净。

对比相邻宽度的截图时，标记任何出现以下变化的宽度：
- **列数变化**（3 列 → 2 列 → 1 列网格）
- **导航模式切换**（完整导航 → 汉堡菜单，或相反）
- **侧边栏出现/消失**（内容宽度跳跃）
- **网格重排**（卡片换行到下一行）

报告每个过渡发生的**精确宽度范围**：

| 过渡 | 从 | 到 | 宽度范围 |
|-----------|------|-----|-------------|
| 导航：汉堡 → 完整 | 768px | 1024px | 在 ~960px 切换 |
| 网格：1 列 → 2 列 | 640px | 768px | 在 ~700px 重排 |
| 侧边栏出现 | 1024px | 1280px | 在 ~1100px 显示 |

这告诉开发者应该在何处设置（或修复）CSS 断点。

## 严重程度等级

与 ux-audit 保持一致：

| 严重程度 | 含义 |
|----------|---------|
| **严重** | 布局已损坏 —— 内容不可读、导航不可用、页面无法使用 |
| **高** | 显著布局问题 —— 大量溢出、关键内容隐藏、过渡断裂 |
| **中** | 明显但可用 —— 间距尴尬、轻微溢出、堆叠顺序次优 |
| **低** | 打磨 —— 留白微调、轻微对齐问题、触控目标小幅不足 |

## 自主规则

- **直接执行**：调整视口、截图、分析布局、对比宽度
- **简要确认**：扫描模式前（15 个视口）、并行测试 4+ URL 前
- **先询问**：与表单交互或点击通过认证流程前
