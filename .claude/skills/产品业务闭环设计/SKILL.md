# 产品业务闭环设计 Skill

## 0. 核心防错三条

违反即退回重写：

1. **阶段无边界 → 不进入页面设计。**
2. **字段无归属 → 不进入页面。**
3. **角色无闭环 → 不交付方案。**

页面不是字段容器，页面是某个角色在某个阶段完成某个动作的工具。

## 1. 输出模式

| 模式 | 使用场景 | 输出要求 |
|---|---|---|
| 标准模式 | 常规模块重构 | 关键判断、必要表格、执行动作 |
| 深度模式 | 复杂核心模块 | 完整输出 |
| 焦点模式 | 只改某个页面或字段 | 只围绕指定问题，不扩散 |
| 审查模式 | 检查已有设计 | 只指出问题，不重写全案 |

**能删则删。不要为了完整而堆表格。**

## 2. 业务定位

前置判断，回答：

| 问题 | 回答 |
|---|---|
| 这个模块服务谁？ | |
| 用户为什么要进入这个模块？ | |
| 用户进入时处于哪个业务阶段？ | |
| 当前用户已经知道什么？ | |
| 当前用户此刻能做什么？ | |
| 当前用户不应该看到什么？ | |
| 操作完成后流向哪里？ | |
| 结果如何被验证？ | |

**字段归属表出现 3 个以上「来源不明」→ 退回角色与阶段重来。**

## 3. 角色与权限边界

| 角色 | 使用场景 | 核心动作 | 可见内容 | 可操作内容 | 禁止内容 |
|---|---|---|---|---|---|

角色至少：普通用户 / 员工、管理者 / 组长、负责人、owner / 系统运营者、AI / 系统自动能力。

原则：用户填事实、管理者审权限内内容、AI 只建议不结论、owner 管规则配置审计。

## 4. 阶段流转

先拆状态流转，再设计页面。

| 阶段 | 目标 | 责任人 | 输入 | 输出 | 下一步 |
|---|---|---|---|---|---|

标准链路：用户进入 → 用户提交 → 系统接收 → AI 初判 → 人工审核 → 结果反馈 → 用户下一步 → 结果验证 / 复测 → 资产沉淀

## 5. 字段归属校验

新模块完整输出。审查已有模块：只列问题字段、高风险字段、易混阶段字段。

| 字段 | 所属阶段 | 字段来源 | 填写 / 生成角色 | 展示对象 | 是否允许出现在当前页面 | 原因 |
|---|---|---|---|---|---|---|

7 类来源：

| 来源 | 说明 |
|---|---|
| 用户提交字段 | 用户当前已知事实 |
| 系统生成字段 | 系统计算 / 抓取 / 记录生成 |
| AI 初判字段 | 推测 / 归类 / 建议，不得直接作为最终结论 |
| 管理员审核字段 | 仅审核阶段填写或确认 |
| 结果反馈字段 | 仅审核确认后展示给用户 |
| 复测字段 | 仅二次或后续验证阶段出现 |
| 资产沉淀字段 | 仅进入案例库 / 规则库 / 经验库 |

**字段无阶段归属、无填写角色、无展示对象 → 不允许进入页面。**

## 6. 页面职责设计

每个页面只有一个主动作。

| 页面 | 服务角色 | 所属阶段 | 主动作 | 展示内容 | 可操作内容 | 禁止出现内容 | 下一步 |
|---|---|---|---|---|---|---|---|

职责对应：

| 页面类型 | 主动作 |
|---|---|
| 提交页 | 提交事实和证据 |
| 审核页 | 审核、修改、确认 |
| 反馈页 | 查看结论、执行下一步 |
| 复测页 | 验证改进结果 |
| 资产页 | 沉淀、检索、复用 |

多阶段同一页面：拆分职责区，区分只读 / 可编辑 / 可审核 / 可确认。**禁止用 tab、折叠、隐藏掩盖阶段混乱。**

## 7. 多角色闭环

| 角色 | 第一眼看到什么 | 优先处理什么 | 如何得到 / 下发反馈 | 如何追踪结果 | 如何沉淀资产 |
|---|---|---|---|---|---|

必须回答：普通用户知道下一步做什么？管理者知道该处理谁、处理什么、怎么反馈？负责人看到趋势、异常和结果？owner 能维护权限规则配置审计？AI 失败能降级人工处理？

## 8. AI 与人工边界

| 环节 | AI 能做什么 | 人必须确认什么 | AI 失败时如何降级 |
|---|---|---|---|

AI 原则：可以初判，不能终判。可以生成建议，不能自动处罚。可以提炼案例，不能自动入库。输出可查看依据、可修改、可驳回、可追溯。涉及评价、处罚、权限变更、绩效、对外反馈 → 必须人工确认。**必须允许人工跳过 AI，直接审核和反馈。**

## 9. MVP 取舍

| 类型 | 内容 | 原因 |
|---|---|---|
| V1 必须做 | 只保留跑通最小闭环所需功能 | |
| V1 暂不做 | 不影响闭环但增加复杂度的功能，最多 3 条 | |
| 明确禁止做 | 造成角色、阶段、字段混乱的功能 | |
| V2 再考虑 | 提升效率或体验的增强功能 | |

V1 最小闭环：提交 → 系统接收 → AI 初判 → 人工审核 → 反馈 → 执行 → 验证 → 基础沉淀。**不得默认保留所有功能。**

## 10. 开发交付清单

| 交付对象 | 必须拿到什么 |
|---|---|
| 产品 | 业务阶段、角色权限、字段归属、页面职责、MVP 范围 |
| UI / 交互 | 页面主动作、展示内容、禁止内容、状态流转 |
| 前端 | 页面入口、字段、按钮、状态提示、权限展示 |
| 后端 | 数据实体、字段来源、状态机、权限校验、日志审计 |
| AI 能力 | 输入字段、输出字段、判断依据、人工确认点、失败降级方案 |
| 运营 | 配置项、审计项、案例沉淀规则、异常处理方式 |
| 验收 | 每个阶段的成功标准和失败标准 |

## 11. 最终自检

自检基于第 0 步三条核心防错原则。6 条检查是展开形式：如果某条不通过但三条核心原则仍然满足 → 影响有限，可继续；如果三条核心原则有一条被违反 → 该部分必须重写。

| 自检项 | 不通过如何处理 |
|---|---|
| 是否先判断业务阶段，而不是先设计 UI？ | 重写业务定位 |
| 是否每个角色都有明确动作和边界？ | 重写角色权限 |
| 是否把提交、AI 初判、人工审核、反馈、复测、沉淀拆开？ | 重写阶段流转 |
| 是否每个关键字段都有来源、阶段、角色、展示对象？ | 重写字段归属 |
| 是否每个页面只有一个主动作？ | 拆分页面或职责区 |
| 是否做了 MVP 取舍，而不是默认全保留？ | 重写 MVP 范围 |

## 12. 最终判定

> 这个模块现在是「可用产品模块」，还是「功能外壳」。

| 判定 | 标准 |
|---|---|
| 可用产品模块 | 角色清楚、阶段清楚、字段归属清楚、页面职责清楚、权限边界清楚、结果可验证、开发可执行 |
| 功能外壳 | 只有页面和功能，没有状态流转、字段归属、权限边界、结果验证和开发落点 |
