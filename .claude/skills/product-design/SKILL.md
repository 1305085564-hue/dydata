# 产品工作台设计 V4

核心标准：产出的不是「对的系统」，是「人用它能几下办完事、过程不慌不卡的产品」。正确≠好用，好用≠敢用。本 Skill 同时卡三件——业务对不对、人快不快、人安不安心，缺一退回。

先判断模式再输出，不默认全展开。

| 模式 | 场景 | 输出 |

## 战略层：该不该做、做成什么样

### 0. 场景分流

拿到需求后第一个判断：这是新建还是重构？

**新建**（从无到有加功能/页面）→ 走 1A 证据审计路径
**重构**（已有页面/功能要改）→ 走 1B 现状定性路径

### 1A. 需求证据审计（新建场景）

AI 不判断需求真假，AI 逼你拿证据。

强制追问（缺任何一条标记"未验证需求"）：

| 问题 | 目的 |
|---|---|
| 谁会用？具体是什么角色/什么人？ | 锁定用户 |
| 他现在没这个功能，怎么解决的？ | 判断替代方案是否够用 |
| 多久遇到一次？每天/每周/偶尔？ | 判断频率 |
| 举一个具体卡住的场景 | 验证痛点真实性 |
| 不做会怎样？最坏结果是什么？ | 判断紧迫性 |

判定规则：
- 5条全答得上 → 高置信需求，继续
- 答出3-4条 → 中置信需求，继续但标记"需上线后验证"
- 答不出3条 → 未验证需求，可以往下走但自动降到V2优先级，不进V1

### 1B. 现状定性（重构场景）

已有页面/功能要重构时，先给旧版定档，再决定改法。

用四档判定（见底部判定标准）给当前页面打档，然后回答：
- 当前卡在哪？（业务闭环断？角色越权？首屏没重点？路径断裂？状态缺失？）
- 哪些要保留？（已跑通的闭环、用户已习惯的操作、数据结构）
- 哪些必须砍/改？（低频占核心位置、主次混乱、冗余字段）
- 用户/竞品已有的心智模型是什么？（用户之前用什么解决同样问题？那个东西的交互范式你要沿用还是打破？）

定档结论决定改法：
- 顶级工作台雏形 → 焦点模式，只改具体问题
- 可用产品模块 → 局部重构，保留骨架改交互 and 流转
- 功能外壳 → 大重构，保留数据结构重做产品层
- 能跑的垃圾 → 推倒重来，只保留业务逻辑不保留页面设计

### 2. 成功判据

不定北极星指标，定**页面/模块级的成功标准**。

AI 基于场景建议 3-5 条判据，用户挑选确认。判据要求：
- 具体到可感知（"处理一条反馈从10分钟降到2分钟"，不是"提升效率"）
- 允许定性（"用户不再需要问客服怎么操作"也算）
- 必须有失败标准（"如果上线后X仍然发生，说明没解决"）

示例格式：
| 成功 | 失败 |
|---|---|
| 管理者打开页面5秒内知道先处理谁 | 管理者还是需要翻完整个列表才能找到重点 |
| 用户提交后能看到进度，不再反复追问 | 用户提交完不知道谁在处理、什么时候有结果 |
| 一条反馈从收到到处理完≤3步操作 | 处理一条反馈仍需要跳转3个以上页面 |

**⏸ 硬停点 1：到这里必须停下等用户确认。确认"做不做"和"做成什么样算好"对齐后，才继续。**

## 战术层：怎么设计

### 3. 设计原则锚定

不同项目优先级不同。开始设计前，锚定本项目的 3-5 条设计原则。

AI 根据场景建议，用户确认/调整。常见原则池（选适用的，不全选）：

| 原则 | 含义 | 典型场景 |
|---|---|---|
| 效率优先 | 最少步骤完成任务 | 内部后台、工作台 |
| 信任优先 | 每步都让用户确认掌控 | 金融、医疗、权限操作 |
| 发现优先 | 引导探索，不强制路径 | 内容平台、创作工具 |
| 容错优先 | 任何操作可撤回可恢复 | 数据操作、批量处理 |
| 一致优先 | 同类操作永远同样模式 | 复杂系统、多模块产品 |
| 渐进披露 | 先简后繁，复杂按需展开 | 新手多、功能密集 |

锚定后，后续所有设计决策如果与原则冲突，必须标注并说明理由。

### 4. 业务主线与角色

先定主线再定角色，没主线的页面是功能堆砌。

**主线六问**（一句话答）：
1. 这个页面/模块的定位是什么？
2. 核心闭环是什么？（从什么到什么算完）
3. 用户最高频动作是什么？
4. 管理者最高价值动作是什么？
5. 最重要的结果指标是什么？
6. 最该降级到后台的功能是什么？

**角色只三类**，不因权限差异生造角色：

| 角色 | 核心动作 | 可操作 | 禁止 |
|---|---|---|---|
| 用户 | 提交事实、看反馈、执行改进、验证 | 提交、查看、确认、补充、复测 | 看他人数据、审核他人、改规则 |
| 管理者 | 审核、反馈、处理异常、追踪 | 审核、改反馈、下发、追踪、处理异常 | 越权看非管辖范围、改系统规则 |
| 运营者 | 管权限、配规则、查审计、控系统健康 | 权限管理、规则配置、审计查询 | 代业务角色做日常操作 |

边界规则：组长/负责人/Owner 是权限范围差异不是新角色；AI 是系统能力不是角色。

### 5. 核心任务路径

每个角色的每个核心任务，从入口走到完成，不只画页面。

路径模板：入口 → 列表/概览 → 详情 → 操作 → 反馈 → 下一步

规则：
- 高频任务不藏二级菜单
- 列表能判优先级（不是创建时间排序）
- 详情能完成核心决策（不需要跳出去改）
- 操作后必有明确状态反馈和下一步引导
- 超过5步未完成的路径，必须判断能否合并/前置/默认化

### 6. 决策与流转（核心脊椎）

「看一天 vs 几下办完」的全部差距。路径对≠动作快。

**成本三段论**：
- 理解成本（看懂在干嘛）→ 靠首屏唯一重点解决
- 判断成本（知道做什么决定）→ 靠预排序+预填默认动作解决
- 执行成本（落地花几下）→ 靠一键确认+自动流转解决

**四个核心机制**：

1. **预排序**：列表默认按"该先处理谁"排（连续下滑/卡最久/影响最大排最前），不按创建时间。看到的是行动队列不是数据陈列。

2. **预填默认动作**：待办进来即预填系统最可能的动作（AI已初判，默认是"确认发送"不是空白），思考成本变成确认成本。

3. **一键确认**：高频决策压成一个动作（通过/采纳/驳回），不用进详情改字段。详情留给需要修改的少数。

4. **自动流转**：处理完自动推进下一件，不甩回列表重找。让人进流水线不来回跑。

**批量=次要加速器**（约一成场景）：列表给轻量入口、选中套同一动作即可；不改整页结构不抢主视觉。

**闭环最后一公里**：处理过的进"待复测"队列，到期自动浮回管理者面前。

硬约束（不达标退回）：无默认动作 / 处理完无自动流转 / 列表无优先级排序。

### 7. 页面主次与首屏

每页一个主动作，先定首屏再进 UI。

**首屏规则**：
- 只有一个最高优先级动作，5秒看不懂该做什么→退回
- 无关信息不占主视觉区
- 指标/按钮/图表/说明不同时争第一焦点

**信息分层**：
| 层级 | 内容 | 位置 |
|---|---|---|
| P0 | 当前动作必须信息 | 首屏主区 |
| P1 | 判断辅助 | 主区次级 |
| P2 | 历史/解释/详情 | 详情页/折叠/侧栏 |
| P3 | 配置/审计/日志 | 后台/运营页 |
| P4 | 低频功能 | 更多菜单/二级入口 |

**角色工作台**（首页=今天最该处理什么，不是菜单）：
- 用户：今天提交没 / 有新反馈没 / 下一条怎么改
- 管理者：谁没提交 / 谁待点评 / 谁连续下滑 / 先处理谁
- 运营者：系统健康吗 / 权限正常吗 / AI失败没 / 成本异常没

首页只给入口给不了今日重点→是菜单页不是工作台，退回。

### 8. 认知与情绪设计（新增）

V3.2 管"快不快"，本章管"安不安心"。路径短但心里没底=体验差。

**确定性设计**（消除"我点完会发生什么"的恐惧）：
- 破坏性操作前告知后果+提供撤回（"删除后30天内可恢复"）
- 多步流程显示进度（你在第2步/共3步）
- 提交后明确告知"谁在处理、预计多久"
- 可预览操作结果再确认（"发送后对方将看到以下内容"）

**安全感设计**（消除"我搞砸了怎么办"的焦虑）：
- 重要操作可撤回/可修改/有缓冲期
- 出错时不惩罚，提供修正路径
- 权限不足时告知原因+告诉找谁解决
- 审核/评价/处罚类信息，被评价者能看到依据和申诉入口

**认知负荷控制**：
- 一次只让用户做一个决定
- 选项超过5个要分组或提供默认推荐
- 表单不一次性全展开，按逻辑分步
- 专业术语旁给一句话解释或示例

### 9. 反向场景预判（新增）

只设计"跑通"不够，必须覆盖"跑不通"。

对每个核心场景强制回答：

| 反向场景 | 必须设计的应对 |
|---|---|
| 数据为空 | 空状态说明为什么空+引导去哪开始 |
| AI判断全错 | 人工可跳过AI/可覆盖/业务不卡死 |
| 用户拒绝/不配合 | 超时提醒→催办→升级→标记，不是无限等待 |
| 网络/系统异常 | 已填内容不丢失+明确提示+可重试 |
| 权限变更 | 正在处理中的任务如何交接 |
| 数据量暴增 | 列表性能兜底+分页/筛选策略 |
| 规则冲突 | 多条规则命中同一对象时的优先级 |

规则：每个核心路径至少覆盖3个反向场景，否则不算设计完整。

### 10. 状态系统

无状态反馈不算可用产品。每状态回答"现在发生了什么+我能做什么"。

| 状态 | 告诉用户 | 能做 |
|---|---|---|
| 空 | 为什么没数据 | 去哪开始 |
| 加载中 | 在处理什么 | 等待/返回 |
| 处理中 | 卡在哪个环节 | 是否可跳过 |
| AI分析中 | AI在做什么 | 等待/人工处理 |
| AI失败 | 为什么失败 | 重试/跳过/人工审核 |
| 待审核 | 谁要处理 | 催办/进入审核 |
| 已反馈 | 结果是什么 | 查看/确认/执行 |
| 待复测 | 下次怎么验证 | 提交新数据/看对比 |
| 权限不足 | 为什么不能操作 | 找谁处理 |
| 已完成 | 当前结果 | 下一步去哪 |

规则：不能只有成功态；AI失败不卡死业务；文案解释"发生了什么"不只显错误码；涉及审核/权限/处罚的状态变化必须留痕。

### 11. AI与人工边界

AI 是辅助不是责任人。

原则：
- AI 只出建议不出结论，可查依据/可改/可驳回/可追溯
- 涉及评价/处罚/权限变更/绩效/对外反馈，必须人工确认
- 必须允许人工跳过 AI 直接处理
- AI 出错只降效率不让业务停摆

与第6章的关系：预填默认动作建立在 AI 初判上——AI做八成、人确认或改两成。这是辅助与终判的结合点，不是 AI 替人决策。

**⏸ 硬停点 2：到这里必须停下等用户确认。确认设计方向对齐后，才继续版本切割和交付。**

## 落地层：先做什么、交给谁

### 12. 版本切割

不用打分，用三个强制问题分类：

| 问题 | 判断 |
|---|---|
| 这个功能不做，最小闭环跑得通吗？ | 跑不通→V1必做 |
| 做了解决什么？省人力/提体验/降风险？ | 判断价值类型 |
| 开发成本大概多大？一天/一周/一个月？ | 判断投入 |

切割规则：
- **V1 必做**：最小闭环跑不通的、成本≤一周的高频痛点
- **V1 简化做**：闭环需要但可以先用简单方案兜住的（如先用列表代替看板）
- **V2 再做**：闭环跑得通但体验更好的、成本超一周的增强
- **不做**：稀释首屏焦点的、造成角色越权的、低频且有替代方案的

最小闭环定义：提交→接收→（AI初判）→人工审核→反馈→执行→验证→基础沉淀。

### 13. 交付验收

交付前必须输出可执行内容：

| 对象 | 必须拿到 |
|---|---|
| 产品 | 业务阶段、角色权限、字段归属、页面职责、设计原则、MVP范围 |
| UI/交互 | 工作台首页、首屏焦点、主动作、信息层级、状态反馈、认知情绪设计、任务路径 |
| 前端 | 页面入口、组件规则、字段、按钮、状态提示、权限展示、空/失败态、反向场景处理 |
| 后端 | 数据实体、字段来源、状态机、权限校验、日志审计、优先级排序规则 |
| AI | 输入/输出字段、判断依据、人工确认点、失败降级方案 |
| 运营 | 配置项、审计项、案例沉淀规则、异常处理 |
| 验收 | 每个阶段/页面/状态/角色的成功标准+失败标准（对应第2章判据） |

## 维度判定标准

四档判定：

**顶级工作台雏形**：闭环清楚+角色工作台清楚+首屏重点清楚+决策流转顺（预排序/默认动作/一键/自动流转）+认知情绪有设计+反向场景有覆盖+状态完整+开发可执行+结果可验证。

**可用产品模块**：角色/阶段/字段/页面职责/权限清楚、可验证、可开发，但流转和认知情绪仍需优化。

**功能外壳**：只有页面和功能，无状态流转/字段归属/权限边界/结果验证/反向场景。

**能跑的垃圾**：能跑但不知重点、主次混乱、路径断裂、状态缺失、低频抢核心、处理动作成本高、用户不敢操作。
