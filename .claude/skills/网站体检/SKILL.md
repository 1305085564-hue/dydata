# 网站体检

对指定页面做一次标准性能体检。目的：让每个一次性接手的 AI 都先获得全局视野，再动局部，不靠救火式乱修。

## 前置（必做，不许跳）

1. 先完整读 `docs/架构与性能总纲.md`。它是全站唯一总控。
2. 确认体检目标页面（阿禅没指定就问一句）。
3. 在总纲「三、架构地图」里找到该页面依赖的加载器、表、接口。

## 体检五步

### 第 1 步 分领域
按总纲「二、岗位分诊」判断：这次慢先归「性能」领域分诊。初判是前端锅（请求太多/太早/串行）还是后端锅（接口本身慢/数据库慢）。写下初判。

### 第 2 步 实测请求
用浏览器实测该页面（优先 playwright / browser 工具，登录后打开目标页）：
- 数打开时发了多少个请求，对比总纲「四」的首屏请求数上限。
- 找出最慢的接口和耗时，对比单接口上限。
- 判断请求是并行还是一个等一个（串行瀑布）。

### 第 3 步 实测接口耗时
先确认目标页面的加载器已有分步骤 `[perf]` 埋点。若没有或只埋了总耗时 → 先补细粒度埋点（用 `measureAsync` 包住每个独立查询），否则不许声称「已实测某步骤耗时」，只能记「无埋点，未实测」。
补齐埋点后本地跑 `DYDATA_PERF_LOG=1`，看控制台 `[perf]` 日志，定位加载器里最慢的步骤。读对应 `src/lib/loaders/*` 文件，检查：
- 有没有独立查询被串行 `await`（应改 `Promise.all` 并行）。
- 有没有同一份数据查两遍。
- 有没有首屏拉了当前没显示的数据（应延后成前端补充接口）。
- 有没有该缓存却没缓存的重复计算。

### 第 4 步 归因 + 最小改动
把问题归到唯一一类：前端请求太多/太早 · 后端接口慢 · 数据库缺索引/重复查 · 缺缓存。
只改这一类，守住领域边界：
- 前端类 → 产出改动或提示词，派 Antigravity。
- 后端/数据库类 → 派 Codex，或本会话按后端规范改加载器/接口/加 migration 索引。
- 不允许一次体检里既改样式又改数据库。

### 第 5 步 复测 + 记账
- 改完用同样方法复测同样指标，确认达到总纲「四」的红线。
- 在总纲「六、实测台账」最上面加一行：日期/页面/首屏/请求数/最慢接口/归因/改了什么/执行。
- 若改到了加载器、依赖表、首屏接口，同步更新总纲「三、架构地图」。
- 更新总纲顶部「最后核对代码日期」。

## 交付

体检结束给阿禅一句话结论（大白话）：这页原来多慢、慢在哪、改了什么、现在多快、是否已生效还是要上线后才生效。附一键回滚指令。

## 边界

- 只管「慢」。白屏/报错/登录异常走 debugging-and-error-recovery skill。
- 只做性能领域改动，业务规则和数据含义不动。
- 不批量优化全站；一次体检一个页面，记账后再下一个。
