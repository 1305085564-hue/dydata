# 违规话术系统 V1 — 施工方案（可执行版）

## 目标

全员共享违规/非违规话术经验，把个人踩坑变成公司可查数据资产。

## V1 范围（最小闭环）

只做 4 件事：
1. 全员案例列表 `/violations`
2. 提交页 `/violations/submit`（含 dashboard 轻入口）
3. 详情页 `/violations/[id]`（含追加测试记录）
4. 后台复核页 `/admin/violations`（管理员确认/驳回/标注结论）

V2 再做：统计面板、AI 分析报告、周报自动生成。

## 一、数据模型

### 核心设计原则

- 关联现有 `accounts.id`，同时冗余 `account_name_snapshot`（提交时快照，防账号改名后丢失上下文）
- 关联 `teams.id` 而非存团队名文本
- 截图存 Storage 对象路径（非 URL），读取时生成 signed URL
- 案例有状态流转：submitted → verified / rejected / archived
- 管理员结论是独立字段，员工侧直接可见
- 权限走 `manage_violations` 权限键（新增），owner 天然拥有，admin 需授权

### 表 1：`violation_cases`（话术案例主表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid PK | gen_random_uuid() |
| created_at | timestamptz | 提交时间 |
| submitted_by | uuid FK → profiles.id | 提交人 |
| script_text | text NOT NULL | 话术原文（原封不动） |
| is_violation | boolean NOT NULL | true=违规，false=已验证可用 |
| category | text NOT NULL | '下粉' / '直播' / '短视频' / '其他' |
| account_id | uuid FK → accounts.id | 出现问题的账号（选填，从用户已有账号选） |
| account_name_snapshot | text | 提交时账号名快照（防改名后丢上下文） |
| team_id | uuid FK → teams.id | 来自哪个团队（自动从 profile 取） |
| scene_description | text | 配套画面/简介/导粉方式描述 |
| screenshot_paths | text[] DEFAULT '{}' | Storage 对象路径（非 URL） |
| result | text | 结果描述：如"限流3天""正常过审" |
| tags | text[] DEFAULT '{}' | 标签 |
| pass_count | int DEFAULT 0 | 通过次数 |
| fail_count | int DEFAULT 0 | 未通过次数 |
| **status** | text DEFAULT 'submitted' | 'submitted' / 'verified' / 'rejected' / 'archived' |
| **risk_level** | text | 管理员标注：'high' / 'medium' / 'low'（复核后填） |
| **admin_conclusion** | text | 管理员结论文字（如"公司层面禁止此类画面"） |
| **suggested_action** | text | 建议动作（如"换成XX画面""删除此话术"） |
| **reviewed_by** | uuid FK → profiles.id | 复核人 |
| **reviewed_at** | timestamptz | 复核时间 |
| is_deleted | boolean DEFAULT false | 软删除 |

### 表 2：`violation_test_records`（多号测试记录）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uuid PK | gen_random_uuid() |
| case_id | uuid FK → violation_cases.id | 关联案例 |
| tested_by | uuid FK → profiles.id | 测试人 |
| tested_at | timestamptz DEFAULT now() | 测试时间 |
| account_id | uuid FK → accounts.id | 测试账号 |
| passed | boolean NOT NULL | 是否通过 |
| note | text | 备注 |

### RLS 策略

权限检查不直接用 `is_admin()`，而是新增 `has_violation_permission()` 函数，
检查 `role = 'owner' OR (role = 'admin' AND permissions->>'manage_violations' = 'true')`。

```sql
-- violation_cases
SELECT: auth.uid() IS NOT NULL AND is_deleted = false
INSERT: auth.uid() = submitted_by
UPDATE: public.has_violation_permission()  -- 管理员复核用
DELETE: public.has_violation_permission()  -- 软删除

-- violation_test_records
SELECT: auth.uid() IS NOT NULL
INSERT: auth.uid() = tested_by
        AND EXISTS (SELECT 1 FROM violation_cases WHERE id = case_id AND status NOT IN ('archived') AND is_deleted = false)
DELETE: public.has_violation_permission()
```

### Storage

- Bucket: `violation-screenshots`（private）
- 上传路径: `{user_id}/{date}/{timestamp}_{filename}`（不含 case_id，因为上传时案例尚未创建）
- 提交案例时将已上传的路径列表写入 `screenshot_paths`
- 读取: API route `/api/violations/screenshot/[...path]/route.ts`（catch-all）生成 signed URL（有效期 1 小时）
- 限制: image/png, image/jpeg, image/webp, max 5MB
- 清理: 孤立文件（上传后未关联案例）可定期清理，V1 不做

## 二、状态流转

```
员工提交 → submitted（待复核）
    ↓
管理员复核 → verified（已确认）/ rejected（已驳回）
    ↓
过期/失效 → archived（已归档）
```

### 员工侧显示

| status | 显示 | 颜色 |
|--------|------|------|
| submitted | 待验证 | 黄色 |
| verified + is_violation=true | 已确认违规 | 红色 |
| verified + is_violation=false | 已确认可用 | 绿色 |
| rejected | 已驳回 | 灰色 |
| archived | 已归档 | 灰色 |

只有 `verified` 状态的案例才能进入未来的 AI 分析输入。

## 四、核心页面设计

### 4.1 案例列表 `/violations`

**布局：**
- 顶部：标题 + "提交新案例"按钮
- 筛选栏：状态（全部/已确认违规/已确认可用/待验证）| 分类 | 团队
- 搜索框：话术内容全文搜索
- 列表卡片：
  - 话术原文（截断 80 字）
  - 状态徽章（颜色区分）
  - 通过率（有测试数据时显示）
  - 管理员结论（verified 状态时显示，醒目标注）
  - 账号 | 团队 | 时间
- 默认排序：verified 在前（按 reviewed_at 倒序），submitted 次之（按 created_at 倒序）
- 分页：每页 20 条

### 4.2 提交表单 `/violations/submit`

**字段：**
1. 话术原文（textarea，必填）
2. 是否违规（单选：违规 / 未违规可用）
3. 分类（下拉：下粉 / 直播 / 短视频 / 其他）
4. 账号（下拉，从当前用户的 accounts 列表选；选中后自动快照 account_name_snapshot；选填，可留空表示"不确定哪个号"）
5. 配套画面/导粉方式描述（textarea，选填）
6. 截图上传（最多 5 张）
7. 结果描述（text，选填）

> 注意：不允许手动输入新账号名。如果员工的账号不在列表里，需要先在 dashboard 的"添加账号"功能创建，再回来选。这保证 account_id 始终有效。

**预填逻辑：**
- URL 参数 `?account_id=xxx` → 自动选中账号
- team_id 从当前用户 profile 自动填入，不需要手选

**提交后：** 入库 status=submitted → 跳转详情页 → Toast "已提交，等待管理员确认"

### 4.3 案例详情 `/violations/[id]`

**内容：**
- 话术原文（完整，大字体）
- 状态徽章 + 管理员结论（如果有，用卡片高亮展示）
- 元信息：分类 / 账号 / 团队 / 提交人 / 时间 / 结果
- 截图画廊（signed URL，点击放大）
- 通过率区域：
  - 进度条 + 百分比 + 可信度（1-2次=样本不足，3-5=初步可信，6+=可靠）
  - 测试记录列表
  - "我也测了"按钮 → 弹窗：选账号 + 通过/未通过 + 备注
- 建议动作（如果管理员已填，醒目展示）

### 4.4 后台复核 `/admin/violations`

**布局：**
- 默认显示 status=submitted 的案例（待复核）
- 可切换查看全部状态
- 每条案例展开后显示复核面板：
  - 确认/驳回 按钮
  - 风险等级（high/medium/low 单选）
  - 管理员结论（textarea）
  - 建议动作（textarea）
- 确认后：status → verified，填入 reviewed_by + reviewed_at
- 驳回后：status → rejected，可填驳回原因到 admin_conclusion

## 五、API 接口

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | /api/violations | 列表（筛选/搜索/分页） | 登录 |
| POST | /api/violations | 提交新案例 | 登录 |
| GET | /api/violations/[id] | 详情（含测试记录） | 登录 |
| POST | /api/violations/[id]/test | 追加测试记录 | 登录 |
| PATCH | /api/violations/[id]/review | 复核操作 | manage_violations |
| DELETE | /api/violations/[id] | 软删除 | manage_violations |
| POST | /api/violations/upload | 截图上传 → 返回 path | 登录 |
| GET | /api/violations/screenshot/[...path]/route.ts | 生成 signed URL（catch-all） | 登录 |

### API 校验规则

**POST /api/violations（提交案例）：**
- `script_text` 必填，不为空
- `category` 必须在 ['下粉', '直播', '短视频', '其他'] 内
- `account_id` 如果提供，必须校验属于当前用户（`accounts.profile_id = auth.uid()`）
- `account_name_snapshot` 由后端自动从 accounts 表查询填入，前端不传
- `team_id` 由后端自动从当前用户 profile 取，前端不传
- `submitted_by` 强制设为 auth.uid()，前端不可覆盖

**POST /api/violations/[id]/test（追加测试）：**
- 校验 case 存在且 `status NOT IN ('archived')` 且 `is_deleted = false`
- `account_id` 如果提供，必须属于当前用户
- `tested_by` 强制设为 auth.uid()
- 同一用户 + 同一账号 + 同一案例，24 小时内不能重复提交（防误操作）

**PATCH /api/violations/[id]/review（复核）：**
- 用 `hasPermission(role, permissions, 'manage_violations')` 校验
- 只允许修改：status, risk_level, admin_conclusion, suggested_action
- 自动填入 reviewed_by = auth.uid(), reviewed_at = now()

### 截图上传流程

1. 前端选择文件 → POST `/api/violations/upload`（multipart/form-data）
2. 服务端校验文件类型（image/png|jpeg|webp）和大小（≤5MB）
3. 服务端用 service_role 上传到 Storage，路径：`{user_id}/{YYYY-MM-DD}/{timestamp}_{filename}`
4. 返回对象路径（string），前端暂存到本地数组
5. 提交案例时将路径数组写入 `screenshot_paths`
6. 读取时前端调 `/api/violations/screenshot/[...path]` 获取 signed URL（1 小时有效）

## 六、触发器

```sql
-- 自动更新通过率
CREATE OR REPLACE FUNCTION update_violation_pass_rate()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE violation_cases
  SET
    pass_count = (SELECT count(*) FROM violation_test_records WHERE case_id = COALESCE(NEW.case_id, OLD.case_id) AND passed = true),
    fail_count = (SELECT count(*) FROM violation_test_records WHERE case_id = COALESCE(NEW.case_id, OLD.case_id) AND passed = false)
  WHERE id = COALESCE(NEW.case_id, OLD.case_id);
  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_violation_pass_rate
AFTER INSERT OR DELETE ON violation_test_records
FOR EACH ROW EXECUTE FUNCTION update_violation_pass_rate();
```

## 七、完整 Migration SQL

文件：`supabase/migrations/055_violation_system.sql`

```sql
-- ============================================================
-- 055: 违规话术系统 V1
-- ============================================================

-- 0. 权限检查函数
CREATE OR REPLACE FUNCTION public.has_violation_permission()
RETURNS boolean
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = public
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.profiles
    WHERE id = auth.uid()
      AND (
        role = 'owner'
        OR (role = 'admin' AND (permissions->>'manage_violations')::boolean = true)
      )
  );
$$;

-- 1. 话术案例主表
CREATE TABLE IF NOT EXISTS public.violation_cases (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  created_at timestamptz NOT NULL DEFAULT now(),
  submitted_by uuid NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  script_text text NOT NULL,
  is_violation boolean NOT NULL DEFAULT true,
  category text NOT NULL DEFAULT '其他'
    CHECK (category IN ('下粉', '直播', '短视频', '其他')),
  account_id uuid REFERENCES public.accounts(id) ON DELETE SET NULL,
  account_name_snapshot text,
  team_id uuid REFERENCES public.teams(id) ON DELETE SET NULL,
  scene_description text,
  screenshot_paths text[] DEFAULT '{}',
  result text,
  tags text[] DEFAULT '{}',
  pass_count int NOT NULL DEFAULT 0,
  fail_count int NOT NULL DEFAULT 0,
  status text NOT NULL DEFAULT 'submitted'
    CHECK (status IN ('submitted', 'verified', 'rejected', 'archived')),
  risk_level text CHECK (risk_level IN ('high', 'medium', 'low')),
  admin_conclusion text,
  suggested_action text,
  reviewed_by uuid REFERENCES public.profiles(id),
  reviewed_at timestamptz,
  is_deleted boolean NOT NULL DEFAULT false
);

-- 2. 多号测试记录表
CREATE TABLE IF NOT EXISTS public.violation_test_records (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id uuid NOT NULL REFERENCES public.violation_cases(id) ON DELETE CASCADE,
  tested_by uuid NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  tested_at timestamptz NOT NULL DEFAULT now(),
  account_id uuid REFERENCES public.accounts(id) ON DELETE SET NULL,
  passed boolean NOT NULL,
  note text
);

-- 3. 索引
CREATE INDEX idx_vc_status ON violation_cases(status) WHERE is_deleted = false;
CREATE INDEX idx_vc_category ON violation_cases(category) WHERE is_deleted = false;
CREATE INDEX idx_vc_team ON violation_cases(team_id) WHERE is_deleted = false;
CREATE INDEX idx_vc_created ON violation_cases(created_at DESC) WHERE is_deleted = false;
CREATE INDEX idx_vc_script_search ON violation_cases
  USING gin(to_tsvector('simple', script_text)) WHERE is_deleted = false;
CREATE INDEX idx_vtr_case ON violation_test_records(case_id);

-- 4. 触发器
CREATE OR REPLACE FUNCTION update_violation_pass_rate()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE violation_cases
  SET
    pass_count = (SELECT count(*) FROM violation_test_records
                  WHERE case_id = COALESCE(NEW.case_id, OLD.case_id) AND passed = true),
    fail_count = (SELECT count(*) FROM violation_test_records
                  WHERE case_id = COALESCE(NEW.case_id, OLD.case_id) AND passed = false)
  WHERE id = COALESCE(NEW.case_id, OLD.case_id);
  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_violation_pass_rate
AFTER INSERT OR DELETE ON violation_test_records
FOR EACH ROW EXECUTE FUNCTION update_violation_pass_rate();

-- 5. RLS
ALTER TABLE violation_cases ENABLE ROW LEVEL SECURITY;
ALTER TABLE violation_test_records ENABLE ROW LEVEL SECURITY;

-- violation_cases
CREATE POLICY "vc_select" ON violation_cases
  FOR SELECT USING (auth.uid() IS NOT NULL AND is_deleted = false);

CREATE POLICY "vc_insert" ON violation_cases
  FOR INSERT WITH CHECK (auth.uid() = submitted_by);

CREATE POLICY "vc_update" ON violation_cases
  FOR UPDATE USING (public.has_violation_permission());

CREATE POLICY "vc_delete" ON violation_cases
  FOR DELETE USING (public.has_violation_permission());

-- violation_test_records
CREATE POLICY "vtr_select" ON violation_test_records
  FOR SELECT USING (auth.uid() IS NOT NULL);

CREATE POLICY "vtr_insert" ON violation_test_records
  FOR INSERT WITH CHECK (
    auth.uid() = tested_by
    AND EXISTS (
      SELECT 1 FROM violation_cases
      WHERE id = case_id
        AND status NOT IN ('archived')
        AND is_deleted = false
    )
  );

CREATE POLICY "vtr_delete" ON violation_test_records
  FOR DELETE USING (public.has_violation_permission());
```
