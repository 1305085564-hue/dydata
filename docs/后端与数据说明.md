# DYData 后端与数据说明

> 本文件是"后端与数据"领域的入口分诊页：接了接口/数据相关的活，动手前先读这里，知道该看哪几处、边界在哪、有哪几条死规矩、出问题按什么顺序查。
> 它只做串联和立规矩，不重抄数据口径和数据库规范——那些在现成资源里，本文件只指路。
> 用途：任何 AI 动"接口对不对、数据准不准、快不快"的活之前，先在这里对齐领域边界和必读清单。
> 面向阿禅：你不用懂代码。这份是给做后端的 AI 立的规矩和指路，保证它接手时不乱查、不改坏数据。术语一律就地用大白话解释。
> 面向 AI：这里是后端与数据领域的分诊入口，事实已按当前代码核实。与代码冲突一律以代码为准并当场修本文件。

## 〇、这个领域管什么（给阿禅）

网站分七个"岗位领域"，这份管其中一个——后端与数据。用大白话说，就是管三件事：接口对不对（点一下按钮，后台给的数据对不对）、数据准不准（那些播放量、涨粉率算得对不对）、快不快（接口响应慢不慢，含 AI 接口）。它不管页面长什么样（那是前端体验领域）。

---

## 一、领域边界（这活归不归我、哪部分要转派）

管：接口逻辑、数据查询、数据准确性、数据库结构（表和字段）、AI 接口的调用与配置。主要执行是 Codex。

不碰、要转派的：
- 页面样式、交互、动效 → 转前端体验领域（`docs/美学规范.md`）。后端任务不改页面长相。
- 权限规则本身（谁有资格看/改、角色和开关怎么定）→ 转权限与安全领域（`docs/权限与安全说明.md`）。但注意：后端查询普遍直接用权限领域算好的 `visibleUserIds`（能看见哪些人）做范围过滤，"数据查不出/查错人"的后端排查绕不开它，不能简单转走——见第四节。边界是"权限规则归权限领域，范围过滤的落地在后端查询里，两边都要看"。
- 页面慢、首屏慢、请求数多 → 转性能领域（`docs/架构与性能总纲.md`）。后端只管单个接口/查询快不快，整页性能走总纲。
- 出线上故障（白屏/登录挂/报错）→ 先走 `docs/运维排查手册.md` 急救，再回本领域查根因。

一个任务只落一个领域，跨领域先拆、产出提示词转派，不越界顺手改。

## 二、动手前必读（散在多处的现成资源，这里串成入口）

后端与数据的知识刻意复用现成资源，不单独造百科。接活前按需读：

- `docs/数据口径.md`：6 个核心指标（24h 播放、互动率、粉转率、导粉率、主页访问率、爆款系数）的定义、公式、代码来源、爆款判定标准。凡涉及指标怎么算，先读它，别自己推。
- skill `supabase-postgres-best-practices`：数据库和 Postgres 的写法规范（查询、迁移、索引等）。写数据库相关代码前对照。
- skill `nextjs-supabase-auth`：Next.js + Supabase 的登录认证写法。碰认证相关接口时读。
- `docs/架构与性能总纲.md` 第三节"架构地图"：核心页面 → 加载器 → 表 → 接口的依赖图。查"这个接口连哪张表、改了连累谁"。
- `docs/全站模块地图.md`：哪张表连哪些页面、哪些接口是活的还是废的。动一张表前先查它的连累面。
- `docs/权限与安全说明.md`：service role（绕过数据库门禁的超级权限）和 RLS 的规矩。写用 service role 的接口前必读。

只读这些、不在本文件重抄它们的内容。

## 三、数据库访问死规矩（原来散着，收口到这里）

这些是从代码和 CLAUDE.md 收口的硬规矩，不是新造，写接口/查询时必须守：

- 选对数据库连接（`src/lib/supabase/` 四种）：`client`（浏览器端）、`server`（服务端，受数据库门禁 RLS 约束）、`admin`/`service`（同一把超级钥匙，绕过 RLS）。用 `admin`/`service` 前必须先在应用层鉴权，细节见权限说明第四节。
- 只查线上真实存在的字段。`select` 里写的字段，必须是线上数据库已经有的；不确定先查真实表结构和已跑过的 migration，别凭旧记忆写字段。
- 缺字段要降级、别崩。读可能缺列的表时做兼容降级（查不到就返空/默认值），不要让整个页面因为一个缺列直接报错。
- migration 只增不改。数据库结构变更只能新增 migration 文件，禁止改任何旧 migration。
- 不留幽灵字段。前端传给后端、但后端没有接收逻辑的字段，要么补后端接收、要么删掉，不留半截。

## 四、数据/接口问题排查顺序

聚焦数据层，和运维手册的"线上急救"分工（急救先看那份）：

1. 数据算得不对（口径类）：先翻 `docs/数据口径.md` 的公式，再对 `src/lib/video-metrics.ts` 的实现，确认是口径理解错还是计算错。
2. 数据查不出/查错人：分三类查，别只看前两类——(a) 权限规则（RLS/service role，转权限领域）；(b) 字段（查真实表结构和已跑 migration）；(c) 范围过滤，最常被漏：查询是否按 `visibleUserIds` 过滤、团队/组范围对不对、RPC 的范围参数（如 `p_visible_user_ids`）传对没、业务筛选条件是否把数据滤掉了。数据"凭空消失"多半在第三类。
3. 接口报错：去 Vercel 后台看这个接口的运行日志（Functions/Logs），报错信息在那；再顺模块地图找它依赖哪张表。
4. AI 接口相关：配置有多条并行路径，排障别只盯 `admin/ai-config` 中枢——还有环境变量兜底（`AI_BASE_URL`/`AI_API_KEY`/`AI_MODEL`，见 `src/lib/ai/client.ts`）、旧表 `ai_feature_config` 和旧接口 `/api/admin/ai-rewrite` 仍在读写。先确认这次调用实际走的是哪条路径，再查配置对不对。相关表和接口见模块地图 AI 配置中枢一节。
5. 单接口慢：查询本身的慢属本领域（看有没有该加的过滤、该走的 RPC）；整页慢转性能总纲。

## 五、维护规则

- 新增/删除接口或表、改数据口径 → 同步更新 `docs/全站模块地图.md`（模块归属）和 `docs/数据口径.md`（口径），本文件只在"必读清单""死规矩"有变时更新。
- 新增数据库死规矩或访问约定 → 补进第三节，并与 CLAUDE.md 保持一致。
- 与代码冲突一律以代码为准并立即修本文件。
- 本文件管"后端该读哪、守哪些规矩、怎么查"；指标口径在 `docs/数据口径.md`，依赖图在 `docs/架构与性能总纲.md`，模块归属在 `docs/全站模块地图.md`，权限在 `docs/权限与安全说明.md`。
- 顶部"最后核对代码日期"每次核对后更新。
