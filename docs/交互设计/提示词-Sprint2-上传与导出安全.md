# 提示词：Sprint 2 — 上传与导出安全

## 你的身份
你是 DYData（抖音数据日报平台）的资深全栈开发工程师，精通 Next.js App Router + Supabase + TypeScript + Tailwind CSS。

## 任务背景
项目存在两个数据层面的安全隐患：
1. 文件上传无客户端大小预校验，用户选择超大文件后才发现报错
2. 数据导出功能将所有匹配行加载到内存生成 XLSX，大数据量时存在内存溢出风险

## 任务清单（按顺序执行）

### 任务 1：截图上传增加客户端大小预校验

**目标**：用户选择文件后，如果超过限制，立即在前端提示，不上传。

**涉及文件**：
- `src/components/screenshot-upload.tsx`（如果有）
- `src/app/(app)/dashboard/video-submit-form.tsx` 中的截图上传逻辑
- `src/app/(app)/violations/submit/submit-form.tsx` 中的截图上传逻辑
- 任何其他包含 `<input type="file">` 且用于上传的组件

**要求**：
1. 定义上传限制常量：
   ```typescript
   const UPLOAD_LIMITS = {
     screenshot: 8 * 1024 * 1024,      // 8MB
     violationScreenshot: 5 * 1024 * 1024, // 5MB
     ocr: 8 * 1024 * 1024,             // 8MB
   };
   ```
2. 在文件选择事件（`onChange`）中，第一时间检查 `file.size`
3. 超过限制时：
   - 不上传该文件
   - 显示 `feedbackToast.error("文件超过 XMB 限制，请压缩后重试")`
   - 清空 input 值（允许用户重新选择同一文件）
4. 如果支持多文件选择，逐个检查，超限的文件跳过并提示，未超限的正常上传
5. 在 UI 上显示限制提示（如上传按钮下方小字「支持 JPG/PNG，最大 8MB」）

**验收标准**：
- 选择 10MB 文件时，立即 toast 报错，不上传
- 选择 5MB 文件时，正常上传
- 多文件选择时，超限的跳过，未超限的继续
- 提示文案清晰，用户知道怎么解决

### 任务 2：导出功能改为流式 + 大小限制

**目标**：大数据量时保护服务端内存，给用户明确反馈。

**文件**：`src/app/api/export/route.ts`（或等效导出 API 文件）

**要求**：
1. **大小限制**：查询前先做 `count` 估算，如果超过 10000 条，返回：
   ```json
   { "error": "数据量过大", "message": "当前筛选条件匹配 X 条数据，超过 10000 条限制，请缩小时间范围或增加筛选条件后重试", "count": X }
   ```
2. **流式生成**：小数据量（<= 10000 条）时，使用流式方式生成 XLSX，不要一次性把所有数据加载到内存
3. **前端适配**：导出按钮点击后显示 loading 状态，收到「数据量过大」错误时显示具体提示
4. **进度反馈（可选加分项）**：如果实现复杂，可先不做，但预留接口

**验收标准**：
- 导出 100 条数据，正常下载
- 导出 15000 条数据，前端收到错误提示，不触发服务端内存问题
- 导出过程中按钮显示 loading，不可重复点击

### 任务 3：增加 API 速率限制

**目标**：防止接口被恶意刷爆。

**文件**：`src/middleware.ts` 或新建 `src/lib/rate-limit.ts`

**要求**：
1. 基于 IP 地址做简单速率限制
2. 规则：同一 IP 在 10 秒内最多 20 次请求
3. 超限返回 429 Too Many Requests，响应头带 `Retry-After: 10`
4. 使用内存存储（`Map<string, { count: number; resetTime: number }>`），不需要 Redis
5. 以下路径**不参与限流**：
   - `/login`、`/register`、`/api/auth/*`（登录相关）
   - 静态资源（`.js`、`.css`、图片等）
6. 限流逻辑放在 middleware 中，所有请求统一处理

**验收标准**：
- 正常用户操作不受影响
- 用脚本 1 秒内发送 30 次请求，第 21 次起返回 429
- 10 秒后再次请求，恢复正常
