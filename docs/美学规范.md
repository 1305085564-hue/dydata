# DYData 美学规范

> 适用：阿禅名下所有项目（DYData 及延伸产品）
> 核心哲学：**简单有深度**——第一眼就知道高级、容易看懂；细品之后，发现还有惊喜和洞天。
> 阅读方式：从头到尾读一遍建立直觉，之后按需查阅。设计时脑子里装着"设计灵魂"那段画面，而不是逐条对规则。

---

## 0. 设计灵魂

你是 DYData 的 UI 设计师。在你动手之前，先在脑子里看到这幅画面：

> 一间光线充足的工作室，大面积的暖白色桌面，工具放在顺手的位置。窗外的光从左侧洒进来，在桌面上留下柔和的明暗过渡。桌面整洁但不空旷——几支笔、一个打开的笔记本、一杯还冒着热气的咖啡。
>
> 这里的一切都是经过设计的，但不刻意。你注意到马克杯的釉面有微妙的渐变，笔记本的纸张有轻微的纹理，窗框投下的阴影边缘柔软而不锐利。这些细节不会在你第一眼时跳出来——它们在你坐下来工作十分钟后，才被你不经意地发现，然后你会觉得：嗯，这个地方待着真舒服。

**这就是 DYData 的气质**：温暖、清楚、高级、有细节。不是冷硬的科技后台，不是极简到冷淡的留白，不是堆满装饰的花哨面板。是一个**让人愿意待一整天的、有品位的工作环境**。

**设计信条**：

- **美学是上限，效率是底线。** 不会为了美牺牲可用性，但更不会为了"效率优先"而主动放弃美。两者冲突时，先找同时满足的方案；实在不行，保底线但要心疼上限。
- **好用和好看是一件事的两面。** 一个好看的界面让人更愿意学习它的用法，一个好用的界面让人觉得它更好看。不需要为美找功能理由——美本身就是功能。
- **克制不等于寡淡。** 克制是"每个元素都有存在的理由"，不是"尽可能少放东西"。一杯好茶是克制的，但它不寡淡。

## 1. 三层深度（核心方法论）

"简单有深度"不是口号，它是一个具体的设计方法——**每个页面都应该有三层体验**：

### 第一层：第一眼清晰（3 秒内）

用户扫一眼就知道：**这是什么页面、我该看哪里、我该做什么。**

- 视觉层级一眼可辨：最重要的东西最大、最亮或位置最突出
- 页面不拥挤：信息分组清楚，有呼吸空间
- 主操作显而易见：不需要找，它就在那里

> 这一层决定用户的第一印象："哦，这很清楚。"

### 第二层：使用中顺滑（持续操作）

用户开始操作后感受到：**工具在顺手的位置，反馈即时，流程不绕弯。**

- 交互元素看起来就是可交互的
- 操作有反馈：点击有响应，加载有进度，成功有确认
- 错误处理温和且就地解决，不打断工作流
- 相关功能放在一起，不需要来回跳转

> 这一层决定用户的工作感受："用起来很顺手。"

### 第三层：细品有惊喜（发现细节）

用户使用一段时间后，不经意间发现：**原来这里还有这样的细节。** 具体实现手法见第 3 章深度工具箱。

> 这一层决定用户的长期印象："这个产品有品位。"

**关键原则：第一层是必须做到的，第二层是应该做到的，第三层是区分好和卓越的。但设计时，三层同时考虑。**

## 2. 视觉基因（全站一致性）

> 全站风格一致是唯一的硬性要求。以下定义 DYData 的视觉 DNA——无论哪个页面，用户都能感受到"这是同一个产品"。
>
> 本产品为纯亮色模式设计。所有视觉方向、色彩选择和氛围营造均基于亮色环境。

### 2.0 空间层级（Surface Elevation）

> 一切视觉决策的上游。动手之前先定：这块内容在空间中处于哪一层？

**骨架**

**L0 空间** — 工作室的墙壁与画框。顶栏、侧边栏及其底色 `bg-stone-50`（`#FAFAF9`），围合出舞台但自身不承载业务内容。

**L1 主画面** — 铺满内容区的整块白色画板。`bg-white`，占满 L0 围合后的全部剩余空间（`min-h-screen`），内容瀑布式向下生长。**一个页面只有一个 L1。** 所有留白都是这张画板自身的呼吸，不靠露出底色制造间距。

**L3 悬浮** — 飘在空中的便签与菜单。`shadow-lg`，操作完即消失，不计入嵌套深度。

**三条纪律**

1. **桌面与物件** — L1 直接铺满 L0 围合的空间，不在中间再嵌套"大盒子壳"。
2. **凸起与凹陷** — 方向即语义。凸 = 行动，凹 = 从属。
3. **两层即止** — 静态布局只有 L0 + L1。L1 内部靠六种表达形态做层级，不靠嵌套白框。

**L1 画面上的六种表达形态**

L1 是统一的白色画板，内部所有视觉层级变化靠以下方式"加工"。可叠加、不互斥。

| # | 名称 | 物理隐喻 | 视觉实现 | 何时使用 |
|---|---|---|---|---|
| ① | 凹槽 Inset | 画板表面刻出的浅槽 | `bg-stone-100/60`，无边框 | 展开详情、历史数据、次级筛选 |
| ② | 印记 Flat Print | 墨水直接印在纸面 | 字号 + 色深阶梯 + 留白 | 标题正文、数据阵列、分割线 |
| ③ | 贴纸 Badge | 角落贴上的彩色便签 | 胶囊圆角 + 色底 `/0.12` + 同色实字 | 状态标签、分类标签、计数器 |
| ④ | 浮雕 Raised | 微微凸起的物理按键 | 暖橙实底 + `shadow-sm`，hover 上浮 | 主 CTA、激活态 Tab（每区至多 1–2 处） |
| ⑤ | 光影 Glow | 釉面散发的温暖微光 | 功能色光晕 `opacity 0.02–0.08`，无硬边 | 核心 KPI 重量感、异常氛围 |
| ⑥ | 附着 Docked | 磁铁吸在白板上的小工具 | `shadow-md`，紧贴内容边缘，带过渡 | hover 工具栏、吸附 Tooltip |

> 空间层级是决策入口，2.1–2.4 及第 3 章是各形态的参数手册。

### 2.1 色彩世界

DYData 的色彩是**温暖而丰富的**。不是只有灰色的世界，而是以暖灰为底色、功能色为路标、氛围色为空间感的完整世界。全站中性色使用 stone 色系，保持温暖一致的基调。画布底色（L0 `stone-50` / L1 `white`）由空间层级（2.0）定义。

#### 品牌性格色

**暖橙 `#D97757`** 是 DYData 的性格——温暖、可亲、值得信赖。用户看到暖橙就知道：**这里可以点。**

暖橙按浓度分层使用，越重要的操作越浓：

| 层级 | 用途                   | 表现                                   |
| ---- | ---------------------- | -------------------------------------- |
| 最浓 | 每屏唯一的主 CTA 按钮  | 暖橙实底 + 白字                        |
| 中等 | 次要操作按钮、重要图标 | 暖橙文字或描边                         |
| 轻触 | 正文内链接、低频操作   | `stone-700` + 下划线，hover 时亮为暖橙 |
| 最淡 | 工具栏图标、辅助入口   | `stone-500`，hover 时 `stone-700`      |

这样暖橙像聚光灯一样集中在主操作上，不会泛滥成页面上到处的橙色斑点。

#### 功能语义色

| 名称   | 色值      | 含义           | 用户心声     |
| ------ | --------- | -------------- | ------------ |
| 暖橙   | `#D97757` | 行动/可交互    | "这里可以点" |
| 石青   | `#5F82A8` | 选中/当前/定位 | "我在这里"   |
| 琥珀   | `#D99E55` | 提醒/待处理    | "需要留意"   |
| 矿物绿 | `#6FAA7D` | 成功/通过/正常 | "搞定了"     |
| 矿物红 | `#C9604D` | 异常/错误      | "出问题了"   |

这五个是**语义色（路标）**，锁的是"用途"而非"数量"：暖橙/绿/红硬锁不可挪用，琥珀/石青语义锁定。它们回答"这告诉用户什么"，可作卡片轻淡底色、图表主色、标签背景，**要有存在感**，不是蜻蜓点水的色块。

#### 表达色池

语义色之外的开放色层。**不承载状态语义**，专供三类用途：图表多线/多类、分类标签、氛围底色。存在的意义：不让全站被五个路标困住，同时"开放但成套"——统一从池子里取，不随手调色，才不会花。

| 名字   | 色值      | 气质       |
| ------ | --------- | ---------- |
| 靛青   | `#4F5E96` | 沉的蓝紫   |
| 藕紫   | `#7E5C99` | 实的紫     |
| 陈玫   | `#A85A6B` | 灰调玫红   |
| 青瓷   | `#3E8479` | 沉静青绿   |
| 苔橄榄 | `#7A7A3E` | 大地橄榄   |

规则：这是**实色基准**（明度按上表由深到浅错档，色相拉开，全部避开五个语义色）。图表直接取实色当线色；分类标签用"淡底 `/0.12` + 同色实字"；**氛围**时才把实色调到 `/0.08` 左右当背景光晕——氛围的淡是"从实色调下去"，不是一开始就淡。禁止：把表达色用在承载状态语义的地方（成功/错误/待处理仍归语义色）。

#### 氛围色

不承载信息语义，只营造空间感和情绪——工作室里的窗外光线。极淡（`opacity 0.02–0.08`），与 stone 色系和谐，每个页面最多一种氛围基调。具体实现见空间层级（2.0）光影型⑤。

#### 色彩使用节奏

功能色要有存在感——不要只用在最小面积上。图表和数据可视化是色彩的主要舞台，大胆使用完整光谱。页面整体保持白底 → 功能色亮点 → 白底的冷暖交替韵律。

### 2.2 文字系统

全站无衬线字体：`Inter, -apple-system, PingFang SC, system-ui, sans-serif`。数字使用 `tabular-nums` 保证数据列对齐。

文字通过**字号和色深**建立层级，而不是靠粗体堆叠：

- 字号分四档：页面标题（24px）→ 卡片标题（18px）→ 正文（13px）→ 辅助说明（12px）
- 色深分三阶：`stone-900`（标题/核心数据）→ `stone-700`（正文）→ `stone-500`（辅助信息，全站最浅文字）
- 字重以 `normal` 和 `medium` 为主。核心 KPI 每屏至多一处可用 `semibold`
- 中文行高 `leading-[1.6]`，保证阅读舒适

允许在需要**节奏对比**的场景灵活运用字重——比如标签用 `font-medium` 的小号字"压实"成独立元素，与旁边正文形成节奏差。

> 具体字号阶梯和字重字色参数表见《美学规范补充》。

### 2.3 形态语言

**圆角是语义**：越圆越"独立"（标签、徽章用胶囊），越方越"容器"（卡片、面板用大圆角）。按钮居中——圆角大小随按钮高度递增。

**边界与层级**：遵循空间层级（2.0）纪律——L1 内部分区优先用留白和单边细线，不用闭合边框围成子容器。阴影只给浮雕型（④）和悬浮层（L3）。

**图标**：全站统一 Lucide Icons，stroke-width 1.5px。默认 `stone-500`，hover `stone-700`。

> 圆角阶梯、阴影参数、间距档位见《美学规范补充》。

### 2.4 间距与密度

间距不只是排版工具，它是**信息关系的视觉表达**——靠得近的元素用户认为有关系，离得远的认为是两件事。

核心节奏：紧密关联（≤8px）→ 同模块（16px）→ 模块之间（≥24px）。

## 3. 深度工具箱

> 以下工具帮助你创造"第三层"体验——那些让用户细品后觉得"这个产品有品位"的细节。
>
> 不是每个页面都要用所有工具。像调味料一样使用它们——恰到好处。

### 3.1 光与影

- **环境光**：页面顶部或侧边的极淡渐变，暗示光源方向，给页面"空间感"（氛围色 `opacity 0.02–0.05`）
- 焦点光 → 见空间层级（2.0）光影型⑤
- 交互影 → 见空间层级（2.0）浮雕型④
- 深度影 → 见空间层级（2.0）悬浮层 L3

### 3.2 动态与节奏

- **进入动画**：页面内容不要硬切出现，用极短的 fade+轻微位移（150–250ms）让内容"浮现"
- **数据动画**：数字变化时用缓动过渡而非瞬间替换，让用户感受到"变化的过程"
- **交互反馈**：按钮点击的轻微 scale、切换的 spring 弹性、拖拽的跟随感
- **Stagger**：列表项、卡片网格依次进入（间隔 30–50ms），创造"一个接一个铺开"的节奏感
- **状态过渡**：loading → 数据 → 空状态之间用 crossfade 衔接，不要突然跳变

动效的原则：**让界面感觉"活着"，但不要让它"动个不停"。** 动效应该在用户触发交互时出现，而不是自顾自地表演。

### 3.3 材质与质感

- **玻璃质感**：仅用于 L3 悬浮层（顶部导航栏、浮动工具条）——`backdrop-blur` + 半透明背景暗示"浮在内容上方"

> 纹理暗示、渐变面等低频手法见《美学规范补充》。

### 3.4 状态演绎

每种 UI 状态都是展现品位的机会：

- **空状态**：不是冷冰冰的"暂无数据"，而是一个有性格的空间——可以有轻量插画、友好的引导文案、甚至一点幽默
- **加载态**：不只是转圈。可以是内容骨架屏带呼吸动画，或关键数字的模糊→清晰过渡
- **成功态**：让用户感到"做完了，舒服"——一个短暂的勾选动画、一条温暖的提示
- **错误态**：不是警报，是帮助。用矿物红标记问题，但语气温和，重点放在"怎么修"而不是"你错了"

## 4. 场景气质

> 每个页面类型有自己的情感目标。同一套视觉基因，可以因场景不同而呈现不同气质——就像同一个人穿不同场合的衣服，始终是同一个人。

### U1 提交（表单/编辑）

**情感**：专注、高效、轻松

用户的心态是"我有东西要提交，帮我快速搞定"。

- 页面应该像一张整洁的工作桌——工作区明确（白卡），工具位置固定
- 高频字段直接显示，低频选项折叠但可发现
- 操作路径短：填完就能提交，不需要翻页或跳转
- 错误就地显示，不用弹窗打断思路
- 成功反馈在上下文内，让用户知道"搞定了"并自然地继续下一件事

**深度机会**：字段之间的微妙间距节奏让表单看起来不像表格；提交按钮的点击反馈让"完成"有仪式感。

### U2 查阅（列表/搜索）

**情感**：通透、快速、可控

用户的心态是"我在找东西，别挡我路"。

- 列表行通透，不要用独立卡片包裹每一行——卡片会让页面变得厚重
- 行点击区域覆盖整行，不要让用户精确瞄准某个小链接

**深度机会**：行 hover 时轻微的背景色变化和信息浮现；搜索结果高亮匹配词；滚动时的轻柔阴影暗示"上面还有内容"。

### U4 分析（数据/图表）

**情感**：掌控、洞察、安定

用户的心态是"我需要看懂趋势，做出判断"。

- 这是**指挥室，不是告警台**。核心数据大而稳，趋势图平滑流畅
- 异常数据用颜色"托出来"，不是用闪烁"警报"——应该是"嗯，这里需要留意"，不是"警告！！！"
- 每屏一个核心 KPI 做视觉主角，其他数据甘当配角
- 图表是色彩的舞台——这里是使用丰富色彩的最佳场景。图表配色应独立于状态语义色，用同色调渐变展开（如暖色系从浅到深），避免红色趋势线被误读为异常

**深度机会**：数据加载时的数字滚动动画；图表 hover 时的详细数据浮层；KPI 数字旁的微妙趋势箭头带缓动动画；关键阈值附近颜色的渐变过渡。

### M1 审核（逐条决策）

**情感**：节奏感、判断力、成就感

用户的心态是"我要逐条过，快速决策"。

- 行内完成决策——通过/拒绝不需要打开新页面
- "通过"是常态，视觉上安静（矿物绿）；"拒绝"是例外，hover 时才暴露（矿物红）
- 批量操作需要确认（防误操作），但确认过程简洁
- 已处理项和待处理项视觉上有明确区分

**深度机会**：处理一条后的轻微退场动画（fade + 高度收缩）；批量处理完成后的计数弹跳动画；审核队列减少时的进度感。

### M2 配置（设置/调参）

**情感**：精确、安心、掌控

用户的心态是"我要精确地设置一些东西，设完就忘"。

- 表单清晰分组，每组有明确的标题说明"这是在设置什么"
- 变更有即时反馈（保存成功/失败）
- 危险操作（删除、重置）视觉上与普通操作明确区分，有二次确认
- 未保存的修改离开时有提示

**深度机会**：开关切换的 spring 动画；保存成功后按钮的短暂"完成"状态变化；危险操作确认框的适度紧张感（不是恐吓，是提醒）。

### M3 库管（素材/资源管理）

**情感**：整洁、高效、可整理

用户的心态是"我要找到、整理、归类这些东西"。

- 删除操作的视觉模式是撤销而非恐吓——轻量的 undo toast，不是红色警告弹窗
- 支持灵活的视图切换（列表/网格），两种视图保持相同的视觉气质

**深度机会**：拖拽排序的跟随感和落位动画；删除后的撤销 toast 带倒计时进度条；网格视图的封面图带微妙的 hover 放大。

### M4 总览（仪表盘）

**情感**：全局感、安心、发现异常

用户的心态是"让我快速扫一眼，一切还好吗"。

- 仪表盘是"鸟瞰图"——信息密度高但层级清晰
- 正常状态安静（灰阶为主），异常模块自动获得最高视觉优先级
- 每个模块提供下钻入口——总览只给结论，细节按需展开

**深度机会**：进入仪表盘时模块卡片的 stagger 进入动画；异常模块边缘的微妙呼吸光晕；模块间的视觉节奏（大卡+小卡交替排列）。

## 5. 效率底线

> 以下 6 条是不可逾越的底线。不是因为它们比美学重要，而是因为没有它们，美学就失去了载体——用户用不了的界面，再好看也没人看。

1. **3 秒定位主操作**：每屏的主操作按钮必须在 3 秒内被找到。如果两个按钮长得一样亮，用户会犹豫。

2. **输入框可辨识**：输入框必须与背景形成可辨识的边界。用户 3 秒内能定位"我在哪里打字"。实现方式自由——底色差、边框、轻阴影都可以，但必须做到。

3. **错误可见且就地**：错误状态必须在当前视口内可见，不需要滚动寻找。错误信息就近显示在出问题的地方。

4. **导航一致**：全站导航结构、位置、交互方式保持一致。用户在任何页面都能用同样的方式找到"去其他页面的路"。

5. **可交互元素看起来可交互**：按钮看起来像按钮，链接看起来像链接，可点击的行看起来可以点击。不要让用户猜"这个能不能点"。

6. **全站风格一致**：每个页面的色彩、字体、圆角、间距都属于同一套视觉语言。用户从 A 页面跳到 B 页面时，不应该觉得"这是另一个产品"。

## 6. 挑战条款

如果你认为某条规范在特定场景下会让界面更丑、体验更差、或与更重要的目标冲突：

→ 可以自行采用更优方案
→ 说明挑战理由（一句话，包含为什么你的方案更好）
→ 效率底线 6 条不可挑战

**优先级**：设计灵魂（整体感受）> 场景气质（情感目标）> 视觉基因（一致性）> 效率底线（可用性）

如果规范没有覆盖某个维度，回到"设计灵魂"的画面——那个温暖的工作室——问自己：这个决策放在那个空间里，违和吗？
