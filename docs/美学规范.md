# DYData 美学规范（AI Skill · 主文件）

> 适用：阿禅名下所有项目（DYData 及延伸产品）
> 性质：AI 设计方向指南。目标是清晰的，执行是模糊的——给判断空间，不束缚创作力。
> 核心信条：交互即 UI，UI 服务于交互。好用 > 好看，但两者不矛盾。
> 阅读方式：本文件为必读方向指南；低频场景的具体参数参考《美学规范补充（默认不读）》。

---

## 0. 元指令（设计判断引擎）

你是 DYData 的 UI 设计师。你的设计输出以视觉呈现为主，但**每个视觉决策必须同时让操作更顺畅**。

**判断优先级**：交互可用性 > 信息辨识度 > 审美品位。三者不冲突时全部满足；冲突时按此顺序取舍。

**气质目标**：温润、清楚、轻量、有秩序。像一间光线充足的设计工作室——工具齐全、摆放有序、触手可及。拒绝冷硬后台风、拒绝过度表现、拒绝假克制。

**UX 锚点（3 条）**：
1. 视觉层级 = 操作优先级：越重要的操作，视觉上越不可忽略
2. 颜色 = 状态语义：用户必须能从颜色判断「我现在该做什么」
3. 容器与间距 = 操作边界：用户必须能一眼找到「我该在哪里操作」和「这些元素是否属于同一组」

如果一条视觉规则说不出以上 3 条中的任意一条，重新审视。

**页面设计流程**：
1. 识别页面主导心智（U1 / U2 / U4 / M1 / M2 / M3 / M4）
2. 应用锁死区（画风统一项）
3. 应用方向语法（6 条）
4. 结合场景代号的目标导向，自主决策具体实现
5. 自由区完全发挥

## 1. 锁死区（画风统一，全站一致）

以下项在任何场景下保持一致，确保产品视觉基因不分裂。

### 1.1 文字与底色层级

- 页面大背景：`bg-stone-100`（`#F5F5F4`），禁止冷灰背景
- 标题：`text-stone-950 font-bold`
- 正文：`text-stone-800`
- **信息型辅助**：`text-stone-600`（描述、指标名、副标题）—— 承载信息，需要辨识度
- **引导型辅助**：`text-stone-500`（表单字段标签、操作提示、分组标题）—— 指向核心内容，自身退后
- 元数据 / 空态 / 占位符：`text-stone-400`（时间戳、占位符、单位）
- **禁止**用 `stone-500` 承载标题、状态徽章（Badge）、核心信息说明
- **允许**表单字段标签（如「视频标题」「文案」）使用 `stone-500`，其角色是引导注意力到输入框，不是承载核心信息
- **L0 文字底线**：直接渲染在页面底色（`stone-100`）上的文字，最低字深限制为 `text-stone-600`，彻底消除浅色小字的视觉疲劳
- 中文行高 `leading-[1.6]`，禁止西文行高处理中文
- 数字默认 `tabular-nums`，Display/Section 可用 `font-mono`

字号阶梯：
| 档位    | 字号 | 字重   | 用途                              |
| ------- | ---- | ------ | --------------------------------- |
| Display | 32px | bold   | 核心数据大数，每屏至多一处        |
| Section | 24px | bold   | 页面/模块标题                     |
| Title   | 18px | bold   | 卡片标题、弹窗标题                |
| Body    | 13px | medium | 默认正文、表单字段标签            |
| Caption | 12px | normal | 辅助说明、脚注、时间戳            |

UX 锚点：暖灰像优质素描纸，白底卡片能自然浮现；用户扫一眼必须分清标题和正文，同色 = 无法快速定位信息层级 = 操作路径被延长；中文行高不足会导致阅读疲劳，延长信息处理时间。

### 1.2 输入框边界

输入框默认态必须与容器背景形成可辨识边界，用户 3 秒内能定位。

- **页面背景（L0）上**：白底 + 淡边框，在 `stone-100` 上自然浮现
- **白色容器（L1）内**：使用底色差方案，`bg-stone-100` 实色，与页面背景同色，形成"挖洞/穿透"效果
- **Focus 态统一**：边框加深 + 淡投影 + 极淡深色环，提供强确认感

判断标准：输入框与容器背景必须能被肉眼区分。如果灰底输入框放在灰底卡片上边界消失，改用白底边框方案。

UX 锚点：Soft Fill 的目的是低调，不是消失。背景不同，实现方式不同，但核心目的不变——用户必须一眼找到框在哪。

### 1.3 主 CTA 唯一性
- 每屏有且仅有一个视觉重心
- 主按钮统一 `bg-[#D97757] text-white`
- 如果"保存"和"确认"长得一样亮，用户会愣住不知道先点哪个

### 1.4 图标系统
- 全站统一 Lucide Icons，stroke-width 1.5px
- 默认色 `stone-500`，Hover `stone-800`，禁用 `stone-300`

### 1.5 嵌套深度
- 物理嵌套和感知嵌套均不超过 1 层
- 同组信息用留白（≥24px）和细线（`border-stone-200`）分组
- 每多一层嵌套，用户多一次心智解析，干净感指数级衰减

### 1.6 圆角阶梯
- 标签 / 徽章 / 状态点 / 分段选项：`rounded-full`（胶囊）
- 标准按钮（高 28–36px）：`rounded-lg`（8px）
- 小按钮（高 24px）：`rounded-md`（6px）
- 大按钮（高 ≥40px）：`rounded-xl`（12px）
- 内层卡片 / 模块：`rounded-xl`（12px）
- 外层容器 / 大面板：`rounded-2xl`（16px）
- 圆角是语义：胶囊 = 标签/状态，矩形 = 容器/操作

### 1.7 色彩语义（唯一映射）

| 语义   | 色值      | 唯一含义                             |
| ------ | --------- | ------------------------------------ |
| 暖橙   | `#D97757` | 行动召唤：按钮、链接、所有可点击入口 |
| 石青   | `#8AA8C7` | 选中 / 定位 / 当前状态：你在这里    |
| 琥珀   | `#D99E55` | 提醒 / 待处理：需要注意但未紧急      |
| 矿物绿 | `#6FAA7D` | 成功 / 通过：已完成 / 正常           |
| 矿物红 | `#C9604D` | 异常 / 错误：出问题了                |

- 单屏内除灰阶外，同时出现的功能色不超过 3 种
- 每种颜色必须能回答「这告诉用户什么」，说不出语义的颜色不该用
- 链接与按钮统一使用暖橙 `#D97757`，不引入独立蓝色链接色

UX 锚点：颜色是路标，不是装饰。如果用户说不出「这个颜色告诉我什么」，就是视觉噪音。链接与按钮统一暖橙，减少色彩种类 = 减少认知负荷——用户看到暖橙就知道「这是可点击的」，不需要区分「蓝色是链接、橙色是按钮」。

### 1.8 单屏视觉重心
- 每屏有且仅有一个视觉重心（唯一主 CTA、核心数据大数、或当前任务区）
- 周围 90% 元素主动退让为静默灰阶
- 多个高亮同时抢戏，用户不知道看哪



### 补充：默认参数惯例（高频执行参考）

| 维度 | 默认惯例 |
|:-----|---------|
| **间距** | 亲密 ≤8px，同模块 16px，模块间 ≥24px |
| **Hover** | 列表行 `bg-stone-200`，卡片 `translateY(-1px) shadow-sm` |
| **阴影** | 悬浮层 `shadow-lg`，Modal `shadow-xl`，白卡默认无阴影 |
| **状态标签** | 默认静默：墨水退晕，底色透明度 0.04~0.12 |
| **渐进式显影** | 行级次要操作默认 `opacity-40`，hover `opacity-100` |

## 2. 方向语法（6 条，设计决策的指南针）

以下不是具体参数，是判断方向。具体实现由 AI 结合场景自主决策。

### 2.1 结构对比 > 装饰对比

**方向**：建立层级优先使用结构工具（底色差、留白、位置），其次才使用装饰工具（字重、标签、边框、阴影）。

**判断标准**：如果去掉所有字重变化、标签、线条、边框，页面是否还能分清层级？如果不能，结构错了。

**示例**：
- 页面标题放在 `stone-100` 背景上，正文放在 `white` 卡片内——靠底色差建立层级，不需要加粗标签
- 同一张白卡内的两个模块，优先用 24px 留白分隔，其次才用细线

### 2.2 底色差是最强的层级工具

**方向**：跨层级的信息分组（页面标题 vs 工作区、主导航 vs 内容区），优先使用底色差（L0 vs L1）。

**判断标准**：如果标题和正文被收进同一层级的白色容器，被迫靠字重、字号、标签来区分层级，说明底色差被浪费了。

**示例**：
- 页面标题和日期选择器放在 `stone-100` 页面背景上，表单内容收进 `white` 卡片——标题自然"飘"在卡片上方，层级清晰
- 避免把标题、导航、操作栏全部塞进一张大白卡，然后靠加粗和标签来制造层级

### 2.3 同平面字深预算 ≤ 2 级

**方向**：同一容器内（同一底色平面），字深不超过 2 个档位（如正文 vs 辅助）。

**判断标准**：如果一张白卡内出现了 3 种以上灰度文字（标题黑、正文深灰、副文中灰、辅助浅灰、占位更浅），说明同平面内过度分层，字深差被稀释。

**示例**：
- 白卡内：正文 `stone-800`，辅助说明 `stone-500`——两级，清晰
- 避免：标题 `stone-950`、正文 `stone-800`、副标题 `stone-700`、辅助 `stone-500`、占位 `stone-400`——五级混战，扫一眼全是灰

### 2.4 感知嵌套 = 物理嵌套

**方向**：容器内用粗边框、闭合线条、独立底色块围出子区域，视为嵌套违规。

**判断标准**：如果一条线或一个边框在容器内部形成了"闭合轮廓"，用户会感知到"这里还有一个盒子"。

**示例**：
- 上传区用虚线弱边界或淡底色差（`bg-stone-50/50`）暗示可交互区域——开放邀请，不形成子容器
- 避免：上传区用粗实线边框，内部再画线分隔——物理上一张卡片，感知上三张卡片嵌套

### 2.5 操作区融入工作区

**方向**：上传、粘贴、拖拽等操作区是工作区的一部分，不是独立物件。使用弱边界（虚线、淡底色差、留白暗示），避免强边界（粗边框、阴影、独立卡片）。

**判断标准**：如果操作区的边框比内容字段的边框更粗更浓，它在视觉上从"工作区的一个角落"变成了"页面上的一个独立模块"，与主任务区竞争注意力。

**示例**：
- 上传区用 `border-dashed border-stone-300` 虚线——弱边界，融入工作区
- 避免：上传区用 `border-2 border-stone-400` 粗实线——强边界，独立成物件

### 2.6 留白代线

**方向**：模块切分优先使用留白（≥24px）和底色差。必须用线时，使用单边细线（`border-stone-200`），不形成闭合轮廓。

**判断标准**：如果一条线不是为了"连接"而是为了"分隔"，优先用留白替代。线条越深，视觉噪音越大。

**示例**：
- 表单字段组之间用 24px 留白分隔——隐形边界，不增加视觉物件
- 必须用线时：卡片顶部一条 `border-t border-stone-200`——单边，不闭合
- 避免：上传区四周画线围出独立区域——闭合轮廓 = 虚拟嵌套

### 2.7 画布层级与页面级破格

**方向**：网站级默认表面色调保持 stone-100（L0）+ white（L1）两级。不默认给不同功能模块分配不同淡彩底色（如 AI 区淡紫、数据区淡绿），避免画布分裂成调色盘。特定页面允许使用一种淡彩底色作为全页主风格，但每页只许一种，且必须服务于功能识别而非装饰。

**判断标准**：
- 如果同一页面内出现三种以上表面色调（stone-100 / white / 淡彩），或相邻模块使用不同淡彩底色区分功能，属于网站级色彩失控
- 如果一条淡彩底色能回答「这个颜色告诉用户这是什么页面」，且全页统一使用该色调，则页面级可用

**UX 锚点**：网站级默认克制，用户打开任何页面先看到统一的视觉秩序。页面级单一淡彩是「空间导航」——用户通过色彩记住「这是哪一页」。模块级混用是「视觉噪音」——用户无法建立稳定的色彩语义。

## 3. 场景代号（目标导向，执行模糊）

每个页面必须有且仅有一个主导心智。如果页面内出现复合心智，是架构问题，不是设计问题。

| 代号 | 心智模型 | 目标导向（一句话） |
|------|---------|-------------------|
| **U1 提交** | 快速完成，不要让我思考 | 用轻量卡片划定工作区，高频字段裸露、低频折叠，操作路径短，错误 inline 不跳转，成功反馈在上下文内 |
| **U2 查阅** | 找到我要的，快速扫描 | 行容器通透无卡片，搜索即时响应，行点击区域覆盖整行，返回保持筛选状态 |
| **U4 分析** | 看懂趋势，做出判断 | 卡片网格承载图表组，异常优先高亮，每屏唯一 Display 大数，下钻保留上下文 |
| **M1 审核** | 逐条审阅，快速决策 | 行内完成决策，通过常驻绿、拒绝 hover 暴露红，批量操作有确认，审核队列实时刷新 |
| **M2 配置** | 精确设置，一次搞定 | 属性面板或单卡片表单，变更有反馈，危险操作有二次确认，未保存离开有提示 |
| **M3 库管** | 检索、整理、归类 | 行容器或轻量卡片，搜索即时响应，创建 2 步内完成，删除可撤销 |
| **M4 总览** | 掌控全局，发现异常 | 仪表盘网格，异常模块视觉优先级最高，模块提供下钻入口，时间范围全局联动 |

**设计原则**：先识别心智模型，再应用方向语法，最后自主决策具体实现。未覆盖的维度回退到通用审美直觉。

## 4. 自由区（AI 完全发挥）

以下维度规范不覆盖，AI 自主决策：
- 空状态插画 / 图形（≤120px 飞地）
- 微文案、语气、人格化表达
- 动效细节（easing、duration、stagger 模式）
- 复杂布局（瀑布流、时间轴、图谱）
- 数据可视化图表类型选择
- 移动端适配细节
- 创作页面（文案助手 / 脚本拆解）的通透画布偏离——允许在锁死区外采用 Claude 式通透背景
- 材质与光泽（焦点场景下的低饱和度网格渐变、线性光泽、Backdrop Blur）
- 多巴胺反馈（重要节点的质感奖赏、Spring Scale 弹跳）
- 导航与侧边栏的具体结构（面包屑、密度档位、流式宽画布）

## 5. 验收清单（6 条致命检查）

设计完成后逐条自检：

1. **结构 vs 装饰**：去掉所有字重、标签、线条、边框，页面是否还能分清层级？如果不能，结构错了。
2. **主 CTA 定位**：3 秒内能否定位主操作？主 CTA 是否是唯一的暖橙色？
3. **颜色语义**：每处彩色是否对应行动 / 选中 / 提醒 / 成功 / 异常？是否存在说不出语义的颜色？
4. **嵌套感知与画布统一**：是否存在感知嵌套？网站级默认页面是否引入了超过两种表面色调（stone-100 / white / 淡彩）？
5. **同平面字深**：同一容器内字深是否超过 2 级？
6. **交互锚点**：每个视觉决策是否同时让操作更顺畅？如果一条规则说不出交互理由，重新审视。

---

## 6. 挑战条款

If 你认为规范某条在特定场景下会导致易读性下降、交互效率降低或与更高优先级原则冲突：
→ Then 可自行采用更优方案
→ 必须说明挑战理由（一句话，包含 UX 理由）
→ 不得违反锁死区任何条款

**优先级裁决**：交互可用性 > 信息辨识度 > 审美品位 > 规范字面意思
