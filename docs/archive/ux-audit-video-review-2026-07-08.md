# 视频审核 / 产量对账模块 UX 审计报告

> 审计人：产品设计师（审查官）
> 日期：2026-07-08
> 范围：`/video-review` 及其子页面（page、submit、exemption、archive、manage）
> 依据：《美学规范.md》及补充文件
> 说明：本报告以“非技术用户能看懂”为原则，所有问题均给出“白话解释 + 规范条款 + 修改方向”。未涉及代码实现细节，仅描述改什么、为什么这么改。

---

## 一、总体判断（一句话）

**当前 video-review 模块“能跑”，但远未到达“心流顺畅”的交付标准。**

最核心的问题是：**同一模块里混进了三套完全不同的心智模型**——普通成员每日交凭证、管理员审批请假、管理员审核待发稿。三套流程被强行折叠在同一个入口和同一批弹窗里，导航链路还有死链。如果用户不是非常熟悉后台逻辑，很容易迷路、点错、找不到该进的页面。

下面的问题按 **P0（必须修，否则用户会错/找不到/心断）/ P1（明显摩擦，建议修）/ P2（精致度，有余力再修）** 分类。

---

## 二、P0 阻塞级问题（不打回不能归档）

### P0-1 子页面 `/submit`、`/exemption`、`/archive` 全部死链（重定向回首页）

- **位置**：路由文件 `submit/page.tsx`、`exemption/page.tsx`、`archive/page.tsx`
- **现状**：这三个地址都直接 `redirect("/video-review")`，但代码里大量按钮和链接仍然指向它们。
  - `approved-list-data-container.tsx` 的「上传待审稿」按钮指向 `/video-review/submit`
  - `approved-row.tsx` 的「补交截图」链接指向 `/video-review/submit?edit=...`
  - `video-review-tabs.tsx` 的「提交作品」「申请豁免」「已发案例」三个 Tab 也指向这三个地址
  - 顶部全局导航「视频审核」点进来就是首页，下面却有一排看上去能点的 Tab
- **痛点**：用户看到按钮/链接，点完被踢回首页，会产生“系统坏了”或“我点错了”的混乱。这是典型的「可发现性零容忍」违反。
- **方案**：要么恢复这三个子页面（让它们真正承载对应功能），要么彻底删掉这些入口和 Tab，不要把用户骗到重定向里。如果功能已经在弹窗里完成，就不要留子路由的入口。
- **涉及规范**：【方向语法 2.5 操作区融入工作区】【可发现性零容忍】

### P0-2 「管理审核台」（待审稿件队列）没有入口

- **位置**：`/video-review/manage` 页面及 `ManageShell`
- **现状**：管理审核台是真正处理“稿件通过/打回”的页面，功能完整（左侧队列、右侧详情、J/K 快捷键、通过/打回），但从视频审核首页没有任何按钮或链接能进去。管理员只能记住网址或者靠导航猜测。
- **痛点**：一个核心管理页面藏在深巷，违反“好的功能必须能被主动发现”。
- **方案**：在管理员视角的「管理 ▾」下拉菜单里增加「管理审核台」入口，并给它一个与“产量看板”“审批”并列的位置；或者在首页顶部给出一个明显的「审核队列 · N」文字按钮。
- **涉及规范**：【可发现性零容忍】【锁死区 1.3 主 CTA 唯一性】（管理入口不宜过多，但必须存在）

### P0-3 同一页面混合三套心智模型，信息架构混乱

- **位置**：`/video-review` 首页（`video-review-workbench.tsx`）
- **现状**：首页同时承载：
  1. 普通成员每日交凭证（顶部「上传作品凭证」弹窗）
  2. 已发案例案例库（中间卡片网格，其实是“已通过稿件”）
  3. 管理员产量看板（弹窗）
  4. 管理员请假审批（弹窗）
- **痛点**：一个页面既是 U1 提交、又是 U2 查阅、又是 M4 总览、又是 M1 审核。不同角色打开同一张页面，看到的东西主次不清。普通成员最该看到“今天交了几条/还差几条”，但现在这些信息被压成顶部 header 里的一行小字；管理员最该看到“红灯/待审”，却要先进弹窗。
- **方案**：
  - 普通成员首页：今日目标/已交/差额大数卡 + 快速提交区 + 今日提交历史（U1 主导）。
  - 管理员首页：保留「产量看板」和「审批」入口，但把核心指标（全队目标/已交/红灯数）直接展示在页面上，而不是全部藏在弹窗后。
  - 已发案例库可以独立为子页面 `/archive`，或在首页作为次要模块下沉。
- **涉及规范**：【场景代号 U1/U2/M1/M4】【锁死区 1.8 单屏视觉重心】

### P0-4 管理审核台通过/打回按钮颜色语义错误

- **位置**：`review-detail.tsx` 底部操作区
- **现状**：「通过 · 入库」按钮使用暖橙色 `#D97757`（主行动色），「打回 · 改进」使用白底灰边。
- **痛点**：在 M1 审核场景下，通过应当使用矿物绿 `#6FAA7D`，打回/拒绝应当使用矿物红 `#C9604D`。暖橙色在这里会跟“点击去提交/点击去申请”等主 CTA 混淆，管理员无法通过颜色快速判断“这是通过操作还是跳转操作”。
- **方案**：
  - 「通过」改为矿物绿实心或绿底白字；
  - 「打回」改为矿物红，或在 hover/激活时显红；
  - 两者并列常驻时，必须让绿/红语义一眼可辨。
- **涉及规范**：【锁死区 1.7 色彩语义】【补充 1.4 M1 审核】

### P0-5 待审队列当前选中指示器颜色错误

- **位置**：`queue-list.tsx`
- **现状**：当前激活的队列项左侧有一条 2px 的暖橙色竖线。
- **痛点**：暖橙色 = 行动/可点击，而这里表达的是「当前位置/选中」。应当是石青色 `#8AA8C7`。
- **方案**：激活态竖线改为石青色，同时行背景 hover 使用 `bg-stone-100` 即可。
- **涉及规范**：【锁死区 1.7 色彩语义】

### P0-6 「只看自己」开关使用暖橙色表示激活

- **位置**：`approved-list.tsx`
- **现状**：开启「只看自己」后，按钮变成暖橙底/橙字。
- **痛点**：这是一个状态开关（选中/未选中），不是主 CTA。暖橙色会让用户误以为“点击它会触发一个动作”。选中态应使用石青色 `#8AA8C7`。
- **方案**：激活态改为 `bg-[#8AA8C7]/10 text-[#8AA8C7] border-[#8AA8C7]`。
- **涉及规范**：【锁死区 1.7 色彩语义】

### P0-7 已发案例卡片「我的」标签滥用暖橙色

- **位置**：`case-card.tsx`
- **现状**：用户自己提交的卡片左上角有一个暖橙色实心标签「我的」，带阴影。
- **痛点**：暖橙色在这里不是行动召唤，而是“归属标记”。长期会让用户形成错误映射：橙色 = 我的东西，而不是橙色 = 可点击。
- **方案**：「我的」标签改用中性处理——例如白底黑字细边框，或淡石青底，或仅用头像/名称旁的小字标注。不要让它成为页面最跳的颜色。
- **涉及规范**：【锁死区 1.7 色彩语义】【锁死区 1.3 主 CTA 唯一性】

### P0-8 已发案例列表行（`approved-row.tsx`）同样滥用橙色标记“我的”

- **位置**：`approved-row.tsx`
- **现状**：自己的行左侧有 3px 暖橙竖线 + 浅橙背景，提交者名字也变成橙色。
- **痛点**：与 P0-7 同理，把行动色用成了归属标记。
- **方案**：
  - 如果“我的”确实需要高亮，使用石青色（表示“当前与我相关”）或干脆只用左侧一个 subtle 的灰点；
  - 不要让整行背景变色去抢视觉重心。
- **涉及规范**：【锁死区 1.7 色彩语义】

## 三、P1 高摩擦问题（建议第二批修）

### P1-1 首页核心大数字号过小，Display 档位缺失

- **位置**：`dashboard-dialog.tsx` 顶部三指标、`production-dashboard.tsx` 四指标卡、`submit-workbench.tsx` 三指标卡
- **现状**：数字多为 18px，少数 28px，均未到规范 Display 档位 32px。
- **痛点**：对于 M4 总览/U4 分析页面，核心数据必须一眼可见。18px 和正文拉不开差距，用户扫一眼找不到重点。
- **方案**：核心大数字统一改为 `text-[32px] font-bold font-mono tabular-nums`。每屏只设一个 Display 重心，其余数字降级为 18px 辅助。
- **涉及规范**：【锁死区 1.1 文字与底色层级】【补充 1.7 M4 总览】

### P1-2 多处输入框 Focus 环颜色不统一

- **位置**：`submit-form.tsx`、`submit-workbench.tsx`、`exemption-workbench.tsx`、`production-dashboard.tsx`
- **现状**：这些页面的输入框 focus 是 `focus:ring-stone-950/5 focus:border-stone-200`（灰环），而 `submit-dialog.tsx`、`exemption-dialog.tsx`、`approved-list.tsx` 搜索框是暖橙环。
- **痛点**：同一产品里，输入框聚焦时有的发橙光、有的发灰光，用户无法建立“我激活了这个框”的稳定预期。
- **方案**：全站输入框 focus 统一为暖橙色环（主 CTA 色），即 `focus:ring-[#D97757]/20 focus:border-[#D97757]/40`。
- **涉及规范**：【锁死区 1.2 输入框边界】

### P1-3 多处输入框在白容器内边界不清（挖洞感不足）

- **位置**：`submit-form.tsx`、`submit-workbench.tsx`、`exemption-workbench.tsx`、`production-dashboard.tsx`
- **现状**：输入框使用 `border-0 bg-stone-100/70`（半透明灰底、无边界），放在白卡片里时，某些屏幕/浏览器下边界会“化开”。
- **痛点**：规范要求白容器内输入框应形成「挖洞/穿透」效果，即实色 `bg-stone-50` + 可辨识边界。半透明灰底在对比度差的屏幕上会跟白卡片糊在一起。
- **方案**：白卡片内的输入框统一改为 `bg-stone-50` 实色，可保留细边框或明确底色差。弹窗内已实现的部分（`submit-dialog.tsx`）作为标杆。
- **涉及规范**：【锁死区 1.2 输入框边界】

### P1-4 标签/小标题过度使用大写 + 超宽字距

- **位置**：`video-review-workbench.tsx`（"Fulfillment & Quota"）、`manage/page.tsx`（"Review Workbench"）、`submit-form.tsx`（"Step 0" / "Review"）、`review-detail.tsx`（"话术原文" / "截图"）
- **现状**：大量 `uppercase tracking-[0.25em]` 的英文/中文小标题，字间距拉得很开。
- **痛点**：这种“设计工作室感”的排版在中文界面里显得造作、降低扫描效率，也不符合“温润清楚轻量”的气质。对普通用户来说，这些标签不像系统语言，像装饰。
- **方案**：
  - 删除所有 `uppercase tracking-[0.25em]` 处理；
  - 中文标签保持正常字距，使用 `text-stone-400` / `text-stone-500` 即可；
  - 英文标注（如确需）改为句子大小写（Sentence case），不要全大写。
- **涉及规范**：【锁死区 1.1 文字与底色层级】【气质目标】

### P1-5 空状态处理不一致，多处只是干瘪文字

- **位置**：
  - `production-dashboard.tsx`："暂无相关成员数据"
  - `exemption-workbench.tsx`："暂无历史申请记录"
  - `submit-workbench.tsx`："今天还没有提交发片凭证"
  - `approval-dialog.tsx` 已审历史："暂无历史审批处理记录"
- **现状**：只有文字，没有图标、没有引导按钮。
- **痛点**：用户看到空白页面不知道下一步该做什么，容易以为功能坏了。
- **方案**：统一空状态三件套——Lucide 图标 + 一句说明 + 暖橙 CTA。例如：
  - 无成员数据 → 提示「当前筛选条件下无数据」+ 「清除筛选」按钮
  - 无历史申请 → 提示「还没有申请过豁免」+ 「去申请」按钮
  - 无提交记录 → 提示「今天还没交凭证」+ 「上传凭证」按钮
- **涉及规范**：【4. 状态反馈】【补充 1.1 U1 提交】

### P1-6 管理审核台空状态缺少刷新入口

- **位置**：`manage-shell.tsx`
- **现状**：队列为空时显示「队列已清空」，但没有刷新按钮或返回提示。
- **痛点**：如果用户刚处理完几条，想确认是否真的清空，只能刷新浏览器。
- **方案**：在空状态加「刷新列表」次按钮，或自动刷新提示。
- **涉及规范**：【4. 状态反馈】【补充 1.4 M1 审核】

### P1-7 错误兜底仅显示琥珀色文字，无重试按钮

- **位置**：`review-queue-data-container.tsx`、`approved-list-data-container.tsx`
- **现状**：加载失败时只展示一段琥珀色文字 `errorMessage`。
- **痛点**：用户遇到错误时只能刷新整页，没有原地恢复路径。
- **方案**：错误提示增加「重新加载」按钮，点击后重新获取数据。
- **涉及规范**：【4. 状态反馈】【边界场景兜底】

### P1-8 弹窗标题字号/字深不统一

- **位置**：多个 Dialog
- **现状**：
  - `submit-dialog.tsx`、`exemption-dialog.tsx`、`approval-dialog.tsx`、`dashboard-dialog.tsx`：18px / `text-stone-950` ✅
  - `case-detail-dialog.tsx`：16px / `text-stone-800` ❌
  - `production-dashboard.tsx` 内的作品详情 Dialog：18px / `text-stone-800` ❌
  - `submit-workbench.tsx` 内的放大预览 Dialog：无标题
- **痛点**：弹窗标题字深和字号跳来跳去，同一产品像出自不同人之手。
- **方案**：所有 Dialog 标题统一为 `text-[18px] font-bold text-stone-950`（Title 档位）。
- **涉及规范**：【锁死区 1.1 文字与底色层级】

### P1-9 主页面 h1 标题过小

- **位置**：`video-review-workbench.tsx`、`manage/page.tsx`
- **现状**：页面大标题使用 `text-[18px]`。
- **痛点**：页面标题应当使用 Section 档位 24px，18px 是弹窗/卡片标题的级别。
- **方案**：页面 h1 统一为 `text-[24px] font-bold text-stone-950`。
- **涉及规范**：【锁死区 1.1 文字与底色层级】

### P1-10 阴影使用不克制

- **位置**：`case-card.tsx`、`submit-workbench.tsx`、`exemption-workbench.tsx`、`review-detail.tsx`、`video-review-tabs.tsx` 等
- **现状**：卡片 hover 带 shadow-sm、主按钮带 shadow-sm、标签带 shadow-sm、Tab 选中带 shadow-sm。
- **痛点**：规范要求白卡默认无阴影，阴影只用于悬浮层/弹窗。过多阴影会让页面产生“廉价后台系统感”。
- **方案**：
  - 白底卡片默认态去掉 shadow；
  - 列表/卡片 hover 用微位移 + 极淡阴影或不加；
  - 按钮默认不加 shadow，只有悬浮层（Dropdown/Popover/Modal）使用 shadow-lg/xl。
- **涉及规范**：【补充 2.4 阴影与边框】【锁死区 1.5 嵌套深度】

### P1-11 多处圆角语义混乱

- **位置**：全模块
- **现状**：
  - 标准按钮（h-9/h-10）有的用 `rounded-lg` ✅，有的用 `rounded-xl` ❌（如 `exemption-workbench.tsx` 提交按钮 h-11 rounded-xl）
  - 大按钮/主 CTA 规范应为 `rounded-xl`（高 ≥40px），但部分 h-9 按钮也用了 rounded-xl
  - 小缩略图有的 `rounded-lg` ✅，有的 `rounded-xl` ❌
- **痛点**：圆角是语义，混用会让组件失去辨识度。
- **方案**：全站按规范阶梯执行：
  - 标签/徽章：rounded-full
  - 标准按钮 h-9：rounded-lg（8px）
  - 大按钮 h-11：rounded-xl（12px）
  - 内层卡片：rounded-xl
  - 外层大面板/弹窗：rounded-2xl
- **涉及规范**：【锁死区 1.6 圆角阶梯】

### P1-12 管理审核台布局无移动端适配

- **位置**：`manage-shell.tsx`
- **现状**：使用 `grid grid-cols-[320px_1fr]`，没有任何响应式处理。
- **痛点**：在 1024px 及以下屏幕，左侧 320px 固定宽度会挤压右侧审核区，甚至导致水平滚动；375px 移动端基本不可用。
- **方案**：
  - 大屏：左侧 320px 队列 + 右侧详情；
  - 中屏（<1280）：左侧 280px；
  - 小屏（<1024）：改为上下堆叠或抽屉式队列，保证详情区可读。
- **涉及规范**：【16. 响应式检查】【移动端适配】

### P1-13 产量看板弹窗筛选栏固定宽度可能导致换行断裂

- **位置**：`dashboard-dialog.tsx`
- **现状**：日期输入固定 `w-32`，团队/小组下拉固定 `w-36`，「只看红灯」按钮 `ml-auto`。
- **痛点**：在中等宽度弹窗或移动端，这些固定宽度的控件可能换行不自然，导致筛选行高度不均、按钮被挤到第二行。
- **方案**：使用更弹性的宽度（如 `min-w-[8rem] flex-1`），并在小屏时垂直堆叠筛选控件。
- **涉及规范**：【16. 响应式检查】【方向语法 2.6 留白代线】

### P1-14 删除提交记录使用浏览器原生 `confirm()`

- **位置**：`submit-workbench.tsx` `handleDeleteSubmission`
- **现状**：点击删除直接弹浏览器确认框。
- **痛点**：原生确认框文案由浏览器控制，无法说明“删除后不可恢复”，也不够温润；且与全站使用 toast/Dialog 的反馈风格不一致。
- **方案**：改为行内二次确认（例如点击后按钮变为「确认删除？」红色按钮，或弹一个统一风格的 Dialog）。
- **涉及规范**：【8. 破坏性信心】【4. 状态反馈】

### P1-15 作品详情弹窗内元数据区域形成感知嵌套

- **位置**：`case-detail-dialog.tsx`
- **现状**：文案块用 `rounded-xl bg-stone-50 p-4`，元数据又用 `rounded-xl border border-stone-100 p-3 bg-stone-50/20`。
- **痛点**：在白底弹窗里再画一个闭合的浅色盒子，形成二级感知嵌套，增加解析负担。
- **方案**：
  - 元数据改为无框列表，用 `border-b border-stone-100 last:border-0` 分隔；
  - 或改为 2 列极简文字，靠留白分组。
- **涉及规范**：【方向语法 2.4 感知嵌套 = 物理嵌套】【锁死区 1.5 嵌套深度】

### P1-16 管理审核台历史反馈区形成感知嵌套

- **位置**：`review-detail.tsx` `FeedbackHistorySection`
- **现状**：在主卡片内又套了一个 `rounded-xl border border-stone-200 bg-stone-50/60` 的盒子。
- **痛点**：审核详情本身已经是一个大卡片，历史反馈再包一个盒子，出现“盒中盒”。
- **方案**：历史反馈区去掉外边框和背景，改用顶部细线 `border-t border-stone-100` + 留白分隔。
- **涉及规范**：【方向语法 2.4 感知嵌套】

### P1-17 提交表单 ReviewStep 汇总区形成感知嵌套

- **位置**：`submit-form.tsx` `ReviewStep`
- **现状**：确认提交内容用一个 `rounded-xl bg-stone-100/50 p-4` 的灰底盒子包裹。
- **痛点**：在 StepWizard 的白底工作区内再包一层灰盒，层次冗余。
- **方案**：去掉灰底盒，直接用分组文字 + 单边细线或留白分隔。
- **涉及规范**：【方向语法 2.4 感知嵌套】

### P1-18 产量看板/提交页指标卡过度使用独立卡片

- **位置**：`production-dashboard.tsx`、`submit-workbench.tsx`
- **现状**：3~4 个指标每个都是 `rounded-2xl border bg-white p-5 h-[100px/110px]` 的独立大卡片。
- **痛点**：卡片过多会让页面变成“卡片马赛克”，尤其这些数字本可以合并到一个轻量指标条里。
- **方案**：
  - 改为顶部一条横向指标条（无卡片边界，用留白/细竖线分隔）；
  - 或最多用一个统一容器装下所有指标，内部用细线分隔，禁止每个指标独立成卡。
- **涉及规范**：【方向语法 2.4 感知嵌套】【补充 1.7 M4 总览】

### P1-19 产量看板弹窗小组折叠头仍是盒子嵌套

- **位置**：`dashboard-dialog.tsx`
- **现状**：小组标题使用 `bg-stone-50/50 p-2.5 rounded-xl border border-stone-100`。
- **痛点**：在白色弹窗内容区里，每个小组头又画了一个浅灰闭合盒子，造成嵌套感。
- **方案**：小组头改为纯文字 + 箭头，用 24px 留白与上方内容分隔，不要用背景框。
- **涉及规范**：【方向语法 2.4 感知嵌套】【方向语法 2.6 留白代线】

### P1-20 图片放大查看器不统一

- **位置**：`production-dashboard.tsx`、`submit-workbench.tsx` vs `image-lightbox.tsx`
- **现状**：主工作台和管理审核台使用统一的 `ImageLightbox` 组件，但产量看板详情和提交页却各自用 Dialog 包了一个 `<img>` 放大。
- **痛点**：同一动作（看大图）有三种不同实现，关闭方式、遮罩颜色、ESC 行为可能不一致。
- **方案**：所有大图查看统一复用 `ImageLightbox`，确保体验一致。
- **涉及规范**：【视觉一致性】【13. 浏览器调试】

## 四、P2 精致度问题（第三批再修）

### P2-1 加载骨架屏缺少语义结构

- **位置**：`loading.tsx`
- **现状**：一个大方块 + 一个横条 + 6 个小方块脉冲。
- **痛点**：没有暗示出“这是标题区、这是筛选区、这是列表区”，用户等待时无法预判内容结构。
- **方案**：按真实页面结构做骨架屏：顶部 header 区、搜索/操作栏、卡片网格占位。
- **涉及规范**：【4. 状态反馈】

### P2-2 加载 Spinner 颜色不统一

- **位置**：`production-dashboard.tsx`（橙边）、`approval-dialog.tsx`（灰）、`dashboard-dialog.tsx`（灰）
- **现状**：有的 spinner 是 `border-t-[#D97757]`，有的是 `text-stone-400`。
- **方案**：全站 loading spinner 统一为 `text-stone-400` 或统一为主色，不要混用。
- **涉及规范**：【视觉一致性】

### P2-3 部分自定义按钮缺少 focus-visible 样式

- **位置**：`approved-list.tsx`、`submit-dialog.tsx`、`approval-dialog.tsx` 等使用原生 `<button>` 的地方

- **痛点**：键盘用户无法看到焦点位置，可访问性差。
- **方案**：所有交互按钮补充 focus-visible 环。
- **涉及规范**：【13. 浏览器调试】【无障碍】

### P2-4 禁用态透明度不一致

- **位置**：多处按钮
- **现状**：有的 `disabled:opacity-60`，有的 `disabled:opacity-50`。

- **涉及规范**：【视觉一致性】

### P2-5 日期输入在部分浏览器下样式不可控

- **位置**：所有 `type="date"` 输入框
- **现状**：直接使用原生 date input，无自定义图标。
- **痛点**：iOS/Safari 下原生日期选择器样式可能与设计不符。
- **方案**：如要极致一致，可封装日期选择组件；若保留原生，至少确保边框/背景色在所有浏览器下可见。
- **涉及规范**：【自由区】【移动端适配】

### P2-6 `approved-row.tsx` 疑似已弃用

- **涉及规范**：【减法哲学】
