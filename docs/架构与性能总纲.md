# DYData 架构与性能总纲

> 本文件是全站架构与性能的唯一总控。任何性能 / 架构 / 慢页面类任务，动手前必须先读本文件。
> 本文件与代码冲突时，一律以代码为准，并当场修正本文件（见「维护规则」）。
> 最后核对代码日期：2026-07-16

---

## 〇、这份文件怎么用

面向 AI：接到「某页面慢 / 优化性能 / 改架构 / 加接口」类任务时，按顺序走：

1. 读本文件「二、岗位分诊」，判断这件事归哪个领域，明确不能越界动什么。
2. 读本文件「三、架构地图」，找到目标页面依赖哪些加载器、表、接口。
3. 实测现状（见「五、排查流程」），不凭本文件里的旧数字下结论。
4. 只做该领域内的最小改动。
5. 把实测结果和改动记进「六、实测台账」。

面向阿禅：你不用读代码。你只需要用「岗位语言」指挥，例如「这是前端性能问题，别动数据库」「这是代码治理问题，别顺手改功能」。分诊表就是你的指挥手册。

## 一、技术底座（不变的事实）

| 项 | 现状 |
|---|---|
| 框架 | Next.js App Router，服务端组件（RSC）为主 |
| 样式 | Tailwind + shadcn/ui |
| 数据库 | Supabase（新加坡 sin1 区），新增选题库后约 80 张表 |
| 部署 | Vercel，仅 `main` 分支 push 触发线上 |
| 数据加载 | 大部分页面走集中加载器（`src/lib/loaders/`）。当前仍未收口的主要例外是 `/violations`、`/admin/settings`；`/video-review` 首页已收口到 `loaders/video-review-page`，但历史案例图仍走截图签名链路 |
| 性能埋点 | `src/lib/perf.ts` 的 `measureAsync`，靠环境变量 `DYDATA_PERF_LOG=1` 打开 |
| 包体优化 | `next.config` 已开 `optimizePackageImports`（recharts/lucide/framer-motion）+ bundle-analyzer |
| 缓存 | 分两层：①页面/接口级用得少（无 `unstable_cache`/`revalidate`/react-query，仅 4 接口做进程内 60 秒内存缓存：content/list、dashboard-alerts、sidebar-badges、cockpit/queue-overview）→ 主要优化空间；②请求级已用 React `cache()` 对权限查询去重（permissions.ts、current-permission-context.ts）。别说"全站没缓存" |

术语（给阿禅）：
- 服务端组件 = 页面数据在服务器上先查好再发给浏览器，首屏更快。
- 加载器 = 每个页面专属的「采购清单」，一个文件集中管这个页面要哪些数据。
- 埋点 = 代码里埋的秒表，能测出每步花了多久。
- 缓存 = 把算过的结果先存着，下次不用重算，是最大的提速手段，目前几乎没用。

## 二、岗位分诊（出问题先对号入座）

| 领域 | 管什么 | 别越界动什么 | 主要执行 |
|---|---|---|---|
| 产品与信息架构 | 做不做、哪个页面重要、流程顺不顺 | 不碰具体代码 | 阿禅拍板 + Claude |
| 前端体验 | 长相、交互、好不好用 | 不碰接口和数据库 | Antigravity |
| 性能 | 快不快：首屏、请求数、接口耗时 | 不改业务规则和数据含义 | Claude 分诊 → 派前端/后端 |
| 后端与数据 | 接口对不对、数据准不准、查得快不快（含 AI 接口） | 不碰页面样式 | Codex |
| 权限与安全 | 谁能看、谁能改、后台权限 | 不靠临时补丁 | Codex |
| 部署运维 | Vercel/Supabase/域名/回滚/监控 | 不动业务逻辑 | Claude + Codex |
| 代码治理 | 文件分类、模块边界、改动能否一起发 | 不顺手改功能 | Claude 定规范 → Codex 执行 |

分诊原则：
- 「慢」几乎总要先由「性能」领域分诊，因为它常横跨前后端（这次慢就是首屏偷拉大接口，前端后端都踩）。先判断是前端的锅（请求太多/太早）还是后端的锅（接口本身慢），再派活。
- 一个任务只落一个领域。跨领域先拆，不允许一个任务里既改样式又改数据库。

模块全貌（有哪些模块、活/废、连累关系）见 `docs/全站模块地图.md`，本表只服务性能/依赖排查。

读法（给阿禅）：
- 加载器 = 这页打开时服务器一次性查好的东西，越重首屏越慢。
- 前端补充接口 = 页面打开后再慢慢拉的数据，可以延后、可以缓存，不该抢首屏。
- growth 当前走服务端单契约，不再做 initial/full 二次补载；其他页面是否分段加载必须按实测请求和首屏需求判断，不直接照搬旧链路。

架构约束（改代码时守住）：
- 首屏只放加载器里真正首屏要显示的数据；次要数据走前端补充接口，别塞进加载器。
- 目标是页面组件不直接写 supabase 查询、走 `loaders/` 保持「一页一采购清单」；但现状仍有例外（violations/admin-settings 仍页面内直查），这是待收口项，不是既成规则。新页面按目标做，改老页面顺手收口。
- 加载器内多个独立查询要用 `Promise.all` 并行，禁止一个 `await` 接一个 `await` 串行等。

## 四、性能红线（死规矩，基本不变）

每个核心页面的验收标准。达不到就算不合格，不看感觉。

| 页面 | 首屏目标 | 完整加载 | 首屏请求数上限 | 单接口上限 |
|---|---|---|---|---|
| `/dashboard` | 2 秒内 | 3 秒内 | 15 个 | 600ms |
| `/growth` | 2 秒内 | 3.5 秒内 | 12 个 | 首屏链路 600ms；补全链路 1200ms |
| `/admin/content` | 2.5 秒内 | 5 秒内 | 20 个 | 1200ms |
| 其他登录后页面 | 2.5 秒内 | 4 秒内 | 20 个 | 800ms |

术语：首屏 = 打开网页最先看到那一屏；请求数 = 网页向服务器要数据的次数，越多越慢；单接口上限 = 单个数据请求最多花多久。

通用红线：
- 加载器内串行 `await` 超过 2 个独立查询 → 必须改并行。
- 同一份数据一次加载里被查两遍 → 必须合并。
- 首屏拉了页面当前根本没显示的数据 → 必须延后。

## 五、排查流程（慢了先查什么）

固定顺序，别跳步、别凭猜：

1. 分领域：先按「二、岗位分诊」判断归谁，是前端锅还是后端锅。
2. 实测请求：浏览器开发者工具 Network 面板，数一数打开这页发了多少请求、哪个最慢、是不是一个等一个（串行）。
3. 实测接口耗时：本地跑时设 `DYDATA_PERF_LOG=1`，看控制台 `[perf]` 日志，定位最慢的加载器步骤。
4. 分类归因：把问题归到「前端请求太多/太早」「后端接口本身慢」「数据库缺索引/重复查」「缺缓存」之一。
5. 最小改动：只改归因那一类。前端派 Antigravity，后端/数据库派 Codex。
6. 复测 + 记账：改完重测同样指标，填进「六、实测台账」。

白屏 / 报错另走 debugging skill；本文件只管「慢」。

## 七、维护规则（这份文件不过期的保证）

因为阿禅不懂代码、无法核对本文件对错，所以「文件不过期」是 AI 的硬责任：

- 改到核心页面的加载器、依赖表、首屏接口 → 必须当场同步更新「三、架构地图」。
- 本文件任何描述与代码不符 → 一律以代码为准，并立即改本文件，不许将错就错。
- 每次性能优化 → 必须补「六、实测台账」一行。
- 新增核心页面 → 补进「三」和「四」。
- 顶部「最后核对代码日期」每次核对后更新。

配套机制：
- 硬规则写在项目根 `CLAUDE.md`，强制任何性能/架构任务先读本文件。
- 一键体检：Claude Code 环境有 skill「网站体检」自动走「五、排查流程」并回填台账；其他 AI 无此 skill 时，按「五」手动逐步执行，效果一致。
