# DYData 交互设计文档

> 角色：高级交互设计师  
> 范围：抖音数据日报平台全站交互审计  
> 版本：v1.0（基于 2026-05-18 代码快照）

## 一、用户核心路径与页面流转

### 1.1 用户生命周期路径

| 阶段 | 路径 | 关键页面 | 退出点 |
|------|------|----------|--------|
| 获客 | 外部链接/直接访问 → 注册 | `/register` | 团队列表加载失败时无重试 |
| 激活 | 注册 → 登录 → 首次填报 | `/login` → `/dashboard` | 首次填报无引导，直接暴露完整表单 |
| 留存 | 每日填报 → 查看趋势/排行榜 | `/dashboard` → `/growth` | 连续未填报无提醒（仅 cron 催交） |
| 管理 | 审批入队 → 权限配置 → 数据审核 | `/admin/*` | 权限变更无实时同步提示 |
| 退出 | 注销/被移除 | `/login` | 被移除后下次操作才感知 |

### 1.2 页面流转图（核心链路）

```
┌─────────┐    ┌─────────┐    ┌──────────┐    ┌─────────┐
│ /login  │───→│/register│───→│/dashboard│───→│ /growth │
└─────────┘    └─────────┘    └────┬─────┘    └─────────┘
      ↑                            │
      └────────────────────────────┘ (每日循环)
                                   │
                              ┌────┴────┐
                              │ /admin  │
                              │  cockpit│
                              └────┬────┘
                                   │
                    ┌──────────────┼──────────────┐
                    ↓              ↓              ↓
              ┌─────────┐   ┌──────────┐   ┌──────────┐
              │/analytics│   │/videos   │   │/conversion│
              └─────────┘   └──────────┘   └──────────┘
```

### 1.3 关键入口与跳转规则

| 来源 | 目标 | 触发条件 | 备注 |
|------|------|----------|------|
| `/login` | `/dashboard` | 登录成功 | 硬跳转，无渐进过渡 |
| `/register` | `/login` | 注册成功 | 需手动重新登录 |
| `/dashboard` | `/login` | 无 auth cookie | middleware 拦截 |
| `/admin/*` | `/login` | 无 auth cookie | middleware 拦截 |
| `/admin/ai-features` | `/admin/ai-channels` | 任何访问 | 301 重定向（遗留路由兼容） |
| `/demo` | `/dashboard` | 点击「退出演示」 | 清除 DEMO_VIEWER 状态 |

### 1.4 权限驱动的页面可见性

| 角色 | 可见页面 | 隐藏页面 |
|------|----------|----------|
| owner | 全部 | 无 |
| team_admin | 全部（除 AI 配置） | `/admin/ai-channels`（若未授权） |
| group_leader | `/dashboard`, `/growth`, `/analytics` | `/admin/*`（若未授权） |
| member | `/dashboard`, `/growth` | `/analytics`, `/admin/*` |
| DEMO_VIEWER | `/demo` 硬编码数据 | 所有真实数据页 |

**设计缺陷**：权限不足时页面直接 404 或白屏，无「申请权限」引导路径。

## 二、各页面交互行为拆解

### 2.1 登录页 `/login`

**表单字段**：邮箱（带记住功能）、密码

**交互细节**：
- 记住邮箱：localStorage 键 `dydata.rememberedEmail`，勾选后下次自动填充
- 提交状态：`useFormStatus.pending` 显示「登录中」，按钮 disabled
- 错误反馈：密码字段清空，`feedbackToast.error` 顶部弹出
- 无「显示密码」切换按钮
- 无「验证码防暴力破解」机制

**边界情况**：
- 网络中断：无超时处理，按钮永久处于「登录中」
- 密码错误 5 次：无账户锁定提示

### 2.2 注册页 `/register`

**表单字段**：邮箱、密码（带强度指示器）、确认密码、选择团队

**交互细节**：
- 密码强度：弱 `#D99E55` / 中 `#8AA8C7` / 强 `#6FAA7D`，基于长度/大小写/数字/特殊字符评分
- 团队列表：异步加载 `/api/register-teams`，加载中提交按钮 disabled
- 空团队列表：按钮保持 disabled，无「联系管理员创建团队」引导

**边界情况**：
- 团队 API 超时：无重试，无限 loading
- 邮箱已注册：后端返回错误，前端仅 toast 提示

### 2.3 仪表盘 `/dashboard`

**双表单并存**（技术债）：

#### 2.3.1 旧版 DashboardForm（日报填报）

**核心交互**：
- 截图导入弹窗：点击「导入截图」→ 选择图片 → OCR 识别 → 字段自动填充
- 提交：浮动操作栏（桌面）/ 固定底部栏（移动端，含 `safe-area-inset-bottom`）
- 成功反馈：字段旁显示「已提交 HH:MM」2 秒后消失（`data-success` + `setTimeout`）
- 修改模式：同表单，成功提示变为「已修改 HH:MM」

**状态流转**：
```
[空表单] → [填写中] → [提交中] → [成功 2s] → [锁定/可修改]
                ↓
           [OCR 导入中] → [字段填充]
```

#### 2.3.2 新版 VideoSubmitForm（视频提交）

**截图槽位状态机**（3 个槽位独立）：
```
empty → uploading → recognizing → pending_confirm → confirmed
                          ↓
                        failed（可重试）
```

**提交前校验链**（硬阻断，无分步保存）：
1. 必传截图已上传？
2. 元数据字段完整？
3. 话题标签已选（干货/复盘）？
4. `follower_convert > 0` 时 `script_text` 非空？

**AI 质检流程**：
- 提交成功后自动调用 `/api/dashboard/sample-quality-check`
- 返回 AI 标签（如「画面模糊」「标题过长」）
- 通过 `onSubmitted()` 回调回显到父组件

**设计缺陷**：
- 截图识别失败时仅槽位变红，无具体错误原因说明
- AI 质检结果无「忽略/申诉」入口，仅展示标签
- 无草稿自动保存，刷新页面全部丢失

### 2.4 成长分析 `/growth`

**交互模式**：纯数据展示页，图表 + 排行榜

**关键交互**：
- 时间范围切换：日/周/月，触发数据重载
- 排行榜排序：点击表头切换升/降序
- 个人趋势图：hover 显示具体数值 tooltip

**边界情况**：
- 无数据时段：显示空状态插图
- 加载中：骨架屏（`Skeleton` 组件）

### 2.5 数据分析 `/analytics`

**交互模式**：多维度数据看板

**关键交互**：
- 筛选器联动：日期范围 + 团队/组 + 指标类型
- 图表下钻：点击柱状图某列可查看明细（若实现）
- 导出：触发 `/api/export`，全量加载到内存生成 XLSX

**设计缺陷**：
- 导出无进度指示，大数据量时按钮无反馈
- 导出无大小限制，存在内存溢出风险

### 2.6 管理驾驶舱 `/admin`

**实时数据轮询**：`useSafeFetch(url, 60_000)` 每 60 秒刷新

**状态卡片**：
| 卡片 | 数据来源 | 空状态文案 |
|------|----------|------------|
| 待筛视频 | `/api/admin/pending-videos` | "今天的视频都已打标且无异常" |
| 待审违规 | `/api/admin/pending-violations` | "当前没有需要复核的违规案例" |
| 待催交成员 | `/api/admin/pending-reminders` | "所有在岗成员今天都已交报" |
| 待审豁免 | `/api/admin/pending-exemptions` | "暂无待审核的豁免申请" |

**交互细节**：
- 卡片 hover：轻微阴影提升
- 数字点击：跳转到对应管理子页
- 401 错误：整页显示错误状态，不自动跳转登录

### 2.7 入队审批 `/admin/join-requests`

**乐观更新模式**：
```
用户点击「同意」→ 行立即从列表消失 → toast「已同意」→ 后台请求
                                          ↓
                                    请求失败 → 行恢复 + toast 错误
```

**拒绝流程**：点击「拒绝」→ 展开 textarea 输入备注 → 确认

**设计缺陷**：
- 批量操作缺失：逐条审批效率低
- 无「稍后处理」状态，只有同意/拒绝二选一

### 2.8 违规复核 `/admin/conversion-hub` → ViolationsTab

**筛选器**：状态（待复核/全部/已确认/已驳回/已归档）+ 分类 + 关键词搜索

**复核卡片交互**：
- 风险等级选择：下拉选择高/中/低
- 管理员结论：文本输入，必填
- 建议操作：文本输入
- 提交复核：乐观更新状态，失败回滚

**状态流转**：
```
待复核 → [填写结论] → 已确认 / 已驳回
   ↓
已归档（独立操作）
```

**设计缺陷**：
- 无「暂存」功能，必须一次性完成复核
- 无历史修改记录，谁改的、什么时候改的不可追溯
- 驳回时无模板原因，每次手动输入

### 2.9 AI 管理助手 `/admin/ai-assistant`

**消息类型系统**：
| 类型 | 交互行为 | 示例 |
|------|----------|------|
| text | 纯展示 | 日常回复 |
| confirmation | 显示「确认/取消」按钮 | 删除成员、重置密码等敏感操作 |
| result | 展示操作结果 | 权限更新成功 |
| error | 红色高亮 + 重试按钮 | 请求失败 |

**确认流程**：
```
AI 发送 confirmation 消息 → 用户点击「确认」→ 发送确认请求 → 显示 result
                        → 用户点击「取消」→ 发送取消请求 → AI 重新提示
```

**重试机制**：失败消息旁显示「重试」按钮，点击重新发送该消息

**设计缺陷**：
- fetch 无 `AbortController` 超时，网络卡顿时永久 loading
- 无消息历史持久化，刷新页面对话丢失
- 无「复制消息」功能

### 2.10 权限管理 `/admin` → PermissionManager

**Sheet 弹层编辑**：
- 点击成员行 → 右侧滑出 Sheet
- Sheet 内标签页：编辑权限 / 变更角色 / 重置密码 / 移除成员

**权限编辑**：
- 复选框网格，按模块分组
- 实时保存：点击即触发 `updatePermissions` action
- 乐观更新：勾选后立即生效，失败回滚

**危险操作二次确认**：
- 重置密码：`ConfirmDialog` 确认
- 移除成员：`ConfirmDialog` 确认

**设计缺陷**：
- 权限变更后，被修改用户无实时通知（需刷新页面才感知）
- 无「权限模板」功能，每个成员逐条勾选

### 2.11 AI 渠道配置 `/admin/ai-channels`

**复杂状态管理**：
- 渠道列表（左栏，260px）+ 渠道详情表单（右栏）
- URL 驱动选中：`?channel=xxx` 或 `?channel=__new__`

**自动保存机制**（功能绑定区域）：
```
用户编辑功能配置 → 800ms debounce → 自动保存 API
                              ↓
                        保存中：显示「保存中…」
                        成功：显示「已保存」2s 后恢复
                        失败：显示「保存失败」+ toast 错误
```

**版本控制**：每个功能独立版本号，防止竞态条件（旧响应覆盖新数据）

**渠道操作**：
- 测试连接：按钮触发，显示测试结果
- OCR 测试：独立测试按钮
- 启用/停用：Toggle 开关
- 删除：`ConfirmDialog` 二次确认

**设计亮点**：
- 编辑后未保存离开无提示（无 `beforeunload` 拦截）
- 自动保存失败时保留错误状态，不自动重试

### 2.12 市场环境录入 `/admin/market`

**表单字段**：日期、是否交易日、三大指数涨跌幅、市场情绪、热门板块

**交互细节**：
- 日期选择：日历组件，选择已有数据日期自动填充（编辑模式）
- 涨跌幅：文本输入，前端 `Number.isNaN` 校验
- 热门板块：逗号分隔字符串，保存时 split + trim
- Upsert 逻辑：`onConflict: "context_date"`，同日期的数据覆盖

**成功反馈**：新增显示「市场环境已保存」，编辑显示「市场环境已更新」

## 三、表单交互设计

### 3.1 表单验证模式

项目**未使用 Zod**，全部采用手动验证：

| 验证层级 | 实现方式 | 示例 |
|----------|----------|------|
| 客户端预检 | 提交前 if 检查 | `if (!canSubmit()) toast.error("请先上传必传截图")` |
| 服务端校验 | Server Action 内检查 | 返回 `{ error: string }` |
| 字段级校验 | 部分有，部分无 | 密码强度实时计算 |

### 3.2 错误反馈模式

```
[客户端预检错误] → feedbackToast.error("具体错误文案")，表单保持打开
[服务端错误]     → feedbackToast.error(result.error)，部分清空字段（如密码）
[网络错误]       → feedbackToast.error("请求失败，请稍后重试")
```

**缺陷**：错误 toast 3 秒后自动消失，无常驻错误区域，用户可能错过。

### 3.3 提交状态管理

| 模式 | 使用场景 | 实现 |
|------|----------|------|
| `useFormStatus` | 简单表单（登录/注册） | 按钮显示「登录中」 |
| `useTransition` | 复杂表单（日报/视频提交） | 整表单 loading 遮罩 |
| 手动 isLoading | 异步操作（AI 渠道测试） | 按钮局部 loading |

### 3.4 文件上传交互

| 场景 | 大小限制 | 交互流程 |
|------|----------|----------|
| 日报截图 OCR | 8MB | 选择 → 上传 → 识别中 → 字段填充/失败 |
| 视频提交截图 | 8MB | 拖放/选择 → 上传 → 识别 → 待确认 |
| 违规案例截图 | 5MB | 选择 → 预览缩略图 → 可删除/重排 |

**缺陷**：
- 无客户端预校验，超限文件提交后才报错
- 上传进度无百分比，只有「上传中」状态
- 无断点续传，大文件网络中断需重新上传

## 四、状态与反馈系统

### 4.1 Toast 通知系统

**封装**：`feedbackToast` 基于 sonner

| 类型 | 颜色 | 使用场景 | 持续时间 |
|------|------|----------|----------|
| success | `#6FAA7D` | 操作成功 | 3s |
| error | `#C9604D` | 操作失败/校验不通过 | 持久（需手动关闭） |
| warning | `#D99E55` | 需要关注但不阻断 | 3s |
| loading | `#8AA8C7` | 异步操作中 | 手动 dismiss |

**设计缺陷**：
- 无 toast 历史记录，错过即消失
- 移动端 toast 可能遮挡底部操作栏
- 无「撤销」操作（如删除后的撤销）

### 4.2 加载状态体系

| 组件 | 使用场景 | 视觉表现 |
|------|----------|----------|
| `Skeleton` | 页面初始加载 | 灰色脉冲块 |
| `Spinner` | 按钮/操作加载 | 旋转圆圈 |
| `LoadingOverlay` | 表单提交 | 半透明遮罩 + 文字 |
| `ProgressBar` | 暂无使用 | — |

### 4.3 空状态体系

| 页面 | 空状态文案 | 是否有行动按钮 |
|------|------------|----------------|
| 待筛视频 | "今天的视频都已打标且无异常" | 否 |
| 待审违规 | "当前没有需要复核的违规案例" | 否 |
| 待催交成员 | "所有在岗成员今天都已交报" | 否 |
| 渠道列表 | "暂无渠道" | 是（添加按钮） |
| 成员列表 | 暂无成员 | 是（邀请按钮） |

### 4.4 错误状态体系

| 层级 | 表现 | 恢复方式 |
|------|------|----------|
| 全局错误 | ErrorBoundary 白屏 | 刷新页面 |
| 页面错误 | `ErrorState` 组件 | 点击「重试」按钮 |
| 组件错误 | 内联错误提示 | 自动重试或手动刷新 |
| API 错误 | Toast 通知 | 用户重新操作 |

## 五、管理后台特殊流程

### 5.1 SOP 检查点系统

**5 个检查点**：
1. 数据上报（11:00）
2. 早盘早会（11:15）
3. 选题第一关（15:00）
4. 文案第二关（18:00）
5. 审片发布（20:00）

**状态流转**：
```
idle（灰色）→ pending（黄色，到点未开始）→ done（绿色，已完成）→ late（红色，逾期）
```

**触发机制**：`/api/sop/check-overdue` 被动调用，无主动推送
**缺陷**：管理员不刷新页面就不知道谁逾期了

### 5.2 豁免申请流程

```
组员打开日历 → 选择日期 → 填写原因 → 提交申请
                                          ↓
                                   待审核（组员不可再申请同日）
                                          ↓
                                   管理员审批（同意/拒绝）
```

**阻断规则**：有待审申请时，该日期不可再次申请
**缺陷**：组员看不到申请进度，只能猜

### 5.3 每日催交流程

**触发**：Vercel Cron，每天 11:15 调用 `/api/remind`
**执行**：
1. 查询未填报成员
2. 发送飞书 webhook 通知
3. 无重试机制，失败即丢失

**缺陷**：
- 催交记录不可查，管理员不知道催了谁
- 无「已读回执」，不知道成员看到没

### 5.4 AI 功能权限控制

**权限开关**：
- `ai_copywriting`：AI 文案生成
- `ai_management`：AI 管理助手

**控制粒度**：成员级，通过 `permissions` 表字段
**生效方式**：前端隐藏入口 + 后端 API 校验

## 六、当前交互中的困惑与矛盾点

### 6.1 认证层矛盾

**问题**：`hasSupabaseAuthCookie()` 只检查 cookie 存在，不检查 token 有效性。

**表现**：
- 用户 token 过期但 cookie 还在 → middleware 放行 → 页面加载 → API 401 → 白屏或错误
- 无自动刷新机制，用户必须手动退出重新登录

**建议**：middleware 增加 token 有效性校验，或至少增加「会话过期」统一处理。

### 6.2 双表单并存矛盾

**问题**：`/dashboard` 同时存在 `DashboardForm`（旧日报）和 `VideoSubmitForm`（新视频提交）。

**表现**：
- 用户不清楚该填哪个
- 两个表单数据可能冲突
- 维护成本高，bug 易遗漏

**建议**：明确切换机制或合并为统一表单，废弃旧版。

### 6.3 乐观更新与回滚的矛盾

**问题**：多处使用乐观更新，但回滚体验不一致。

**表现**：
- 入队审批：行消失 → 失败 → 行恢复（位置可能变化，因排序）
- 权限编辑：勾选生效 → 失败 → 勾选恢复（用户可能已离开页面）
- 违规复核：状态变更 → 失败 → 状态恢复（但结论文本不恢复）

**建议**：统一乐观更新规范，失败时除恢复数据外，增加视觉提示说明「刚才的操作已撤销」。

### 6.4 自动保存与离开提示的矛盾

**问题**：AI 渠道配置有自动保存，但无离开拦截。

**表现**：
- 用户正在编辑 → 未触发 debounce → 点击其他渠道 → 数据丢失
- 浏览器关闭/刷新 → 正在保存中的数据可能丢失

**建议**：增加 `beforeunload` 提示，或切换渠道前检查未保存状态。

### 6.5 演示模式的身份混淆

**问题**：`/demo` 路由使用 `DEMO_VIEWER` 角色，但无明确视觉标识。

**表现**：
- 用户可能分不清自己在看演示还是真实数据
- 演示数据与真实数据格式相同，更易混淆

**建议**：全局增加演示模式标识（如顶部横幅），所有操作按钮禁用并提示「演示模式不可操作」。

### 6.6 催交与豁免的闭环缺失

**问题**：催交和豁免是两个独立系统，无联动。

**表现**：
- 已申请豁免的成员仍收到催交通知
- 管理员不知道成员是否已申请豁免

**建议**：催交逻辑增加豁免状态过滤，豁免申请页面显示催交记录。

### 6.7 导出功能的黑箱体验

**问题**：数据导出无进度、无限制、无结果通知。

**表现**：
- 用户点击导出 → 无反馈 → 突然开始下载 / 或永远没反应
- 大数据量时服务端内存溢出，用户端无错误提示

**建议**：增加导出队列，提供进度条，限制单次导出条数，完成后 toast 通知。

## 八、给开发 AI 的关键摘要

### 8.1 技术约束

- **框架**：Next.js App Router，Server Components 为主，交互部分用 Client Components
- **样式**：Tailwind CSS + shadcn/ui，颜色令牌见环境变量
- **状态**：无全局状态库，用 `useState` + `useTransition` + 乐观更新
- **表单**：无 Zod，手动验证 + `useActionState`/`useFormStatus`
- **通知**：统一用 `feedbackToast`（sonner 封装），不要用原生 alert

### 8.2 必须遵循的交互模式

**乐观更新标准模板**：
```typescript
// 1. 立即更新本地状态
setLocalState(nextState);
feedbackToast.success("已更新");

// 2. 在 transition 中发请求
startTransition(async () => {
  const result = await serverAction(payload);
  if (result.error) {
    // 3. 失败时恢复状态
    setLocalState(previousState);
    feedbackToast.error(result.error);
  }
});
```

**危险操作标准模板**：
```typescript
// 必须用 ConfirmDialog 二次确认
<ConfirmDialog
  open={Boolean(deleteTarget)}
  title="删除确认"
  description="确定删除「xxx」吗？此操作不可撤销。"
  confirmText="删除"
  cancelText="取消"
  destructive        // 红色确认按钮
  loading={isDeleting}
  onConfirm={handleDelete}
/>
```

**轮询数据标准模板**：
```typescript
// 用 useSafeFetch，不要自己写 setInterval
const { data, error, isLoading } = useSafeFetch(
  "/api/admin/pending-videos",
  60_000,  // 60 秒间隔
  null     // 初始值
);
```

### 8.3 常见陷阱

1. **不要假设 auth cookie = 有效登录**：middleware 只检查存在性，API 层必须独立校验
2. **不要无超时发 fetch**：所有 fetch 必须包 `AbortController`，建议 30s 超时
3. **乐观更新必须可回滚**：任何本地先更新的操作，失败时必须有恢复逻辑
4. **不要直接操作 DOM**：移动端底部栏用 `createPortal`，不要用 `position: fixed` 硬写
5. **权限校验要前后端双验**：前端隐藏入口不够，API 必须独立校验权限

### 8.4 新增页面的交互 checklist

- [ ] 加载状态：初始用 Skeleton，操作中用 Spinner/LoadingOverlay
- [ ] 空状态：数据为空时显示 EmptyState，带引导操作
- [ ] 错误状态：API 错误用 toast，页面级错误用 ErrorState
- [ ] 表单校验：客户端预检 + 服务端校验双保险
- [ ] 成功反馈：操作成功用 `feedbackToast.success`
- [ ] 危险操作：删除/重置/移除必须用 `ConfirmDialog`
- [ ] 移动端：底部操作栏用 `createPortal` 到 body，含 `safe-area-inset-bottom`
- [ ] 轮询数据：用 `useSafeFetch`，间隔 >= 30s
- [ ] 自动保存：debounce >= 800ms，显示保存状态（保存中/已保存/失败）
- [ ] 权限控制：前端隐藏 + 后端校验，无权限时显示「无权访问」而非白屏
