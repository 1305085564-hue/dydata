# DYData 运维排查手册

> 这份是"出事时的急救包"。网站白屏、登录挂了、功能报错、线上出问题时，照着这里的固定步骤走，不靠临时摸索。
> 面向阿禅：你不用懂代码，出事先翻到对应症状那一节，照着念给 AI 听，或自己按"你能做的一步"操作。
> 面向 AI：这里是排查的固定顺序和回滚规范，按症状分诊，别跳步、别凭猜。
> 与代码/平台实际冲突时以实际为准并当场修本文件。最后核对日期：2026-07-08

---

## 〇、出事先做这一件事：判断影响面

先问三句，决定慌不慌：

1. 是全站打不开，还是只有某一个页面/功能出问题？（全站 = 紧急，单功能 = 不紧急）
2. 是所有人都这样，还是只有你一个人？（只有你 = 大概率你本地缓存/登录问题，先清缓存）
3. 是刚上线一次改动之后开始的吗？（是 = 八成是那次改动的锅，优先考虑回滚）

第 3 问最关键：**只要是"上线后立刻坏"，先回滚，再慢慢查**。回滚方法见第一节。

---

## 一、回滚：最重要的急救动作（阿禅能自己做）

DYData 只在 Vercel 上部署，`main` 分支一 push 就自动上线。所以"上线后坏了"的最快解药不是改代码，是**在 Vercel 后台一键退回上一个好版本**：

1. 打开 Vercel 后台 → 进 dydata 项目 → 点 Deployments（部署记录）。
2. 找到出事那次之前、最后一个正常的部署（状态是 Ready、时间更早那个）。
3. 点它右边的 `...` → 选 Promote to Production（提升为正式）或 Instant Rollback（即时回滚），确认。
4. 等一两分钟，线上就退回那个好版本了。**这一步不碰代码，最安全，你自己能操作。**

术语：部署 = 一次上线；Promote/Rollback = 把某个旧的好版本重新设成当前线上版本。

回滚只是止血，不是修好。退回去之后，把"哪次改动导致的、什么现象"告诉 AI，让它在代码里查根因、修好再重新上线。代码层的回滚（撤销某次提交）由 AI 操作，且必须提醒你确认后才重新 push，禁止私自 push main。

## 二、按症状急救

### 症状 A：网站白屏 / 打不开
先分诊：

1. 只有你打不开？→ 先清你本地缓存：网址后面加 `?__clear_site_data=1` 打开一次（项目内置的清缓存开关），或换个浏览器/无痕窗口试。好了就是你本地的事，不用惊动线上。
2. 所有人都打不开？→ 紧急。先看是不是刚上线过：是 → 立刻按第一节回滚。
3. 回滚后仍白屏？→ 大概率是外部依赖挂了，往下查：
   - 看 Vercel 后台最近部署状态：是 Error（构建失败）还是 Ready？构建失败线上会停在旧版，页面反而是好的；如果 Ready 还白屏，往下。
   - 查 Supabase 是否正常（数据库挂了页面会转圈/报错）：见第三节健康检查。

注意：本项目目前**没有自定义全局错误页**，代码一旦在渲染时抛错，用户看到的可能是浏览器原始报错或空白，没有友好兜底。这是已知隐患，属"前端体验+代码治理"领域的待补项。

### 症状 B：登录异常（登不进、一直跳回登录页）
鉴权由 `src/middleware.ts` 统一把关。按顺序查：

1. 只有你？→ 先 `?__clear_site_data=1` 清缓存重登，或无痕窗口。登录态多是本地 cookie 过期。
2. 所有人登不进？→ 查两样：
   - Supabase 是否正常（登录依赖它）：见第三节。
   - 环境变量 `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` 是否被动过（改错登录直接崩）。
3. 提示被限流/请求太频繁？→ middleware 里有限流机制（rate-limit），短时间大量请求会被挡，等几分钟再试；持续异常交后端领域查 `src/lib/rate-limit.ts`。

### 症状 C：某个功能报错 / 点了没反应
这是单功能问题，不紧急。给 AI 三个定位信息，它就能查：

1. 哪个页面、点哪个按钮、期望什么结果、实际什么现象。
2. 让 AI 去 Vercel 后台看这个接口的运行日志（Functions/Logs），报错信息一般在那。
3. 归领域：数据不对 → 后端与数据；页面样式/交互坏 → 前端体验；看不到本该有权限看的东西 → 权限与安全。分领域后按各自规矩修。

### 症状 D：数据不显示 / 数据不对
1. 先确认是"查不出"还是"查错了"。查不出可能是权限（RLS）或字段问题，查错了是业务逻辑。
2. 让 AI 顺着模块地图找这个功能依赖哪张表（`docs/全站模块地图.md`），再查那张表的真实数据和字段。
3. 权限类（有人看到不该看的、或该看的看不到）→ 查 RLS 策略和 service role 调用链，属权限与安全领域。

### 症状 E：全站变慢 / 某页面卡
走性能领域，不在本手册细化。用体检 skill：说一句"体检 /某页面"，或读 `docs/架构与性能总纲.md` 按排查流程走。

## 三、健康检查：怎么确认各部分还活着

出事时逐个确认哪一环挂了：

- 网站本身（Vercel）：Vercel 后台 Deployments，当前正式部署是不是 Ready 状态；Functions/Logs 有没有大量报错。
- 数据库（Supabase）：Supabase 后台看项目是否 Active（新加坡区，免费版长期不用会被暂停，本项目靠每天保活接口防止）；或让 AI 调用保活接口探活。
- 保活接口：`/api/supabase-keepalive`，正常返回 `ok:true`，未授权返回 401。它每天由 Vercel cron 自动跑一次，作用就是戳一下数据库别让它休眠。

术语：Supabase 免费版数据库长时间没请求会自动"暂停"，暂停后网站就连不上数据。保活接口就是每天定时戳它一下，防止暂停。

## 四、关键设施速查

| 设施 | 事实 | 出问题查 |
|---|---|---|
| 部署平台 | 仅 Vercel，`main` push 自动上线；仓库里 Cloudflare/OpenNext 配置是历史残留，不是正式方案 | Vercel 后台 Deployments |
| 定时任务 | Vercel cron 只有 2 个：保活（每天）、清过期通知（每天）。其他带 cron 鉴权的接口是手动/外部触发，不是自动在跑 | `vercel.json` |
| cron 鉴权 | 兼容 `CRON_SECRET` 和 `REMIND_SECRET`，靠网址 `?secret=` 参数校验 | `src/lib/cron-auth.ts` |
| 鉴权入口 | 所有登录校验、限流、清缓存都在 `src/middleware.ts` | 登录/限流问题查这里 |
| 数据库连接 | supabase 客户端分四种：`client`(浏览器) / `server`(服务端) / `admin`·`service`(超级权限，绕过 RLS) | `src/lib/supabase/` |
| 环境变量 | 公开的 `NEXT_PUBLIC_*` 改错登录直接崩；服务端 key 改错接口崩。禁止直接改线上 env | `.env.example` 对照 |
| 清缓存开关 | 网址加 `?__clear_site_data=1` 可强制清用户本地缓存和登录态 | 单用户异常时先用它 |

## 五、排查分层（AI 用：归到哪一类）

线上问题先看 Vercel 最近部署，再区分五类，别混着查：

1. 代码：刚上线就坏 → 先回滚，再查那次改动。
2. 数据库：数据查不出/连不上 → 查 Supabase 状态、真实字段、已执行 migration。
3. 权限：能不能看/改的问题 → 查 RLS 策略、helper、service role 调用链。
4. 缓存：改了数据但页面没变 → 查那 4 个有 60 秒内存缓存的接口，或用户本地缓存。
5. 部署/环境：构建失败、env 改动 → 查 Vercel 构建日志和环境变量。

## 六、维护规则

- 部署平台、cron、鉴权方式、健康检查入口有变 → 当场更新第四节。
- 新增能自助操作的急救动作 → 补进对应症状节，用阿禅能懂的话写。
- 与平台实际或代码冲突一律以实际为准并立即修本文件。
- 本文件管"出事怎么救"；慢的问题在 `docs/架构与性能总纲.md`，模块归属在 `docs/全站模块地图.md`。
- 顶部"最后核对日期"每次核对后更新。
