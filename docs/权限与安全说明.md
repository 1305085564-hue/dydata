# DYData 权限与安全说明

> 本文件回答一个问题：谁能看什么、谁能改什么、判断"会不会越权"时先读哪里。
> 用途：任何 AI 动"谁能看/谁能改/是否越权"相关的活之前，先在这里查清角色、权限开关和判定链路，别从旧记忆抄。
> 面向阿禅：你不用懂代码。想知道"某个人能不能看到某数据、能不能改"，看第一节的角色表就够；术语一律就地用大白话解释。
> 面向 AI：这里是权限模型和鉴权链路的权威地图，所有结论已按当前代码核实。与代码冲突一律以代码为准并当场修本文件。
> 最后核对代码日期：2026-07-08

## 〇、先搞懂两个"角色"（给阿禅）

系统里"角色"有两套说法，别混：

- 数据库角色（`role`）：存在数据库里的原始身份，只有三种——`owner`（老板）、`admin`（管理员）、`member`（普通成员）。
- 业务角色（`businessRole`）：程序真正拿来判断权限用的，四种——`owner`、`team_admin`（团队管理员）、`group_leader`（组长）、`member`。

为什么要两套：数据库只记了粗身份，但同样是 `admin`，有的能管全团队成员、有的只是带一个小组的组长，权限差很多。所以程序会拿数据库角色 + 几个开关，现算出更精细的业务角色。判断一个人能干什么，永远看业务角色，不看数据库角色。

## 一、角色映射表（谁是谁、能到哪）

程序怎么把数据库角色算成业务角色（逻辑在 `src/lib/business-role.ts` 的 `resolveBusinessRole`）：

| 业务角色 | 成立条件 | 数据可见范围 | 默认权限 |
|---|---|---|---|
| owner | 数据库 `role = owner` | 全公司（all） | 全开 |
| team_admin | `role = admin` 且开关 `manage_members = true` | 本团队（team） | 全开 |
| group_leader | `role = admin` 且是某个组的 `leader_user_id`（组长） | 本团队（team） | 部分开（见第二节） |
| member | 其他所有情况 | 只有自己（self） | 全关 |

判定顺序有先后：先看是不是 owner；再看是不是"admin + 能管成员"（team_admin）；再看是不是"admin + 带组"（group_leader）；都不是就是 member。所以一个 admin 只要 `manage_members` 开着，就直接算 team_admin，即使他也带组。

数据可见范围（`data-access-scope.ts` 里的 `visibleUserIds`）落地成一份"能看见哪些人的数据"的用户清单：owner 看全部人；team_admin/group_leader 看本团队所有人；member 只看自己。所有按范围过滤的查询都拿这份清单去筛。

术语：`leader_user_id` = 某个小组表里记着的"这个组的组长是谁"。`visibleUserIds` = 程序给你算出的"你有权看到数据的那批人"。

## 二、权限开关清单（PERMISSION_KEYS）

所有细粒度权限开关统一定义在 `src/types/index.ts` 的 `PERMISSION_KEYS`，共 11 个，分两组。每个人的 `permissions` 字段就是这 11 个开关的开/关状态。

管理类 9 个（`ADMIN_PERMISSION_KEYS`）：

| 开关 | 中文名 | 管什么 |
|---|---|---|
| view_all_data | 查看所有数据 | 分析类入口的准入开关之一；不是全站统一的"全范围"总开关（见下方说明） |
| edit_data | 编辑/删除数据 | 能否改删数据 |
| export_data | 导出数据 | 能否导出 |
| view_analytics | 数据分析 | 能否进数据分析页 |
| manage_members | 管理成员状态 | 能否管成员（也是 team_admin 的判定开关） |
| manage_violations | 违规话术复核 | 能否复核违规 |
| view_conversion_hub | 查看转化中心 | 能否看转化数据 |
| view_content_review | 查看批改台 | 能否进批改台 |
| manage_video_assets | 管理素材库 | 能否管视频素材 |

AI 类 2 个（`AI_PERMISSION_KEYS`）：

| 开关 | 中文名 | 管什么 |
|---|---|---|
| use_ai_copywriting | AI 文案助手 | 能否用文案改写工具 |
| use_ai_management | AI 管理助手 | 能否用后台 AI 助手做查询/执行 |

各业务角色的默认开关（`business-role.ts` 里的默认表）：

- owner / team_admin：11 个全开。
- group_leader：开 `export_data`、`view_analytics`、`view_content_review`、`use_ai_copywriting`，其余关。
- member：11 个全关。

重要规则：这些默认值只是"没单独设置时的兜底"。程序读权限时（`normalizePermissionsForBusinessRole`）逐个开关看——数据库里这个人显式设了就用他设的，没设才用角色默认。所以给单个人开/关某个权限，是覆盖默认值。owner 是硬编码全通过（`hasBusinessPermission` 里 owner 直接 return true），不看开关。

关于 `view_all_data` 的坑：它不是"全站统一放大数据范围"的总开关。真正决定"你能看到哪些人的数据"的主链路是 `data-access-scope.ts`，靠 `businessRole` 和 `accessLevel`（all/team/group/self）判定，`view_all_data` 只是在推断 accessLevel 时（admin 且开了它才算最高级 4）和部分分析入口（`analytics-access.ts` 的分析页准入、"看全成员"判断）里被单独读到。所以不能把它当成"开了就能看全部数据"，改动范围逻辑要回到主链路看。

## 三、鉴权是怎么把关的（三层）

权限不是靠数据库一层，是应用层为主、数据库为辅。理解这个分工是安全判断的关键。

第一层 · 登录闸机（`src/middleware.ts`）：所有受保护页面（dashboard/admin/growth/violations/video-review/content-tools）进门先过 middleware，它只管三件事——校验有没有有效登录态（没有就踢回登录页）、限流（防刷）、清缓存开关。它不判细粒度权限，只判"登没登录"。

第二层 · 接口鉴权（多种入口，`requireAdminActor` 是其中之一）：后台管理接口的权限关卡。`src/app/api/admin/auth-helper.ts` 的 `requireAdminActor` 做四步——确认已登录、读出 profile、算出业务角色和权限、比对本接口要求的权限（`requiredPermission`），不满足返回 403；约 9 个 route 文件直接用它。**但接口鉴权不是只有这一种**：还有的接口走 `getUserPermissions` 自行判权限、走 `requireAuth` + 对象归属校验（如改写接口用 `requireConversationOwner` 校验会话归属）、或走 cron 密钥（定时任务接口）。判断一个接口安不安全，不能只看它有没有 `requireAdminActor`，要看它有没有"某种等价的登录 + 权限/归属/范围校验"。`requireAdminActor` 不传 `requiredPermission` 时，退化成"有任意一个管理类权限就放行"（`hasAnyAdminPermission`）。

第三层 · 数据范围过滤（`current-permission-context.ts` + `data-access-scope.ts`）：确认"能进这个接口"之后，还要限定"只能看到你范围内的人"。它算出 `visibleUserIds` 清单，查询按这份清单筛，member 就只筛出自己。owner 范围是 all，不筛。

性能补充：第一、三层的权限查询用 React `cache()` 做了"同一次请求内去重"（`permissions.ts` 的 `loadUserPermissions`、`current-permission-context.ts` 的 `getCurrentPermissionContext`），同一个请求里重复问权限不会重复查数据库。

## 四、service role：绕过所有权限的"万能钥匙"（安全重点）

术语：正常情况下数据库有一层叫 RLS（行级安全）的门禁，规定每个登录用户只能读到自己该读的行。但有一把"服务端密钥"（service role key）能完全绕过 RLS，用它连数据库等于以超级管理员身份操作，谁的数据都能读能改。

关键事实（`src/lib/supabase/admin.ts`）：
- `createAdminClient()` 用环境变量 `SUPABASE_SERVICE_ROLE_KEY` 建连接，绕过 RLS。`service.ts` 里的 `createServiceClient` 只是它的别名，同一把钥匙。
- 全站约 61 个后台 route 文件在用这把钥匙。这是刻意的架构选择：后台接口不靠数据库 RLS 控权，而是先在应用层挡住没权限的人（用第三节的接口鉴权），进来之后用 service role 直连数据库。
- 连权限本身的读取（`permissions.ts`、`data-access-scope.ts`）都用 service role 读 profiles/groups——因为要在"判断你是谁"这一步先绕过 RLS 才能读到完整信息。

安全含义（务必守住）：
- service role 用得对不对，全压在应用层鉴权上。任何用了 `createAdminClient`/`createServiceClient` 的接口，必须先有等价的鉴权——登录 + 权限校验（`requireAdminActor` 或 `getUserPermissions`），或对象归属校验（如改写接口的 `requireConversationOwner`），或按 `visibleUserIds` 的范围过滤。三者一个都没有、直接拿 service role 返数据，才是越权漏洞。别一看到"没走 requireAdminActor"就判成漏洞——很多接口走的是等价的另一种校验。
- service role key 是服务端机密，绝不能出现在发到浏览器的代码里，绝不写进前端，绝不 log。
- 判断一个后台接口安不安全，先看两点：有没有某种等价鉴权（`requireAdminActor` 传对 `requiredPermission`，或 `getUserPermissions` 自判，或对象归属校验，或 cron 密钥）；返数据时有没有按 `visibleUserIds` 限范围。

## 五、数据库 RLS：第二道纵深防御

虽然后台接口普遍用 service role 绕过 RLS，RLS 仍是重要的兜底——面向浏览器直连的普通客户端（`client`/`server`）仍受 RLS 约束。**RLS 策略散落在多份 migration 里，不止下面几份**，改任何表的读写权限前，先扫一遍 `supabase/migrations/` 里所有涉及该表 `create policy` / `enable row level security` 的文件，别只看这几份早期核心的：

- `002_invite_rls.sql`：邀请码表的 RLS（注册时按规则可读、只能标记自己用过）。
- `007_owner_role_and_permissions.sql`：加 owner 角色，并建数据库端 helper 函数 `is_admin()`（admin 或 owner 视为管理级）、`is_owner()`、`has_permission(perm)`（owner 恒真，admin 看 jsonb 开关）。这套函数供 RLS 策略调用。
- `033_rls_policies.sql`：一大批表的读写策略，普遍模式是"按范围读、仅管理员写/改/删"（脚本文档、AI 输入包、AI 洞察、团队、分组、豁免申请/授予、审计日志等）。
- `035_screenshot_slot_roles.sql`：截图槽位相关的角色策略。
- 后续 migration 还在陆续给新表加 RLS/策略，例如产量对账（`daily_quota_config`、`work_submissions`）、案例库重构、履约申诉等（文件名带日期戳的那批）。这类表改权限时，务必找到对应最新 migration。

规矩：RLS 策略变更只能新增 migration，禁止改旧 migration。数据库 helper（`is_admin`/`has_permission`）和应用层判定（`business-role.ts`）是两套实现，改权限规则时要留意两边是否需要同步，别只改一边导致数据库和应用层判定不一致。

## 六、改权限先查哪几处（AI 用）

要改任何"谁能看/谁能改"的逻辑，先按这条链查全，别只改一处：

1. 权限开关本身：`src/types/index.ts`（`PERMISSION_KEYS` / 默认权限）。加/减开关从这里起。
2. 业务角色与默认值：`src/lib/business-role.ts`（`resolveBusinessRole` 判定、各角色默认开关表、`hasBusinessPermission`）。
3. 权限读取入口：`src/lib/permissions.ts`（`getUserPermissions`，请求内缓存）。
4. 数据范围：`src/lib/data-access-scope.ts`（`visibleUserIds` 怎么算）+ `current-permission-context.ts`。
5. 接口关卡：`src/app/api/admin/auth-helper.ts`（`requireAdminActor`）——顺带确认相关接口传了正确的 `requiredPermission`。
6. 数据库端：对应 RLS migration 和 helper 函数，判断是否要同步新增 migration。

## 七、常见越权/权限问题排查顺序

- 该看的看不到 / 不该看的看到了：先看这个人算出来的业务角色对不对（`resolveBusinessRole`），再看 `visibleUserIds` 范围算得对不对（`data-access-scope.ts`），最后看具体接口有没有正确按范围过滤。
- 某后台接口谁都能进：查它有没有等价鉴权。常见合格链路包括 `requireAdminActor` 并传对 `requiredPermission`、`getUserPermissions` 自行判权限、`requireAuth` + 对象归属校验、cron 密钥、或按 `visibleUserIds` 做范围过滤。用了 service role 又完全没有这些校验，才是漏洞。
- 改了权限没生效：留意请求内 `cache()` 去重（同一请求不重查）和是否 profiles 里显式开关覆盖了角色默认值。
- 数据库直连（非后台接口）读不到数据：查对应表的 RLS 策略和 helper 函数。RLS 分散在多份 migration，不只 `033_rls_policies.sql`；先扫 `supabase/migrations/` 里所有涉及该表的 `CREATE POLICY` / `ENABLE ROW LEVEL SECURITY`。
- owner 相关异常：owner 在应用层是硬编码全通过、在数据库端靠 `is_owner()`/`has_permission()`，两处都要对得上。

## 八、维护规则

- 新增/删除权限开关、改角色判定或默认值 → 当场更新第一、二节，并按第六节把链路各处同步改全。
- 新增用 service role 的后台接口 → 确认已有等价鉴权（`requireAdminActor` / `getUserPermissions` / `requireAuth` + 对象归属 / cron 密钥 / `visibleUserIds` 范围过滤），并在第四节安全含义里守住"先鉴权或限范围，后返回 service role 数据"的铁律。
- RLS 策略或数据库 helper 有变 → 更新第五节，且只能新增 migration。
- 与代码冲突一律以代码为准并立即修本文件。
- 本文件管"谁能看/谁能改/怎么鉴权"；模块归属在 `docs/全站模块地图.md`，出事急救在 `docs/运维排查手册.md`，配合使用。
- 顶部"最后核对代码日期"每次核对后更新。
