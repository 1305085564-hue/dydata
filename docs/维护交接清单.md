# 维护交接清单

这份清单是给“接手 DYData 的维护者”用的。目标不是讲架构理论，而是让对方最快接起来，不在第一天踩坑。

## 1. 先交什么

必须一起交：

- GitHub 仓库访问权限
- Vercel 项目访问权限
- Supabase 项目访问权限
- Cloudflare / 域名访问权限
- 飞书 webhook 的管理方式
- AI key 的管理方式

如果只给代码，不给上面这些，对方只能看代码，不能真正维护。

## 2. 代码仓库

- 仓库地址：`git@github.com:1305085564-hue/dydata.git`
- 默认分支：`main`
- 部署方式：push `main` 自动触发 Vercel

交接前建议确认：

- 仓库里最新修复已经 push
- `git remote -v` 指向正式仓库
- `git config user.email` 为 `1305085564@qq.com`

## 6. 硬规则

- 不要把部署方式改成 Cloudflare Workers
- 不要改旧 migration，只能新增 migration
- 不要直接动线上 `.env`
- 不要看到仓库里有历史残留配置，就擅自改部署方式
- 配置类改动前先读原文

## 7. 日常发布流程

推荐流程：

1. 新建分支修复
2. 本地自测
3. 跑最小必要 lint / test
4. 合并到 `main`
5. 看 Vercel 部署是否成功
6. 验证线上关键链路

至少验证这几条：

- 登录
- 日报提交
- 上传链路
- 管理后台能打开
- 提醒接口没有异常

## 8. 数据库变更流程

正确做法：

1. 在 `supabase/migrations/` 新增 migration
2. 本地检查 SQL
3. 推到远端 Supabase
4. 再验证受影响页面

不要做：

- 修改旧 migration
- 直接在线上手写 SQL 却不回填仓库
- 不验证就上线

## 9. 交接前最后确认

建议逐条确认：

- 代码已推 GitHub
- Vercel 权限已加给维护者
- Supabase 权限已加给维护者
- Cloudflare 权限已加给维护者
- 环境变量来源已说明
- 飞书 webhook 来源已说明
- AI key 来源已说明
- README 已能指导本地启动
- 维护者知道“部署只走 Vercel”
- 维护者知道“数据库变更只增不改”
