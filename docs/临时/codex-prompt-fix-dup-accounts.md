## 用户的核心诉求
1. **昵称账号（猎龙侦探、股建贝聿铭、金田路道哥）是员工自己注册的"真账号"**
2. **真名账号（王东菊、张玥晗、江季芸）是后来为了管理方便后加的**

4. **目标：把"真账号"作为正式账号留在分组里，清理掉重复的"后加账号"**

## 当前的核心矛盾（需要 Codex 解决）
后加的真名账号（王东菊5条、张玥晗5条、江季芸17条）**并非0数据**，无法直接删除而不丢失数据。同时两个账号在多个日期有冲突数据（同一天两人都有 daily_report 记录）。

用户曾提出两种方案，但未明确最终选哪个：
- 方案A：把"有数据的账号名称改成前面的真名"（即把昵称账号的 name 改成真名，删掉真名账号）
- 方案B：直接删掉后加的真名账号（但用户又强调"只删除没有数据的账号"）

## 需要 Codex 确认并执行的操作
**第一步：向用户确认处理策略**
- 后加的真名账号（王东菊5条、张玥晗5条、江季芸17条）都有历史数据，不能直接删除
- 两个方案：
  1. **改名合并**：把昵称账号的 `name` 字段改为真名（如"猎龙侦探"→"王东菊"），然后删除原来的真名账号。此方案会导致真名账号的 daily_reports 记录变成"孤儿数据"（user_id 指向已删除的 profile）。
  2. **数据迁移+删除**：先把真名账号的 daily_reports 的 user_id 改为昵称账号的 id（冲突日期需要处理），再删除真名账号。此方案数据不会丢失，但需要处理"同一天两个账号都有数据"的冲突。
- **请用户明确选择方案**

**第二步：执行用户选择的方案**
- 如果选方案1（改名）：更新昵称账号的 `name` 为真名，删除真名账号的 `profiles` 记录
- 如果选方案2（迁移数据）：合并 daily_reports（非冲突日期直接改 user_id，冲突日期需要用户决定保留哪条），再删除真名账号
- 无论选哪个方案，最终分组状态应保持：每组只有一个该员工的账号（用真名显示）
