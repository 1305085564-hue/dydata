- 用户反馈：页面割裂，交互繁琐，不够精致

2. 再用 `/frontend-design` skill 设计产品级 UI/UX

## 用户核心诉求

> "我要的不是能运行的网站，而是可以上线的产品级作品，每一处细节都需要打磨。"

关键词：
- **产品级** — 交互流畅、视觉克制、信息层级清晰
- **细节打磨** — 间距、圆角、动效、状态反馈、加载、错误处理
- **轻量交互** — 弹窗优先，减少页面跳转

## 场景与交互流程

### 场景 1：组员每日提交作品凭证

**用户画像：**
- 30 人短视频团队的普通组员
- 每天要发 4-6 个视频（本周 4 个/天，下周起 6 个/天）
- 需要提交文案 + 截图证明自己发了片

**典型流程：**
1. 打开首页 `/video-review`
2. 看到全员已发作品的**卡片流式布局**（瀑布流或网格）
   - 卡片上部：视频截图（主视觉）
   - 卡片下部：文案内容（截断显示，1-2 行）
   - 鼠标悬停：显示"查看详情"、"复制文案"快捷操作
3. 顶部搜索框：可以搜文案关键词，找到好案例直接复制
4. 右上角"上传文案和截图"按钮
5. 点击按钮 → 弹出**提交弹窗**
   - 显示今日指标："今天要交 4 条，已交 2 条，还差 2 条"
   - 上传截图区（支持多张，拖拽上传，实时预览）
   - 文案输入框
   - 提交按钮
6. 提交成功 → toast 提示 → 弹窗自动关闭（带淡出动效）
7. 首页自动刷新，新提交的卡片出现在列表顶部

**边界场景：**
- 今天还没交够，且没请假 → 右上角按钮显示红点徽章 + "还差 N 条"
- 今天已交够 → 按钮变为普通样式，仍可继续提交（超额完成）
- 今天请假了 → 按钮显示"已豁免"，仍可提交但不计入考核

### 场景 2：组员申请豁免（请假）

**触发时机：**
- 预知今天发不够（生病、出差、设备故障、限流）
- 或者昨天忘记请假，今天补申请

**典型流程：**
1. 首页顶部或侧边有"申请豁免"入口（醒目但不抢戏）
2. 点击 → 弹出**豁免申请弹窗**
   - 豁免类型下拉菜单：
     - 请假 1 天
     - 请假 3 天
     - 请假 4 天
     - 请假 5 天
     - 补昨日请假
     - 自定义日期范围
     - 永久豁免（特殊情况）
   - 日期选择器（根据类型自动填充或手动选择）
   - 原因说明（必填，多行文本框）
3. 提交 → toast 提示"等待审批" → 弹窗自动关闭
4. 弹窗底部显示**申请历史列表**（折叠式，默认收起）
   - 状态标签：待审批（琥珀色）、已通过（绿色）、已拒绝（红色）
   - 点击展开查看详情和审批备注

---

### 场景 3：管理查看产量看板

**用户画像：**
- 团队管理者（owner/team_admin/group_leader）
- 需要每天快速对账：谁交了、谁没交、谁请了假

**典型流程：**
1. 打开首页 `/video-review`
2. 看到和组员一样的**卡片流式布局**（已发案例）
3. 右上角多了一个**"产量看板"按钮**（管理专属）
4. 点击 → 弹出**产量看板弹窗**（全屏或大尺寸 Dialog）
   - 顶部：今日概览卡片
     - 全队目标总数 / 已提交总数 / 完成率
     - 红灯人数（未达标且无豁免）
   - 筛选器：日期选择、团队筛选、小组筛选
   - 列表：按小组折叠展开
     - 小组头：小组名 + 完成率
     - 成员行：头像 + 姓名 + 目标/已交/缺额 + 状态灯（红/黄/绿）
   - 点击某个成员 → 二级弹窗或抽屉展开
     - 显示该成员今日提交的所有截图缩略图
     - 点击截图 → 第三级弹窗放大预览（或图片查看器）

**关键视觉：**
- 红灯（未达标 + 无豁免）：必须最醒目，扫一眼就能找到
- 黄灯（豁免待审）：次级提醒
- 绿灯（达标或已豁免）：克制低调

---

### 场景 4：管理审批豁免申请

**触发时机：**
- 组员提交豁免申请后，管理需要尽快审批

**典型流程：**
1. 右上角**消息中心**（铃铛图标）出现红点徽章 + 数字（待审数量）
2. 点击消息中心 → 下拉面板或弹窗
   - "待办事项"分组
   - "豁免审批"项显示："3 条待审"（醒目高亮）
3. 点击"豁免审批" → 弹出**审批弹窗**
   - 待审列表：
     - 每条显示：申请人 + 豁免类型 + 日期范围 + 原因摘要
     - 展开查看完整原因
     - 单条操作：通过 / 拒绝（可填审批备注）
   - 批量操作：多选后批量通过（一键操作）
   - 底部折叠区：已处理记录（默认收起，可展开查看）
4. 审批后 → toast 提示 → 列表自动刷新 → 徽章数字更新

**关键交互：**
- 红点徽章：必须醒目，管理不能错过
- 批量通过：常用操作，必须简单高效（一键完成，不要多次确认）
- 审批后反馈：toast 提示 + 数字动效更新

## 设计约束与美学要求

### 布局层级（严格遵守 `docs/美学规范.md`）

- **L0 背景**：`bg-stone-100`（全局底色）
- **L1 卡片**：`bg-white`（内容卡片）
- **L2 浮层**：弹窗、抽屉、下拉面板
- **间距**：24px 基准，用留白分割，不用边框线

### 卡片流式布局

- 一行 4-5 张卡片（响应式：`grid-cols-4 xl:grid-cols-5`）
- 卡片宽高比：16:10 或 4:3（截图展示区）
- 卡片圆角：12px（`rounded-xl`）
- 卡片悬停：轻微上浮 + 阴影加深（`hover:-translate-y-1 hover:shadow-lg`）
- 截图：`object-cover` 填充，避免拉伸变形

### 弹窗交互

- 弹窗尺寸：
  - 提交作品：中等（`max-w-2xl`）
  - 豁免申请：中等（`max-w-2xl`）
  - 产量看板：大尺寸（`max-w-6xl` 或全屏）
  - 审批中心：大尺寸（`max-w-4xl`）
- 弹窗动效：淡入 + 缩放（`animate-in fade-in-0 zoom-in-95`）
- 关闭方式：
  - 右上角 X 按钮
  - 点击遮罩层关闭
  - ESC 键关闭
  - 提交成功后自动关闭（1 秒延迟）

### 颜色规范

- **CTA 按钮**：暖橙色 `#D97757`
- **状态灯**：
  - 红灯：`bg-rose-500` + `ring-1 ring-white`（静态实心点，不要 ping 动画）
  - 黄灯：`bg-amber-400` + `ring-1 ring-white`
  - 绿灯：`bg-emerald-500` + `ring-1 ring-white`
- **徽章数字**：`bg-rose-500 text-white`（红底白字）
- **禁止使用**：
  - `focus:ring-blue/indigo/sky`
  - `animate-pulse/ping`（除了 loading 骨架屏）
  - `bg-red/blue/green-400/500/600`（用上面的语义色替代）

### 数字规范

- 所有数据数字：`font-mono tabular-nums`（等宽对齐）
- 大数字（32px+）：`text-[32px] font-mono tabular-nums`
- 中数字（24px）：`text-[24px] font-mono tabular-nums`

### 反馈与动效

- **Toast 提示**：使用 `sonner`（不用旧的 `use-toast`）
  - 成功：`toast.success("提交成功")`
  - 错误：`toast.error("提交失败")`
  - 自动消失：3 秒
- **加载状态**：
  - 按钮 loading：显示 spinner + 文字变为"提交中..."
  - 列表 loading：骨架屏（`animate-pulse` 仅用于此场景）
- **动效时长**：
  - 快速：150ms（悬停、按下）
  - 标准：300ms（弹窗、抽屉）
  - 慢速：500ms（页面过渡）

## 技术实现要求

### 组件结构（推荐）

```
/video-review
  ├── page.tsx                    // 首页（卡片流 + 顶部操作区）
  ├── components/
  │   ├── case-card-grid.tsx      // 卡片流容器
  │   ├── case-card.tsx           // 单张卡片
  │   ├── case-detail-dialog.tsx  // 案例详情弹窗
  │   ├── submit-dialog.tsx       // 提交作品弹窗
  │   ├── exemption-dialog.tsx    // 豁免申请弹窗
  │   ├── dashboard-dialog.tsx    // 产量看板弹窗
  │   └── approval-dialog.tsx     // 审批弹窗（或集成到消息中心）
```

### 状态管理

- 简单场景：`useState` + `useOptimistic`（乐观更新）
- 表单状态：`react-hook-form` + `zod` 校验
- 弹窗状态：shadcn/ui Dialog 的 `open` + `onOpenChange`

### 数据获取

- Server Components 预取数据（首屏快）
- Client Components 用 `fetch` + `useTransition` 更新
- 提交后乐观更新列表，不等接口返回

### 响应式

- 桌面端：4-5 列卡片
- 平板端：2-3 列
- 移动端：1-2 列
- 弹窗在移动端自动全屏

## 注意事项

1. **产品级细节**：
   - 空状态：数据为空时的引导文案和插图
   - 加载状态：每个异步操作都要有 loading 反馈
   - 错误处理：网络错误、权限不足、数据异常的友好提示
   - 成功反馈：每次操作成功都要有明确的 toast 或视觉反馈

2. **性能优化**：
   - 卡片流懒加载（虚拟滚动或分页）
   - 图片懒加载 + 占位符
   - 弹窗内容按需加载（不要一次性渲染所有弹窗）

3. **无障碍**：
   - 键盘导航（Tab / Shift+Tab / Enter / ESC）
   - 焦点管理（弹窗打开时聚焦到第一个输入框）
   - ARIA 标签（按钮、弹窗、状态灯）

4. **移动端适配**：
   - 卡片在移动端自动变为 1-2 列
   - 弹窗在移动端全屏显示
   - 触摸手势（滑动关闭抽屉）
