# 交互优化提示词使用说明

## 文件清单

| 文件 | 用途 |
|------|------|
| `DYData-交互设计文档.md` | 全站交互审计报告，含现状、问题、建议 |
| `交互优化排期计划.md` | 6 个 Sprint 的执行计划，含时间和依赖关系 |
| `提示词-Sprint1-认证与请求安全.md` | Sprint 1 的 AI 提示词（4 个任务） |
| `提示词-Sprint2-上传与导出安全.md` | Sprint 2 的 AI 提示词（3 个任务） |
| `提示词-Sprint3-核心表单体验.md` | Sprint 3 的 AI 提示词（4 个任务） |
| `提示词-Sprint4-管理后台提效.md` | Sprint 4 的 AI 提示词（4 个任务） |
| `提示词-Sprint5-反馈与可见性.md` | Sprint 5 的 AI 提示词（4 个任务） |
| `提示词-Sprint6-催交与豁免闭环.md` | Sprint 6 的 AI 提示词（3 个任务） |

## 使用流程

### 1. 分发任务

将对应 Sprint 的提示词文件，完整复制给执行 AI（如 Kimi、Codex、Gemini）。

**示例**：
```
请按照以下提示词完成任务，完成后返回修改文件列表和问题记录：

[粘贴 提示词-Sprint1-认证与请求安全.md 的完整内容]
```

### 2. 并行策略

| 可并行 | 不可并行（有依赖） |
|--------|-------------------|
| Sprint 1 + Sprint 2 | Sprint 3 依赖 Sprint 1 的 fetchWithTimeout |
| Sprint 4 + Sprint 5 | Sprint 6 依赖 Sprint 2 的 API 改动熟悉度 |
| Sprint 3 + Sprint 4 | — |

**建议第一波并行**：Sprint 1 + Sprint 2（两个独立 AI 同时执行）

### 3. 复核流程

执行 AI 完成后，返回：
1. 修改文件列表
2. 问题记录
3. 需要特别关注的地方

**复核者（我）的工作**：
- 检查修改是否符合提示词要求
- 运行 `npm run build` 验证编译
- 检查是否有副作用（改 A 坏 B）
- 确认测试通过
- 决定是否可以合并

### 4. 合并与迭代

每个 Sprint 完成后：
1. 复核通过 → 合并到主分支
2. 复核不通过 → 记录问题 → 返回执行 AI 修复 → 重新复核
3. 更新《交互优化排期计划.md》中的完成状态

## 执行优先级（如果资源有限）

只做 **快速启动版 5 件事**：
1. Sprint 1 任务 1：middleware token 有效性
2. Sprint 1 任务 3：fetchWithTimeout
3. Sprint 3 任务 1：草稿自动保存
4. Sprint 2 任务 1：上传大小预校验
5. Sprint 5 任务 3：演示模式标识

对应提示词文件中的相关任务，可单独提取执行。

## 注意事项

1. **不要同时修改同一文件**：两个 AI 不要同时改 `middleware.ts`，会冲突
2. **先读原文件再改**：执行 AI 必须先读取目标文件的当前内容，不要凭记忆写
3. **保持编码风格一致**：使用项目现有的引号风格、缩进、命名规范
4. **TypeScript 严格模式**：任何类型错误都必须修复，不能 `@ts-ignore`
5. **不要改线上配置**：`.env`、数据库 migration、Vercel 配置等需要人工复核
