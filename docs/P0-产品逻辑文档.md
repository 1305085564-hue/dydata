## P0-1：卡点状态实时追踪

### 产品目标
每个达人的 5 个卡点状态，所有人实时可见。组长看自己组，你看全局。

### 数据流

```
达人操作                          系统自动                    组长/负责人看到
─────────                        ─────────                  ─────────────
① 填报数据 → 提交            
   → sop_daily_status.data_report_status = 'SUBMITTED'
   → sop_checkpoint_submissions 插入一条 DATA_REPORT 记录
                                                       → 卡点①变绿 ✅

② 早会（暂不需达人在网站操作）
   → sop_daily_status.morning_review_status = 'APPROVED'（组长手动确认或自动）
                                                       → 卡点②变绿 ✅

③ 提交选题 → 达人填写选题文本 → 提交
   → sop_daily_status.topic_status = 'SUBMITTED'
   → sop_checkpoint_submissions 插入一条 TOPIC 记录
                                                       → 卡点③变黄（待审核）🟡

④ 组长审核选题 → 通过
   → sop_daily_status.topic_status = 'APPROVED'
                                                       → 卡点③变绿 ✅

④' 组长审核选题 → 打回
   → sop_daily_status.topic_status = 'REJECTED'
   → sop_checkpoint_submissions.review_status = 'REJECTED'
                                                       → 卡点③变红 ❌ + 打回原因

⑤ 提交文案 → 达人填写文案 → 提交 → 组长审核（同上逻辑）
⑥ 提交成片 → 达人填视频链接 → 提交 → 组长审核（同上逻辑）
```

### 前端展示

**5 个节点的视觉改造**（现在只是图标按钮，要改成有状态的流程条）

每个节点显示：
- 图标 + 名称 + 状态色
- 状态色规则：
  - 灰色 = IDLE（还没到这一步）
  - 蓝色闪烁 = PENDING（该做了，还没交）
  - 黄色 = SUBMITTED（已提交，等审核）
  - 绿色 = APPROVED（通过）
  - 红色 = REJECTED（被打回）
  - 红色脉冲 = OVERDUE（超时未交）

**全局看板**（所有人的卡点状态一览表）

```
         | 数据报表 | 早会 | 选题 | 文案 | 成片
─────────|─────────|─────|─────|─────|─────
阿禅     |   ✅    | ✅  | 🟡  | 灰  | 灰
张三     |   ✅    | ✅  | ✅  | 🟡  | 灰
李四     |   ❌    | 🔴  | 灰  | 灰  | 灰
```

### 需要新建的 API

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/sop/status` | GET | 查询某日所有人的卡点状态（按组过滤） |
| `/api/sop/status` | PATCH | 更新某个卡点的状态（提交/审核通过/打回） |
| `/api/sop/submit` | POST | 提交某个卡点的内容（选题/文案/成片） |

### 需要改的前端

| 组件 | 改动 |
|------|------|
| 5 个节点按钮 | 改成有状态色的流程节点，可点击进入对应卡点 |

## P0-3：组长日报

### 产品目标
组长每天 20:30 前在网站提交日报，你一页看 5 份，不用翻微信群。

### 数据库

**新建表 `leader_daily_reports`**

```sql
create table public.leader_daily_reports (
  id uuid primary key default gen_random_uuid(),
  leader_user_id uuid not null references public.profiles(id),
  group_id uuid not null references public.groups(id),
  report_date date not null,

  -- 固定 4 个字段（对应管理设计文档 §5.2）
  topic_feedback text,        -- 选题：今日组内选题方向亮点/问题（举例）
  opening_feedback text,      -- 开头：哪条开头写得好/哪条开头有问题
  script_feedback text,       -- 脚本文案：组内整体写作水平观察、典型问题
  video_feedback text,        -- 视频：成片质量观察、突出案例/待改进案例

  submitted_at timestamptz default now(),
  updated_at timestamptz default now(),

  unique (leader_user_id, report_date)
);
```

## P0-4：异常自动标红

### 产品目标
异常情况不用人盯，系统自动推到组长和你的视野里。

### 异常判定规则

| 异常类型 | 触发条件 | 推给谁 |
|---------|---------|--------|
| 超时未交 | 卡点截止时间后仍未提交 | 组长 + 你 |
| 连续数据下滑 | 播放量连续 3 天下降（对比前 3 天均值） | 组长 |
| 断更 | 当日未提交数据且未申请豁免 | 组长 + 你 |
| 审核积压 | 待审核超过 30 分钟未处理 | 组长 |

### 时间节点（对应管理设计文档）

```
卡点① 数据上报  截止 11:15
卡点② 早会      截止 12:00
卡点③ 选题      截止 15:30
卡点④ 文案      截止 18:00
卡点⑤ 成片      截止 20:00
```

### 实现方式

**方案：定时任务 + 前端轮询**

1. 后端定时任务（每 5 分钟跑一次）：
   - 检查 `sop_daily_status` 中各卡点状态
   - 到了截止时间还是 IDLE/PENDING → 改为 OVERDUE
   - 查询近 3 天播放量，连续下滑 → 标记异常

2. 前端轮询（每 60 秒）：
   - 调 `/api/sop/alerts` 获取当前异常列表
   - 有新异常 → 顶部显示红色通知条

### 异常展示位置

| 位置 | 展示方式 |
|------|---------|
| 组长看板顶部 | 红色告警条，列出本组所有异常 |
| 你的全局看板 | 红色告警条，列出所有组的异常 |
| 5 个节点流程条 | 超时的节点红色脉冲动画 |
| 导航栏 | 异常数量角标（如"审核中心 3"） |

### 需要新建的 API

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/sop/alerts` | GET | 返回当前用户可见范围内的异常列表 |
| `/api/sop/check-overdue` | POST | 手动触发超时检查（定时任务调用） |

## 给 Claude Code 的指令要点

1. 所有 SOP 相关 API 统一用 `SUPABASE_SERVICE_ROLE_KEY`，不用 anon key
2. `sop_daily_status` 的状态流转要严格按 IDLE → PENDING → SUBMITTED → APPROVED/REJECTED 走，不能跳步
3. 组长看板的权限过滤：`group_id = (select group_id from profiles where id = auth.uid())`
4. 5 个节点组件要支持所有状态色，不要只做 IDLE 和 SUBMITTED
5. 异常检查用 Supabase Edge Function 或 Next.js API Route + cron，不要在前端算

## 给 GPT-5.5/Codex 的指令要点

1. 先跑通 P0-1 的后端 API，这是其他功能的基础
2. `sop_checkpoint_submissions` 的 `topic_text`、`script_text`、`video_url` 三个字段，不同卡点只填对应的字段，其他留 null
3. 审核打回时必须写 `rejection_reason`，前端强制校验
4. 异常检查定时任务用 Vercel Cron，间隔 5 分钟，接口兼容 `CRON_SECRET ?? REMIND_SECRET`
