# DYData 代码治理手册

> 本文件回答一个问题：文件该放哪、模块之间的边界在哪、哪些改动不能混在一起发，才能不让代码越修越乱。
> 用途：任何 AI 新建文件、拆分模块、准备提交/上线前，先在这里对规矩，避免各自为政把项目改花。
> 面向阿禅：你不用懂代码。这份是给 AI 立的"施工规矩"，保证不同 AI 接力干活时房子不塌。术语一律就地用大白话解释。
> 面向 AI：这里是目录职责、命名、边界、发布纪律的权威约定，已按当前代码结构核实。与代码冲突一律以代码为准并当场修本文件。
> 最后核对代码日期：2026-07-15

---

## 〇、为什么要有这份（给阿禅）

网站被几十次接力改过，每个 AI 各按各的习惯放文件、起名字，时间长了同一件事有好几套写法、废代码没人删。这份手册就是定一套统一规矩：东西放哪、谁不许碰谁、哪些改动必须分开发。守住它，房子才不会越修越歪。

---

## 一、目录职责表（东西放哪）

`src` 顶层结构与各自职责：

| 目录/文件 | 职责 | 边界 |
|---|---|---|
| `src/app` | 页面 + 接口（Next.js App Router）。页面在 `(app)`/`(auth)` 分组下，接口在 `app/api` 下 | 页面只管渲染和取数据，重逻辑下沉到 lib |
| `src/components` | 可复用 UI 组件（含 `components/ui/*` 基础组件） | 新组件默认只做展示与交互，不直连数据库；存量 `nav-bar.tsx` 是服务端组件直查 Supabase 的例外，后续治理时再收口 |
| `src/hooks` | 复用的 React hooks | 客户端逻辑复用 |
| `src/lib` | 业务逻辑、数据加载、工具函数（上百个条目，最杂，见第五节债务） | 服务端/通用逻辑的落点 |
| `src/types` | 全局类型和常量（如权限开关 `PERMISSION_KEYS`） | 只放类型和常量 |
| `src/styles` / `app/globals.css` | 样式 | — |
| `src/middleware.ts` | 全站登录闸机、限流、清缓存 | 只判登没登录，不判细粒度权限 |

`src/lib` 下已成型的子目录（按领域归拢，新代码优先归进对应子目录，别堆散文件在 lib 根）：`loaders`（页面数据加载器）、`supabase`（数据库客户端）、`ai` / `admin-ai`（AI 相关）、`admin-tools`、`alert-sources`、`case-library`、`conversion-hub`、`notifications`、`publish-drafts`、`rewrite`、`team-join`、`violations`。

## 二、数据加载约定（loaders）

约定：页面首屏数据统一走 `src/lib/loaders/` 里的加载器（一个页面一个 loader，首屏一次查好），别在页面组件里散着查数据库。这样性能能集中优化、依赖能集中看。

已收口的页面 loader（在 `loaders/` 下有对应文件）：dashboard、growth、admin-content、admin-videos、analytics、fulfillment、guidance、admin-advice、admin-modules、content-tools 等。

例外（页面内直接查、未走独立 loader）：`video-review` 子页、`violations`、`admin/settings`。`/video-review` 首页已有 loader，但 `/submit`、`/exemption`、`/archive` 仍各自取数；这些属于历史遗留，后续收口到 loaders 要走独立任务，别混进别的改动里（见第四节）。

规矩：新页面必须建对应 loader；不要给上面三个例外之外的页面再引入"页面内直查"的新写法。

## 三、命名规范

呼应 CLAUDE.md 的既定约定：

- 文件与文件夹名：除非影响路径解析或运行稳定性，一律用中文（文档、记忆、规则文件尤其如此）。
- 组件：内部用英文 PascalCase 命名，导出时再起中文别名。
- 现状是中英混用（历史遗留），新代码按上面规矩写，别扩大混乱；不要求一次性重命名存量文件。
- 当前入口命名：员工主导航保留「今日工作台 / 成长大盘 / 文案助手」；`/violations` 保留独立页但通过工作台/成长大盘次级入口进入，`/video-review` 保留直接访问但不放员工主导航；管理端 `/admin/content` 用户可见名为「视频复盘」，路由不改。
- 表单切换：用 `key` 重建组件，禁止用 `useEffect` 同步 props（CLAUDE.md 硬规则）。
- 用项目封装的 UI 组件（`components/ui/*`）前，先看它真实的 Props，别套原版 shadcn/Radix 的写法。

## 四、发布纪律：哪些改动不能混在一起

一次提交/一次上线只做一件事、只落一个领域。`main` 一 push 就自动上线，混着发风险最大。禁止混在同一个提交/上线里的组合：

- 前端样式改动 + 接口/数据库改动：分属不同领域（前端 vs 后端），必须拆开。前端体验任务不碰接口和数据库，后端任务不碰页面样式。
- 功能改动 + 大范围重命名/搬文件：重命名单独一个提交，功能单独一个，出问题好回滚。
- 业务逻辑改动 + 性能优化：性能任务不改业务规则和数据含义，混着发无法判断是谁导致的变化。
- "顺手清理"混进功能提交：清理废代码、清理债务走独立提交，别塞进功能改动。
- 任何东西 + 数据库结构变更：migration 单独走，且只能新增、禁改旧 migration。

提交时的硬约束（血泪教训）：当前工作区可能压着别的会话的大量未提交前端改动。提交自己的活时只精确 `git add` 自己碰过的文件，禁止 `git add .` 把别人的脏改动一起提交。是否 push `main` 必须阿禅明确同意，禁止私自 push。

## 五、已知待清理债务清单

这些是当前已知的乱点，可作为"代码治理"领域的独立清理任务（逐项验证后再清，别顺手删），不要在做别的功能时顺带乱动：

| 债务 | 现状 | 治理方向 |
|---|---|---|
| lib 根目录偏杂 | `src/lib` 上百个条目，散文件多 | 按领域归进子目录，收敛根目录散文件 |
| rewrite 膨胀功能残留（注意命名陷阱） | 正式页 `(app)/content-tools/rewrite/page.tsx` 渲染的是 V3 组件 `RewriteWorkbenchV3`（`components/content-tools/rewrite-v3/`）。`components/content-tools/rewrite/` 旧组件目录与 `rewrite-v3` 路由壳已清理。`src/lib/rewrite/` 不是可直接删除的旧层，当前 `/api/rewrite/generate`、会话、文档、技能等仍依赖它；已归档的是旧 `/api/content-tools/rewrite/chat` 与 `/chat/stream` | 后续清理只能从已隐藏的低价值入口继续推进：SkillCabin、TimelineDiff、导出、定稿阅览、参数轮数剪枝、变体生成等。任何删除 `src/lib/rewrite/*` 或 `/api/rewrite/*` 前，必须逐路由确认前端调用、内部调用和测试覆盖 |
| 加载器未收口 | video-review / violations / admin-settings 页面内直查，未走 loader | 抽到 `lib/loaders/` |
| 中英文件名混用 | 存量文件命名不统一 | 新代码按规范；存量不强制重命名 |
| 无全局错误页/404 | 无 `app/error.tsx`、`app/not-found.tsx`、`app/global-error.tsx`，渲染出错用户看到浏览器原始报错或空白 | 补友好错误兜底页（属前端体验+代码治理交叉） |

清理接口是高危动作：判断一个接口能否删，必须逐项查 route 定义、前端调用、内部调用、测试、外部/手动触发，少查一项都不能判死（曾用 grep 搜不到就误判 6 个在用接口为闲置）。见 `docs/全站模块地图.md` 第五节。

## 六、部署与环境约束（治理硬边界）

- 部署平台只有 Vercel，`main` push 自动上线。仓库里的 OpenNext/Cloudflare 配置是历史残留/实验，不是正式方案，别当部署入口改。
- 数据库结构变更只能新增 migration，禁改旧 migration。
- 禁改线上 `.env`。环境变量分公开（`NEXT_PUBLIC_*`，改错登录直接崩）和服务端 key（改错接口崩），以 `.env.example` 和实际 `.env.local` 为准，别从旧记忆抄。
- Git remote 默认 SSH（`git@github.com:1305085564-hue/dydata.git`），禁改 HTTPS；`git config user.email` 必须是 `1305085564@qq.com`。

## 七、维护规则

- 新增 lib 子目录、改目录职责、收口新的 loader → 当场更新第一、二节。
- 清掉一项债务 → 从第五节删掉那行，并同步 `docs/全站模块地图.md` 相关标注。
- 命名或发布纪律有变（且与 CLAUDE.md 一致）→ 更新第三、四节。
- 与代码冲突一律以代码为准并立即修本文件。
- 本文件管"文件放哪、边界、怎么发"；模块归属在 `docs/全站模块地图.md`，性能红线在 `docs/架构与性能总纲.md`，权限在 `docs/权限与安全说明.md`。
- 顶部"最后核对代码日期"每次核对后更新。
