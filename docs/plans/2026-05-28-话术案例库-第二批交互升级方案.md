## 二、新发现的核心痛点（用户反馈，2026-05-28）

> 阿禅原话："话术上传那里应该是点击以后自动下一步，手动选没必要，提升此类的交互体验非常重要"

### 痛点

1. **「选择类型」步骤多余：** 用户已点了卡片就是确认，再让他点一次「下一步」是浪费动作。这种"点击即确认"的场景，应该自动推进。
2. **`step-wizard` 的进度指示器是「线性 4 段」展示：** 但实际上只有 3 步（类型选择 → 核心内容 → 详情补充 → 提交），4 步的视觉感让人误以为还要填好多。提交按钮应该更近。
3. **Step 切换动效过于温吞：** 当前 `x: 20 → 0` 的 0.2s 平移，缺少弹性反馈。
4. **截图上传是「拖拽框 + 文件名 chip」：** 缺少缩略图预览，传完看不到内容。
5. **置信度 / 转化率反馈是静态文字：** 数字变化时没有动效，缺少"在变好"的体感。

### 其他散点

- 抽屉详情虽然加了一键复制，但**整体阅读节奏**偏紧。截图 gallery 的 6 格固定阴影位置略显呆。
- 卡片在没有截图时使用 `from-zinc-50 to-white` 渐变，配合右上角浮动指标块的位置感不一致。
- 管理视角 `task-inbox-v2` 的 BulkActionBar 出现/消失时位置在底部 24px，移动端容易和 tab 栏冲突。

## 四、顶级交互体验的关键原则（落实到代码）

1. **"零冗余确认"：** 凡是点击=表态的场景（类型选择、平台选择、删除单图），下一步推进或反馈必须自动。手动「下一步」只用于需要 review 的场景。
2. **"物理感反馈"：** 所有按钮/卡片 hover 和 active 必须有视觉位移（哪怕 1px），spring 动效优先于 tween。
3. **"焦点闭环"：** 每一步切换时光标自动落到第一个可输入控件；提交成功时光标回到主 CTA。
4. **"状态完备性"：** 每个交互元素都有 default / hover / active / focus / disabled / loading 5 态。`task-inbox-v2` 的 busy 状态已经有 spinner，但 BulkActionBar 没全 disabled，要补。
5. **"反馈即时性"：** API 调用立即给 optimistic UI（按钮文字变「已通过」），失败再回滚。当前是 await 完才 toast，慢。
6. **"键盘可达"：** 全流程能用键盘走完。Enter/Esc/Cmd+Enter 三个键覆盖 90% 场景。
