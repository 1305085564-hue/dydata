# 选题库体验完善 · Antigravity 前端规格

> 日期：2026-07-23
> 范围：选题库前端第二轮体验完善，共 9 项改动的**前端部分**。
> 配套文件：后端规格见 `2026-07-23-选题库体验完善-Codex后端规格.md`（接口字段以它为准）。
> 视觉基线：全程遵守 `docs/美学规范.md`；颜色遵守「红涨绿跌」（红 `#C9604D`＝高于均值/正向，绿 `#6FAA7D`＝低于均值/负向）。

---

## 零、总原则

1. **不额外增加录入负担。** 任何新交互都不能逼用户多填字段才能用；选填就是选填，默认折叠、默认不填也能走完。
2. **推荐必须带理由。** 凡是系统给的候选/推荐/对比结论，旁边都要有一句"为什么"（依据、样本数、数据来源），不给黑盒结论。
3. **颜色纪律。** 涉及涨跌/高低于均值一律红涨绿跌；中性信息用 stone-* 灰阶；强调色 `#D97757`。禁止用绿色表示"好"、红色表示"差"的通用互联网直觉。
4. **样本少要示弱。** 后端返回 `lowConfidence` 或样本<3 的对比/推荐，必须灰示"样本少仅供参考"，不能让人误当结论。
5. **改的是正式前端，不是临时前端。** 下列涉及的页面/组件都是现役产品代码，在其基础上迭代，不推倒重写。

## 二、逐项前端规格

### item 1 · 子题编辑 / 删除（后端已就绪，纯前端）

**入口**：子题详情页 `[id]/page.tsx` 顶部操作区、子题卡 `sub-topic-card.tsx` 展开态。
**权限门控**：仅当 `detail.created_by === currentUserId` 时显示"编辑""删除"，否则不渲染（后端已返回 `created_by`）。

- **编辑**：点击后弹出与录入弹窗同结构的表单（复用 item 6 的字段布局），预填 `title/hook/emotion_tag/source/audience`，提交走 `PATCH /api/topics/sub-topics/[id]`。
- **删除**：二次确认弹窗。若后端返回 `409`（存在已关联作品），不弹通用错误，改为明确文案：「该选题已有 N 条作品关联，删除会切断数据回流。请先处理关联作品，或改为编辑。」并只给"取消/去看关联作品"两个出口，不给强删。
- 成功后：详情页返回选题池并 toast；卡片内编辑则原地刷新，派发既有 `refresh-topics` 事件。

### item 2 · 上传关联"给候选不预选"（纯前端）

**痛点**：`video-submit-form.tsx:664-676` 会把 suggest 返回的 `data[0]` 自动绑成 `topicId`，界面直接跳"已关联"绿标，用户没得选就被替他选了。
**改法**：

- **删除自动绑定**：不再调用 `shouldAutoSelectSuggestedTopic` 自动 setState `topicId`。suggest 回来后**只填充候选列表，不写 `topicId`**。
- 未选中状态下，展示 **2–3 个候选按钮**（现有 `suggestions.slice(0, 2)` 改为 `slice(0, 3)`），每个候选下方补一行极小字理由（母题名，如「美妆 · 系统按标题匹配」）。
- 用户点某个候选 → 才写 `topicId` → 进"已关联"态。仍保留"暂无对应""新建选题""换一个"三个出口。
- URL 带 `topicId`（从认领跳来）的场景不变，仍锁定。

### item 3 · 冷启动 / 空态卡片过渡（纯前端）

**场景**：今日工作台和选题池在"还没认领 / 还没作品 / 还没录入"时，目前是干瘪空白。
**改法**：给每个区块设计**引导型空态**（复用既有 `EmptyState`），文案指向下一步动作而非单纯"暂无数据"：

- 今日工作台无认领：「还没认领选题。去选题池挑一个开始，或直接新建。」+ 主按钮"去选题池" + 次按钮"新建选题"。
- "值得再做"（item 5）无数据：「还没有跑出成绩的选题。发几条作品、积累数据后，这里会推荐值得复刻的选题。」
- 选题池整体为空（母题下无子题）：引导"新建选题"。
- 过渡：加载中用骨架屏（`Skeleton`），不要 spinner 闪烁；空态与骨架屏切换用 framer-motion 淡入，避免跳变。

### item 4 · 候选上限 5 的"替换哪一个"弹窗（纯前端）

**场景**：认领候选达 5 条上限后再点认领，后端返回 `409`。现在只是报错，用户被卡死。
**改法**：捕获 `409` → 弹出**替换选择弹窗**：

- 列出当前 5 条 `candidate` 状态认领（复用 item 7 的 `/claims` 或既有"我的认领"数据），每条显示子题标题 + 认领时间。
- 用户选一条"放回" → 先调 `POST /return` 释放该条 → 成功后自动 `POST /claim` 认领新的 → 一步到位。
- 文案：「候选位已满（5/5）。放回一条腾出位置，再认领这条？」按钮"确认替换 / 取消"。
- 若用户取消，保持原状，不误删。

### item 5 · 今日工作台向"有数据的选题"倾斜

**依赖**：后端 `GET /api/topics/active` 新增 `worthRedoing` 列（见后端规格 item 5）。
**改法**：`today/page.tsx` 现在是三列等权（recentlyClaimed / recentlyWorked / recentlyCreated）。重排为**主次分明**：

- **主区（顶部、更大）**：新增"值得再做" `worthRedoing`——已跑出成绩、值得复刻的选题。每张卡显示：子题标题 + 母题标签 + `qualifiedWorkCount` 达标作品数 + `averagePlayCount` 平均播放（红涨绿跌：高于全库均值标红、低于标绿）+ 一句 `bestCopy` 摘要。这是阿禅进来第一眼想看的"哪个还能再榨"。
- **次区（下方，等权三列或折叠）**：原三列降为辅助信息，标题弱化。"我最近认领"保留醒目（提醒未完成），"最近录入""团队最近在做"可收窄。
- 候选上限进度条（x/5）保留在主区顶部。
- 冷启动（`worthRedoing` 为空）走 item 3 空态。

### item 6 · 录入弹窗选填字段折叠 + hook/title 分离（纯前端，后端已就绪）

**痛点**：`global-topic-create.tsx:117-119` 把 `hook` 直接 `= title`，且只有一个输入框，导致子题卡两行重复、`emotion_tag/audience` 永远空。
**改法**：

- **主输入**：保留一个必填框，明确 label「选题标题」。
- **新增 hook 框**：第二个框「一句话钩子（选填）」，不填时**留空提交 `hook: null`**（不要再 `hook = title`）。子题卡遇 `hook` 为空只显示标题，不再重复两行。
- **选填折叠区**：默认收起，一行"添加更多（情绪/来源/受众）"文字按钮展开。展开后三个选填：`emotion_tag`（情绪标签）、`audience`（目标受众）、`source` 前端不暴露、固定传 `"manual"`。
- 折叠动效用 framer-motion 高度展开，符合《美学规范》节奏。
- 提交 payload 改为：`{ title, hook: hook.trim() || null, emotion_tag, audience, topic_id: selectedTopicId }`。

### item 7 · 认领动态 / 防撞车换服务端接口（修 bug）

**痛点**：详情页 `fetchAllClaims`（`[id]/page.tsx:196-218`）用浏览器直连查 `sub_topic_claims`，普通成员命中 RLS 永远 `claimsLoadFailed`。
**改法**：

- 删除浏览器直查，改调后端新接口 `GET /api/topics/sub-topics/[id]/claims`（见后端规格 item 7）。
- 渲染返回的 `claims`：`scripting`（脚本中）用更强的警示色（`#C9604D` 红，撞车风险高），`candidate`（候选）用中性灰；显示认领人名 + 状态 + 认领时间。
- 顶部一句汇总：「已有 N 人在做（含 M 人脚本中）」，数字用 `candidateCount/scriptingCount`。
- 加载失败才显示重试；正常情况普通成员也能看到，不再一律失败。

### item 8 · "哪类选题在哪个账号最能打"横向对比

**依赖**：后端 `GET /api/topics/comparison`（见后端规格 item 8）。
**入口**：选题池 `topics/page.tsx` 顶部加"横向对比"标签页或抽屉入口（与母题筛选并列）。
**视图**：

- **母题维度**（`dimension=topic`）：一张排行表，列＝母题 / 作品数 / 达标率 / 平均播放 / 最高播放。达标率与平均播放按红涨绿跌着色（高于全体中位标红、低于标绿）。默认按达标率倒序。
- **账号维度**（`dimension=account`）：选定一个母题后，看各账号在该母题下的表现（同样红涨绿跌）；或母题×账号矩阵热力表。
- **样本示弱**：`lowConfidence` 或样本<3 的行灰示 +「样本少仅供参考」角标。
- 时间范围切换（近 7 / 14 / 30 天）复用既有 Select 样式。
- 结论旁给一句人话解读，如「美妆在 A 号达标率 37%，明显高于 B 号 12%，同题材优先投 A 号」。

### item 9 · AI 选题建议并入选题库（"系统推荐"）

**依赖**：后端 `GET /api/topics/recommendations` + `POST /api/topics/sub-topics/from-recommendation`（见后端规格 item 9）。
**位置**：选题池 `topics/page.tsx` 顶部新增独立"系统推荐"区（或标签页），与手动录入的选题池并列，视觉上明确区分"这是系统基于爆款样本给的、还没进库"。
**卡片**：复用 content-tools 的 `TopicSuggestionItem` 结构展示——标题 / 母题 / 切入角度 / 预期表现 / 数据依据 / 参考视频。每张卡带：

- **"采纳进库"主按钮** → 调 `POST .../from-recommendation`（把 `title/angle→hook/category` 传上去）→ 成功后 toast「已存入选题池」并刷新，可直接去认领。
- **"忽略"次按钮** → 本次会话隐藏该条（前端记忆即可，不必落库）。
- 顶部一句依据汇总（`evidenceSummary` + 样本数 + 热点日期），呼应"推荐带理由"。
- 生成中用骨架屏；无爆款样本时空态引导"先积累作品数据"。
- **视觉区隔**：系统推荐区用与手动选题不同的底色/描边（如浅 `#D97757` 描边），让人一眼分清"系统给的"和"团队自己录的"。

## 三、验收要点（交付前自查）

1. item 2：上传时输入标题，出现 2–3 个候选**但不自动选中**，界面停在"待选择"，点了才进"已关联"。
2. item 6：新建选题只填标题、不填钩子，子题卡只显示一行（不再重复）；展开选填能填情绪/受众。
3. item 7：用**普通成员账号**看别人做过的子题，撞车动态正常显示、不再一律"加载失败"。
4. item 4：认领满 5 条后再认领，弹替换弹窗而非死错误，替换一步完成。
5. item 5：今日工作台"值得再做"在最显眼位置，平均播放红涨绿跌着色正确。
6. item 8/9：对比结论、系统推荐都带"为什么"，样本少的灰示"仅供参考"。
7. 全程颜色符合红涨绿跌，视觉符合《美学规范》，未引用任何 Codex 一次性前端代码。

## 四、回滚

本轮前端全部是**页面/组件内的增量改动**，无新增依赖、无路由结构变更。回滚方式：`git revert` 对应前端 commit 即可，后端新增接口不被调用不影响旧行为。上线方式：前端改动需 **push 后经 Vercel 部署才生效**（阿禅注意：本地改完不等于线上生效）。
