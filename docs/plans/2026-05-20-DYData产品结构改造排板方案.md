# DYData 产品结构改造排板方案

## 一、改造结论

DYData 的核心问题不是页面数量，而是页面没有按真实工作流分层。产品应该按“三个闭环”组织：组员执行闭环、管理者日常管理闭环、系统治理闭环。

当前主方向是：员工每天围绕“今日工作台”执行；组长和负责人围绕“团队管理”处理日常管理；负责人和 owner 通过“系统设置”处理成员、分组和 AI 配置；“文案助手”继续保留为独立工具，用于员工单独改文案；“话术案例库”承接话术案例、转化中心和合规审核。

第一二阶段的目标不是完成全部训练闭环，而是先把网站整理成“可理解、可使用、可上线”的结构整理版。第三、四阶段涉及视频复盘、AI 初诊、反馈卡、复验、成长训练和案例沉淀，短期不混进上线版，后续单独开分支设计和验收。

页面上用户可见名称全部使用中文，不出现 Dashboard、Admin、AI Assistant 等英文入口。内部路由可以暂时保留，优先保证用户看到的产品结构清楚。

## 二、改造后的主入口

| 主入口 | 面向角色 | 定位 |
|---|---|---|
| 今日工作台 | 全员 | 每天提交数据、查看状态，后续承接反馈卡和下一条动作 |
| 个人成长 | 全员 | 查看个人趋势、短板和后续训练方向 |
| 话术案例库 | 全员；合规审核仅授权管理者 | 查看话术案例、转化中心内容；管理者处理合规审核 |
| 文案助手 | 全员或授权成员 | 单独改文案、润色文案、生成不同版本 |
| 团队管理 | 组长、负责人、owner | 处理今日待办、经营分析、视频素材、内容复盘 |
| 系统设置 | 负责人、owner | 负责人处理成员权限和团队分组；owner 额外处理 AI 配置 |

这 6 个入口对应两个高频工作流和一个低频治理入口：员工高频用“今日工作台”；管理者高频用“团队管理”；负责人和 owner 低频用“系统设置”。导航形态以左侧边栏为主，顶部区域只放账号、状态和轻操作，不承载主结构。

## 四、不同角色看到什么

| 角色 | 可见主入口 |
|---|---|
| 普通成员 | 今日工作台、个人成长、话术案例库、文案助手 |
| 组长 | 今日工作台、个人成长、话术案例库、文案助手、团队管理 |
| 负责人 | 今日工作台、个人成长、话术案例库、文案助手、团队管理、系统设置 |

系统设置不是只给 owner。负责人也需要处理本团队成员权限和团队分组，所以负责人应看到系统设置；但 AI 配置只能 owner 使用。组长只处理本组日常管理，不进入系统设置。

## 五、增加、减少、合并、迁移

### 需要增加

1. 增加“团队管理”作为管理者日常入口。
2. 增加“系统设置”作为负责人和 owner 的治理入口。
3. 在“话术案例库”增加三个清晰分区：话术案例、转化中心、合规审核。
4. 第三阶段再增加视频复盘闭环：待复盘任务、AI 初诊、人工复核、反馈卡、复验。

### 需要减少

1. 减少主侧边栏里的技术化入口。
2. 减少团队管理里的系统配置入口。
3. 减少团队管理里的话术审核入口。
4. 减少“AI”作为独立概念反复出现，只保留“文案助手”和系统设置里的“AI 配置”。

### 需要合并

1. 后台管理 + 视频资产 + 内容复盘 + 经营分析，合并进“团队管理”。
2. 违规库 + 转化中心 + 违规复核，合并进“话术案例库”。
3. 权限模块 + 团队分组 + AI 配置，合并进“系统设置”。

### 需要迁移

1. 转化中心从团队管理迁移到“话术案例库 → 转化中心”。
2. 违规复核从团队管理迁移到“话术案例库 → 合规审核”。
3. 成员权限和团队分组迁移到“系统设置”，负责人和 owner 可见。
4. AI 配置迁移到“系统设置”，仅 owner 可见。
5. 模块开关暂不显示，等有真实内容后再决定是否恢复。

## 六、分批排板建议

### 第一批：入口改名与角色降噪

目标：先让网站看起来像一个能用的产品，让不同角色知道自己该去哪里。

改动：
- 数据填报 → 今日工作台。
- 成长分析 → 个人成长。
- 违规库 → 话术案例库。
- AI 助手 → 文案助手。
- 后台管理 → 团队管理。
- 增加系统设置入口。
- 页面可见入口全部中文化。

验收：
- 普通成员只看到今日工作台、个人成长、话术案例库、文案助手。
- 组长能看到团队管理，但看不到系统设置。
- 负责人和 owner 能看到系统设置。
- 侧边主入口不再出现英文产品名。

### 第二批：团队管理、系统设置、话术案例库归位

目标：让管理者日常工作、系统治理、话术资产三件事分开。

改动：
- 团队管理只保留：今日待办、经营分析、视频素材、内容复盘。
- 团队管理首页不再放卡片式二级导航，直接展示今日处理概览和待办队列。
- 系统设置承接：成员权限、团队分组、AI 配置。
- AI 配置仅 owner 可见。
- 模块开关暂不显示。
- 话术案例库承接：话术案例、转化中心、合规审核。
- 转化中心复用原真实页面内容：话术、周报、分析、建议。
- 合规审核复用原真实审核工作台：确认、驳回、暂存、模板。

验收：
- 团队管理侧边栏只出现今日待办、经营分析、视频素材、内容复盘。
- 转化中心和合规审核不再出现在团队管理。
- 负责人在系统设置里能处理成员权限和团队分组，但看不到 AI 配置。
- owner 在系统设置里能看到成员权限、团队分组、AI 配置。
- 话术案例库三个 tab 都是真实内容，不只是改名。

### 第三批：视频复盘闭环

目标：把“数据上传”真正变成“反馈提升”。这一批不跟一二阶段一起上线。

改动：
- 成员提交视频数据后，生成待复盘任务。
- 管理者看到待 AI 初诊、待人工复核、已下发反馈。
- AI 只做初诊草稿，人来确认最终反馈。
- 反馈以“反馈卡”形式下发给成员。
- 成员在今日工作台看到反馈卡和下一条动作。
- 下一条视频提交后，系统能做复验对比。

验收：
- 组长可以完成：看视频 → 看 AI 初诊 → 人工修改 → 下发反馈。
- 成员可以在今日工作台看到反馈和下一条动作。
- AI 不直接替代人，只辅助管理者。
- 不引入半成品数据库表和不可用入口。

### 第四批：个人成长训练化和案例沉淀

目标：让成员不是只填报，而是持续被训练。

改动：
- 个人成长增加短板、趋势、训练建议、动作追踪。
- 视频复盘里的优秀内容可以沉淀到话术案例库。
- 话术案例库进一步区分优秀案例、风险案例、可复用模板。
- 让案例库从资料展示页升级为团队学习资产。

验收：
- 成员能知道自己最近最该补哪一项。
- 管理者能把好案例沉淀下来。
- 案例库不再像处罚页，而像团队学习资料库。

## 七、当前优先级判断

当前优先级是先把第一批和第二批做成可上线版本。判断标准不是“功能是否全部先进”，而是“员工和管理者是否能看懂入口、权限边界是否安全、团队管理是否不再混乱”。

第三批是产品价值中枢，但它牵涉数据表、接口、权限、AI 稳定性和反馈状态流转，不能夹在一二阶段里赶工。正确做法是单独开分支，先设计业务流和数据结构，再做前后端。

第四批依赖第三批。只有视频复盘和反馈卡跑通后，个人成长训练化和优秀案例沉淀才有真实来源，否则会继续变成页面堆积。

## 八、最终产品定位

DYData 不应该被定位成“数据分析后台”，也不应该只是“AI 工具”。它应该是一个短视频团队训练与管理系统。

当前一二阶段上线版的定位是“结构整理和权限安全版”：让网站入口清楚、角色清楚、团队管理清楚、话术案例库归位。

后续三四阶段完成后，产品才会进入“完整训练闭环版”：员工提交数据，管理者复盘反馈，AI 辅助初诊，成员按反馈改下一条，结果再复验，优秀案例沉淀到话术案例库。

## 十三、补充记录：话术案例库内容迁移修正

### 1. 根因：上一轮只是外壳迁移

第十二节做完后实际只完成了 tab 名称从“话术案例 / 转化中心 / 合规审核”切到了 /violations 页面，但 `/violations?perspective=conversion` 还是普通转化话术卡片列表，`/violations?perspective=review` 还是普通 CaseCard 列表，无法完成确认 / 驳回操作。

本节是对内容做真正迁移：把原 `/admin/conversion-hub` 和原 `/admin/violations` 的真实页面内容嵌入到话术案例库三个 perspective tab 下，不是另起炉灶写假页面。

### 2. 话术案例库 三 tab 实际承接

| perspective | 用户标题 | 实际内容 |
|---|---|---|
| violation | 话术案例库 | 原话术案例列表 + 筛选器 + 提交新案例（CaseCard / CaseFilters） |

- 把 `ConversionHubShell` 抽成可参数化组件（`basePath` / `extraQueryParams` / `hideViolationsTab` / `pendingViolationsHref` / `layoutVariant`），既能在 `/admin/conversion-hub` 当独立页用，也能嵌入 `/violations` 当 tab 用。
