# DYData 产品结构改造排板方案

## 一、改造结论

DYData 现在不是缺页面，而是页面关系没有按真实工作流组织。后续改造不以“新增更多功能”为主，而是把现有页面重新分层：员工日常使用、管理者日常使用、owner 系统维护三类入口分开。

最终方向：保留“文案助手”作为独立工具；把“后台管理、视频资产、内容复盘、经营分析”合并成“团队管理”；把“权限、AI 配置、模块开关、系统规则”迁移到“系统设置”。

页面上用户可见名称全部使用中文，不出现 Dashboard、Admin、AI Assistant 等英文入口。内部路由可以暂时保留，不作为第一阶段改造重点。

## 二、改造后的主入口

| 主入口 | 面向角色 | 定位 |
|---|---|---|
| 今日工作台 | 全员 | 每天提交数据、查看反馈、知道下一条视频怎么改 |
| 个人成长 | 全员 | 看短板、趋势、训练方向和历史进步 |
| 话术案例库 | 全员 | 查优秀话术、风险话术、可复用案例 |
| 文案助手 | 全员或授权成员 | 单独改文案、润色文案、生成不同版本 |
| 团队管理 | 组长、负责人、owner | 看提交情况、批改视频、复盘内容、管理视频素材、看经营数据 |
| 系统设置 | owner 或授权负责人 | 成员审批、权限、AI 配置、模块开关、系统规则 |

## 三、原页面处理清单

| 原页面/模块 | 处理方式 | 改后位置 | 具体变化 |
|---|---|---|---|
| 数据填报 | 改名 + 升级 | 今日工作台 | 保留提交数据功能，增加昨日反馈、下一条动作、待处理提醒 |
| 成长分析 | 改名/强化 | 个人成长 | 从单纯图表，升级为短板、趋势、训练建议 |
| 违规库 | 改名 | 话术案例库 | 去掉处罚感，改成优秀话术、风险话术、可复用案例 |
| AI 助手 | 保留 + 改名 | 文案助手 | 作为独立入口保留，专门给员工改文案用 |
| 后台管理 | 改名 + 重做 | 团队管理 | 作为管理者日常入口，优先显示谁没交、谁待复盘、谁要反馈 |
| 视频资产 | 合并 | 团队管理 → 视频素材/视频复盘 | 不再作为独立一级入口，放入团队管理 |
| 内容复盘 | 合并 | 团队管理 → 内容复盘 | 和视频复盘、视频素材放在同一组 |
| 经营分析 | 迁移 | 团队管理 → 经营分析 | 管理者仍需要看，保留在团队管理里 |
| 权限模块 | 迁移 | 系统设置 | 从团队管理里拿走，避免组长看到复杂配置 |
| AI 配置 | 迁移 | 系统设置 | 只给 owner 或授权负责人看 |
| 模块开关/系统规则 | 迁移 | 系统设置 | 低频维护内容，不放在管理者日常入口 |
| 视频复盘台 | 新增/修好 | 团队管理 → 视频复盘 | 做成核心闭环：AI 初诊、人复核、下发反馈卡 |

## 四、不同角色看到什么

| 角色 | 可见主入口 |
|---|---|
| 普通成员 | 今日工作台、个人成长、话术案例库、文案助手 |
| 组长 | 今日工作台、个人成长、话术案例库、文案助手、团队管理 |
| 负责人 | 今日工作台、个人成长、话术案例库、文案助手、团队管理 |
| owner | 今日工作台、个人成长、话术案例库、文案助手、团队管理、系统设置 |

组长和负责人默认不需要看到系统设置。系统设置只处理低频维护，不参与每天的业务管理。

## 五、增加、减少、合并、迁移

### 需要增加

1. 增加“团队管理”这个统一入口。
2. 增加“系统设置”这个 owner 维护入口。
3. 补齐“视频复盘台”的完整业务闭环。
4. 在“今日工作台”增加反馈卡、下一条动作、待处理提醒。

### 需要减少

1. 减少顶部导航里的技术化入口。
2. 减少组长/员工看到的系统配置内容。
3. 减少“AI”作为独立概念反复出现，只保留“文案助手”这个明确工具。

### 需要合并

1. 后台管理 + 视频资产 + 内容复盘 + 经营分析，合并进“团队管理”。
2. 视频资产和内容复盘不再分散成多个一级入口，而是作为团队管理里的功能分区。

### 需要迁移

1. 权限模块迁移到“系统设置”。
2. AI 配置迁移到“系统设置”。
3. 模块开关、系统规则、系统治理类内容迁移到“系统设置”。

## 六、分批排板建议

### 第一批：入口改名与页面降噪

目标：先让网站看起来像一个能用的产品。

改动：
- 数据填报 → 今日工作台。
- 成长分析 → 个人成长。
- 违规库 → 话术案例库。
- AI 助手 → 文案助手。
- 后台管理 → 团队管理。
- 新增系统设置入口，仅 owner 可见。
- 把权限、AI 配置、模块开关从团队管理里拿走，归到系统设置。

验收：
- 普通成员看不到复杂后台配置。
- 组长能看到团队管理，但看不到系统设置。
- owner 能看到团队管理和系统设置。
- 页面可见入口全部是中文。

### 第二批：团队管理重组

目标：让管理者进入后台后，第一眼知道今天该处理什么。

改动：
- 团队管理首页只回答三个问题：谁没交、谁待复盘、谁需要反馈。
- 团队管理内部保留：今日待办、视频复盘、内容复盘、视频素材、经营分析。
- 视频资产、内容复盘、经营分析从分散页面变成团队管理内的分区。

验收：
- 组长进入团队管理后，10 秒内能知道今天要处理谁。
- 负责人能看到团队范围，组长只能看到本组范围。
- 系统配置类内容不再干扰日常管理。

### 第三批：视频复盘台闭环

目标：把“数据上传”真正变成“反馈提升”。

改动：
- 成员提交视频数据后，生成待复盘任务。
- 视频复盘台显示待 AI 初诊、待人工复核、已下发反馈。
- AI 只做初诊，人来确认最终反馈。
- 反馈以“反馈卡”形式下发给成员。
- 成员在今日工作台看到反馈卡和下一条动作。

验收：
- 组长可以完成：看视频 → 看 AI 初诊 → 人工修改 → 下发反馈。
- 成员可以在今日工作台看到反馈。
- AI 不直接替代人，只辅助管理者。

### 第四批：个人成长和话术案例库强化

目标：让成员不是只填报，而是持续被训练。

改动：
- 个人成长增加短板、趋势、训练建议。
- 话术案例库区分优秀案例、风险案例、可复用模板。
- 视频复盘中优秀内容可以沉淀到话术案例库。

验收：
- 成员能知道自己最近最该补哪一项。
- 管理者能把好案例沉淀下来。
- 案例库不再像处罚页，而像团队学习资料库。

## 七、当前优先级判断

第一优先级不是大改 UI，而是先改信息结构。先做第一批和第二批，让入口和管理逻辑变清楚。

第三批是产品价值中枢，需要谨慎做。当前项目里视频复盘台已经有半成品，但字段和状态需要先校准，再继续补闭环。

第四批等前三批跑通后再做，否则会继续变成页面堆积。

## 八、最终产品定位

DYData 不应该被定位成“数据分析后台”，也不应该只是“AI 工具”。它应该是一个短视频团队训练与管理系统。

员工端负责执行和提升；管理端负责批改和复盘；owner 端负责系统治理。只要这三层分清，页面就不会继续乱。

## 九、补充记录：第一二批合并验收结果

记录时间：2026-05-20。

第一批“入口改名与页面降噪”和第二批“团队管理重组”已合并执行并通过验收。本次不改原方案正文，以下作为实际执行后的补充口径。

### 1. 已落地的入口结构

| 入口 | 实际结果 |
|---|---|
| 今日工作台 | 已替代原“数据填报”入口 |
| 个人成长 | 已替代原“成长分析”入口 |
| 话术案例库 | 已替代原“违规库”入口 |
| 文案助手 | 已保留为独立入口，给员工改文案使用 |
| 团队管理 | 已替代原“后台管理”，承载日常业务管理 |
| 系统设置 | 已新增为 owner 独立入口 |

页面用户可见的旧名称已清理：数据填报、成长分析、AI助手、后台管理等不再作为用户入口出现。英文入口和主要英文 eyebrow 已完成中文化。

### 2. 团队管理实际结构

团队管理只保留日常管理入口：

| 分区 | 实际入口 |
|---|---|
| 日常管理 | 今日待办、经营分析、内容复盘、视频素材、转化中心、违规复核 |

系统设置类入口不再出现在团队管理侧边栏中：

| 原入口 | 实际处理 |
|---|---|
| 成员权限 | 不在团队管理侧边栏显示 |
| 团队分组 | 不在团队管理侧边栏显示 |
| AI 配置 | 仅在系统设置/直接路由中给 owner |
| 模块开关 | 归入系统设置 |

### 3. 权限边界最终口径

| 角色 | 可见入口 |
|---|---|
| 普通成员 | 今日工作台、个人成长、话术案例库、文案助手 |
| 组长 | 今日工作台、个人成长、话术案例库、文案助手、团队管理 |
| 负责人 | 今日工作台、个人成长、话术案例库、文案助手、团队管理 |
| owner | 今日工作台、个人成长、话术案例库、文案助手、团队管理、系统设置 |

系统设置 `/admin/settings` 只给 owner。AI 配置 `/admin/ai-channels` 只给 owner。

历史兼容点：`/admin/modules` 仍保留负责人直接访问能力，用于延续原有负责人管理成员的权限。但它不再通过团队管理侧边栏暴露，不作为本轮普通日常管理入口。

### 4. 数据范围验收结果

团队管理相关待办和徽标已按数据范围过滤：

| 角色 | 数据范围 |
|---|---|
| owner | 全局 |
| 负责人 | 本团队 |
| 组长 | 本组 |
| 普通成员 | 无团队管理入口 |

已覆盖范围：团队管理首页待办、待审违规列表、侧边栏 `cockpit` 徽标、`conversion_hub` 徽标。

### 5. 技术验收结果

已通过：
- 关键测试：29/29。
- TypeScript 检查通过。
- 生产构建通过。

构建仍有两个既有提醒，不作为本轮阻断：
- Next.js 提醒 `middleware` 文件约定后续迁移到 `proxy`。
- 部分图表容器在静态生成阶段出现宽高警告。

### 6. 下一步建议

第一二批已完成，建议暂停扩大同类改动，先进入第三批“视频复盘台闭环”的单独设计和施工。

第三批开始前需要先明确三件事：
1. 视频复盘任务如何生成：由视频提交自动生成，还是管理者手动加入复盘。
2. AI 初诊结果是否只作为草稿，不直接发给成员。
3. 反馈卡下发后，成员在今日工作台看到什么，以及下一条视频如何复验。

## 十、补充记录：一二阶段上线版调整

记录时间：2026-05-20。

由于第三批、第四批涉及视频复盘、反馈卡、复验、案例沉淀等完整业务逻辑，短期落地难度较大，当前决定先上线第一批和第二批。上线目标从“继续推进视频复盘闭环”调整为“先让现有网站结构清楚、权限安全、入口可理解”。

本次上线版保留：

| 模块 | 上线处理 |
|---|---|
| 今日工作台 | 保留现有提交、通知、数据展示能力，使用中文产品命名 |
| 个人成长 | 保留并完成入口改名 |
| 话术案例库 | 保留并完成入口改名 |
| 文案助手 | 保留独立入口，继续作为员工改文案工具 |
| 团队管理 | 保留今日待办、经营分析、内容复盘、视频素材、转化中心、违规复核 |
| 系统设置 | 保留 owner 独立入口，系统设置和 AI 配置不进入团队管理侧边栏 |

本次上线版暂不带入：

| 第三/四阶段内容 | 暂缓原因 |
|---|---|
| 独立视频复盘台 `/admin/video-review` | 业务闭环未完整设计，暂不上线半成品入口 |
| `video_review` 数据表迁移 | 不把未成熟业务表带入生产数据库 |
| AI 初诊、人工复核、反馈卡、复验状态 | 需要单独设计状态流转和权限边界 |
| 优秀案例自动沉淀到话术案例库 | 属第四批增强能力，等复盘闭环稳定后再做 |

调整后的变更路径：

1. 第一二阶段先上线，解决“页面看不懂、入口太乱、权限边界不清”的问题。
2. 第三阶段只保留在方案中，不在本次上线代码里暴露入口或数据库迁移。
3. 等团队实际使用一二阶段后，再单独开第三阶段分支，重新设计视频复盘台的数据表、接口、权限、页面和验收标准。

上线版判断：可以作为阶段性可用版本发布，但它的定位是“结构整理和权限安全版”，不是“完整视频训练闭环版”。
