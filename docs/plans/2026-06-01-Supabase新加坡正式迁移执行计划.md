## 4. /admin/content 卡住的收尾修复

迁库后又发现一个真实收尾问题：

- 用户浏览器里可能还保留着旧孟买项目的 Supabase 登录 cookie。
- 切到新加坡后，这些旧 cookie 不再对应当前项目，但浏览器仍会带上它们。
- 结果就是后台路由容易表现成“像登录了但又进不去”，体感上就是卡住。

这次已补上兼容清理逻辑：

- 中间件现在会识别所有 Supabase 登录 cookie。
- 如果发现是旧项目遗留 cookie，而不是当前项目 cookie，会自动清掉。
- 清掉后会跳回 `/login?expired=1`，提示重新登录，而不是继续卡在后台页。

这次修复的目标不是改业务逻辑，而是把“切库后的旧登录态残留”处理干净。

## 6.2 追加发现：部分老账号 Auth 服务掉队

在迁移收尾后的真实登录排障中，又确认了一类 Auth 层遗留问题：

- 数据库里的 `auth.users` 有 58 条，但新加坡正式库的 Auth 后台实际只识别出 34 条。
- 也就是说，有 24 个账号属于“数据库里有、`profiles` 也有，但 Auth 服务后台不认”的半残状态。
- 这类账号会表现为：
  - 页面上像是有账号
  - `profiles.status` 也是 `active`
  - 但登录时报 `Invalid login credentials`
  - 管理员接口 `getUserById` / `updateUserById` 一开始会报 `User not found`

本次已实锤并修复的样本账号：

- `1035658919@qq.com`
- `766510408@qq.com`

真实根因：

- 这类账号迁移后在新加坡 `auth.users` 表中仍有记录，但关键内部字段不完整。
- 以 `1035658919@qq.com` 为例：
  - `auth.users.instance_id` 是 `null`
  - `auth.identities` 缺失对应 `email` provider 身份记录
- 正常账号（如 owner）在新加坡正式库中：
  - `instance_id = 00000000-0000-0000-0000-000000000000`
  - `auth.identities` 存在完整 `email` 身份记录

修复方法已经验证有效：

1. 先把失联账号补回正常 Auth 口径：
   - `auth.users.instance_id` 补成 `00000000-0000-0000-0000-000000000000`
   - 补齐 `auth.identities` 的 `email` provider 记录
2. 再通过 Auth 管理接口重置一次密码
3. 然后直接实测密码登录，确认 `last_sign_in_at` 能正常写回

本次 `1035658919@qq.com` 的修复结果：

- Auth 后台已能重新识别
