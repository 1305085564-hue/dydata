# 转化中心 — 设计方案 V3（最终版）

> V1 初稿 → V2 Codex 技术审核 → V3 产品发散 + 入口/流程修正 + 分工确认

---

## 背景

一家财经自媒体公司（多个分公司），核心业务是通过抖音短视频导粉到公众号。

**痛点：**
1. 导粉话术很多，但不知道哪种转化高、哪种容易违规限流
2. 各分公司各自摸索，没有统一的数据沉淀和分析
3. 违规原因不明确，被限流后只能靠经验猜测
4. 好的话术没有被识别和推广，差的话术没有被及时淘汰
5. 抖音平台审核规则持续升级，合规策略需要持续迭代

**现有系统基础：**
- 违规话术案例库 V1（全员提交、截图、管理员复核、通过率统计）
- 员工日报系统（每天填报播放量、导粉数等数据）
- 生产管控系统 SOP（选题→脚本创作→成片审核）
- AI 洞察能力（已接入 Claude API）
- 技术栈：Next.js App Router + Tailwind + shadcn/ui + Supabase + Vercel

**关键约束：**
- 只有约 20% 的员工涉及导粉工作，转化中心对他们是核心工具，对其他人是可选模块
- 数据填报是"第二天上传昨天的数据"，不是实时的
- 被限流的视频往往没有数据（播放量极低或为零），不能走正常数据填报流程

---

## 目标

将违规库升级为**转化中心**，一个模块同时解决：
- **避坑**：哪些话术/画面容易违规限流，原因是什么
- **提效**：哪些话术/画面转化率高，为什么高，怎么复用
- **行动**：每周产出"推广/继续测/淘汰/禁用"四类清单，驱动管理决策

本次目标：完成度 90%，实用性 80%。预留扩展空间但基本成型。

---

## 核心原则

1. **数据驱动**：加权转化率 + 账号基准校正，不靠经验判断好坏
2. **零额外成本采集**：嵌入日报填报流程，不多填一张表
3. **AI 辅助不替代人**：AI 给建议，管理员拍板
4. **合规导向**：找到合规的高转化表达，不是教人绕过审核
5. **可扩展**：当前两个方向（违规 + 转化），架构支持未来加新方向
6. **视觉隔离**：导粉相关功能对非导粉员工不可见或折叠

## 入口设计（核心）

**只有两个入口，不搞第三个。**

### 入口 1：日报填报页（正常情况）

现有数据填报表单的"指标录入"区域：

- 导粉数字段变成开关逻辑
- **导粉数 > 0** → 动画展开"导粉话术"区域（文案输入框 + 未来可扩展区域）
- **导粉数 = 0 或未动** → 动画收缩隐藏
- 文案输入框与左侧"抖音视频链接"行对齐
- 提交时，后端一次操作写两张表（日报表 + script_usage_records）

**交互细节：**
- 展开/收缩用平滑动画
- 导粉话术区域内预留可迭代空间（组件化，字段不写死）
- 文案为必填（导粉数 > 0 时）
- 未来可在此区域内增加：导粉截图上传、话术标签选择、AI 预检等

### 入口 2：违规上报（异常情况）

现有 `/violations/submit` 升级为转化中心的违规上报入口：

- 被限流/违规的视频没有正常数据，走这个入口
- 填：话术文案 + 违规截图 + 处罚信息 + 疑似原因标签
- 不需要填播放量/导粉数（因为被限流了没数据）

### 不做的

- ~~合规预检独立入口~~（预留口子，以后在脚本创作卡点做）
- ~~日报里单独贴违规上报模块~~（会把事情搞复杂）

## 数据模型

### 1. violation_cases 表 — 新增字段

表名不改，新增字段全部带安全默认值，不破坏现有 API。

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| purpose | text NOT NULL | `'violation'` | 方向：violation / conversion |
| script_format | text NOT NULL | `'oral'` | 形式：oral / visual / mixed |
| ai_analysis | jsonb | NULL | AI 分析结果（含 schema_version / status / model / analyzed_at） |
| total_views | bigint NOT NULL | 0 | 聚合：总播放量 |
| total_follows | int NOT NULL | 0 | 聚合：总关注数 |
| usage_count | int NOT NULL | 0 | 聚合：使用次数 |
| weighted_conversion_rate | numeric(10,8) | NULL | total_follows / total_views |
| script_hash | text | NULL | md5(trim(lower(script_text)))，去重提示用 |
| parent_id | uuid | NULL | FK → self，话术变体血缘 |

CHECK 约束：
- `purpose IN ('violation', 'conversion')`
- `script_format IN ('oral', 'visual', 'mixed')`

**旧数据迁移：** 所有旧数据 `purpose = 'violation'`。旧的 is_violation=false 只是违规库里的安全案例，不等于高转化。

## 排行榜逻辑

**上榜门槛：** 最少 3 次使用 + 总播放 ≥ 1000

**排序指标：**
- 主指标：加权转化率（total_follows / total_views）
- 辅助指标：相对提升率 = 话术转化率 / 使用账号的历史平均转化率

相对提升率 > 1 = 这条话术比账号平时表现好。

## AI 能力边界

| AI 能做好的 | AI 做不到的 |
|-------------|-------------|
| 文案风险识别（联系方式/承诺收益/荐股/夸大） | 精准判断抖音为什么限流 |
| 文案结构拆解（痛点/利益点/行动引导） | 预测某条视频一定违规或一定爆 |
| 给出合规替换表达 | 替代管理员复核 |
| 截图 OCR 初筛 | 只靠截图判断完整视频风险 |
| 周报总结 + 四类清单草稿 | 教员工绕过平台审核 |
| 相似话术聚类 | 精准归因单条视频的导粉效果 |

**AI 输出格式固定为：** 风险点 + 证据 + 置信度 + 建议动作。

---

**背景：**

我们在做一个抖音数据日报平台（dydata.cc），现在要在"数据填报"页面里加一个导粉话术采集区域。

**现有页面结构：**

数据填报页面分左右两栏：
- 左栏：截图上传（2张必传截图 + 1张可选）、抖音视频链接、视频标题、话题标签、内容标签、发布时间
- 右栏：指标录入（核心数据：播放量/涨粉数/导粉数、互动数据：点赞/评论/分享/收藏、完播留存：均播时长/跳出率/完播率）

**新增需求：**

在右栏"指标录入"区域下方，新增一个"导粉话术"区域：

- 触发条件：导粉数字段 > 0 时，动画展开；= 0 或未动时，动画收缩隐藏
- 展开后显示：一个文案输入框（多行文本，用于粘贴视频中的导粉文案）
- 文案输入框的顶部与左栏的"抖音视频链接"输入框大致对齐（视觉平衡）
- 文案在导粉数 > 0 时为必填

**设计约束：**

- 展开/收缩要有平滑过渡动画
- 这个区域要有视觉隔离感（和上方的指标录入有区分），但不能太突兀
- 预留可扩展空间：未来可能在文案框下方增加"导粉截图上传""话术标签选择""AI 合规预检结果"等模块，设计时考虑这些未来元素的容纳空间
- 只有约 20% 的员工涉及导粉，这个区域对他们是核心功能，对其他人是隐藏的
- 整体风格与现有页面保持一致（现代、克制、信息密度适中）

**请你给出：**
1. 这个"导粉话术"展开区域的视觉设计方案
2. 展开/收缩的动画交互思路
3. 如何在视觉上与上方指标录入做区分但不割裂
4. 未来扩展时的布局预留策略

**任务：** 为转化中心实现后端数据层

**技术栈：** Next.js App Router + Supabase (PostgreSQL) + TypeScript

**需要实现：**

### 1. 新 Migration 文件

创建 `supabase/migrations/056_conversion_hub.sql`：

a) violation_cases 表新增字段（全部带默认值）：
- purpose text NOT NULL DEFAULT 'violation' CHECK (purpose IN ('violation', 'conversion'))
- script_format text NOT NULL DEFAULT 'oral' CHECK (script_format IN ('oral', 'visual', 'mixed'))
- ai_analysis jsonb DEFAULT NULL
- total_views bigint NOT NULL DEFAULT 0
- total_follows int NOT NULL DEFAULT 0
- usage_count int NOT NULL DEFAULT 0
- weighted_conversion_rate numeric(10,8) DEFAULT NULL
- script_hash text DEFAULT NULL
- parent_id uuid REFERENCES violation_cases(id) ON DELETE SET NULL

b) 新建 script_usage_records 表（字段见方案）
- conversion_rate 用 GENERATED ALWAYS AS (CASE WHEN views > 0 THEN follows::numeric / views ELSE NULL END) STORED
- CHECK: views >= 0, follows >= 0, follows <= views

c) 新建 violation_reason_tags 表 + violation_case_reason_tags 中间表
- 插入初始标签数据

d) 新建 violation_events 表

e) 新建 script_purposes 配置表 + 初始数据

f) 新建 weekly_decisions 表

g) 增量聚合 trigger：script_usage_records INSERT/UPDATE/DELETE → 更新 violation_cases 的 total_views/total_follows/usage_count/weighted_conversion_rate

h) RLS 策略：
- script_usage_records: SELECT 本人 OR has_violation_permission()；INSERT recorded_by=auth.uid() AND (account_id IS NULL OR owns_account(account_id)) AND 案例未归档
- violation_events: SELECT 全员；INSERT reported_by=auth.uid()；UPDATE/DELETE has_violation_permission()
- violation_reason_tags: SELECT 全员；INSERT/UPDATE/DELETE has_violation_permission()

i) 索引：
- script_usage_records: (case_id, used_at DESC), (recorded_by, created_at DESC), (account_id, used_at DESC)
- violation_events: (account_id, occurred_at DESC), (case_id)

j) 旧数据回填：UPDATE violation_cases SET script_hash = md5(trim(lower(script_text))) WHERE script_hash IS NULL

### 2. API 路由

a) `POST /api/conversion-hub/usage-records` — 创建使用记录
- 输入：case_id, account_id, used_at, views, follows, note, daily_report_id(可选)
- 如果 case_id 为空但有 script_text，先创建话术案例再关联
- 验证：follows <= views, views >= 0

b) `GET /api/conversion-hub/usage-records?case_id=xxx` — 查询使用记录
- 本人看自己的，管理员看全部

c) `POST /api/conversion-hub/events` — 创建违规事件
- 输入：account_id, event_type, occurred_at, platform_notice, screenshot_paths, suspected_reason, case_id(可选)

d) `GET /api/conversion-hub/stats` — 聚合统计
- 返回：话术总数、平均转化率、TOP 话术、本周新增等

e) 修改现有日报提交 API：
- 当 follows > 0 且提交了 script_text 时，自动创建 script_usage_records 记录
- source = 'daily_report'，daily_report_id 关联

### 3. 类型定义

在 `src/lib/conversion-hub/types.ts` 定义所有新类型。

- 所有新 API 都要验证登录状态
