## 已知风险

1. 这次已经完成数据库迁移演练和本地不登录冒烟，但还没有做真实账号登录后的浏览器全流程。
2. `get_leaderboard_rows` 这类函数依赖真实登录态里的 `auth.uid()`，service role 只读 smoke 不能完全代表用户态表现。
3. Supabase Auth 的 Site URL、Redirect URLs 还需要在新加坡项目 Dashboard 中核对，尤其是：
   - `https://dydata.cc`
   - `https://dydata.vercel.app`
   - `/auth/callback?next=/reset-password`
4. 正式切换前必须重新做最终备份和最终导入，不能直接拿这次演练库当最终库使用。
5. 切换后如果有新数据写入新库，再回滚旧库，需要人工补回这段新数据。
6. 新旧 Supabase 项目的 JWT secret 不同，正式切换后老登录态大概率失效，用户需要重新登录。
