# 一刻：个人行动规划台产品与后端方案

## 一、判断结论

“一刻”第一阶段不应做成任务管理器，也不应并入 DYData 现有“日报填报、团队管理、内容批改”链路。它应作为 dydata.cc 的独立分支模块：复用登录体系，独立数据模型，独立入口，独立后端服务。

核心定位是个人行动规划台。它解决的不是“我有哪些任务”，而是“我现在最该做哪一个”。所以 V1 的后端优先级不是项目管理完整性，而是四件事：快速写入、状态判断、自动排序、当前执行连续推进。

本方案只覆盖产品功能、后端、数据库、接口、状态流转、权限和验收。不写前端 UI、视觉、动效和交互稿。Kimi 后续只需要按这里的功能契约接页面。

重要边界：本模块最高目标是减少决策成本，不是替用户做决定。系统可以排序、推荐、提醒、自动整理，但不能在用户无感知的情况下把 `planned` 改成 `doing`。状态变化代表用户判断，必须由用户确认，或发生在用户完成当前任务后的明确连续动作中。

## 二、必要纠偏

第一个纠偏：新增“一刻”不是在首页加一个小按钮，而是新增一个独立产品模块。如果后端复用现有团队日报数据表，会把个人行动台拖成团队任务系统，直接违反“第一阶段只个人使用、不做协作权限”的原则。

第二个纠偏：状态流里的“项目”不能等于“项目状态”。项目表不设置状态字段；状态栏里的“项目”只表示“项目下一步任务的推进卡”。状态来自下一步任务，不来自项目本身。这样既保留“性质排序：任务 > 项目 > 备忘”，又不做项目状态、甘特图、里程碑和进度百分比。

第三个纠偏：快速输入不应该把“前 4 个字”硬落成唯一标题。后端应保留完整原文，标题可以由前端建议或用户后改；数据库层只保证 raw_input、title、note 都可承载，不把短标题规则写死。否则后续搜索、拆分和回看都会变弱。

## 三、V1 产品边界

V1 名称：一刻。

V1 页面名：今日行动台。

V1 模块目标：让用户用一句话记录事项，打开后直接看到 1 个主任务、2 个候选任务、1 个进行中项目建议，处理完一件后能明确接下一件，而不是被甩回列表。

V1 成功标准：新增事项不超过 3 秒；首页 5 秒内知道现在做什么；计划做不压迫，只返回前 10 条和剩余数量；项目可以只写名字先保存，但要进入执行区必须有下一步任务；备忘可以一直保留，也可以转任务或拆任务。

V1 明确不做：完整日历、四象限、番茄钟、习惯追踪、复杂标签、甘特图、团队协作、账号分配、通知对方、AI 自动排程、项目状态管理、领域阶段目标、重要性字段、手动置顶。

## 四、入口与模块归属

公共站点入口：dydata.cc 首页在“登录工作台 / 观看演示站”旁增加“一刻”入口。

登录后入口：`/yike`，进入个人今日行动台。

演示入口：`/demo/yike`，仅用于演示数据，不写正式库。

未登录访问 `/yike`：跳转到登录页，登录后回到 `/yike`。

已登录用户第一次访问：后端自动创建个人 workspace，不要求用户配置。

权限边界：所有登录用户都可使用自己的“一刻”；owner/admin 不默认拥有查看别人“一刻”的权限。第一阶段这是个人工具，不是管理后台。

## 五、核心对象

| 对象 | 后端定义 | 是否直接执行 | V1 边界 |
|---|---|---:|---|
| 领域 | 用户长期方向，只有名称、排序、关联事项 | 否 | 不做目标、进度、验收 |
| 项目 | 有目标说明、验收标准、任务列表、下一步任务的事项集合 | 通过下一步任务执行 | 不设项目状态 |
| 任务 | 单个可执行动作 | 是 | 是排序和流转主角 |
| 备忘 | 低成本记录，可模糊、可保留、可转任务、可拆任务 | 可选 | 不强制清算 |
| 负责人 | 用户自己的跟进对象名片 | 否 | 不绑定账号、不通知、不授权 |
| 当前执行槽 | 1 主任务 + 2 候选任务 + 1 项目推进卡 | 是 | 推荐下一步，但不偷改状态 |

## 六、状态系统

事项状态只保留四个，使用英文枚举落库，中文由前端映射。

| 中文 | 枚举 | 含义 | 可进入对象 |
|---|---|---|---|
| 计划做 | `planned` | 候选待办 | 任务、备忘、项目下一步任务 |
| 正在做 | `doing` | 当前注意力 | 任务、项目下一步任务 |
| 别人做 | `delegated` | 个人跟进清单 | 任务、项目下一步任务 |
| 做完了 | `done` | 已完成 | 任务、备忘、项目下一步任务 |

允许流转：

| 从 | 到 | 说明 |
|---|---|---|
| `planned` | `doing` | 用户明确开始做，或完成当前任务后明确选择继续这一件 |
| `planned` | `delegated` | 移交给别人，进入个人跟进 |
| `planned` | `done` | 小事直接完成 |
| `doing` | `done` | 完成后返回下一步推荐，不无感改其他事项状态 |
| `doing` | `delegated` | 做到一半交给别人 |
| `delegated` | `done` | 验收完成 |
| `delegated` | `planned` | 对方没做完，收回自己计划 |
| `done` | `planned` | 允许撤销完成 |

禁止新增状态：暂停、取消、卡住、归档中、审核中、待确认、已延期。卡住、截止、加急都只做提醒字段，不成为状态。

## 七、复杂度、性质、时间

复杂度固定四级：

| 中文 | 枚举 | 排序权重 |
|---|---|---:|
| 深度 | `deep` | 1 |
| 专注 | `focus` | 2 |
| 小事 | `small` | 3 |
| 随手 | `quick` | 4 |

快速输入不要求选择复杂度。后端默认 `small`，但接口允许后改。

性质固定三类：

| 中文 | 后端来源 | 排序权重 |
|---|---|---:|
| 任务 | `yike_items.item_type = task` 且无项目推进语义 | 1 |
| 项目 | 项目下一步任务的 computed card | 2 |
| 备忘 | `yike_items.item_type = memo` | 3 |

时间只做五个桶：

| 中文 | 枚举 | 说明 |
|---|---|---|
| 今天 | `today` | 默认桶 |
| 明天 | `tomorrow` | 次日自动变今天 |
| 本周 | `this_week` | 当前周内 |
| 本月 | `this_month` | 当前月内 |
| 以后 | `later` | 不压进近期 |

快速输入不要求选择时间。后端默认 `today`，因为默认处理比前置填写更符合效率目标。

截止与加急：

| 字段 | 作用 | 是否参与排序 |
|---|---|---:|
| `due_date` | 截止轻提醒 | 否 |
| `is_urgent` | 加急重提醒 | 否 |

## 八、统一排序规则

所有状态栏内部使用同一排序，不按创建时间优先，不允许手动置顶。

排序顺序：

1. 领域排序：`areas.sort_order ASC`，无领域排最后。
2. 复杂度排序：`deep > focus > small > quick`。
3. 性质排序：`task > project > memo`。
4. 稳定兜底：`created_at ASC, id ASC`。

截止和加急只随数据返回提醒标记，不能改变排序位置。

计划做列表默认只返回前 10 条，同时返回 `hiddenCount`。后端仍支持分页或游标加载，但首页首屏不一次性压出全部计划，也不展示大量历史、归档、项目管理和领域管理内容。

## 九、当前执行区后端规则

当前执行区采用 1 + 2 + 1：

| 槽位 | 数量 | 数据来源 | 后端规则 |
|---|---:|---|---|
| 主任务 | 1 | 已确认的执行槽或 `doing` 最高排序任务 | 完成后返回候选与推荐 |
| 候选任务 | 2 | 已确认候选，或从 `planned` 推荐的待确认项 | 推荐项必须标记 `requiresConfirmation` |
| 进行中项目 | 1 | 有下一步任务的项目推进卡 | 显示项目名称 + 下一步任务 |

执行槽是局部当前工作状态，不是全局手动置顶。它只解决“现在做什么”，不改变状态栏的统一排序规则。

主任务完成后的事务：

1. 把主任务状态改为 `done`，写入 `completed_at`。
2. 如果请求里带了用户明确选择的 `continueWithItemId`，服务端把该事项转为 `doing` 并放入主任务。
3. 如果没有明确连续动作，只返回候选任务和计划做推荐，不改变任何其他事项状态。
4. 候选不足时，从 `planned` 按统一排序返回推荐卡，标记 `requiresConfirmation=true`。
5. 如果没有可推荐事项，候选返回空槽。
6. 返回新的 workbench payload，前端不用跳回列表。

项目推进槽规则：

1. 项目表本身无状态。
2. 项目进入推进槽前必须有 `next_task_id`。
3. 项目推进槽展示 `project.name + nextTask.title`。
4. 完成项目下一步任务后，如果项目还有未完成任务，系统推荐下一条作为新的 `next_task_id`。
5. 如果没有未完成任务，返回 `needsNextTask=true`，提示创建或选择下一步；这不是项目状态，只是执行提醒。
6. 只有具备 `next_task_id` 的项目才能进入项目推进槽；只有名字、没有下一步的项目可以保存，但不能占首页执行位。

## 十、快速输入规则

接口必须支持只提交一句话：

```json
{
  "rawText": "把一刻的数据库方案整理给 kimi"
}
```

后端默认生成：

| 字段 | 默认值 |
|---|---|
| `item_type` | `memo` |
| `status` | `planned` |
| `time_bucket` | `today` |
| `complexity` | `small` |
| `memo_granularity` | `unknown` |
| `title` | 从 rawText 生成可读标题，保留完整 rawText |
| `note` | rawText 原文 |

可选增强字段：`areaId`、`projectId`、`status`、`timeBucket`、`complexity`、`dueDate`、`isUrgent`、`assigneePersonId`。这些都不能成为必填项。

备忘默认进入 `today`，但 read model 必须按性质排序把备忘放在任务和项目之后，并给前端 `visualWeight=light`。首页不能因为大量备忘变成备忘垃圾堆。

防重复：快速输入支持 `clientRequestId`，同一用户同一 `clientRequestId` 只创建一次，防止网络抖动重复提交。

输入限制：`rawText` 最大 2000 字；标题最大 120 字；备注最大 5000 字；所有输入在 API 边界做校验。

## 十一、备忘规则

备忘数量感：

| 中文 | 枚举 | 作用 |
|---|---|---|
| 单件事 | `single` | 可直接转任务 |
| 多件事 | `multiple` | 只返回轻提示 |
| 不确定 | `unknown` | 默认 |

备忘可做三类操作：

| 操作 | 后端行为 |
|---|---|
| 保留 | 不改变类型，不强制清算 |
| 转任务 | `item_type` 从 `memo` 改为 `task` |
| 拆任务 | 创建多条 task，记录 `source_memo_id`，原备忘可保留或归档 |

多件事备忘只返回 `suggestSplit=true`，不弹强制流程、不阻断创建、不从排序中提前。

## 十二、项目规则

项目字段只保留：名称、目标说明、下一步任务、任务列表、验收标准、领域归属、归档时间。

项目创建接口只要求 `name`，允许先保存一个只有名字的项目。项目要进入当前执行区或项目推进卡时，必须有 `nextTaskId` 或同步创建 `nextTaskTitle`；否则只出现在项目抽屉/项目列表的“缺下一步”提醒中。

项目任务本质仍是 `yike_items` 中的 task，通过 `project_id` 归属。

项目没有以下字段：`status`、`progress`、`milestone`、`start_date`、`end_date`、`priority`、`is_current_mainline`。

项目缺下一步的处理：项目表不设状态，只在 read model 中返回 `needsNextTask=true`，用于提醒。

## 十三、领域规则

领域字段只保留：名称、手动排序、归档时间。

领域排序由用户拖动后写入 `sort_order`。领域本身不设目标、不设阶段、不设验收。

领域可关联任务、备忘、项目。关联方式是各对象持有 `area_id`，不是单独复杂标签系统。

领域下一个要推进的任务由后端计算：取该领域下 open item 或项目下一步任务中排序最高的一条。

领域只是方向和排序权重，不承担执行，不显示进度条、阶段目标、完成状态。它可以显示“这个方向下一个要做的任务”，但点击推进的仍然是具体任务或项目下一步。

## 十四、别人做规则

“别人做”是个人跟进清单。

负责人是本地对象，不绑定 `profiles`，不创建账号，不发通知，不做协作权限。

别人做事项可记录：

| 字段 | 说明 |
|---|---|
| `assignee_person_id` | 本地负责人 |
| `delegated_note` | 我要验收什么 |
| `follow_up_bucket` | 什么时候看一眼，仍用今天/明天/本周/本月/以后 |

负责人可以直接在事项里新增；重名不强拦，但同一用户下建议前端提示。

## 十五、数据库设计

表名前缀统一使用 `yike_`，避免和现有日报、批改、SOP、通知表混淆。

### 1. `yike_workspaces`

每个用户一个个人工作区。

| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | uuid | PK |
| `user_id` | uuid | NOT NULL, UNIQUE, references auth.users(id) |
| `name` | text | NOT NULL default `'一刻'` |
| `created_at` | timestamptz | NOT NULL |
| `updated_at` | timestamptz | NOT NULL |

索引：`unique(user_id)`。

### 2. `yike_areas`

| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | uuid | PK |
| `workspace_id` | uuid | FK |
| `user_id` | uuid | NOT NULL |
| `name` | text | NOT NULL |
| `sort_order` | int | NOT NULL default 1000 |
| `archived_at` | timestamptz | nullable |
| `created_at` / `updated_at` | timestamptz | NOT NULL |

索引：`(user_id, sort_order, id) where archived_at is null`。

唯一建议：同一用户下未归档领域名唯一，使用 partial unique index。

### 3. `yike_projects`

| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | uuid | PK |
| `workspace_id` | uuid | FK |
| `user_id` | uuid | NOT NULL |
| `area_id` | uuid | nullable FK |
| `name` | text | NOT NULL |
| `goal_note` | text | nullable |
| `acceptance_criteria` | text | nullable |
| `next_task_id` | uuid | nullable FK to `yike_items` |
| `archived_at` | timestamptz | nullable |
| `created_at` / `updated_at` | timestamptz | NOT NULL |

项目不设 `status`。

索引：

| 索引 | 用途 |
|---|---|
| `(user_id, archived_at, created_at)` | 项目列表 |
| `(user_id, area_id)` where `archived_at is null` | 领域关联 |
| `(next_task_id)` where `next_task_id is not null` | 下一步任务回查 |

`next_task_id` 数据库层允许为空，是为了支持“先创建项目，再在同一事务里创建第一条项目任务，最后回写 next_task_id”的落库顺序。产品层不允许活跃项目长期缺下一步；read model 必须把这类项目返回到 `projectsMissingNextTask`。

### 4. `yike_items`

统一承载任务和备忘。项目本身不进此表，项目任务进此表。

| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | uuid | PK |
| `workspace_id` | uuid | FK |
| `user_id` | uuid | NOT NULL |
| `item_type` | text | `task` / `memo` |
| `status` | text | `planned` / `doing` / `delegated` / `done` |
| `title` | text | NOT NULL |
| `note` | text | nullable |
| `raw_input` | text | nullable |
| `area_id` | uuid | nullable FK |
| `project_id` | uuid | nullable FK |
| `complexity` | text | `quick` / `small` / `focus` / `deep` |
| `time_bucket` | text | `today` / `tomorrow` / `this_week` / `this_month` / `later` |
| `bucket_anchor_date` | date | nullable |
| `due_date` | date | nullable |
| `is_urgent` | boolean | NOT NULL default false |
| `memo_granularity` | text | `single` / `multiple` / `unknown` |
| `assignee_person_id` | uuid | nullable FK |
| `delegated_note` | text | nullable |
| `follow_up_bucket` | text | nullable |
| `source_memo_id` | uuid | nullable FK to `yike_items` |
| `client_request_id` | text | nullable |
| `completed_at` | timestamptz | nullable |
| `archived_at` | timestamptz | nullable |
| `created_at` / `updated_at` | timestamptz | NOT NULL |

Check 约束：

| 约束 | 说明 |
|---|---|
| `item_type in ('task','memo')` | 固定类型 |
| `status in ('planned','doing','delegated','done')` | 固定状态 |
| `complexity in ('quick','small','focus','deep')` | 固定复杂度 |
| `time_bucket in ('today','tomorrow','this_week','this_month','later')` | 固定时间桶 |
| `memo_granularity in ('single','multiple','unknown')` | 固定备忘数量感 |

索引：

| 索引 | 用途 |
|---|---|
| `(user_id, status, archived_at)` | 状态栏 |
| `(user_id, status, area_id, complexity, item_type, created_at, id)` where `archived_at is null` | 状态排序 |
| `(user_id, project_id, status)` where `archived_at is null` | 项目任务列表 |
| `(user_id, area_id)` where `archived_at is null` | 领域关联 |
| `(user_id, due_date)` where `due_date is not null and archived_at is null` | 截止提醒 |
| `(user_id, is_urgent)` where `is_urgent = true and archived_at is null` | 加急提醒 |
| `unique(user_id, client_request_id)` where `client_request_id is not null` | 防重复创建 |

### 5. `yike_people`

| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | uuid | PK |
| `workspace_id` | uuid | FK |
| `user_id` | uuid | NOT NULL |
| `name` | text | NOT NULL |

| `archived_at` | timestamptz | nullable |

不引用 `profiles`，不做账号协作。

### 6. `yike_execution_slots`

用于保证处理完一件自动推进，不把用户甩回列表。

| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | uuid | PK |
| `workspace_id` | uuid | FK |
| `user_id` | uuid | NOT NULL |
| `slot_key` | text | `primary_task` / `candidate_1` / `candidate_2` / `project_focus` |
| `item_id` | uuid | nullable FK |
| `project_id` | uuid | nullable FK |
| `filled_reason` | text | `auto` / `manual` |
| `filled_at` | timestamptz | NOT NULL |
| `created_at` / `updated_at` | timestamptz | NOT NULL |

唯一约束：`unique(workspace_id, slot_key)`。

规则：`primary_task/candidate_1/candidate_2` 必须有 `item_id`；`project_focus` 必须有 `project_id`，并返回项目 `next_task_id`。

### 7. `yike_item_events`

轻量事件表，记录关键流转，便于排查和撤销。

| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | uuid | PK |
| `workspace_id` | uuid | FK |
| `user_id` | uuid | NOT NULL |
| `item_id` | uuid | nullable FK |
| `project_id` | uuid | nullable FK |
| `event_type` | text | NOT NULL |
| `before_json` | jsonb | NOT NULL default `{}` |
| `after_json` | jsonb | NOT NULL default `{}` |
| `created_at` | timestamptz | NOT NULL |

事件类型建议：`created`、`updated`、`status_changed`、`completed`、`memo_converted`、`memo_split`、`slot_auto_filled`、`project_next_task_changed`、`rolled_over`。

### 8. Migration 实现注意

新增 schema 必须通过 `supabase migration new <name>` 创建迁移文件，不手写旧编号，不修改旧 migration。

Postgres 不支持 `add constraint if not exists`。如果后续 migration 需要补约束，用 `do $$ begin if not exists (...) then alter table ... add constraint ...; end if; end $$;`。

所有外键列必须建索引，尤其是 `workspace_id`、`user_id`、`area_id`、`project_id`、`next_task_id`、`assignee_person_id`。

所有时间字段使用 `timestamptz`，日期桶锚点使用 `date`。字符串用 `text` + check 约束，不为枚举单独创建 Postgres enum，便于后续新增但不破坏迁移。

所有 `yike_` 表复用或新增 `updated_at` trigger，保持 `updated_at` 自动更新。

不建议为 V1 建公开 view。若后续为了 read model 建 view，必须使用 `security_invoker = true` 或放在不暴露 schema，避免 view 绕过 RLS。

## 十六、RLS 与权限

所有 `yike_` 表启用 RLS。

每张业务表都冗余 `user_id`，避免 RLS 每行 join workspace，提升性能并降低漏数风险。

基础策略：

| 操作 | 策略 |
|---|---|
| select | `user_id = (select auth.uid())` |
| insert | `user_id = (select auth.uid())` |
| update | `user_id = (select auth.uid())` |
| delete | V1 不开放物理删除，使用归档 |

grant：

| role | 权限 |
|---|---|
| authenticated | select / insert / update |
| service_role | all |
| anon | 无 |

后端如果使用 service role，必须显式加 `user_id = actor.userId` 过滤，不能依赖前端传 userId。

owner/admin 第一阶段不拥有读取他人“一刻”的产品权限。后续如要做组织级任务系统，必须另起 V2 协作方案。

`POST /api/yike/focus/complete`

输入：

```json

  "itemId": "uuid",

}
```

### 7. 替换执行槽

```json
}
```

替换执行槽属于用户明确动作。若 item 仍是 `planned`，服务端可以把它转为 `doing`，并写入 `yike_item_events`，事件来源标记为 `user_confirmed`。

### 8. 备忘转任务

`POST /api/yike/items/:id/convert-to-task`

只改变 `item_type`，不强制改领域、项目、时间。

### 9. 拆分备忘

`POST /api/yike/items/:id/split`

输入：

```json

    { "title": "确认接口契约" }
  ],
  "archiveSourceMemo": false
}
```

创建多条 task，记录 `source_memo_id`。

| 原状态 | 昨日未完成后的处理 |
|---|---|

| `doing` | 留在 `doing`，执行槽可继续 |
| `delegated` | 留在 `delegated` |
| `done` | 不处理 |

## 十九、并发与事务

执行槽相关操作必须避免两个请求同时写入造成重复或错位。

推荐做法：

1. `complete focus`、`replace slot`、`confirm recommendation` 走服务端事务。
2. 事务内锁定当前 workspace 对应的 `yike_execution_slots` 行。
3. 若用 SQL RPC，函数内部必须校验 `auth.uid()` 或由服务端 service role 传入 actor user id 后过滤。
4. 不把排序和推荐逻辑散落在前端。

V1 可以先在服务层串行完成；如果出现并发冲突，返回 409，前端重新拉取 workbench。

## 二十、读模型排序实现

后端 read model 需要返回统一结构，前端不自己算排序。

排序映射：

```ts
const complexityRank = {
  deep: 1,
  focus: 2,
  small: 3,
  quick: 4,
};

const natureRank = {
  task: 1,
  project: 2,
  memo: 3,
};
```

项目推进卡的 nature 是 `project`，但其状态、复杂度、时间桶来自 `nextTask`。

无领域事项的 `areaRank` 设为 999999。

加急和截止只返回：

```json
{
  "reminder": {
    "isUrgent": true,
    "dueState": "due_today"
  }
}
```

不能参与排序函数。

## 二十一、安全与数据保护

输入全部在 API 边界校验，不信任前端。

禁止客户端传 `userId` 决定归属；后端从 Supabase session 取当前用户。

所有用户输入按文本保存，不允许作为 HTML 渲染源；如后续前端需要 Markdown，也必须白名单渲染。

service role key 只允许服务端使用，不能出现在 `NEXT_PUBLIC_`。

物理删除 V1 不开放，归档即可。事件表保留必要变更轨迹。

不记录敏感个人隐私。负责人只是用户自己输入的名称，不绑定真实账号体系。

## 二十三、给 Kimi 的前端对接边界

Kimi 只需要消费这些后端能力：

1. `GET /api/yike/workbench` 渲染今日行动台数据。
2. `POST /api/yike/items/quick` 完成一句话新增。
3. `POST /api/yike/focus/complete` 完成主任务并拿新执行区。
4. `POST /api/yike/focus/replace` 替换当前执行槽。
5. `POST /api/yike/items/:id/transition` 做状态流转。
6. 领域、项目、负责人、备忘转换接口按需接入。

Kimi 不应在前端重算排序，不应自行决定状态变更，不应把截止/加急混进排序，不应给项目增加状态字段。

## 二十五、最终验收标准

V1 完成后必须满足：

1. 新用户首次打开 `/yike` 自动拥有个人 workspace。
2. 输入一句话能创建备忘，不要求领域、项目、时间、复杂度。
3. 今日行动台 payload 第一层直接返回当前执行区，不让前端从列表里自己拼。
4. 当前执行区始终按 1+2+1 返回，空槽也明确返回。
5. 主任务完成后，服务端返回下一步推荐；只有用户明确继续时，才把推荐项转为正在做。
6. 所有状态栏内部排序一致：领域 > 复杂度 > 性质。
7. 截止和加急不改变排序，只出现在 reminders。
8. 项目表没有状态、进度、里程碑字段。
9. 项目推进卡必须有项目名称和下一步任务。
10. 领域只有名称、排序、关联事项。
11. 备忘可保留、转任务、拆任务，多件事只轻提示。
12. 时间桶自动顺延符合规则。
13. 别人做只作为个人跟进，不触发账号、通知、权限。
14. RLS 保证用户只能访问自己的“一刻”数据。
15. 后端测试覆盖排序、状态、推荐确认、顺延和权限。

## 二十六、暂不确认也可默认的产品决策

默认路由使用 `/yike`，中文产品名显示“一刻”。

默认新事项进入 `planned + memo + today + small`。

默认使用独立 `yike_` 表，不复用现有日报、SOP、通知、违规库表。

默认 owner/admin 也不能看别人的“一刻”。这是个人行动规划台，不是团队任务管理。

默认 V1 不接 AI。后续如果要做“自然语言识别领域/复杂度/时间”，只能做建议，不自动改排序和状态。
