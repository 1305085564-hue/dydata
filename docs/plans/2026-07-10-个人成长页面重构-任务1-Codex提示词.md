现有 /growth 页面靠**凭空编造的假数据**撑场面，且存在**两套打架的六维口径**。本次 100% 重构，只保留业务逻辑（指标计算口径），推倒页面。

## 三、必做任务（逐条落地）

### 任务 A：砍假数据
- 删掉 `src/lib/loaders/growth-page.ts` 中的 `generateVirtualReports`（约 L191-222）及其唯一调用点（约 L482，`needsVirtualData = myAllReports.length < 3` 那段逻辑一并删）。
- 全仓搜索确认没有其它地方还依赖虚拟报告；有则一并清。
- 结果：数据少就返回真实的那几条，绝不补造。

### 任务 B：统一六维口径（写死一处）
- **决策已定**：六维统一采用「能力维度」口径 A —— `开头留人 / 中段跳出 / 整体完播 / 增长转化 / 互动吸引 / 话题爆点`（现在 `src/lib/growth-page.ts` 的 `dimensionMeta` L87-94、`getDimensionValues` L134-169 就是这套）。
- **废掉口径 B**：前端 `diagnosis-card.tsx` 里那套「播放量/涨粉/点赞/完播率/5s/2s」维度 + 自算逻辑（`avgDim`/`WEAK_THRESHOLD` 等）**不再由前端计算**。你要把「最弱项识别 + 规则版诊断句 + 规则版改法 + 该学谁」全部收口到后端逻辑层算好，通过数据契约输出。前端只渲染，不算第二套。
- 把口径 B 里有价值的规则文案（`WEAK_ADVICE`/`STRONG_ADVICE` 那套「怎么改」的句子）迁移/合并进后端的规则改法生成逻辑，按能力维度重新对应。
- 口径、阈值、评级规则**只准写死在一处**（后端逻辑层），注释标明这是唯一口径来源。

### 任务 C：可信度分级（不设硬门槛）
- 按当前用户**真实样本条数**返回三档标签：`< 3 → "样本不足，仅供参考"`、`3-10 → "样本累积中"`、`>= 10 → "样本充足"`。
- 分级只影响标签文案，不挡人看数据。契约里用枚举字段返回（见契约）。

### 任务 D：规则版体检结论（本页核心）
后端算出并通过契约返回：
- **最弱项**：六维里 ratio 最低（对比团队均值）的那一项。
- **一句话诊断**：模板句，例「你的 5 秒完播 28%，团队均值 41%，前 3 秒没留住人」。用真实数字填充。
- **一句话改法**：按最弱维度预设的「一句可照做的动作」，例「下一条开头 3 秒先抛冲突或结果，别先讲背景」。
- **分析来源标记**：本棒全部是 `"rule"`（规则版）；契约预留 `source: "rule" | "ai"` 字段给 V2 用。

### 任务 E：同事对标（实名，规则版「该学谁」）
- 找出最弱维度上团队里该项**最高**的真实同事，返回其**真实姓名 + 该维度指标 + 文案片段**（对标块需要的字段）。
- 隐私：走服务端取数，**只返回对标块要展示的字段**（姓名 / 该维度指标 / 文案片段），不顺带带出同事其它隐私字段。
- 降级：团队里找不到可对标的人 → 返回团队均值兜底标记（契约里用状态字段表达），不报错。

### 任务 F：脚本/文案降级
- 脚本表（`content_item` / `script_document` / `script_segment`）缺数据或 schema 缺失时 → 契约里 `scriptBreakdown` 返回明确的「无脚本」状态标记，前端据此整块隐藏。已有 `isMissingContentScriptSchemaError` 降级逻辑，沿用。

### 任务 G：死代码清理（前端以外的部分）
- **可直接删的孤儿组件文件**（全仓零引用，已确认）：`benchmark-card.tsx`、`sample-library.tsx`、`capability-grid.tsx`、`advice-panel.tsx`、`ai-advice.tsx`、`growth-insight-panel.tsx`、`weakness-benchmark-grid.tsx`（均在 `src/components/growth/`）。
- **疑似废弃接口**：`src/app/api/growth-advice/route.ts` 只被死组件 `ai-advice.tsx` 调用。删组件后此接口若全仓再无引用，一并删；删前 grep 确认无其它引用。
- ⚠️ **正在使用、本棒别动 UI**：`diagnosis-card.tsx` 的口径 B 逻辑要废，但**文件本身的删除/重写属于前端棒（Antigravity）**——你只需保证「诊断数据由后端契约提供」，不要去改这些正在渲染的前端组件的 UI。

## 四、核心交付物：数据契约文档（给下一棒前端用）

**这是本棒最重要的产物。** 下一棒 Antigravity 从零写前端，只看这份契约 + 产品设计稿，不看你的代码。契约越精确，前端越不返工。

请单独产出一份 `docs/plans/2026-07-10-个人成长页面重构-数据契约.md`，内容包含：

1. **加载器入口**：函数名、参数、返回类型名。
2. **完整返回结构**：每个字段的「字段名 + 类型 + 含义 + 可能的取值/枚举 + 空时返回什么」。至少覆盖下面这些块（字段名你按实际实现定，但含义要齐）：

```
{
  identity:      { profileName, accountCount, reportCount }        // 身份与规模
  credibility:   { level: "low" | "mid" | "high", label: string,   // 全局可信度
                   sampleCount: number }
  verdict:       {                                                  // P0 体检结论卡（首屏焦点）
    weakestDimension: string,      // 最弱维度名（六维之一）
    diagnosis: string,             // 一句话诊断（已填真实数字）
    prescription: string,          // 一句话改法（可照做的动作）
    source: "rule" | "ai",         // 本棒恒为 "rule"
    metric: { self: number, teamAvg: number, unit: string }  // 诊断依据的数字
  }
  radar:         [ { dimension, self, teamAvg, rating: "strong"|"mid"|"weak" } ]  // P1 六维雷达（统一口径）
  metricsOverview: [ { label, value, trend, unit } ]               // P1 核心指标概览
  benchmark:     {                                                 // P2 该学谁（实名对标）
    state: "ok" | "fallback_team_avg" | "none",
    peer?: { name, dimensionValue, scriptSnippet },
    teamAvg?: number
  }
  scriptBreakdown: {                                               // P2 文案拆解
    state: "ok" | "none",
    segments?: [ { type, order, content } ]
  }
  trend:         [ ... ]                                           // P2 成长趋势（真实历史）
  emptyState:    { isEmpty: boolean, reason?: string }             // 0 数据时前端走引导页
}
```

3. **七种状态各返回什么**（前端要按这些分支渲染，务必写清每种状态下上面哪些字段是什么值）：
   - 空（0 数据）：`emptyState.isEmpty = true`，其余块给空/null。
   - 样本不足（1-2 条）：真实数据照返 + `credibility.level = "low"`。
   - 样本累积中（3-10）/ 样本充足（≥10）：对应 credibility。
   - 无脚本：`scriptBreakdown.state = "none"`。
   - 对标找不到人：`benchmark.state = "fallback_team_avg"` 或 `"none"`。
   - （AI 相关状态是 V2，本契约预留 `source` 字段即可，不实现。）
4. **口径说明**：六维每一维怎么算、阈值多少、评级规则（strong/mid/weak 的 ratio 边界）、可信度三档边界——把唯一口径写清楚，注明代码位置。
5. **隐私边界**：对标块返回了哪些同事字段、为什么只返回这些。

## 五、一次性糙前端（只为验证链路，用完即弃）

- 目的：把「后端 → 加载器 → 页面」整条链路真的渲染出来，眼见为实确认七种状态都有正确数据，不是纸面通。
- 做法：用最朴素的 HTML/文字把契约里每个字段直接打印出来即可（`<pre>{JSON.stringify(data)}</pre>` 级别都行），**不要做任何审美、配色、布局设计**。
- **必须在页面顶部和代码注释里显眼标注**：
  ```
  ⚠️ 一次性验证前端，Antigravity 重写时整块删除，禁止参考此处任何结构/样式/命名。
  ```
- 为什么这么严格：下一棒前端若参考你的糙前端，审美和结构会被带偏、越改越乱（阿禅已实测）。所以糙前端只证明数据通，不承担任何设计价值。
- 验证要覆盖：正常样本充足、样本不足(1-2条)、0 数据、无脚本、对标找不到人——这几种状态分别构造/找到真实用户跑一遍，确认契约返回符合预期。
