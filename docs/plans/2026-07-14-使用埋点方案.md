# 2026-07-14 使用埋点方案

## 目标

用最轻量的方式，在两周后回答这三个问题：

1. 哪些登录后页面真的有人打开。
2. 哪些核心功能真的有人完成过。
3. 哪些页面 / 功能几乎没人用，可以进入砍功能候选。

本次方案故意收得很窄：

- 不接第三方 SDK。
- 不做可视化后台。
- 只统计登录后业务页。
- `page_view` 只记页面级路径，不记 query。
- 动态详情页把 UUID 统一折叠成 `[id]`，避免 `/violations/每个案例` 被拆碎。

## 数据结构

表：`public.usage_events`

字段只保留 4 个业务字段：

| 字段 | 说明 |
| --- | --- |
| `user_id` | 谁触发的 |
| `path` | 归属页面路径 |
| `event_type` | `page_view` 或关键动作名 |
| `created_at` | 发生时间 |

补充约束：

- `user_id` 走登录态，未登录页面不计入这套“用户使用”口径。
- `path` 必须是站内路径。
- `event_type` 只允许白名单里的 9 个值。

## 埋点范围

### 页面打开

只统计这些登录后页面族：

- `/dashboard`
- `/growth`
- `/violations`
- `/video-review`
- `/content-tools`
- `/admin`

说明：

- 公开页如 `/login`、`/register` 不纳入本轮，因为没有稳定登录用户口径。
- `/violations/{uuid}` 会归一成 `/violations/[id]`。

### 核心动作

本次只挑 8 个“完成一次真实工作流”才会上报的动作：

| event_type | 归属页面 path | 代表什么 |
| --- | --- | --- |
| `submit_daily_report` | `/dashboard` | 提交日报 |
| `apply_exemption` | `/video-review/exemption` | 提交豁免申请 |
| `submit_work_submission` | `/video-review/submit` | 提交作品凭证 |
| `submit_review_draft` | `/video-review/submit` | 发起批改 / 提交审核稿 |
| `submit_violation_case` | `/violations/submit` | 提交违规 / 案例记录 |
| `review_violation_case` | `/violations/[id]` | 管理员完成审核 |
| `rewrite_generate` | `/content-tools/rewrite` | 真正发起一次文案改写生成 |
| `mark_fulfillment_status` | `/admin/fulfillment` | 履约标记（单条或批量） |

## 实现口径

### 前端

- 在登录后布局层监听路由切换，异步上报 `page_view`。
- 关键动作只放在“成功返回”分支后上报。
- 上报使用浏览器原生 `sendBeacon`，失败再退到 `fetch + keepalive`。
- 整个上报链路不等待返回，不阻塞用户操作。

### 后端

- 新增极简接口：`POST /api/usage-events`
- 只做三件事：读请求、验白名单、写 `usage_events`
- 任一步失败都返回 `202`，前端不依赖结果

## 两周后直接可查的 SQL 模板

### 1. 页面 PV 排名

```sql
select
  path,
  count(*) as pv,
  count(distinct user_id) as uv,
  round(count(*)::numeric / nullif(count(distinct user_id), 0), 2) as pv_per_user
from public.usage_events
where event_type = 'page_view'
  and created_at >= now() - interval '14 days'
group by path
order by pv desc, uv desc;
```

### 2. 核心功能调用排名

```sql
select
  event_type,
  path,
  count(*) as calls,
  count(distinct user_id) as users,
  round(count(*)::numeric / nullif(count(distinct user_id), 0), 2) as calls_per_user
from public.usage_events
where event_type <> 'page_view'
  and created_at >= now() - interval '14 days'
group by event_type, path
order by calls desc, users desc;
```

### 3. 哪些功能零调用

```sql
with expected(event_type, path, feature_name) as (
  values
    ('submit_daily_report', '/dashboard', '提交日报'),
    ('apply_exemption', '/video-review/exemption', '提交豁免申请'),
    ('submit_work_submission', '/video-review/submit', '提交作品凭证'),
    ('submit_review_draft', '/video-review/submit', '发起批改'),
    ('submit_violation_case', '/violations/submit', '提交违规/案例'),
    ('review_violation_case', '/violations/[id]', '审核违规案例'),
    ('rewrite_generate', '/content-tools/rewrite', '文案改写生成'),
    ('mark_fulfillment_status', '/admin/fulfillment', '履约标记')
)
select
  expected.feature_name,
  expected.path,
  expected.event_type,
  count(ue.*) as calls_14d,
  count(distinct ue.user_id) as users_14d
from expected
left join public.usage_events ue
  on ue.event_type = expected.event_type
 and ue.path = expected.path
 and ue.created_at >= now() - interval '14 days'
group by expected.feature_name, expected.path, expected.event_type
order by calls_14d asc, users_14d asc, expected.feature_name asc;
```

### 4. “多久用一次” 的粗口径

这个口径不追求复杂，只看两周内“每个用户用了几天”：

```sql
with user_day as (
  select
    event_type,
    path,
    user_id,
    date_trunc('day', created_at) as active_day
  from public.usage_events
  where created_at >= now() - interval '14 days'
  group by event_type, path, user_id, date_trunc('day', created_at)
)
select
  event_type,
  path,
  count(distinct user_id) as users_14d,
  count(*) as active_user_days_14d,
  round(count(*)::numeric / nullif(count(distinct user_id), 0), 2) as avg_active_days_per_user_14d
from user_day
group by event_type, path
order by avg_active_days_per_user_14d desc, users_14d desc;
```

## 已知边界

1. 这套方案优先服务“砍功能判断”，不是精细增长分析。
2. 不统计公开页，也不区分未登录访客。
3. 不做停留时长；两周后先靠“打开频次 + 完成动作频次”判断够不够用。
4. 某些功能存在多个入口时，本次统一归到同一个 `event_type`，先判断“有没有人在用”，不细拆入口差异。

### 管理员两周后怎么查

当前 `usage_events` 的 RLS 只允许用户查自己的记录，这对埋点写入是安全的；但做“全站统计”时，最稳妥做法不是改 RLS，而是：

- 直接去 Supabase 后台的 SQL Editor；
- 用项目管理员权限执行本文件上面的 SQL 模板。

这样有三个好处：

1. 不需要额外放开前台或业务角色的读权限。
2. 不会把“全站埋点明细”暴露给普通登录用户。
3. 两周后临时查一次数据即可，不值得为此新增长期权限面。

所以本次不改 RLS。
