> 日期：2026-07-23
> 范围：选题库（`/topics/*` 页面 + `/api/topics/*` 接口）第二轮体验完善，共 9 项改动的**后端部分**。

## 零、总原则

1. **不重写既有能力。** 下面「一」列出的项，后端已经全部就绪，Codex 只需确认现状、不要改动。改动集中在「二」的 4 个接口。
2. **权限模型沿用 `DataAccessScope`。** 所有涉及"看别人数据"的聚合/动态接口，必须走服务端 admin client（`service_role`）+ `scope`（`scope.kind === "all"` 或 `visibleUserIds` 过滤），绝不能让浏览器直连带 RLS 的表。这是本轮后端最核心的一条。
3. **三条铁律不变**（轻录入 / 推荐带理由 / 绝不让人额外维护）。新接口不得引入需要人工填写才能用的字段。
4. **错误返回统一** `{ "error": "中文说明" }`，状态码沿用既有约定（401 未登录 / 403 越权 / 404 不存在 / 409 冲突 / 400 参数错）。

## 二、需要后端新增/修改的项（本轮后端工作量集中在这 4 项）

### item 5 · 今日工作台向"有数据的选题"倾斜

**痛点**：`loadActiveTopics`（`service.ts:616`）返回的三列（recentlyClaimed / recentlyWorked / recentlyCreated）全部只按时间倒序，冷启动或数据积累后都无法突出"这个选题已经跑出成绩、值得再做一条"。

**改动**：在 `loadActiveTopics` 返回值里**新增一列** `worthRedoing`（值得再做）。

- 取所有 `videos.topic_id` 非空、`lifecycle_state='active'`、按 `scope` 过滤后的作品，按子题聚合。
- 复用既有 `calculateTopicWorkSummary`（play≥1000 为达标）与 `loadTopicSummaries` 逻辑，只保留 `qualifiedWorkCount >= 1` 的子题。
- 按 `averagePlayCount` 倒序，取前 `limit` 条。
- 每条返回：子题基本信息（id/title/topic 母题名/group）+ 内嵌 `summary`（`qualifiedWorkCount / averagePlayCount / bestCopy`）。
- 返回结构追加：`{ recentlyClaimed, recentlyWorked, recentlyCreated, worthRedoing }`（前三列保持不变，只新增第四列，前端向后兼容）。

**接口**：沿用 `GET /api/topics/active`，无需新增路由，仅扩展响应体。

---

### item 7 · 认领动态 / 防撞车（本轮后端第一优先级）

**痛点（这是个真 bug）**：详情页 `fetchAllClaims`（`topics/[id]/page.tsx:196-218`）用浏览器 `createClient()` 直查 `sub_topic_claims`，命中 RLS，普通成员根本读不到别人的认领记录 → `claimsLoadFailed=true`，"谁在做这个选题"对非管理员**永远显示失败**。防撞车能力对最需要它的普通成员是坏的。

**根因**：撞车信息本质是"看别人的动态"，必须服务端 admin client + scope，不能浏览器直连带 RLS 的表。

**改动**：新增服务端接口，绕过 RLS、按 scope 返回。

```
GET /api/topics/sub-topics/[id]/claims
```

- 鉴权：`requireTopicsContext`（登录必需），拿到当前用户 `scope`。
- 用 **admin client（service_role）** 查 `sub_topic_claims`，`sub_topic_id = :id` 且 `status != 'returned'`。
- 按 `scope` 过滤：`scope.kind === "all"` 返回全部；否则只返回 `user_id ∈ visibleUserIds` 的认领。
- 关联 `profiles`（或既有取名字的方式）返回认领人显示名。
- 返回：
  ```jsonc
  {
    "claims": [
      { "userId": "...", "displayName": "小王", "status": "candidate", "claimedAt": "..." },
      { "userId": "...", "displayName": "小李", "status": "scripting", "claimedAt": "..." }
    ],
    "candidateCount": 3,
    "scriptingCount": 1
  }
  ```
- 排序：`scripting` 在前（更接近发布、撞车风险更高），再按 `claimed_at` 倒序。

**注意**：`scope` 外的人认领了同一子题时，出于隐私不返回其身份，但**计数仍应包含**（`candidateCount/scriptingCount` 为全量口径），这样"已有 3 人在做"的警示不会因为可见范围而漏报。若隐私要求更严，可与阿禅确认是否连计数也按 scope 收窄——**默认按全量计数、只对身份脱敏**。

### item 8 · "哪类选题在哪个账号最能打"横向对比

**痛点**：现在没有任何接口能回答"美妆母题在 A 号 vs B 号谁的完播更好""哪个母题类型整体胜率最高"。这是阿禅明确的新建重点。

**改动**：新增聚合接口。

```
GET /api/topics/comparison?dimension=topic|account&days=30&topicId=<可选>
```

- 鉴权：`requireTopicsContext` + `scope`；**admin client + scope 过滤**（要跨人看数据）。
- 数据源：`videos`（`lifecycle_state='active'`、`topic_id` 非空、`uploaded_at` 在 `days` 窗口内）join `video_metrics_snapshots(play_count)` 与 `sub_topics(topic_id → topics.name)`、`accounts(name)`。
- 聚合维度：
  - `dimension=topic`：按**母题**（`topics.id/name`）聚合，输出每个母题的作品数、达标数（play≥1000）、达标率、平均播放、最高播放。
  - `dimension=account`：按**账号 × 母题**交叉聚合（传 `topicId` 则锁定单母题看各账号；不传则给一张母题×账号矩阵）。
- 返回（`dimension=topic` 示例）：
  ```jsonc
  {
    "dimension": "topic",
    "windowDays": 30,
    "rows": [
      { "topicId": "...", "topicName": "美妆", "workCount": 24, "qualifiedCount": 9,
        "qualifiedRate": 0.375, "avgPlayCount": 18200, "bestPlayCount": 96000 }
    ],
    "sampleTotal": 120
  }
  ```
- 样本可信度：某维度 `workCount < 3` 时，返回里带 `lowConfidence: true`，前端灰示"样本少仅供参考"（复用既有阈值心智，别让人误当结论）。
- 排序：`rows` 默认按 `qualifiedRate` 倒序，同率按 `avgPlayCount` 倒序。

### item 9 · AI 选题建议并入选题库（"系统推荐"）

**痛点**：`content-tools` 里的 `topic_suggest`（`route.ts:314`）已经能基于爆款样本产出结构化建议（标题/母题/切入角度/参考视频/预期表现/依据），但它是个**孤儿只读 tab**——建议看完即弃，无法一键存进选题库变成可认领的子题。

**改动分两步**：

**(a) 复用生成逻辑，暴露给选题库调用**
既有 `handleTopicSuggest` 逻辑不动，新增一个选题库侧入口（二选一，前端只认一个）：
- 方案 A（推荐）：新增 `GET /api/topics/recommendations?days=14&accountId=<可选>`，内部复用 `handleTopicSuggest` 的样本→prompt→`parseTopicSuggestions` 链路，返回 `TopicSuggestionItem[]` + `evidenceSummary`。这样选题库不必依赖 content-tools 的 POST 语义。
- 返回沿用既有 `TopicSuggestionItem` 结构（`content-tools/types.ts:19`），前端直接复用类型。

**(b) 新增"采纳建议存库"接口**
```
POST /api/topics/sub-topics/from-recommendation
body: {
  title: string,          // AI 建议标题
  angle: string,          // 切入角度 → 存入 hook
  category?: string,      // AI 给的母题名，用于匹配 topics.id
  emotion_tag?: string,   // 可选
  audience?: string       // 可选
}
```
- 鉴权：登录必需，`created_by = 当前用户`。
- `category` 按名称匹配 `topics`：命中则填 `topic_id`；不命中则挂到"常规母题"或留空（与手动录入未选母题时行为一致）。
- **关键**：写入时 `source = "ai_recommendation"`（区别于手动录入的 `"manual"`），便于后续统计"AI 来源子题的转化"。
- 复用 `createSubTopic` 的校验与写库，不要另起一套。
- 返回新建子题（含 id），前端据此跳转/刷新。

## 三、回滚

本轮后端全部是**新增接口 + 扩展响应字段**，不改表结构、不删既有接口：

- item 5：`loadActiveTopics` 只新增 `worthRedoing` 字段 → 回滚即删除该字段计算与返回。
- item 7/8/9(a)：新增独立路由文件 → 回滚即删除对应 `route.ts`。
- item 9(b)：新增 `from-recommendation` 路由 + `source='ai_recommendation'` 取值 → 回滚删路由；已写入的子题不受影响（`source` 是普通文本列）。

无 migration，无需数据库回滚。若前端未同步上线，新增接口不被调用，旧行为完全不受影响（向后兼容）。

## 四、验收要点（交付前自查）

1. item 7 用**普通成员账号**（非管理员）打开一个别人认领过的子题，`/claims` 能正常返回、不再 `claimsLoadFailed`。
2. item 8 传 `dimension=account&topicId=<某母题>`，返回各账号在该母题下的达标率，样本<3 的行带 `lowConfidence`。
3. item 9(b) 采纳一条 AI 建议后，去选题池能搜到这条子题，且其 `source='ai_recommendation'`。
4. item 5 `/active` 返回体含 `worthRedoing`，且每条带 `summary.qualifiedWorkCount>=1`。
5. 所有新接口未登录返回 `401`，跨可见范围的数据不泄漏。
