## 三、健康度报告（按严重度排序）

### 🔴 严重（影响一致性 / 安全 / 业务）

**S1. audit 表分裂：业务全部走旧表 `audit_logs`，新表 `audit_log` 完全没人用**
- 旧表 `audit_logs`（005）：被 4 处业务代码读写
  - 读：`src/lib/loaders/admin-page.ts:304`
  - 写：`src/app/(app)/admin/actions.ts:49`、`src/app/api/smart-alert/notify/route.ts:107/168`
  - RLS：005 已加 `is_admin()` select/insert policy，054 让 is_admin 认 owner，链路通
- 新表 `audit_log`（032）：建好、加了 RLS、加了 4 条 policy，但没有任何业务代码 INSERT/SELECT
- 风险：新表是设计冗余，不是审计空壳；旧表跑得正常
- 终选：保旧废新（067 落地）

**S2. 020-022 内容填报重构半成品**
- `field_provenance / provisional_tag_text` 业务零调用
- `content_asset` 业务零调用（但 CLAUDE.md / AGENTS.md 仍把它当"截图存储位"引用）
- `content_item / metric_snapshot` 仅 AI insight 路径用，主流仍走 `daily_reports / videos`
- `submission_batch` 业务零 `.from()` 调用，**但被 021_metric_snapshot.sql 多条 RLS policy 通过 EXISTS 子查询引用**
- 风险：项目 CLAUDE.md 把 content_asset 的 OCR 字段当事实记，但实际截图存储在 video_metrics_snapshots
- 终选：维持现状 + 加 `【实验性】` comment + 修文档误导（067 落地 + 文档侧另行修订）

**S3. profiles.role CHECK 约束遗忘了 `team_admin`/`group_leader`**
- 当前 DB 约束：`role IN ('member','admin','owner')`
- 业务代码 `BusinessRole` 表达 4 档：`owner / team_admin / group_leader / member`
- 实际机制：DB role 仍是 3 档，业务用 `manage_members 权限位` + `groups.leader_user_id` 在代码层组合出 4 档
- 风险：不算 bug，但"业务概念 vs DB 概念"不对齐，新人和 Codex 都容易踩
- 决策需要：是把 4 档下沉到 DB（动 CHECK 约束），还是只在文档明确"DB 永远 3 档，4 档在代码层组合"？

### 🟡 中等（运维 / 性能 / 可维护性）

**M1. 053 是 029 的生产环境补丁，旧 migration 还在仓库**
- 029_profiles_team_and_group.sql 在生产没执行过，053 是 idempotent 重写
- 现状：两个 migration 都在仓库，新装库会执行两次（有 IF NOT EXISTS 兜底，不报错但混乱）
- 决策需要：是删掉 029，还是在文件里加注释说明？

**M2. is_admin 函数定义了 2 次（001 + 054）**
- 001 版只认 `admin`
- 054 修复后认 `admin / owner`
- 现状：都用 `create or replace`，没问题，但有历史包袱
- 处理：合并到一个权威定义，或在 001 里直接改成最终版（破坏 migration 不可变原则）
- 建议：保留现状，在新 migration 里整合

**M3. migration 011 缺号**
- 序列：010 → 012，找不到 011 的痕迹
- 风险：低，只是疑团；可能是历史回退
- 处理：在文档备注，或建一个空 011 占位

**M4. RLS policy 重写 94 次**
- 033 是大重写，50+ 表统一过一遍
- 后续 028/030/044/050/053/057/060 等多次 drop + create
- 风险：单条 policy 看不出全貌，需要按表汇总当前生效版本
- 处理：建一个"权威 policy 视图清单"作为参考文档（不入 migration）

**M5. `audit_logs`（旧表）RLS 状态：已确认健康**
- 005 已启用 RLS，加了 `is_admin()` select/insert policy
- 054 已让 is_admin 认 owner/admin
- 067 补上 service_role bypass 兜底

## 五、调优 migration（保守版定稿）

> 2026-05-18 经 Codex 复核，原 a/b/c 路线全部修订为保守路线。文件落地于
> `supabase/migrations/067_one_shot_audit_tuning.sql`。

### Codex 复核纠正的 2 个事实错误

1. **audit_logs 不只是被读，还有 INSERT**：`src/app/(app)/admin/actions.ts:49` 和
   `src/app/api/smart-alert/notify/route.ts:107/168` 都在写。原草稿要把它改名 →
   后台动作 + 智能提醒的审计写入直接断。
2. **submission_batch 被 RLS policy 引用**：`021_metric_snapshot.sql` 多条 policy 通过
   `EXISTS (SELECT 1 FROM submission_batch ...)` 子查询引用本表。改名 → 021 所有 policy
   断引用。原草稿只看了 `from('submission_batch')` 业务调用，漏了 policy 内部 SQL 引用。

### 5 项最终决策（覆盖第四节）

| # | 决策 | 终选 |
| --- | --- | --- |
| 1 | audit 双表 | **保旧表 audit_logs，废新表 audit_log**：补 service_role bypass + 权威 comment；新表加 deprecated comment，不改名不 DROP |
| 2 | 020-022 内容填报重构 | **维持现状只加注释**：所有相关表加 `【实验性】` comment 标明主流填报走 daily_reports/videos，禁改名禁 DROP |
| 3 | businessRole 4 档下沉 | **不下沉**：DB 永远 3 档（owner/admin/member），4 档由代码层 permissions + groups.leader_user_id 组合 |
| 4 | 029 + 053 旧 migration | **保留并加说明**：053 是 029 的生产兼容补丁，二者并存；不删 029，不改旧 migration |
| 5 | 死表归档 | **本轮只加 comment**：`submission_batch / field_provenance / provisional_tag_text` 等保留观察，半年无调用再做 068 清理 |

### 067 实际落地内容（保守路线）

落地范围：

- `audit_logs`（活）：补 `audit_logs_service_role_bypass` policy + grant + 权威 comment
- `audit_log`（废）：加 deprecated comment，不动结构
- `submission_batch / content_item / metric_snapshot / content_asset / field_provenance / provisional_tag_text`：加`【实验性】`comment，明确主流走 daily_reports/videos
- `is_admin()`：重述权威定义（与 054 等价，固化到本审计 migration）
- `profiles.permissions`：兜底回填 NULL → `{}`（007 的 NOT NULL DEFAULT 已经够，这里只补历史脏数据）
- `daily_reports`：补 `(user_id, report_date desc)` 显式索引
- `public` schema comment：移交 Codex 标记

落地不做的：

- 不 DROP / RENAME 任何表
- 不改 029 旧 migration
- 不动 businessRole CHECK 约束
- content_asset 字段 schema 不动（CLAUDE.md / AGENTS.md 的误导表述在文档侧另行修订，不在 067 里）
