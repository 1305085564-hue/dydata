# DYData 第一二批合并执行分工与验收

## 一、执行边界

本次只做第一批和第二批：入口改名、页面降噪、团队管理重组。目标是先让网站结构清楚，不做视频复盘闭环的深度开发，不做个人成长和案例库的深度强化。

本轮不追求新增复杂功能，优先解决“成员不知道点哪里、组长不知道看哪里、owner 配置入口和团队管理混在一起”的问题。

本轮页面可见名称全部使用中文。内部地址可以暂时保留，不作为本轮重点。

## 二、分工原则

| 执行对象 | 负责内容 | 不负责内容 |
|---|---|---|
| Kimi | 前端页面、导航文案、布局分区、交互状态、空状态、中文化 | 权限判断、数据范围、数据库、接口逻辑 |
| 新 Codex 页面 | 权限、角色可见范围、路由保护、后台数据读取、可能影响运行的逻辑 | 视觉风格大改、页面审美细节 |
| 当前对话 | 规划、分工、验收标准、最终审查 | 直接施工 |

## 四、Kimi 执行范围

Kimi 负责“用户能看到、能感知”的部分。

### 1. 顶部导航中文化

需要改成：
- 数据填报 → 今日工作台
- 成长分析 → 个人成长
- 违规库 → 话术案例库
- AI 助手 → 文案助手
- 后台管理 → 团队管理
- 系统设置仅在 owner 可见时出现

验收要求：
- 页面顶部不出现 Dashboard、Admin、AI Assistant 等英文入口。
- “文案助手”保留独立入口。
- 普通成员不出现“团队管理”和“系统设置”。

### 2. 团队管理侧边栏重组

团队管理内部建议分组：

| 分组 | 包含页面 |
|---|---|
| 日常管理 | 今日待办、视频复盘、内容复盘、视频素材、经营分析 |
| 系统设置 | 成员权限、团队分组、AI 配置、模块开关 |

Kimi 只负责前端显示结构。具体哪些角色能看到，由 Codex 处理。

验收要求：
- 组长看到的是“日常管理”。
- owner 额外能看到“系统设置”。
- 不把权限、AI 配置混在日常管理最上面。

### 3. 团队管理首页改成待办页

团队管理首页第一屏只回答三个问题：
- 谁没交？
- 谁待复盘？
- 谁需要反馈？

页面可以先使用已有数据，不要求本轮新增复杂数据模型。

验收要求：
- 管理者进入团队管理后，10 秒内知道今天该处理谁。
- 不再像功能列表或后台工具箱。
- 空状态要说清楚“今天暂无待处理事项”，不能只显示空白。

### 4. 今日工作台轻量整理

本轮只做轻量整理，不做完整反馈闭环。

建议分区：
- 今日提交
- 待处理提醒
- 昨日反馈
- 下一条动作

验收要求：
- 成员进入后知道今天要提交什么。
- 如果暂无反馈，要有中文空状态。
- 不引入还没有后端支撑的假功能。

## 五、新 Codex 页面执行范围

新 Codex 页面负责“会不会跑错、会不会越权、会不会部署失败”的部分。

### 1. 角色可见范围

需要确认并处理：

| 角色 | 可见入口 |
|---|---|
| 普通成员 | 今日工作台、个人成长、话术案例库、文案助手 |
| 组长 | 今日工作台、个人成长、话术案例库、文案助手、团队管理 |
| 负责人 | 今日工作台、个人成长、话术案例库、文案助手、团队管理 |
| owner | 今日工作台、个人成长、话术案例库、文案助手、团队管理、系统设置 |

验收要求：
- 组长不能看到系统设置。
- 普通成员不能进入团队管理。
- owner 能看到团队管理和系统设置。

### 2. 团队管理与系统设置的权限边界

团队管理是业务管理入口，系统设置是 owner 维护入口。

需要迁移或隐藏：
- 权限模块 → 系统设置
- AI 配置 → 系统设置
- 模块开关 → 系统设置
- 系统规则 → 系统设置

验收要求：
- 组长可以做日常管理，但不能改权限、AI 配置、系统模块。
- owner 可以进入系统设置。
- 未授权访问时要跳回合理页面，不出现报错页。

### 3. 数据范围

需要确保：
- 组长只看本组。
- 负责人看本团队。
- owner 看全局。
- 普通成员只看自己。

验收要求：
- 不因前端改名破坏原有权限规则。
- 不把 owner 数据暴露给组长或成员。
- 不把系统设置入口误开放给组长。

### 4. 构建和测试

新 Codex 页面需要负责：
- 检查页面改名后是否有路由、权限、类型错误。
- 跑构建或至少跑必要检查。
- 检查已有视频复盘台半成品，不在本轮深挖，只保证入口不导致页面崩。

验收要求：
- 本轮改动不破坏登录、今日工作台、文案助手、团队管理。
- 构建通过。
- 没有用户可见英文入口。

## 六、不在本轮做的事

以下内容留到第三批或第四批：

1. 不做完整视频复盘闭环。
2. 不新增复杂训练任务系统。
3. 不重做个人成长的数据模型。
4. 不重做话术案例库的沉淀机制。
5. 不改线上环境变量。
6. 不改旧 migration。
7. 不把内部路由全部改成中文路径。

## 七、给 Kimi 的执行提示词

你负责 DYData 第一二批合并改造里的前端部分，只处理页面可见结构、中文文案、导航层级、布局分区、交互状态。不要改权限逻辑、接口逻辑、数据库或路由保护。

目标：让网站入口从“功能堆叠”变成“成员日常、团队管理、系统设置”三层。所有用户可见入口必须是中文。

具体任务：
1. 顶部导航改名：数据填报→今日工作台，成长分析→个人成长，违规库→话术案例库，AI助手→文案助手，后台管理→团队管理。
2. 保留“文案助手”为独立入口，不降级，不隐藏。
3. 团队管理内部重组：日常管理包含今日待办、视频复盘、内容复盘、视频素材、经营分析；系统设置包含成员权限、团队分组、AI配置、模块开关。
4. 团队管理首页第一屏围绕三个问题组织：谁没交、谁待复盘、谁需要反馈。
5. 今日工作台轻量整理为：今日提交、待处理提醒、昨日反馈、下一条动作。没有真实数据时写清楚中文空状态，不做假功能。
6. 页面上不要出现 Dashboard、Admin、AI Assistant 等英文入口。

验收重点：
- 普通成员只看到今日工作台、个人成长、话术案例库、文案助手。
- 组长/负责人看到团队管理。
- owner 额外看到系统设置。
- 页面看起来像短视频团队训练和管理系统，不像后台功能列表。

## 八、给新 Codex 页面的执行提示词

你负责 DYData 第一二批合并改造里的非前端风险部分，只处理权限、角色可见范围、路由保护、数据范围、构建错误和容易导致运行失败的逻辑。不要做大规模视觉重构。

目标：配合前端改名和团队管理重组，保证页面不越权、不跑错、不部署失败。

具体任务：
1. 核对并修正主入口可见范围：普通成员不能看到团队管理和系统设置；组长/负责人能看到团队管理；owner 能看到团队管理和系统设置。
2. 把权限模块、AI配置、模块开关、系统规则类入口归到系统设置，并限制为 owner 或授权负责人可见。
3. 团队管理保留日常业务入口：今日待办、视频复盘、内容复盘、视频素材、经营分析。
4. 确认组长只看本组，负责人看本团队，owner 看全局，普通成员看自己。
5. 检查视频复盘台半成品，只保证本轮入口不崩，不在本轮补完整闭环。
6. 跑构建或必要检查，修复因改名、移动入口、权限判断造成的错误。

验收重点：
- 权限边界不变形。
- 页面入口按角色正确显示。
- 未授权访问不会报错。
- 构建通过。
- 不改旧 migration，不改线上环境变量。
