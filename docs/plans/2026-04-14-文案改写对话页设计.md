# 文案改写对话页设计

日期：2026-04-14

## 目标

给普通员工一个简单直接的 AI 对话页，核心任务只有一个：改写文案。

这版先满足 4 件事：

- 粘贴原文，直接改写
- 在对话里切换模型视图，如 `Gemini`、`Opus`
- 在对话里按需切换模式，如 `更像爆款`、`更口语`、`更专业`
- 记录会话历史，方便继续追问

这版不做的事：

- 不给员工看真实渠道名和密钥
- 不开放系统提示词编辑给员工
- 不做复杂工作流编排
- 不做多工具混合调用

## 页面定位

建议作为员工端的独立对话页，入口可以先挂在内容工具区，后续再决定是否单独进主导航。

建议路由：

- 页面：`/content-tools/rewrite`
- 接口前缀：`/api/content-tools/rewrite/*`

这样不会和后台 AI 助手混淆，也方便后端权限隔离。

## 前端草图

先按“能用、清楚、字段齐”的思路做，前端视觉可以后续重设计。

```text
+----------------------------------------------------------------------------------+

| 副标题：输入原文，选择模式，一键改成更适合发布的版本                              |

| [自动模式: 开] [模型: Gemini v] [模式: 可不选 v] [字数: 精简 v] [新建会话]        |

+-----------------------------------+----------------------------------------------+

|-----------------------------------|----------------------------------------------|
| 今天                              | 助手气泡                                     |

| - 直播口播优化                    |                                              |
| 昨天                              | 用户气泡                                     |
| - 财经短视频标题                  | “把这段改得更像爆款，保留专业感。”           |
```

## 页面分区

### 1. 顶部控制条

员工真正会用到的只有 5 个控件：

- 自动模式开关
- 模型视图切换：`Gemini` / `Opus`
- 模式切换：如 `更像爆款` / `更口语` / `更专业` / `不选`
- 输出长度：`精简` / `标准` / `展开`
- 新建会话

这里的“模型视图”不是数据库里的真实渠道，而是给用户看的抽象层。

例如：

- `Gemini` 下面实际可能挂 2 个 key
- `Opus` 下面实际可能挂 2 个 key

员工只知道自己在用哪个模型家族，不知道内部 key。

### 2. 左侧历史会话

只保留最简单能力：

- 新建会话
- 查看最近会话
- 点击继续聊
- 自动生成标题

列表项只显示：

- 标题
- 最后更新时间
- 当前模式标签
- 当前模型标签

不显示长摘要，不显示调试信息。

### 3. 右侧对话区

对话区只处理“改写”这一个任务，避免做成通用聊天页。

首屏空状态：

- 一句用途说明
- 2 到 3 个快捷按钮
- 一个大输入框

建议快捷按钮：

- 改得更像爆款
- 改得更口语一点
- 保留专业感但更好懂

AI 输出建议不是只回一大段，而是结构化展示：

- `版本 A`
- `版本 B`
- `版本 C`
- `改写说明`
- `继续追问建议`

这样最适合员工直接复制。

## 与后端对齐的真实需求

前端虽然简陋，但后端一开始就要按真实产品需求建。

### 1. 模型视图层

前端需要拿到的不是 `ai_channels` 原表，而是整理后的“可选模型视图”：

```ts
type RewriteModelOption = {
  key: "gemini" | "opus" | string;
  label: string;
  description: string | null;
  feature_key: string;
  default_channel_group: string | null;
  default_model: string | null;
  is_default: boolean;
};
```

注意：

- 一个模型视图可映射到多个真实渠道
- 同一渠道可有多个 key
- 员工端永远不返回真实 `api_key`、`channel_id`

### 2. 模式层

模式不能写死在前端，应该单独配置。

建议表：`rewrite_modes`

核心字段：

- `id`
- `key`
- `name`
- `description`
- `mode_prompt`
- `sort_order`
- `is_enabled`
- `created_at`
- `updated_at`

前端需要：

```ts
type RewriteModeOption = {
  id: string;
  key: string;
  name: string;
  description: string | null;
  is_default: boolean;
};
```

这里的模式不是必选项。

也就是说：

- 用户可以选一个模式
- 也可以不选
- 不选时，就只按系统提示词 + 用户原文 + 用户补充要求来处理

### 3. 会话层

建议表：`rewrite_conversations`

核心字段：

- `id`
- `user_id`
- `title`
- `auto_mode_enabled`
- `selected_model_key`
- `selected_mode_id`（可为空）
- `system_prompt_snapshot`
- `mode_prompt_snapshot`
- `model_snapshot`
- `channel_snapshot`
- `last_message_at`
- `created_at`
- `updated_at`

这里一定要存快照，不要只存引用。

原因：

- 后面系统提示词改了，老会话不能串味
- 模式提示词改了，历史记录要能对得上
- 实际使用的模型也要能追溯

### 4. 消息层

建议表：`rewrite_messages`

核心字段：

- `id`
- `conversation_id`
- `role`：`user` / `assistant` / `system_note`
- `content`
- `version_count`
- `structured_result`
- `created_at`

其中 `structured_result` 建议直接存 AI 返回的结构化 JSON，比如：

```json
{
  "versions": [
    { "title": "版本A", "content": "..." },
    { "title": "版本B", "content": "..." },
    { "title": "版本C", "content": "..." }
  ],
  "notes": ["更像口播", "保留专业词"],
  "follow_up_suggestions": ["再短一点", "更有情绪", "改成标题党一点"]
}
```

这样前端好渲染，也方便后面做复制按钮和快捷追问。

### 5. 自动改写流程层

这页不只是“选一个模型改写一次”，还要支持“一条龙双阶段改写”。

建议单独加一层：`改写流程`

人话理解：

- 用户看到的是一个开关：`自动模式`
- 后台看到的是一条流程：`第一步谁打框架，第二步谁做润色`

建议不要把这件事写死在前端，也不要把它硬编码到单个接口里。

应该做成可配置：

- 你可以新建一个自动流程
- 你可以指定第一步用哪个真实方案
- 你可以指定第二步用哪个真实方案
- 你以后可以把 `Claude -> Gemini` 改成别的组合

建议表：

- `rewrite_workflows`
- `rewrite_workflow_steps`

最少要表达出这些意思：

- 这是一个什么流程
- 默认开还是关
- 总共几步
- 每一步用哪个展示模型或真实执行方案
- 每一步用什么提示词
- 每一步失败了怎么处理

## 双阶段自动改写

### 目标

用户发一次文案，系统自动做两次处理：

1. 第一步：先把结构和框架改出来
2. 第二步：在第一步结果上继续加情绪、加语感、补冲突感，出终版

你现在提的默认链路可以理解成：

- 第一步：`Claude`
- 第二步：`Gemini`

但这个组合建议做成后台可调，不要写死成永远只能这样。

### 推荐方案

我推荐做成 `自动模式`，并且默认开启。

用户体验是：

- 开启时：一发出去就自动跑两步
- 关闭时：回到普通单步对话

这是最适合员工端的方案，因为：

- 员工不用理解太多技术细节
- 平时直接用默认自动模式就行
- 高级用户真想单独用某个模型时，也可以关闭自动模式

### 页面怎么表现

建议顶部增加一个明显但不复杂的控件：

- `自动模式` 开关

打开时：

- 模型下拉可以隐藏，或者变成只读说明
- 页面显示一句小字：`当前自动链路：强逻辑起稿 -> 强语感润色`

关闭时：

- 恢复单模型选择
- 用户可手动选一个展示模型
- 用户进入普通对话
- 不再自动调用第二步

这里要定死一条规则：

- `自动模式开启` = 严格按后台配置好的双阶段标准流程执行
- `自动模式关闭` = 就是普通对话，不走双阶段自动链路

这样页面不会太乱，也能把两种工作方式区分开。

### 后台怎么理解

后台不要只理解成“模型”，而要理解成两类东西：

#### 1. 单步方案

给关闭自动模式时用。

就是一次调用：

- 选一个展示模型
- 如有模式，就带模式提示词
- 如没选模式，就不带模式提示词
- 找到对应真实执行方案
- 调一次

#### 2. 流程方案

给开启自动模式时用。

就是一串调用：

- 第一步先生成框架稿
- 第二步拿第一步结果继续润色
- 最后返回终版

所以自动模式本质上不是“特殊按钮”，而是“选中一条后台流程”。

## 自动模式的后台设计

### 流程配置

建议让你在后台可以配这些东西：

- 流程名称
- 是否默认启用
- 是否对员工开放
- 第一步叫什么
- 第一步走哪个展示模型
- 第一步提示词
- 第二步叫什么
- 第二步走哪个展示模型
- 第二步提示词
- 第二步是否读取第一步结果
- 第二步失败时，是回退到第一步，还是整次报错

最推荐的默认流程可以是：

- `自动标准改写`
  - 第一步：强逻辑模型，先整理结构和节奏
  - 第二步：强语感模型，强化情绪、口语感、冲突感

### 每一步到底怎么找真实模型

这块仍然按你前面定的三层走：

- 员工看到的展示模型
- 展示模型下面挂很多真实执行方案
- 每个真实执行方案 = `渠道 + 密钥 + 具体模型`

所以流程里不直接绑死一个 key，而是先绑“展示模型”。

流程执行时：

1. 先根据步骤找到对应展示模型
2. 再根据展示模型找到可用的真实执行方案
3. 按顺序去试
4. 成功后进入下一步

这样你以后改 key、改具体模型，不用改流程定义。

## 自动模式的返回结果

这块一定要设计清楚，不然前端和历史记录会乱。

我建议：

- 前端默认展示第二步终版
- 但消息里把第一步结果也存下来
- 需要时可以展开看“框架稿”

也就是说，一次用户提问，最终会产生：

- 用户消息
- 助手终版消息
- 助手终版消息里带步骤信息

建议结构化结果至少存：

```json
{
  "workflow": {
    "mode": "auto",
    "workflowName": "自动标准改写"
  },
  "steps": [
    {
      "step": 1,
      "name": "框架改写",
      "modelLabel": "强逻辑模型",
      "status": "success",
      "output": "..."
    },
    {
      "step": 2,
      "name": "情绪润色",
      "modelLabel": "强语感模型",
      "status": "success",
      "output": "..."
    }
  ],
  "final_versions": [
    { "title": "版本A", "content": "..." },
    { "title": "版本B", "content": "..." },
    { "title": "版本C", "content": "..." }
  ]
}
```

这样好处是：

- 前端能展示“终版”
- 以后你想看第一步中间稿，也有记录
- 第二步失败时，也能明确知道失败在哪

## 失败兜底

这里是最容易出事故的地方，必须先定规则。

我推荐默认规则：

- 第一步失败：整次失败，告诉用户“本次改写失败，请重试”
- 第一步成功、第二步失败：先把第一步结果作为可用稿返回，并明确提示“二次润色失败，当前先展示框架版”

不要因为第二步失败，就把整次结果全吞掉。

因为员工最需要的是“有东西可用”，不是完美。

### 前端提示建议

当第二步失败但第一步成功时，页面显示：

- 一个轻提示
- 文案内容仍可复制

比如：

- `自动润色步骤失败，已先返回第一步改写稿`

这样体验最稳。

## 延迟和成本

双阶段自动模式一定比单步慢，也一定更贵。

所以建议：

- 默认开启自动模式
- 但允许关闭
- 接口返回里带本次运行模式：`auto` / `single`
- 后台保留超时控制

建议第一版做 2 条简单限制：

- 单次输入字数限制
- 每次只跑两步，不要支持无限链式编排

不要一开始做成通用工作流引擎，那会过度设计。

## 推荐结论

我推荐的最终方案是：

- 页面有一个 `自动模式` 开关，默认开
- 自动模式本质上对应后台一条“改写流程”
- 流程先跑第一步框架改写，再跑第二步情绪润色
- 自动模式关闭后，回到普通单模型对话
- 模式不是必选项，可以为空
- 前端默认展示终版
- 第二步失败时回退到第一步结果
- 所有步骤结果都存档，方便追溯和后续优化

## 提示词拼装规则

最终调用 AI 时，建议按下面顺序拼：

1. 功能级系统提示词
2. 模型视图附加提示词（可选）
3. 模式提示词
4. 页面固定约束
5. 用户原文和当前要求
6. 最近几轮对话上下文

页面固定约束建议至少包含：

- 输出 2 到 3 个版本
- 保持中文
- 不要解释过长
- 适合直接复制使用
- 如果原文信息不足，先给改写，再补一句提醒

## 接口草案

### 1. 初始化数据

`GET /api/content-tools/rewrite/bootstrap`

返回：

- 可用模型视图
- 可用模式
- 默认模型
- 默认模式

### 2. 会话列表

`GET /api/content-tools/rewrite/conversations`

返回当前员工自己的最近会话。

### 3. 创建会话

`POST /api/content-tools/rewrite/conversations`

入参：

- `modelKey`
- `modeId`
- `firstMessage`

### 4. 获取消息

`GET /api/content-tools/rewrite/conversations/:id/messages`

### 5. 发送消息

`POST /api/content-tools/rewrite/chat`

入参建议：

```ts
type RewriteChatRequest = {
  conversationId: string;
  message: string;
  modelKey?: string;
  modeId?: string;
  lengthPreset?: "short" | "normal" | "long";
};
```

返回建议：

```ts
type RewriteChatResponse = {
  conversationId: string;
  assistantMessage: {
    id: string;
    content: string;
    structuredResult: {
      versions: Array<{ title: string; content: string }>;
      notes: string[];
      followUpSuggestions: string[];
    };
  };
  usage: {
    modelLabel: string;
    modeName: string;
  };
};
```

## 模型与渠道实际选择规则

建议顺序：

1. 员工选了 `modelKey`
2. 后端把 `modelKey` 映射到渠道组
3. 在该渠道组里按优先级选健康 key
4. 如果首个 key 失败，则组内切下一个
5. 如果整组都失败，再决定是否允许降级到默认组

这里的重点是：

- 员工选的是模型家族，不是渠道 id
- 后端选的才是真实 key
- 故障切换发生在后端，不让前端感知

## 权限规则

员工端只允许：

- 使用
- 查看自己的会话
- 删除自己的会话

员工端不允许：

- 改系统提示词
- 改模式提示词
- 改真实渠道
- 看密钥或底层配置

模式、模型映射、系统提示词统一在管理端维护。

## 第一版范围

为了尽快上线，第一版建议只做这些：

- 纯文本输入
- 纯文本输出
- 3 个改写版本
- 模型切换
- 模式切换
- 历史会话
- 继续追问

先不做：

- 图片识别
- 语音转写
- 团队共享模板
- 收藏版本
- 一键同步飞书
- 批量改写

## 给前端的最小交付清单

前端这版只要按下面字段设计就不会返工：

- 顶部：`models[]`、`modes[]`、`lengthPreset`
- 左侧：`conversations[]`
- 中间：`messages[]`
- 助手消息支持 `structuredResult.versions[]`
- 助手消息支持 `followUpSuggestions[]`

如果前端先只画草图，可直接按上面的线框图出稿。

## 推荐结论

这页不要做成“万能聊天页”，而要做成“文案改写工作台”。

这样好处最大：

- 员工一看就会用
- 后端约束清楚
- 提示词和模式更容易稳定
- Gemini / Opus 切换能成立
- 多 key 容灾能藏在后端

下一步建议：

1. 先把这页的数据库表和接口 contract 定下来
2. 再让 Gemini 按这份文档出视觉稿
3. 最后按视觉稿补页面实现

## 落地方案

### 先定一件事

是的，建议让 Gemini 按这套方案先做前端。

但不是“随便先写个页面”，而是按已经定好的后端结构来写前端壳子。

正确顺序是：

1. 我先把后端 contract 定死
2. Gemini 按 contract 写前端静态页和 mock 交互
3. 我再写真实后端
4. 最后联调

这样最快，也最不容易返工。

### 角色分工

#### 我负责

- 定数据结构
- 定接口 contract
- 定权限和隔离规则
- 改 AI 路由逻辑
- 写真实后端接口
- 接会话存储和消息存储

#### Gemini 负责

- 把效果图落成真实前端页面
- 先用 mock 数据把列表、聊天区、结果卡、快捷追问做出来
- 等后端接口出来后再改成真实请求

### 实施顺序

#### 第一阶段：先把协议钉死

这一步先不追求功能都能跑，重点是把前后端说同一种话。

我要先定下这些东西：

- 页面路由：员工默认 AI 页
- 会话数据长什么样
- 消息数据长什么样
- 模型视图字段长什么样
- 模式字段长什么样
- 字数预设字段长什么样
- AI 返回结构长什么样

这一步完成后，Gemini 就可以直接开工，不用等真实后端。

#### 第二阶段：Gemini 先做前端壳子

Gemini 这一步建议只做这些：

- 左侧会话列表
- 顶部模型/模式/字数选择
- 中间聊天区
- 结果卡片
- 快捷追问按钮
- 新建会话交互

这一步先用假数据，接口直接 mock。

重点不是接通，而是保证：

- 页面结构和效果图一致
- 字段名和后端 contract 一致
- 以后切真实接口时不用重画页面

#### 第三阶段：我补真实后端

我这边按协议补这些：

- 会话表
- 消息表
- 模式表
- 字数预设表
- 展示模型表
- 展示模型下的真实执行方案表
- bootstrap 接口
- 会话列表接口
- 会话消息接口
- chat 接口

这一步才会真正把“渠道 + 密钥 + 真实模型”串起来。

#### 第四阶段：联调

最后再把 Gemini 做好的前端，从 mock 数据切到真实接口。

联调重点只有 4 件事：

- 列表能不能正常显示
- 发消息能不能生成 3 个版本
- 会话历史能不能继续追问
- 模型切换、模式切换、字数切换能不能真正生效

### 给 Gemini 的工作方式

Gemini 现在就可以开始，但要按这几个原则：

- 不自己发明接口字段
- 不自己决定真实模型结构
- 不把 `Gemini`、`Opus` 写死成真实渠道
- 不把模式和字数预设写死在代码里当最终方案

可以临时 mock，但字段名要按后端协议来。

### 推荐协作方式

最顺的方式是：

1. 我先再补一版“接口字段清单”
2. 你把这份文档和效果图一起发给 Gemini
3. Gemini 开始写前端静态页
4. 我同时开工后端
5. 最后接起来

### 下一步执行建议

现在最应该做的，不是马上写全量代码，而是先把下面 3 份东西定死：

1. 数据表草案
2. 接口 contract
3. 给 Gemini 的 mock 数据格式

这 3 份一旦定了，前端和后端就能并行。
