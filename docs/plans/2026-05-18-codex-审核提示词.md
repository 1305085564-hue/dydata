### 1. 数据 / RLS（高优）

- `supabase/migrations/068_notifications.sql`
  - 表结构、check 约束、唯一索引（user_id, type, source_type, source_id）能否正确去重？source_id 为 null 时是否会被排除（`where source_id is not null`）？

### 2. 服务端 emit（高优）

- `src/lib/notifications/server.ts`
  - `emit` 用 upsert + `onConflict: "user_id,type,source_type,source_id"`：当 source_id 为 null 时 upsert 退化为重复 insert（会被 dedupe 唯一索引的 partial 条件放行），是否会无限堆积？

- `src/lib/notifications/recipients.ts`
  - `inferBusinessRoleSimple` 不查 groups 表是否会漏掉 `group_leader` 判定？是否应该重用 `getUserPermissions` 的全量推断？

### 3. API 路由

- `src/app/api/notifications/route.ts` / `[id]/read` / `[id]/done` / `mark-all-read`
  - 鉴权统一走 `auth.getUser()`，OK
  - mark-done 接收 body.reason；非法值是否被严格收敛（当前只接受 `"ignored"` 否则默认 `"done"`）

## 审核输出格式

请按以下 4 个分组输出：

1. **必须修（阻塞合入）**：列出能引发数据错乱、安全问题、运行时崩溃的项
2. **建议修（合入前最好处理）**：稳健性 / 一致性 / 简化项
3. **可选改进**：风格、命名、可扩展性
4. **不必改**：明确确认 OK 的设计选择，避免阿禅未来反复

每条都要给出文件:行号 + 一句话原因 + 建议改法。
