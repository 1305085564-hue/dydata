# 申请-审批制注册方案

## 背景

当前注册流程采用**邀请码制**：用户必须持有有效的邀请码才能注册成功。之前已取消登录验证码，但注册邀请码未同步处理，此事做了一半。

目标改为**申请-审批制**：
- 用户注册时**不再需要邀请码**，只需选择想加入的团队
- 注册成功后账号处于**待审核**状态
- 团队管理员（admin/owner）在后台**批准**后，用户才能正式使用系统

---

## 核心变更一览

| 模块 | 当前行为 | 变更后行为 |
|------|---------|-----------|
| 注册页 | 必填邀请码 | 去掉邀请码，保留团队选择 |
| 注册后 | 直接可用，跳转 dashboard | 状态为 pending，提示等待审批 |
| 登录后 | 直接进系统 | middleware 拦截 pending 用户，展示等待页 |
| 管理后台 | 有"邀请码管理"模块 | 新增"成员审批"模块，可批准/拒绝 |
| 数据统计 | 所有成员参与统计 | 排除 pending 用户 |

## 数据库改动

### 1. profiles.status 扩展

当前 `status` 只有两个值：`active | exempt`，需扩展为：

```sql
alter table public.profiles
  drop constraint if exists profiles_status_check,
  add constraint profiles_status_check
    check (status in ('active', 'exempt', 'pending'));
```

> **兼容性说明**：现有数据全部保持 `active`，不受影响。所有现有查询中 `status = 'active'` 或 `status != 'exempt'` 的逻辑不需要改（`pending` 自然被排除在 active 之外）。

### 2. 新建 team_join_requests 表（可选，建议先不做）

如果希望保留完整的申请历史（谁、什么时候、申请加入哪个团队、被谁审批），可以建一张审计表：

```sql
create table if not exists public.team_join_requests (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references public.profiles(id) on delete cascade,
  team_id uuid references public.teams(id),
  request_status text not null default 'pending'
    check (request_status in ('pending', 'approved', 'rejected')),
  reviewed_by uuid references public.profiles(id) on delete set null,
  reviewed_at timestamptz,
  created_at timestamptz not null default now()
);

alter table public.team_join_requests enable row level security;

-- admin/owner 可查看全部
-- 普通用户只能看自己的
create policy "team_join_requests_admin_select"
  on public.team_join_requests for select
  using (public.is_admin() or user_id = auth.uid());

create policy "team_join_requests_admin_all"
  on public.team_join_requests for all
  using (public.is_admin());
```

**优先级：低**。一期可以不做，直接用 `profiles.status = 'pending'` + `audit_logs` 记录审批动作即可。

## 前端改动

### 1. 注册页 `app/(auth)/register/`

#### register-form.tsx
- **删除** "邀请码"输入框及所有相关文案
- 保留：姓名、邮箱、所属团队、密码、密码强度提示
- 提交按钮文案可保持"创建账号"

#### register/page.tsx（Server Action）
- **删除** `validate_invite_code` RPC 调用
- **删除** invite_codes 过期检查和回写逻辑
- `signUp` 成功后，`profiles.insert` 时增加 `status: 'pending'`

```typescript
// 变更前
await supabase.from("profiles").insert({ id: userId, name });

// 变更后
await supabase.from("profiles").insert({
  id: userId,
  name,
  status: "pending",
  team_id: selectedTeam.id,
});
```

- 注册成功后不再 `redirect("/dashboard")`，而是 `redirect("/login?registered=1")`，登录页根据 `registered=1` 显示：
  > "注册申请已提交，等待团队管理员审批后即可登录使用。"

### 2. 登录页 `app/(auth)/login/`

#### login/page.tsx
- 登录成功后（`signInWithPassword` 通过），**额外检查** `profiles.status`
- 如果 `status === 'pending'`：
  - 不跳转 dashboard
  - 返回错误状态：`{ error: "账号正在审批中，请联系团队管理员。", email }`
  - 前端 toast 提示该错误
- 如果 `status === 'exempt' || status === 'active'`：正常按 role 跳转

### 3. Middleware `middleware.ts`

增加 pending 用户拦截：

```typescript
// 在现有 admin 路由检查之后，dashboard 路由检查之前
if (user && (isDashboardRoute || isAdminRoute || isAnalyticsRoute)) {
  const { data: profile } = await supabase
    .from("profiles")
    .select("status")
    .eq("id", user.id)
    .single();

  if (profile?.status === "pending") {
    // 如果访问的是 /pending 页面则放行，否则重定向
    if (pathname !== "/pending") {
      return NextResponse.redirect(new URL("/pending", request.url));
    }
  }
}
```

> 注意：middleware 里不宜做太多查询，但这个查询很轻（单行索引命中），且只在已登录用户访问受保护路由时触发，可接受。

### 4. 新增等待页 `app/pending/page.tsx`

- 仅 `status === 'pending'` 的用户可访问（middleware 已做拦截）
- 页面内容：
  - "你的账号正在审批中"
  - "已申请加入：{teamName}"
  - "请联系团队管理员加快审批"
  - 提供"退出登录"按钮

### 5. 管理后台 `app/(app)/admin/`

#### admin/page.tsx
- 顶部摘要卡片："邀请码存量"可替换为"待审批成员数"
- ActionHub 快捷操作新增"审批新成员"入口

#### 新增组件 `app/(app)/admin/pending-members-manager.tsx`

- 表格展示 `status = 'pending'` 的成员
- 列：姓名、邮箱、申请团队、注册时间
- 操作按钮：
  - **通过** → 调用 `approveMember(userId)`，改 `status = 'active'`，记 audit_log
  - **拒绝** → 调用 `rejectMember(userId)`，删除 auth user + profile，记 audit_log
- 批量操作：可选，一期可不做

#### admin/actions.ts

新增两个 server action：

```typescript
export async function approveMember(targetUserId: string): Promise<{ error?: string }> {
  const perm = await getUserPermissions();
  if (!perm) return { error: "未登录" };
  if (!hasPermission(perm.role, perm.permissions, "manage_members")) return { error: "无权限" };

  const supabase = await createClient();

  const { data: target } = await supabase
    .from("profiles")
    .select("status, name")
    .eq("id", targetUserId)
    .single();

  if (!target) return { error: "用户不存在" };
  if (target.status !== "pending") return { error: "该用户不处于待审批状态" };

  const { error } = await supabase
    .from("profiles")
    .update({ status: "active" })
    .eq("id", targetUserId);

  if (error) return { error: error.message };

  await writeAuditLog(supabase, perm.userId, "approve_member", targetUserId, `批准成员: ${target.name}`);
  revalidatePath("/admin");
  return {};
}

export async function rejectMember(targetUserId: string): Promise<{ error?: string }> {
  const perm = await getUserPermissions();
  if (!perm) return { error: "未登录" };
  if (!hasPermission(perm.role, perm.permissions, "manage_members")) return { error: "无权限" };
  if (targetUserId === perm.userId) return { error: "不能操作自己" };

  const supabase = await createClient();

  const { data: target } = await supabase
    .from("profiles")
    .select("status, name")
    .eq("id", targetUserId)
    .single();

  if (!target) return { error: "用户不存在" };
  if (target.status !== "pending") return { error: "该用户不处于待审批状态" };

  // 先记日志（删了人就记不了了）
  await writeAuditLog(supabase, perm.userId, "reject_member", targetUserId, `拒绝成员: ${target.name}`);

  // 删除 supabase auth 用户（连带级联删除 profiles）
  const adminSupabase = createAdminClient();
  const { error: deleteError } = await adminSupabase.auth.admin.deleteUser(targetUserId);
  if (deleteError) return { error: deleteError.message };

  revalidatePath("/admin");
  return {};
}
```

#### 邀请码模块的处置

**方案 A（推荐，渐进式）**：
- 保留 `InviteCodeManager` 组件和 `generateInviteCode` action 一段时间
- 后台增加提示："邀请码功能即将下线，建议改用成员审批"
- 前端注册页已经去掉邀请码，老邀请码自然失效（无人使用）
- 二期再彻底清理数据库表和相关代码

**方案 B（一次性清理）**：
- 同步删除 `InviteCodeManager`、`generateInviteCode`、`validate_invite_code` RPC
- 删除 `invite_codes` 表（或留着但不再使用）

建议选 **方案 A**，先保证注册流程走通，旧代码不删，降低风险。

## 后端/数据层改动

### 1. 所有统计查询排除 pending 用户

需要检查以下地方，确保 `pending` 用户不混入统计：

| 文件/功能 | 处理方式 |
|----------|---------|
| `get_today_submission_status()` | `where p.status != 'exempt' and p.status != 'pending'` |
| `loadAdminPageData` 中的 `totalProfiles` / `activeProfilesCount` | `activeProfilesCount` 只统计 `active`，`totalProfiles` 建议拆分为"正式成员"和"待审批"两个数字 |
| 排行榜 API | 排除 `pending` |
| 趋势图数据 | 排除 `pending` |
| 飞书 webhook 催交 | 排除 `pending` |
| cron 任务 | 排除 `pending` |

**通用做法**：现有查询里凡是 `role = 'member'` 的，后面加上 `and status != 'pending'`；凡是查 `status != 'exempt'` 的，改为 `status = 'active'`。

### 2. RLS 策略（如需要）

pending 用户理论上不应该能读写任何业务数据。当前已有 `profiles_select_own_or_admin` 等策略，但 `daily_reports` 等表的 insert 策略是 `user_id = auth.uid()`，如果 pending 用户能登录（技术上能拿到 session），他们理论上可以 insert。

**建议**：在 middleware 中拦截 pending 用户，让他们无法进入 dashboard，这样就触不到业务 API。如果需要更严格，可以在各业务表的 RLS 中增加 `and exists (select 1 from profiles where id = auth.uid() and status = 'active')`，但改动面太大。先以 middleware 拦截为主。

### 3. `lib/teams.ts` 中的 `getTeamOptions`

当前 `ensureDefaultTeam` 会自动创建"深圳二部"。新注册流程下，用户注册时选择的团队就是 `team_id`，不需要再回写到 `auth.users.user_metadata`。但现有代码中 `getTeamMeta` 还从 metadata 读取，可以保留作为 fallback，不影响。

## 风险与注意事项

1. **现有用户不受影响**：所有现有用户 `status = 'active'`，登录、统计、权限全部保持原样。

2. **Supabase auth 用户已创建**：pending 状态下 `auth.users` 里已经有记录了，只是应用层限制。如果拒绝，需要调用 `auth.admin.deleteUser` 清理，否则数据库会残留 auth 用户。

3. **middleware 查询性能**：每次已登录用户访问受保护路由都会查一次 `profiles.status`。可以缓存到 cookie 或 JWT claim 中，但当前用户量小，直接查表可接受。

4. **owner 是唯一不受限的**：当前系统 `owner` 是 `1305085564@qq.com`，必须确保 owner 账号永远不会被误标记为 pending。

5. **演示模式**：`/demo` 路由不登录也能进，不受此方案影响。

6. **member_change_log**：approve 时是否记 `add`？建议记，但 `action_type` 需要扩展或复用现有类型。当前 `action_type` 为 `('add','remove','transfer','disable')`，`add` 语义上合适。
