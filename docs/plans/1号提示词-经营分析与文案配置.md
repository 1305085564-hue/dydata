## 任务一：经营分析页面 (Analytics) 优化

### 目标路径
`src/app/(app)/admin/analytics/`

### 现有代码结构（必读，× 猜）
- `page.tsx` — Server Component，调用 `loadAnalyticsPageData()` 加载数据，分发给子组件
- `视频结论卡.tsx` — 4 张核心结论卡片（最值得放大的题材/最强表达形式/最佳发布时间段/需干预人员）
- `analytics-sections.tsx` — Accordion 折叠容器，包含 5 个区块
- `hit-analyzer.tsx` — 爆款分析散点图（X:完播率 Y:播放量 Z:互动量），已有快捷筛选器+提交人筛选+爆款阈值线(10w)
- `personnel-analysis.tsx` — 人员排行卡片（排名/标签/均播对比条/爆款率/互动粘性）
- `time-analysis.tsx` — 7×24 热力图
- `follower-convert-trend.tsx` — 导粉趋势折线图（仅 admin/owner 可见）
- `ai-insight.tsx` — AI 洞察生成器（周度/月度切换，调用 /api/ai/insight/period）
- `video-analytics.tsx` — 死代码，未被引用，× 基于它开发

### 数据加载器
`src/lib/loaders/analytics-page.ts` 返回：
- `filteredReports` — daily_reports 表数据（含 play_count, completion_rate, likes, comments 等）
- `filteredVideos` — videos 表数据（含 accounts(name), profiles(name)）
- `filteredSnapshots` — video_metrics_snapshots
- `filteredVideoTags` — video_tags
- `isPrivilegedUser` — 权限标记（admin/owner = true）
- `submitters` — 提交人列表

### 优化要求

#### 1. 视觉分层（打破数据墙）
当前所有区块平铺在 Accordion 里，缺乏层次感。优化为三层：

**顶层（Hero 指标区）**：在视频结论卡上方，新增一排大字号宏观指标卡片（3-4个）：
- 总视频数 / 总播放量 / 大盘爆款率(>10w占比) / 平均完播率
- 数据源：从 `filteredReports` 聚合计算
- 带趋势箭头（对比上一周期，如果有数据的话）
- 用大字号 + 简洁卡片，一眼扫到关键数字

**中层（保持现有）**：视频结论卡 + Accordion 区块（爆款分析器/人员分析/时间热力图/AI洞察）
- 这些组件已经很好，保持现有功能不变
- 可以微调视觉细节（间距、圆角、阴影等）

**底层（新增明细表格）**：在 Accordion 下方新增一个视频明细表格
- 数据源：`filteredVideos`
- 列：标题 / 发布人 / 发布时间 / 播放量 / 完播率 / 互动量(点赞+评论+分享+收藏)
- 必须设 `max-h-[500px]` + `overflow-y-auto`，× 拉长整个页面
- 播放量 > 10w 的行：加高亮背景色（如浅红色）+ "爆款" Tag
- 支持按播放量/完播率/互动量排序

#### 2. 已知 Bug 修复（顺手修）
- `hit-analyzer.tsx` 的 tooltip 引用了 `cover_url` 字段（第 112 行），但 `analytics-page.ts` 的 loader 没查这个字段。修复方案：在 loader 的 videos select 中加上 `cover_url`（如果 videos 表有这个字段），或者删掉 tooltip 中的封面图渲染块
- `ai-insight.tsx` 第 227 行 "将其设为下期目标" 按钮没有 `onClick` handler → 暂时加 `onClick` 弹一个 toast 提示"功能开发中"

#### 3. 权限保护（× 丢）
- `page.tsx` 必须保持 Server Component，× 加 `"use client"`
- `isPrivilegedUser` 条件渲染必须保留（导粉趋势只对 admin/owner 显示）
- 人员分析的 title 根据权限显示不同文案（"团队成员表现" vs "仅展示我的个人数据"）

## 任务二：文案改写配置页面 (AI Rewrite) 优化

### 目标路径
`src/app/(app)/admin/ai-rewrite/`

### 现有代码结构
- `page.tsx` — Server Component，仅 owner 可访问（有 role !== "owner" 硬性检查）
- `ai-rewrite-client.tsx` — 1768 行巨型客户端组件，包含：
  - 7 种编辑器类型：fixed_mode / model_view / model_route / mode / length_preset / workflow / workflow_step
  - 纯垂直排列：摘要卡片(4列) → 运行规则 → 固定套餐 → 展示模型 → 执行路线 → 普通配置(模式+字数预设) → 自动流程
  - 所有 CRUD 通过 `/api/admin/ai-rewrite` 的 GET/POST/PUT
  - 编辑用 Dialog 弹窗

### 数据模型（× 改，只改展现形式）
- `FixedModeRow` — 固定能力套餐（最多2个），含 name/description/fixed_prompt/model_view_id/length_preset_id
- `ModelViewRow` — 展示模型，含 key/label/description/sort_order/is_enabled/is_default
- `ModelRouteRow` — 真实执行路线，含 model_view_id/workflow_step_id/channel_id/actual_model/priority/weight
- `ModeRow` — 普通模式，含 name/mode_prompt（自由文本，不是枚举）
- `LengthPresetRow` — 字数预设，含 name/length_prompt（自由文本，不是枚举）
- `WorkflowRow` / `WorkflowStepRow` — 自动流程及步骤
- `ChannelRow` — 渠道列表（id/name/model/is_enabled/priority）

### 优化要求

#### 1. 布局改造：左右分栏
将 1768 行的垂直长页面改为左右分栏布局：

**左侧（60% 宽度）— 配置区**：
- 用折叠面板（Collapsible/Accordion）将现有配置分组：
  - 运行规则（is_enabled / output_token_limit / context_message_limit）
  - 固定能力套餐（最多 2 个卡片）
  - 展示模型（表格）
  - 真实执行路线（表格）
  - 普通配置（模式 + 字数预设）
  - 自动流程与步骤
- 默认展开"运行规则"和"固定能力套餐"，其余折叠

**右侧（40% 宽度）— 沙盒测试区（UI 占位）**：
- 固定定位（sticky），不随左侧滚动
- 包含：一个 Textarea 输入测试文案 + "试跑改写" 按钮 + 结果展示区
- 本次只做 UI 占位 + mock 预览效果（点击按钮后显示 loading 动画 → 2秒后显示一段示例改写结果）
- × 建后端 API，后续再接通真实接口
- 右侧顶部显示当前选中的固定套餐/模式名称，让用户知道"我在测试哪个配置"

#### 2. 拆分组件（× 继续 1768 行单文件）
将 `ai-rewrite-client.tsx` 拆分为：
- `ai-rewrite-client.tsx` — 主容器（状态管理 + 布局框架），控制在 300 行以内
- `rewrite-config-panel.tsx` — 左侧配置面板
- `rewrite-sandbox.tsx` — 右侧沙盒测试区
- `rewrite-editor-dialog.tsx` — 编辑弹窗（从主文件提取）
- 各配置区块可以进一步拆分为独立组件（如 `rewrite-fixed-modes.tsx`、`rewrite-model-routes.tsx` 等）

#### 3. 权限保护
- `page.tsx` 的 owner 权限检查必须保留，× 动
