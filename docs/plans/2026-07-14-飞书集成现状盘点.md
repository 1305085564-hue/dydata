# 飞书集成现状盘点

> 盘点日期：2026-07-14
> 目标：摸清本项目现有的飞书(Feishu/Lark)集成到底做到哪一步，无需修改代码。

## 一、已打通能力
目前系统**唯一**真正打通的飞书能力是：**自定义机器人群消息推送（Webhook）**。
系统通过直接向指定的 Webhook URL 发送 HTTP POST 请求，推送 `lark_md` 格式的富文本通知。

**具体集成点及分布文件**：
1. **用户操作通知**：
   - **用户填报通知**：`src/app/(app)/dashboard/actions.ts` (`notifyFeishu` 接口会在用户提交日报后触发推送)。
2. **定时任务推送 (Cron)**：
   - **自动催交提醒**：`src/app/api/remind/route.ts` (基于 Vercel Cron，遍历未交人员发消息)。
   - **数据报告推送**：`src/app/api/report/route.ts`。
3. **系统及业务告警**：
   - **首屏性能告警**：`src/app/api/admin/first-screen-monitor/route.ts`。
   - **系统异常告警**：`src/app/api/admin/system-issues/route.ts`。
   - **智能业务预警**：`src/app/api/smart-alert/notify/route.ts` 与 `src/lib/smart-alert.ts`。
4. **管理后台配置**：
   - 履约工作台包含飞书自动催交的全局开关，通过 `src/app/api/admin/system/settings/route.ts` 存储 `feishu_fulfillment_reminder_enabled` 配置，并在 `src/app/(app)/admin/fulfillment/fulfillment-workbench.tsx` 与 `filter-bar.tsx` 中呈现。
   - 提醒日志弹窗支持显示来源为“飞书”（判断条件为 `channel === "feishu_webhook"`）。

## 二、半成品/空壳
从代码搜索结果（有 `feishu`、`webhook`，但 `app_id`、`app_secret`、`tenant_access_token` 查无匹配）来看：

- **飞书开放平台应用（企业自建 App）未接入**：整个系统没有对接飞书鉴权体系的任何接口，属于纯空白状态。
- **缺失的能力（没有实现任何逻辑代码）**：
  - **飞书登录 (SSO)**：没有引入飞书扫码或 OAuth，当前鉴权仍独立基于 Supabase。
  - **多维表格 (Bitable) 读写**：所有数据均存储和计算在自有 Supabase 数据库中，并未与飞书表格进行读写打通。
  - **审批流 (Approval)**：当前的申诉、复核与豁免均为平台内的审批记录，没有拉起飞书原生的审批流组件。
  - **通讯录同步**：团队架构与成员管理完全在系统内自己维护，没有请求飞书开放接口获取部门或人员架构。
  - **交互式卡片回调**：发出去的消息只是单向通知，没有实现任何能接收飞书卡片按钮点击回调的服务端路由。

## 三、缺失的配置
因为当前能力仅限发消息，所以需要的配置也极其简单：
1. **当前需要的配置**：系统全局仅依赖 `FEISHU_WEBHOOK_URL` 以及部分配套的 `CRON_SECRET`。
2. **位置与完整度**：该值明确存在于 `.env.local` 并在生产环境配置（地址为 `https://open.feishu.cn/open-apis/bot/v2/hook/9bd6cf6a-...`），**目前配置是齐全的**。
3. **是否失效**：只要该飞书群机器人未被管理员移除，Webhook 便长期有效；但对于完整版飞书集成来说，因为压根没有去申请注册“飞书企业自建应用”，所以自然也没有 `app_id`、`app_secret` 等配置项，不能说是“缺失或失效”，而是从一开始就未采用该架构。

## 四、可复用程度
- **通知能力的可复用度（较高）**：由于现有的各类告警和业务推送都已经封装好了基于 `lark_md` 格式的文本排版和请求发送代码，后续如果只增加“单向提醒场景”，可以直接复用这套逻辑。
- **深层协作的可复用度（极低 / 从零开始）**：根据《飞书协同重定位判断》规划，如果未来将看板、日报填写或视频审批等重度协作“退回飞书”或者开展卡片交互，现在的代码**完全无法复用**。必须从零开始：申请飞书 App 凭证、封装请求 `tenant_access_token` 的底层服务，以及实现飞书开放平台所有的加密校验和接口。
