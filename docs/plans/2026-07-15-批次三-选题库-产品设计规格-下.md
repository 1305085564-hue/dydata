# 批次三 · 选题库 · 产品设计规格（下）

> 接上篇 `2026-07-15-批次三-选题库-产品设计规格.md`。上篇讲定位/铁律/数据结构/来源，本篇讲状态流转/数据闭环/页面/协作/MVP/与批次一的接口。

## 4. 状态流转——跟人走，不跟子题走

**核心决策**：子题本身不带状态。状态跟着"**人 × 子题**"走——谁认领了谁有自己的进度，互不干扰。

同一个子题可以多人各自认领各自做（多个号、多个角度），A 在写脚本、B 还在候选，互不影响。

**进度只有两步 + 一个退出动作**：

```
候选 → 脚本中
       ↓（不做了）
    放回选题池
```

- **候选**：运营从池里捞起来了，打算做，还没开始写脚本。每人同时最多 **5 条**候选（防囤积）。
- **脚本中**：正在写脚本/拍摄/发布中（中间不细分，一步到底）。对别人来说=告知"有人在做了"；对自己来说=自己预选的优质选题。
- **放回选题池**：不做了，一键放回。选题回到公共池，别人可以继续捞。

**子题卡上的展示**：只显示"目前有 N 人在做"+ 简要列出是谁、各自到哪步。这是参考信息，不阻塞别人。

**看别人进度的入口**：有，但收在低频位置（子题详情页里展开），不占主界面空间。

> 注意：选题库不管"复盘"。数据反馈和复盘在数据上传那边完成——选题库是选题的**规划**，数据上传是选题的**执行与反馈**。

## 5. 数据闭环——关联前置到认领环节

阿禅拍板方向：**自动归类、不让运营额外动手**；但为防脏数据（塞错/对不上/塞两处），落地为"**系统猜、人点一下确认**"，不做纯静默自动匹配。

两条路：

**首选（天生绑对，不用猜）**：视频本来就是**从选题库挑了子题去做的**（认领环节已绑定子题号）——上传数据时把这个"子题号"一路带过来，天生绑对，零猜测。这就是那条主线（库里挑→写脚本→发布→回数据）跑顺的样子。

**兜底（没走库、临时拍的）**：上传数据页面给一个"**关联选题**"选择框 + **系统智能推荐**（系统默认选中最可能的）——运营**点一下确认或改**；匹配不上就一键"新建选题"或"暂无对应"。运营不用搜索，系统帮他猜好了。

> 关联动作前置到认领环节（走库的路径），而不是放在上传时让运营动脑。上传是最忙乱的时候，能少想一步就少想一步。

发布后的数据（播放/完播/互动/转粉）回填到作品，再聚合到子题：**≥1000 流量的作品取平均播放量**，作为子题卡的汇总数据展示。

## 6. 页面一：今日选题工作台（运营每天开的页）

不要一上来甩一个大列表。首页直接回答"最近什么选题在动"：

**MVP 阶段用"最近活跃"替代智能推荐**（冷启动没有数据喂养算法，硬推=推不出东西或瞎推，运营几天就不来了）：

- **最近活跃位**：最近有人认领的选题 / 最近发布了作品的选题 / 最近新录入的选题。用人工活跃度+时间序保证首页永远有内容。
- 每条可一键：**认领**（捞起来）/ 跳过。
- 目标：运营打开就有东西看、有东西可以捞，不用自己搜半天。

> 后期数据积累够了，再上真正的智能推荐位（按历史表现、撞车检测、内容类型平衡等排序）。

## 7. 页面二：选题池（挑选与管理）

选题池是选题库的主体。提供**时间筛选**（近 3 天 / 本周 / 本月 / 近 3 个月录入的）+ **按母题折叠浏览**（母题→子题树）。

**子题卡展示层级**（见上篇"子题卡信息层级"）：
1. 第一眼：选题好不好（子题名 + 一句话 + 汇总数据）
2. 第二眼：最好/最近的文案 + 对应数据
3. 展开详情：历史所有文案版本 + 各自数据 + 相近选题文案归纳

**默认视图**（MVP 阶段精简，不做一堆空视图）：
- 全部（按录入时间排序）
- 我正在做的（我认领的）
- 我提交的

> 后期可加：系统推荐 / 高潜未使用 / 我收藏的。先验证核心流程跑通再补。

## 8. 脚本环节

**"脚本中"只是状态标记**，告诉别人"有人在做了"，对自己是"我预选的优质选题"。

选题库**不负责跳转文案助手、不自动创建脚本**。运营自己去文案助手写脚本，写完后脚本会通过数据上传时的关联自动挂到作品上。选题库不重造写作器。

## 9. 多运营协作

- **状态跟人走**：每个运营认领子题后有自己独立的进度（候选/脚本中），不影响别人。
- **同一子题多人可做**：不禁止、不互锁。子题卡上只提示"目前有 N 人在做"。
- **看别人的入口**：有，收在子题详情页里（低频功能），不占主界面。
- **放回选题池**：不做了就一键放回，选题回到公共池。
- **不做**占坑/超时释放/复杂权限（人少，口头协调够）。

## 10. MVP（第一版只做这些，先验证真会用）

先跑通"存+选"闭环，别一上来做全套智能系统：

1. **轻量录入**：选母题（高频上低频下）+ 写一句话 = 完成，3 秒。
2. **母题/子题两层 + 8 大预置母题 + 分组自动归**。
3. **认领 + 两步进度 + 放回**（候选→脚本中，不做了放回池），状态跟人走，候选上限 5 条。
4. **选题池**：时间筛选 + 母题折叠 + 三个默认视图。
5. **子题卡文案收纳**：多版本文案 + 对应数据，三级展示（汇总→最好/最近→全部历史）。
6. **今日工作台**：最近活跃位（替代冷启动阶段的智能推荐）。
7. **作品关联**：认领时天生绑定；没走库的上传时"系统猜+人确认"。

**暂不做**（验证用起来了再加）：复杂 AI 推荐、全网热点抓取、精细权限、大量图表、认领超时释放、复盘界面、智能推荐位。

**判断成功的核心指标**：三个月后，**团队发布的作品里，有多大比例是从选题库挑的选题做出来的**。多数还绕开它 = 没真进工作流。

## 11. 与批次一的接口对接点（重要，两批别打架）

选题库的数据闭环依赖批次一的"数据上传"，两批要留一个对接点：

- **批次一的数据上传表单**要**加一个"关联选题"字段**（现批次一规格里没有）——首选带子题号进来（从库认领的路径天生绑对），兜底给"系统推荐+人确认"的选择框（系统默认选中最可能的，运营确认即可）。
- 上传接口 `POST /api/video-submit`（批次一定义）**入参增加 `topic_id`（关联子题，可空）**；空时走兜底匹配。
- 选题库需要读上传后的视频数据（播放/完播等）回填作品——复用批次一/现有视频表数据源。
- 作品的文案内容从视频表的文案字段读取，多次上传同一子题的不同文案自动收纳为多版本。

> 落地顺序：批次一先把"关联选题"字段的坑位和 `topic_id` 入参留好（哪怕先接空值），选题库批次三再接上。现在写清楚，免得批次一做完了才发现要返工。

## 12. 待阿禅确认后再拆分工

本设计确认后，按批次一的模式拆两份执行规格：
- **Codex 后端规格**：三层数据表、状态流转、"猜+确认"匹配、作品数据回填与文案收纳、与批次一 `topic_id` 对接。
- **Antigravity 前端规格**：今日工作台/选题池两页，从零搭（不参考视频审核卡片墙），遵守《美学规范》、红涨绿跌。子题卡三级信息层级是前端核心交互。
