## 四、本轮沉淀的规则

### 4.1 阿禅减法 · 后台专属施工规则（新增，可直接复用）

1. **一个 section 一个标题**：不要 eyebrow + h2 + desc 三连堆。如果主体组件自带标题，外层就别再加。**Codex 最喜欢犯**。
2. **Index 索引单源**：同一页只能有一套索引（用 AdminWorkspaceLayout 的 `indexItems`），不许在 children 内再搞一套。
3. **Index 短标题**：两字内，放不下再用四字（成员 / 团队 / 目录 / 治理）。
4. **装饰卡 vs 结构卡**：
   - 删除：单独包一个 section 只为"好看"的 `rounded-2xl border bg-white shadow-sm`
   - 保留：需要视觉边界区分的多列并排场景（如中控台三列队列 QueueCard）
5. **锚点不要复述标题**：外层 indexItems 是跳转用的，**只要 label**，不要 hint 描述——主体区已经说过一遍了。
6. **口号卡禁区**：Workflow/Step/Flow 等"解释流程"的卡片，阿禅不需要，删。
7. **overflow-y-auto 内滚动慎用**：页面本身能滚就让它滚，内盒子套一个 `max-h-[720px] overflow-y-auto` 会造成"下方大量留白"（left-col 短被 right-col 拉伸）。
8. **双标题清单**：改一个 section 前先 grep 子组件是否自带标题：
   ```bash
   grep -n "h1\|h2\|h3" src/app/(app)/admin/对应子组件.tsx
   ```

### 4.2 Sticky 排查清单（本轮三连挫的通用教训）

遇到 `sticky` 失效，**按这个顺序查**：

1. **查真正的滚动容器**：从被 sticky 的元素往外，找最近的带 `overflow: auto/scroll` 的祖先。这个祖先就是 sticky 的参考系，不是 window。
   ```bash
   grep -rn 'overflow-y-auto\|overflow-auto' src/app/ src/components/
   ```
2. **查断点实际生效宽度**：`xl:block` 基于**视口宽度**，但实际可用主区宽度 = 视口 - sidebar - padding。如果外层有 240px 左 sidebar + 24px padding + 限宽 1400px，视口 1440 下主区内部只有 1152，`xl:`(1280) 触发不到。**常用解法：降一个断点（xl→lg）**。
3. **查父容器高度是否足够**：sticky 元素在父容器内活动，父容器高度不够时 sticky 会"提前结束"。
   - grid 下加 `items-start` 会把 cell 压缩到自然高度——**破坏 sticky**
   - 默认 stretch 让 cell 跟最高兄弟等高——**sticky 有空间**
4. **查 top 值是否被遮挡**：
   - 如果滚动容器是 `<main overflow-y-auto>`，top-0 是 main 内顶部，**main 内部没 NavBar 遮挡，top-8 就够**
   - 如果滚动容器是 window，top 要避开 fixed NavBar，用 `top-[calc(var(--app-top-offset)+1.25rem)]`

### 4.3 debug 反思（本轮失败）

- **第 1 次错判**：直接改 UI 没查外层 layout——应该先查 `/admin/layout.tsx` 以及 `admin-main-area.tsx`
- **第 2 次错判**：加 `xl:items-start` 以为能让 aside 贴顶，其实压缩了父容器，反而破坏 sticky
- **第 3 次错判**：改 `top-[--app-top-offset]` 以为避开 NavBar，但参考系是 main 不是 window
- **根本教训**：**sticky 问题先查滚动容器 → 断点实际生效宽度 → 父容器高度 → top 遮挡**，四步不能跳。
