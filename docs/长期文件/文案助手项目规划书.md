# 文案助手项目规划书

> 版本：V1.1 | 更新时间：2026-06-29 | 维护者：阿禅

---

## 一、产品定位

**一句话定义**：带技能注入的多轮对话文案优化工具。

本质就是一个 AI 对话助手，但可以随时加入"技能"（一段提示词），让 AI 按特定风格、要求改写文案。用户和 AI 像聊天一样对话，直到对最终文案满意为止。

### 不是什么

| 它不是 | 为什么不是 |
|--------|-----------|
| 多步工作流系统 | 不需要"先结构→再润色→再标题"的流水线 |
| 写作 IDE | 不需要代码编辑器级别的补全/修订追踪 |
| 文档协作平台 | 不需要多人在线编辑同一篇文案 |
| 内容资产管理系统 | 不需要发布、投放、复盘的完整闭环 |

### 核心价值

1. **降低改写门槛**：员工不需要会写提示词，选个技能就能用
2. **积累团队经验**：好的改写思路沉淀为技能，团队可复用
3. **保持对话自然**：像跟同事说"再口语化一点"一样简单

---

## 二、目标用户与使用场景

### 用户

| 角色 | 使用方式 |
|------|---------|
| 文案编辑 | 输入原文 → 选技能 → 多轮对话优化 → 复制终稿 |
| 团队主管（阿禅） | 创建平台技能 → 配置模型 → 查看使用统计 |
| 资深员工 | 创建个人技能 → 验证效果 → 共享给团队 |

### 核心场景

1. **日常改写**：拿到一篇原始文案，用"小红书爆款"技能改写
2. **多轮微调**：首稿不满意，继续对话"开头再吸引一点""结尾加个 CTA"
3. **局部修改**：整体满意，但第 2 段太正式了，只改这一段
4. **技能探索**：试试不同技能组合，找到最适合的风格

## 三、功能设计

### 3.1 核心交互模式

```
┌─────────────────────────────────────────────────┐
│  顶部栏：模型选择 | 新对话 | 历史 | 定稿导出      │
├──────────────────────┬──────────────────────────┤
│                      │                          │
│   左侧 45%           │    右侧 55%              │
│   对话区             │    文案画布               │
│                      │                          │
│   用户消息           │    段落 1                 │
│   AI 回复            │    段落 2 [已锁定]        │
│   用户消息           │    段落 3                 │
│   AI 回复（流式）    │    段落 4                 │
│                      │                          │
├──────────────────────┤                          │
│  [技能A] [技能B] ×   │                          │
│  ┌──────────────┐    │                          │
│  │ 输入框 / @   │ ▶  │                          │
│  └──────────────┘    │                          │
└──────────────────────┴──────────────────────────┘
```

### 3.2 技能系统

#### 技能分类

| 类型 | 创建者 | 可见范围 | 说明 |
|------|--------|---------|------|
| 平台技能 | 阿禅/管理员 | 所有人 | 官方推荐，质量有保障 |
| 个人技能 | 任何用户 | 仅自己 | 自建自用 |
| 共享技能 | 任何用户 | 所有人 | 个人技能设为共享后所有人可用 |

#### 技能本质

技能 = 一段提示词（system prompt 注入）。

#### 使用方式

- 输入框输入 `/` 弹出技能选择器
- 可以同时激活多个技能
- 激活的技能显示在输入框上方（Active Skills Bar）
- 点击 × 可以移除技能
- 技能之间的优先级和合并由 AI 自行判断

#### 创建技能

用户自建技能只需要填：
- 名称（必填）
- 提示词主体（必填）
- 描述/备注（选填，方便自己区分，也方便共享后别人理解）

#### 技能与模型的关系

- 技能默认不绑定模型
- 后台可以给技能绑定默认模型（可选）
- 模型优先级：用户手动选择 > 技能默认模型 > 系统默认模型

### 3.3 段落编辑

#### 为什么保留

来自批改台的批改功能，实际验证过好用。允许用户只修改文案的某一部分，而不是每次都全篇重写。

#### 功能范围

| 功能 | 状态 | 说明 |
|------|------|------|
| 段落显示 | ✅ 已实现 | 右侧文案按段落切分展示 |
| 段落选中 | ✅ 已实现 | 点击段落高亮选中 |
| 多段选中 | ✅ 已实现 | 可以选中多个不连续段落 |
| 段落锁定 | ✅ 已实现 | 锁定后 AI 生成时不修改 |
| 局部修改 | ✅ 已实现 | 选中段落后发消息，只改选中部分 |
| 智能局部修改 | ✅ 已实现 | 全局输入"第二段加强情绪""最后两段压短"等指令时，自动识别目标段落并走局部 patch |
| 跨段半句选区 | ❌ 不做 | 太复杂，意义不大 |

#### 智能局部修改规则

目标：用户在左侧全局输入框说"第二段加强情绪"时，右侧只改第二段，其他段落不消失、不流式重打、不白屏。

目标识别优先级：

1. 手动选中段落最高优先级。
2. 其次识别自然语言里的明确段落目标：`第二段`、`第2到3段`、`前两段`、`最后一段`、`最后两段`、`开头/结尾`。
3. 再使用最近聚焦段落作为弱兜底。
4. 模糊指令如"加强情绪"不默认全文重写，提示用户选择段落或写明第几段。
5. 只有明确说"全文/整篇/全部/整体/通篇/重新生成"时，才走全文重写。

实现口径：

- 前端本地先用 `target-plan.ts` 解析目标，不把目标判断完全交给生成模型。
- 解析到段落后，请求 `/api/rewrite/generate` 时带 `targetParagraphIds`。
- 生成过程中不清空 `documentParagraphs`，只让目标段落进入生成态。
- 后端 SSE 会下发 `target_plan` 事件，前端据此高亮目标范围。
- 局部 patch 保留原 `paragraphId`，方便后续撤销、追踪和继续改同一段。

### 3.4 模型选择

- V2 顶部有模型选择器，用户可以随时切换
- 模型列表来自后台配置的 `ai_provider_key_models`

## 五、技术架构

### 5.1 系统总览

| 层 | 技术 |
|----|------|
| 前端 | Next.js App Router + React + TypeScript + Tailwind |
| 后端 | Next.js API Routes（服务端） |
| 数据库 | Supabase（PostgreSQL） |
| AI | 多供应商 + 多 Key + 多模型（三层渠道体系） |
| 部署 | Vercel |

### 5.2 V2 数据库设计

#### 核心表

| 表名 | 作用 |
|------|------|
| `rewrite_conversations` | 会话容器（V1 保留） |
| `rewrite_messages` | 对话消息（V1 保留） |
| `rewrite_skills` | 技能主体 |
| `rewrite_skill_versions` | 技能版本（编辑即新版本） |
| `rewrite_conversation_skills` | 会话技能栈 |
| `rewrite_documents` | 右侧画布容器 |
| `rewrite_document_revisions` | 文档版本 |
| `rewrite_document_paragraphs` | 段落块 |
| `rewrite_variants` | 候选方案（预留） |
| `rewrite_generation_runs` | 生成记录 |

#### 表关系

```
conversation ─────────────────── messages (V1 对话历史)
     │
     ├── conversation_skills ──── skill_versions ── skills
     │
     └── document
            │
            ├── revisions (current_revision_id 指向最新)
            │      │
            │      └── paragraphs
            │
            └── variants (候选方案，预留)
```

#### Revision 状态机

```
pending → completed → (可被新 revision 替代)
pending → failed    → (永不成为 current)
pending → aborted   → (永不成为 current)
```

**硬规则**：只有 `completed` 状态的 revision 才能成为 `current_revision_id`。

### 5.3 AI 渠道体系

#### 三层结构

```
ai_providers        → 供应商（OpenAI、Anthropic、DeepSeek 等）
  └── ai_provider_keys    → 密钥（一个供应商可以有多个 key）
        └── ai_provider_key_models → 模型（一个 key 可以支持多个模型）
```

#### 模型选择优先级

1. 用户手动选择的模型
2. 当前激活 skill 的默认模型
3. 系统默认模型
4. 自动选择健康可用的模型（熔断逻辑）

#### 熔断机制

- `consecutive_failures >= 3` 且 `unhealthy_until > now()` → 跳过该 key
- 失败时自动 bump failure count
- 健康检查后重置

### 5.4 前端组件结构

```
RewriteWorkbench（主工作台）
├── RewriteHistory（左侧历史栏）
├── InstructionFeed（对话流）
├── ChatInputBar（输入框 + 技能选择 + / 命令）
├── target-plan（自然语言段落目标识别）
└── PolishedDocumentCanvas（右侧文案画布 + 段落编辑 + 局部流式展示）
```

**来源**：阿禅与 Codex 的第一轮对话

**核心确认**：
- 本质是"多轮对话优化文案 + 打包好的 skill 技能"

### 6.6 智能局部更新修正（2026-06-29 15:59）

**来源**：阿禅反馈全局输入"第二段加强情绪"时，右侧仍然全篇刷新并重新输出。

**问题根因**：
- 旧逻辑只有手动选中段落时才发送 `targetParagraphIds`。
- 全局输入框没有目标识别能力，所以默认走全文重写。
- 前端发送 V2 请求时曾清空 `polishedText`，无目标时还清空 `documentParagraphs`，直接造成右侧白屏/全篇重打体验。

**最终决定**：
- 不采用"让模型第一行输出 `<targets>` 再动态切换"作为主方案。它依赖模型格式，容易被流式分块和输出污染影响。
- 采用"前端本地目标识别 + 后端块级 patch 协议"。
- 明确段落指令直接转成 `targetParagraphIds`；模糊指令先提示确认，不默认全篇重写。
- 保留全文重写能力，但必须由用户明确表达全文意图。

**已完成实现**：
- `src/components/content-tools/rewrite/target-plan.ts`：本地目标识别。
- `src/components/content-tools/rewrite/target-plan.test.ts`：覆盖第二段、范围、最后两段、全文、模糊指令、锁定段落。
- `useRewriteLogic.ts`：接入目标识别，局部修改时不清空右侧画布。
- `generation.ts`：SSE 增加 `target_plan` 事件。
- `createParagraphPatchRevision`：局部 patch 保留原 `paragraphId`。
- `undo route`：修复 Next 16 动态路由参数类型。

**已通过验证**：
- `npx tsx --test src/components/content-tools/rewrite/target-plan.test.ts src/lib/rewrite/v2-backend.test.ts src/app/api/rewrite/routes.test.ts`

## 七、关键决策记录

### 决策 1：为什么做 V2 而不是修补 V1

**背景**：V1 的代码里同时残留"单步改写"和"多步 workflow"两套逻辑，核心模型混乱。

**决定**：新增 V2 系统，V1 保留兼容但不再扩展。

**理由**：
- V1 的 `rewrite_messages` 同时承担"聊天历史"和"终稿真相"两个职责，耦合太紧
- V2 把"对话"和"文档"分开：messages 管聊天，document/revision 管终稿
- V2 的 skill 系统比 V1 的 fixed_modes 更灵活（支持自建、共享、多选）

### 决策 2：为什么保留段落编辑

**背景**：段落编辑不在初始需求里，是阿禅后面加的。

**决定**：保留。

**理由**：
- 批改台的批改功能验证过好用
- 文案改写经常是"整体满意但某一段不行"，全篇重写浪费
- 段落锁定很实用：定好的段落不让 AI 再动

### 决策 3：为什么删除 workflow

**背景**：V1 有 workflow/autoStep 机制。

**决定**：从主线撤掉，只做兼容读取。

**理由**（阿禅原话）：
> "workflow 只是一个 skill，说白了是后端做砸了出来的产物"

### 决策 4：为什么删除字数预设

**背景**：V1 有 `rewrite_length_presets` 表。

**决定**：不再作为一等功能。

**理由**：
- 实际使用中字数限制效果不稳定（AI 经常不遵守）
- 如果用户真的需要，可以在自然语言里说"控制在 500 字以内"
- 以后可以作为 skill 的可选参数扩展

### 决策 5：技能注入时保存 prompt 快照

**背景**：技能编辑后，旧会话里用的是哪个版本？

**决定**：注入时保存当时的 prompt 快照。

**理由**：
- 技能后来被编辑了，历史会话不应该受影响
- 保证历史可追溯、可复现

### 决策 6：AI 上下文策略——画布快照 + 全量对话 + 可配置条数

**背景**：长对话几十轮，全量带给 AI 会烧钱且干扰。最初设计为"只带最小必要信息"，但用户希望保留连贯的对话上下文。

**最终决定**：
- 画布快照：当前文档的完整状态
- 全量对话历史：保留上下文连贯性
- 用户可配置消息条数上限：默认值待定，用户可在设置中调整
- 活跃技能栈：当前会话注入的所有技能 prompt
- 最新指令：本次用户输入

**理由**：
- 注意力机制天然对最新消息权重更高，老消息影响有限
- 用户需要 AI 记住前几轮的指示（如"保持口语化"），不能每轮丢失
- 配置上限给用户控制权，成本敏感时可手动收紧

### 决策 7：局部更新以块级 Patch 为主，不依赖模型自报目标

**背景**：Gemini 曾提出"单次 LLM 推理目标检测"，让模型第一行输出 `<targets>p-xxx</targets>` 或 `<targets>FULL</targets>`。这个方向看似省一次请求，但稳定性不足。

**决定**：
- 目标识别前置，优先由前端本地规则处理。
- 生成模型只负责"改写目标段落"，不负责决定要改哪里。
- 后端协议显式下发 `target_plan`，前端按目标段落局部展示流式内容。

**理由**：
- 避免模型格式不稳定导致目标解析失败。
- 避免流式第一行被污染后退回全文刷新。
- 让协议可扩展：未来可以把 paragraph 扩展为 title、hook、CTA、scene 等 block 类型。
- 让测试可控：目标识别是纯函数，能用单元测试覆盖。

## 九、风险与注意事项

### 9.1 已知风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 段落切分不准 | 用户选中的段落和实际不匹配 | V1 用双换行切段兜底，后续可优化 |
| 自然语言目标识别漏判 | 用户表达方式太自由，未识别到目标段落 | 模糊时提示选择段落或写明第几段，不默认全文重写 |
| 技能冲突 | 多个互斥技能同时激活，AI 输出混乱 | AI 自行判断优先级；后续可加冲突提示 |
| 流式输出中断 | 网络不稳时生成一半 | revision 状态隔离，失败不污染主画布 |
| V1/V2 双轨维护成本 | 两套系统并存增加复杂度 | 短期接受，中期合并 |

### 9.2 绝对不能做的事

| 禁止事项 | 原因 |
|---------|------|
| 在 V2 里恢复 workflow/autoStep | 已确认是失败产物 |
| 把字数预设做回一等功能 | 效果不好，用自然语言替代 |
| 向用户暴露"版本分叉树"UI | 认知负荷太高，用"基于此版本继续"包装 |
| 全量带聊天历史给 AI | 烧钱 + 幻觉 |
| 失败的 revision 覆盖主画布 | 硬规则：只有 completed 才能成为 current |

### 9.3 每次改动前必须确认的

1. **改动是否偏离核心定位**：这是一个"带技能的对话 AI"，不是别的
2. **改动是否增加用户认知负荷**：员工能 10 分钟内学会使用吗？
3. **改动是否破坏已有功能**：V1 用户能正常使用吗？
4. **改动涉及几个文件**：超过 3 个文件先列清单确认

## 十、给 AI 协作者的说明

### 接手须知

如果你是新接手这个项目的 AI：

1. **先读这份文档**，理解产品定位和核心决策
2. **再看 `docs/rewrite-refactor-phase1-3-report.md`**，了解技术实现细节
3. **不要自作主张加功能**，所有新功能必须阿禅确认
4. **不要恢复已删除的功能**（workflow、字数预设、普通模式）
5. **记住核心定位**：带技能注入的多轮对话文案优化工具。就这么简单。

### 代码入口

| 你想改什么 | 看哪里 |
|-----------|--------|
| 前端界面 | `src/components/content-tools/rewrite/` |
| 状态管理 | `src/components/content-tools/rewrite/useRewriteLogic.ts` |
| 后端 API | `src/app/api/rewrite/` |
| 数据库函数 | `src/lib/rewrite/` |
| AI 调用 | `src/lib/ai/` |
| 数据库迁移 | `supabase/migrations/` |

### 协作规则

- 改动前先读相关代码，不要凭猜测改
- 改完跑 `tsc --noEmit` + `npm run build` 确认不报错
- 不要改 AI provider 的 `baseUrl`（加 /v1 会炸）
- 不要改 provider 的 API 字段格式
