# DYData 全链路数据与批改反馈流设计及开发指南

## 1. 业务链路设计概览
本流线覆盖了抖音视频从创作者端的数据填报上传、管理端的批改诊断、员工端的复盘回传，直至最终作为优质素材沉淀入库的闭环生命周期。其核心目标是解决数据上报不及时、批改反馈断路以及优质素材无法有效沉淀的管理痛点。整个生命周期依次流经：创作者在工作台提交数据与视频（T+0/T+1） -> 系统根据数据基线标记异常 -> 组长在批改台进行视频分析并下发反馈卡 -> 员工在工作台查阅反馈并提交回传复盘 -> 组长将典型案例标记为经验并完成视频资产的分级入库。

## 2. 创作者端：数据上传与填报 (T+0 / T+1)
在创作者端工作台（对应路由 `/dashboard`），用户通过数据填报面板提交视频截图和核心运营指标。界面核心组件为 `video-submit-panel.tsx` 和 `video-submit-form.tsx`。填报过程分为 T+0 基础信息提交与 T+1 的 24小时数据补充两个阶段，且支持多槽位截图识别（`screenshot_1` 为必传发布截图，`screenshot_2` 为必传 24 小时数据与留存曲线截图，`screenshot_3` 为选传截图）。前端调用 `/api/ocr-screenshot` 接口对上传截图进行 OCR 识别，用户确认并补充标题、文案、话题后，整体将向 `/api/video-submit` 发送 POST 请求。

接口 `/api/video-submit/route.ts` 负责数据的最终验证与入库。其数据变更采用严格的回滚机制以保证稳定性，涉及四张表的操作。首先在 `videos` 表创建或更新视频元数据，接着在 `video_metrics_snapshots` 表写入 24 小时指标快照，然后在 `daily_reports` 表生成员工日报记录。在此过程中，接口调用大语言模型（基于 `AI_MODEL` 环境变量配置，如 `claude-sonnet-4-6`）提取视频题材、表达形式、CTA类型等维度标签，并连同手动录入的话题与关键词写入 `video_tags` 表。此外，当检测到粉转数大于 0 且存在脚本文案时，接口还会自动调用转化中心的 `replaceDailyReportUsageRecord` 写入脚本库使用记录，完成向转化中心的关联沉淀。

## 3. 管理端：批改台复核与反馈下发
管理端批改台（对应路由 `/admin/content`）是组长或负责人进行内容诊断的控制中心。其核心页面为 `content-page-client.tsx`，主要的交互容器为详情对话框 `content-detail-dialog.tsx` 与诊断工作台 `content-diagnosis-workbench.tsx`。批改流的触发基于系统自动识别的异常视频。系统根据播放量、爆款系数（计算公式见 `src/lib/video-metrics.ts`）等算法指标，筛选出表现异常的视频，并在批改台列表予以标红。

组长打开待复核视频后，在“分析”页签下查看视频详情、OCR识别数据、前一条视频对比，并可以一键生成 AI 辅助内容诊断报告。诊断结果与历史报告存于 `ai_insight_result` 表。当判定需要给员工下发行动改进要求时，组长会在“反馈”页签中填写两个核心字段，即主要问题（单行概要 `summary.one_line` 和问题标签 `problem_tags`）以及改进反馈（对组员的改进建议 `actions.message_for_member`）。

反馈数据通过调用 `/api/admin/content-feedback-cards/[videoId]` 进行处理。反馈卡表 `content_feedback_cards` 支持状态流转。组长可将其保存为草稿（`card_status` 设为 `draft`），此时仅保存 `draft_payload`；或将其置为已确认（`confirmed`），写入 `confirmed_payload`；最后通过下发操作将 `card_status` 变更为 `sent`，向对应员工开放查阅权限。需要特别注意的是，批改台已完全废弃早期的 `sop_review_scores` 六维度打分机制，严禁在业务逻辑中重新引入该废弃打分结构。

## 4. 员工端：反馈查阅与复盘回传闭环
下发的反馈卡会实时呈现在员工工作台底部的反馈卡区域（对应组件 `feedback-card-section.tsx`）。当员工查阅标记为已下发（`card_status` 为 `sent`）的反馈时，系统会向 `/api/content/feedback/reply` 发送请求，更新反馈卡状态为 `viewed` 并记录查阅时间 `viewed_at`。

查阅之后，员工必须对主要问题和改进建议作出响应。员工可以在输入框中填写复盘总结，并选择回复状态为“收到”（`acknowledged`）或“申诉”（`disputed`）。提交复盘数据会向 `/api/content/feedback/reply` 接口发起 POST 请求，该接口在 `feedback_card_replies` 表中插入一条新的回复日志以保留历史修改记录，同时更新 `content_feedback_cards` 表中的 `employee_reply_status`、`employee_reply_text`、`employee_replied_at` 与 `employee_replied_by` 四个字段，从而形成完整的管理反馈与执行闭环。

## 5. 素材沉淀：分级评估与视频资产管理
在视频诊断过程中，组长可以将典型的视频案例手动沉淀为团队经验。通过点击 `content-detail-dialog.tsx` 中的“标记为经验”按钮，会向 `/api/admin/content-experience-marks` 接口发送 POST 请求。该操作将在 `content_experience_marks` 表中新增或更新经验标记，将视频归类为爆款案例（`hot_case`）、失败案例（`fail_case`）、开头问题（`opening_issue`）等不同经验类型。

视频资产管理台（对应路由 `/admin/videos`）依托数据加载器 `admin-videos-page.ts` 来呈现所有已提交视频的资产状态。系统基于 `src/lib/video-asset-library.ts` 中的 `buildVideoAssetRecord` 规则，动态评估每个视频的资产完整性。视频在资产库中存在三种完整性状态：完整（`complete`，标题、文案、24h快照、标签、分段皆全）、待补（`partial`，缺少部分必要资料）、缺失（`missing`，严重缺少必要资料）。

当且仅当一个视频的资料完整性状态为 `complete`，且在资产库中被人工评定并分配了资产等级（在 `videos` 表中配置 `asset_level` 字段为 S、A、B、C 之一）时，该视频在资产库中的入库状态（`library_status`）才会标记为“已入库”（`ready`），否则将保持为“待整理”（`pending`）。以此实现高质量、多维度标签化视频的规范化归档，为后续内容生产提供优质的参考素材。

## 6. 开发规范与维护守则
进行该链路维护与扩展时，开发者必须遵守以下技术底线。首先，本仓库的所有 SOP 及数据提交接口，若涉及定时任务、外部 Cron 或后台数据同步，必须使用 `SUPABASE_SERVICE_ROLE_KEY` 绕过 Row Level Security (RLS) 策略，而在前端发起的请求中必须严格依赖 RLS 权限判定。其次，严禁修改历史数据库迁移文件，任何表结构或权限策略的修改必须在 `supabase/migrations/` 下创建新的 SQL 迁移文件。最后，在进行业务状态流转或功能合并时，务必对照 `AGENTS.md` 的「废弃机制速查」清单，规避对 `sop_review_scores` 等历史废弃表的错误依赖，确保与线上真实运行的代码逻辑完全对齐。
