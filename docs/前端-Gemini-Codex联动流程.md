# 前端 Gemini + Codex 联动流程

适用场景：
- 前端页面改版
- 后台 UI / UX 优化
- 需要先出方案、再执行、再做代码核查

目标：
- Gemini 负责前端方案与实现
- Codex 负责代码核查
- 我负责把关节奏、纠偏方向、决定是否进入下一轮

## 一句话原则

先让 Gemini 出方案，不直接写代码。

我先看方案，帮你纠偏。

方案过了，再让 Gemini 写代码。

代码写完，再让 Codex 审查。

Codex 的审查结果再交给我，我来判断：
- 直接过
- 先补小问题
- 打回重做
- 继续下一批

## 角色分工

### 1. Gemini
- 先出方案
- 方案确认后再写代码
- 按批次推进，不要一口气重构整页

### 2. Codex
- 只审代码，不审纯方案
- 重点查能力回退、状态风险、边界问题、实现细节

### 3. 我
- 负责“大脑”
- 帮你判断方案能不能开工
- 帮你把 Codex 审查结果翻译成人话
- 决定下一步是补问题还是进下一批

## 推荐节奏

### 阶段 0：先对齐总约束

用途：
- 防止 Gemini 一开始就理解错页面视角
- 明确哪些能力不能删

这一步只要 Gemini 给总方案，不写代码。

### 阶段 1：Gemini 出某一批方案

规则：
- 一次只做一批
- 方案阶段不要写代码
- 必须明确：
  - 改哪些文件
  - 删什么
  - 保留什么
  - 为什么

你把 Gemini 的方案贴给我，我来判断：
- 通过
- 小修后通过
- 不通过

### 阶段 2：Gemini 按确认后的方案写代码

规则：
- 只改这一批允许改的文件
- 不扩大战场
- 不顺手重构无关逻辑

### 阶段 3：Codex 审代码

规则：
- 只在 Gemini 真写了代码之后才让 Codex 审
- 不要让 Codex 去审“纯方案”

你把 Codex 的审查结果贴给我，我来判断：
- 已通过
- 先补小问题
- 仍有阻塞问题

### 阶段 4：决定是否进入下一批

判断规则：
- 有阻塞问题：先修
- 只有轻问题：可补可过，我来帮你定
- 无阻塞问题：进入下一批

## 标准循环

以后前端任务默认按这个循环走：

1. 先让 Gemini 出总方案
2. 把方案贴给我，我帮你把关
3. 我给你新的 Gemini 提示词
4. Gemini 写这一批代码
5. 让 Codex 审查这一批代码
6. 把 Codex 审查结果贴给我
7. 我告诉你：
   - 这批过不过
   - 要不要补问题
   - 下一批怎么发
8. 继续下一轮

## 你发给 Gemini 的总控提示词模板

第一次开一个前端任务时，先发这个：

```text
你现在负责 DYData 某个页面的前端 UI/UX 优化。

要求你按“先方案、后执行、分批推进”的方式工作：

1. 先给总方案，不要直接写代码
2. 方案必须拆批次
3. 每一批都要明确：
- 改哪些文件
- 删什么
- 挪什么
- 保留什么
- 为什么
4. 我会先拿你的方案去确认，通过后你再写代码
5. 你写完代码后，我会再交给 Codex 做核查
6. 没确认前，不要直接开始实现

请先输出：
1. 总体诊断
2. 分批方案
3. 每一批涉及的文件
4. 风险点
```

## 你发给 Gemini 的“某一批方案提示词”模板

当进入某一批时，用这个：

```text
现在只做第 X 批。

要求：
1. 先给方案，不要直接写代码
2. 只允许讨论这一批，不要扩展到下一批
3. 必须明确回答：
- 这一批的目标
- 要改哪些文件
- 每个文件怎么改
- 哪些逻辑不能碰
- 改完后的页面效果会怎样

注意：
- 不要写代码
- 等我确认后你再执行
```

## 你发给 Gemini 的“开始写代码提示词”模板

当我说这批方案通过后，再发这个：

```text
这版方案通过，现在开始写代码。

要求：
1. 只改这批允许改的文件
2. 不要扩大战场
3. 不要新增依赖
4. 不要改后端接口
5. 不要改不属于这批的能力逻辑

写完后请告诉我：
1. 具体改了什么
2. 为什么这样改
3. 哪些能力保持不变
4. 还剩什么风险
5. 是否建议交给 Codex 审查
```

## 你发给 Codex 的代码核查提示词模板

Gemini 真写完代码后，再发这个给 Codex：

```text
请审查 DYData 项目当前工作区里这一批前端改动。

重点不是代码风格，而是有没有在做 UI 调整时误伤现有能力和状态逻辑。

请重点检查：
1. 有没有改到不该动的文件或逻辑
2. 有没有把页面视角理解错
3. 有没有破坏保存、测试、切换、删除等状态链路
4. 有没有假反馈
5. 有没有边界问题、响应式退化、无障碍缺口
6. 有没有 lint / 类型 / 实现细节问题

输出格式：
1. 先列问题，按严重程度排序
2. 每条写清楚文件和行号
3. 每条标注属于哪类：
- 能力回退
- 状态风险
- UI 歧义
- 边界问题
- 实现细节
4. 如果没有严重问题，明确写：
“未发现阻塞性问题”
5. 最后补一句：
“这批修改是否符合本批目标”
```

## 你发给我的两种固定话术

### 1. Gemini 只给方案时

直接对我说：

```text
这是 Gemini 的第 X 批方案，你帮我把关。
请直接告诉我：
1. 能不能过
2. 哪些点要纠偏
3. 下一条我该发给 Gemini 的提示词
```

### 2. Codex 已经审完代码时

直接对我说：

```text
这是 Codex 对第 X 批代码的审查结果，你帮我判断：
1. 这批能不能过
2. 还要不要补问题
3. 如果要补，直接给我发给 Gemini 的提示词
4. 如果能过，直接给我下一批提示词
```

## 什么时候不要让 Codex 出手

以下情况不要找 Codex：
- Gemini 还没写代码，只出了方案
- 你只是想判断思路是否合理
- 你只是想让我帮你压缩提示词

这几种情况直接找我就行。

## 什么时候必须让 Codex 出手

以下情况必须让 Codex 审一次：
- Gemini 已经改了代码
- 涉及状态链路
- 涉及自动保存
- 涉及多文件 TSX 改动
- 涉及“能力不能回退”的页面

## 判定标准

### 可以直接进入下一批
- Codex 明确说“未发现阻塞性问题”
- 只剩低级别文案、无障碍、lint warning 之类的小尾巴
- 页面关键能力没有受损

### 先补问题再说
- 有未保存内容被冲掉
- 有保存状态假成功
- 有当前视角误变成限制作用域
- 有接口契约被误改
- 有明确 lint / 类型 / 构建阻塞

## 这份文件怎么用

以后你只要对我说：

```text
按 docs/前端-Gemini-Codex联动流程.md 走
```

我就会默认按这份流程来：
- 先看方案
- 再把关
- 再让 Gemini 执行
- 再让 Codex 核查
- 最后我判断下一轮

## 当前建议

先把这份文件当作项目内标准流程使用。

如果后面你发现：
- 多个项目都会复用
- 每次都想跨项目调用
- 想让系统自动触发

那时再把它升级成全局 skill 更合适。
