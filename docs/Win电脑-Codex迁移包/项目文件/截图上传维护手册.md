# 截图上传维护手册

这份文档只管一件事：`/dashboard` 里的“截图上传 + 识别 + 提交”链路。

这条链路是 DYData 最关键的入口，出问题会直接影响员工提交、日报落库、视频标签、后续分析，所以后续谁改这里，都先看这份文档。

## 1. 这条链路到底包含什么

用户看到的是“上传截图”，但实际上一共有 4 段：

1. 前端上传截图并调用 OCR
2. OCR 返回结构化识别结果，前端让用户确认
3. 用户补齐标题、文案、话题标签、内容标签
4. 提交到 `/api/video-submit`，一次写入：
   - `videos`
   - `video_metrics_snapshots`
   - `daily_reports`
   - `video_tags`

一句话：

- `/api/ocr-screenshot` 负责“看图识别”
- `/api/video-submit` 负责“最终入库”

## 3. 当前正确行为

### 槽位规则

- `screenshot_1`：必传
- `screenshot_2`：必传
- `screenshot_3`：选传

状态机要求：

- 必传截图没传，不能提交
- 必传截图识别失败，不能提交
- 必传截图还没确认，不能提交
- 标题、文案、内容标签没补齐，不能提交
- 话题标签没选，不能提交

### 内容标签规则

- 最多 3 个
- 每个标签是一个独立词
- 前端会去空格、去重
- 数据库里按 3 条独立 `video_tags` 保存
- 不能把 3 个词拼成 1 条字符串再存

### 标签维度规则

单值维度，同一个视频只能有 1 条：

- `题材`
- `表达形式`
- `CTA类型`
- `内容结构`
- `目标受众`
- `话题`

多值维度：

- `关键词`

也就是：

- 同一个视频可以有 3 条 `关键词`
- 但同一个视频不能有 2 条 `题材`

## 4. 这条链路最容易炸的地方

### 4.1 OCR 识别没问题，但提交时报错

重点先查：

- `/api/video-submit`
- `video_tags`
- `daily_reports`

最常见原因不是图片上传本身，而是最终入库阶段写炸了。

### 4.2 改了数据库约束，但代码还在用旧 `upsert`

这是这条链路最危险的坑之一。

真实事故：

- 之前 `video_tags` 用的是 `onConflict: "video_id,tag_dimension"`
- 后来为了支持“一个视频 3 个关键词”，把数据库唯一规则改成：
  - 非关键词：`(video_id, tag_dimension)` 唯一
  - 关键词：`(video_id, tag_dimension, tag_value)` 唯一
- 如果代码还继续写老 `upsert`，就会报：
  - `there is no unique or exclusion constraint matching the ON CONFLICT specification`

现在的正确做法：

- 不再依赖这类旧 `upsert`
- 改成“删旧标签，再插入新标签”

涉及文件：

- `src/app/api/video-submit/route.ts`
- `src/app/api/video-tags/confirm/route.ts`
- `src/app/(app)/admin/videos/video-detail-dialog.tsx`

### 4.3 同一批 `upsert` 命中同一行两次

真实事故：

- 一个视频提交 3 个关键词
- 代码把 3 个关键词都当成同一个维度 `"关键词"` 去做同批 `upsert`
- 数据库会报：
  - `ON CONFLICT DO UPDATE command cannot affect row a second time`

现在的正确规则：

- 3 个关键词必须是 3 条独立记录
- 不能用“同批相同唯一键”的写法

### 4.4 只改代码，不推数据库 migration

这是第二个高频坑。

截图上传链路不是纯前端功能，它经常同时依赖：

- 前端表单逻辑
- API 校验
- Supabase 表结构 / 约束

如果只 push 代码、不 push Supabase migration，就会出现：

- 本地正常
- 线上数据库规则还是旧的
- 用户一提交就报错

所以涉及截图上传的改动后，必须同时确认：

1. 代码已 push 到 GitHub
2. Vercel 已部署成功
3. Supabase migration 已推远端

## 5. 当前数据库规则备忘

`video_tags` 当前跟截图上传最相关的是这两条：

1. 允许的维度包含：
   - `题材`
   - `表达形式`
   - `CTA类型`
   - `内容结构`
   - `目标受众`
   - `话题`
   - `关键词`

2. 唯一规则：
   - 非关键词：`video_id + tag_dimension`
   - 关键词：`video_id + tag_dimension + tag_value`

对应 migration：

- `043_video_tags_keyword_multi_value.sql`

## 6. 改这里时必须遵守的发布顺序

### 只改前端展示

比如：

- 文案
- 样式
- 交互提示
- 槽位视觉

顺序：

1. 本地验证
2. push `main`
3. 等 Vercel 部署
4. 线上手测

### 改提交流程 / 标签保存 / 数据结构

比如：

- 内容标签规则
- `video_tags` 存储逻辑
- `daily_reports` 写入逻辑
- OCR 返回字段结构

顺序必须是：

1. 新增 migration
2. 本地 lint / test
3. 先推 Supabase migration 到远端
4. 再 push 代码到 `main`
5. 等 Vercel 部署
6. 用真实截图跑一次完整提交

## 7. 每次改完后的必测清单

至少测这 8 项：

1. 只上传 1 张图，页面应提示不能提交
2. 上传 2 张必传图，识别成功后可继续填写
3. OCR 失败时，页面应明确提示，可重试或手填
4. 标题为空时不能提交
5. 文案为空时不能提交
6. 内容标签为空时不能提交
7. 选 3 个内容标签后，提交成功
8. 提交后数据库里能看到：
   - 1 条 `videos`
   - 1 条 `video_metrics_snapshots`
   - 1 条 `daily_reports`
   - 1 条 `话题`
   - 1 到 3 条 `关键词`

额外建议测：

9. 重复编辑同一条视频，标签会正确覆盖旧值
10. 后台视频详情里修改 AI 标签，不报错

## 9. 出故障时的排查顺序

不要上来就猜，按这个顺序查：

1. 先看用户报错是在“上传图片”阶段还是“最终提交”阶段
2. 如果是上传阶段：
   - 查 `/api/ocr-screenshot`
   - 查 AI 功能 `ocr_screenshot` 是否被禁用
   - 查图片格式 / 大小 / 内容类型
3. 如果是最终提交阶段：
   - 查 `/api/video-submit`
   - 查 `video_tags` / `daily_reports` / `videos`
4. 如果报 `ON CONFLICT`：
   - 先怀疑数据库唯一规则和代码写法不一致
5. 如果本地好、线上坏：
   - 先查 Vercel 是否部署到最新 commit
   - 再查 Supabase migration 是否已推远端

## 10. 已知事故记录

### 事故 A：关键词多条导致 `ON CONFLICT DO UPDATE ... affect row a second time`

现象：

- 上传时报错
- 前端看起来像“截图上传坏了”

根因：

- 3 个关键词在一次 `upsert` 里命中了同一个唯一键

修法：

- `关键词` 改为数据库层面支持多条
- 代码层不再把多个关键词塞进同一个旧 `upsert`

### 事故 B：数据库约束已改，但旧 `ON CONFLICT` 代码残留

现象：

- 报错：`there is no unique or exclusion constraint matching the ON CONFLICT specification`

根因：

- 数据库已经不是原来的唯一规则
- 代码还在拿旧冲突键调用 `upsert`

修法：

- 把相关位置全部改成“删旧再插入”

## 11. 后续谁再改这里，必须记这 3 句话

1. 不要把“截图上传问题”只当成前端问题，它常常是最终入库问题。
2. 不要只 push 代码不 push migration。
3. 不要改了 `video_tags` 约束后还沿用旧 `upsert` 写法。
