# DYData 全站模块地图

> 本文件回答一个问题：这个网站到底由哪几块组成，每块是活的还是废的，它们靠什么数据连在一起。
> 用途：任何 AI 接手前，先在这里定位"我要改的东西属于哪个模块、会牵动哪些页面和表"，避免从旧文档里刨出废弃功能重做。
> 与代码冲突时以代码为准，并当场修正本文件。
> 数据来源：`docs/plans/全站模块扫描-底稿.md`（29 页面 / 147 接口全量静态扫描 + 调用链核对）。
> 最后核对代码日期：2026-07-16

## 〇、怎么看这份地图（给阿禅）

把网站想成一栋楼，分四层：

- 门厅层：没登录也能进的（首页、登录注册）。
- 员工层：普通成员登录后用的（看板、成长、避坑案例、过渡视频审核、文案工具）。
- 管理层：你和管理员用的后台（视频复盘、视频素材、履约、分析、成员权限、AI 配置）。
- 地基层：所有楼层共用的水电管线（通知、提醒、定时任务、权限）。

每个模块下面标了三样你能看懂的东西：**它干什么、是不是还在用、坏了会连累谁**。术语一律就地解释。

## 二、员工层模块明细

只列核心表和关键接口，全量接口见底稿。核心表 = 这个模块坏了/慢了，先查这几张表。

### 数据看板 dashboard（核心）
- 页面：`/dashboard`
- 加载器：`loaders/dashboard-page`（首屏一次查好）
- 核心表：daily_reports（日报）、accounts（账号）、profiles（用户）、exemption_request/exemption_grant（豁免）；视频提交可通过 videos.topic_id 关联选题库子题
- 关键接口：`/api/video-submit`（提交日报；异常视频在视频、快照、日报成功后自动生成一条待审核避坑案例；可接收 `topic_id` 关联子题）、`/api/dashboard/submission-calendar`（本月提交/豁免状态）、`/api/dashboard/watch-overview`（看态趋势+排行榜）、`/api/dashboard/trend`（趋势图）、`/api/dashboard/leaderboard`（排行榜）、`/api/dashboard/content-feedback-cards`（批改卡片）
- 连累关系：daily_reports 是全站最核心的表，看板、成长、分析、履约、告警全依赖它，动它影响面最大。

### 成长分析 growth（核心）
- 页面：`/growth`
- 加载器：`loaders/growth-page` 的 `loadGrowthPageContract`，服务端一次返回真实指标、可信度、规则诊断、最小实名对标和脚本状态。
- 核心表：daily_reports、accounts、profiles、content_item、script_document/script_segment（脚本）；`ai_insight_result` 只留给后续 AI V2，不参与当前规则契约。
- 关键接口：当前页面无前端补拉接口；`/api/growth/ai-insight` 留给后续 AI 升级层，不参与 V1 规则页。
- 前端：`growth-data-container.tsx` 把真实契约传给 `growth-client.tsx`；页面覆盖空数据、低样本、正常样本与各模块独立降级。
- 降级：脚本表缺数据、缺表或缺列时，契约返回 `scriptBreakdown.state = "none"`，不拖垮其余体检数据。

### 视频审核 video-review（归档入口，只读）
- 页面：`/video-review` 及 `/submit`、`/manage`、`/exemption`、`/archive`
- 加载器：`loaders/video-review-page` 负责 `/video-review` 首页；`/submit`、`/exemption`、`/archive` 仍各自页面取数；`/manage` 直接重定向到 `/admin/content`
- 核心表：publish_drafts（发布草稿）、work_submissions（作品提交）、exemption_request、profiles、teams、groups；另用 `get_production_dashboard` 等 RPC
- 关键接口：`/api/work-submissions`（GET 历史读取保留，POST/DELETE 已返回归档）、`/api/publish-drafts/*`（读取队列/历史保留，创建/重提/审批/上传已返回归档）、`/api/work-screenshots/upload`（已返回归档）、`/api/exemptions/*`（全局豁免链路，继续保留读写）
- 当前状态：前端已统一加归档提示条并禁用写入按钮/表单；后端已封存 video-review 专属写入接口，历史数据和读取接口保留。本页已从员工主导航移除，仅保留直接访问。注意：`/api/exemptions/apply` 被 `/dashboard` 工作台复用，不能随 video-review 归档一并禁用。

### 避坑案例 violations
- 页面：`/violations`（案例库）、`/violations?view=manage`（审核工作台）、`/violations/[id]`（详情）、`/violations/submit`（上传话术）。用户可见名统一为「避坑案例」，`violations` 只保留为历史技术路由名。
- 加载器：无独立加载器，页面内直接查
- 核心表：violation_cases（违规话术主表；异常视频以 `source_video_id` 幂等写入 `status=submitted`，并保留来源元数据和管理员违规段落标注；员工视角只读 `status=verified` 且未删除数据）、knowledge_cases（高转化/避坑案例主表，员工新提交为 `status=submitted`）、videos（异常案例的来源，历史详情兼容回退）、violation_events、violation_reason_tags、violation_case_visual_tags、visual_tags、script_usage_records；管理视角走 `case_library_*` RPC
- 关键接口：`/api/violations`(违规话术上传写 violation_cases；高转化上传写 knowledge_cases；列表按 purpose 从 violation_cases 查)、`/api/violations/[id]`(详情优先读 violation_cases/knowledge_cases，返回违规段落标注；历史异常视频可回退；PATCH 由 `manage_violations` 管理员编辑已发布话术正文；DELETE 由 `manage_violations` 管理员下架已发布话术并返回撤销快照)、`/api/violations/review/restore`(恢复审核/下架快照，兼容 violation_cases 与 knowledge_cases)、`/api/violations/[id]/review`(仅 `manage_violations` 管理员审核并可保存违规段落标注)、`/api/conversion-hub/*`(转化数据)

### 文案工具 content-tools
- 页面：`/content-tools`、`/content-tools/rewrite`（`rewrite-v3` 路由及旧 `components/content-tools/rewrite/` 组件目录已清理）
- 加载器：`content-tools-page`
- 核心表：一大片 rewrite_* 表（rewrite_documents/conversations/messages/skills/workflows 等）+ ai_* 配置表
- 关键接口：`/api/content-tools`、`/api/content-tools/rewrite/*`、`/api/rewrite/*`（其中 `/api/rewrite/generate` 是当前核心生成链路）。两套 API 仍并存，具体减法需重新沟通后再做。
- 当前状态：正式入口为 `/content-tools/rewrite`，渲染 `components/content-tools/rewrite-v3/RewriteWorkbenchV3`；旧的 `components/content-tools/rewrite/` 组件目录已不存在。2026-07-16 已按阿禅要求撤回前端保守减法和旧 chat 接口归档，现状先完整保留，后续重新沟通方案后再处理。

### 选题库 topics（后端已建，前端待接）
- 页面：前端由 Antigravity 后续实现；本次只交付后端接口。
- 加载器：暂无页面加载器；工作台可按需调用 `/api/topics/active` 获取最近活跃。
- 核心表：topics（母题）、topic_groups（分组）、sub_topics（子题）、sub_topic_claims（人×子题认领状态）、videos.topic_id（作品关联到子题）。
- 关键接口：`/api/topics/pool`（选题池）、`/api/topics/active`（最近活跃）、`/api/topics/sub-topics` 和 `/api/topics/sub-topics/[id]`（子题 CRUD）、`/api/topics/sub-topics/[id]/claim|start-scripting|return`（认领流转）、`/api/topics/sub-topics/[id]/works`（作品与文案收纳）、`/api/topics/sub-topics/suggest`（关键词兜底匹配）。
- 当前边界：只做“存+选”和作品关联；不做 AI 评分、热点抓取、复盘数据接口、文案助手联动、认领超时释放和管理端入口调整。

- 核心表：主接口 `/api/admin/cockpit/queue-overview` 汇总 content_feedback_cards/daily_reports/profiles/videos/violation_cases；告警查一大片表
- 关键接口：`/api/admin/cockpit/*`(待办队列)、`/api/admin/dashboard-alerts`（AI 告警，**有 60 秒缓存**）、`/api/admin/sidebar-badges`（红点，**有 60 秒缓存**）

### 成员/团队/权限 admin/modules
- 页面：`/admin/modules`
- 加载器：`admin-modules`
- 核心表：profiles、teams、groups、admin_actions
- 关键接口：`/api/admin/execute-tool`(执行管理动作)、`/api/admin/modules/team-management`、`/api/admin/member-ai-suggestion`

### AI 配置中枢 admin/ai-config
- 页面：`/admin/ai-config`（旧入口 ai-channels/ai-features 已重定向来此）
- 核心表：ai_channels、ai_providers、ai_provider_keys、ai_provider_key_models、ai_feature_config、ai_feature_bindings、rewrite_model_* 等全套 AI 配置表
- 关键接口：`/api/admin/ai-config`、`/api/admin/ai-rewrite`、`/api/admin/ai-unified/overview`



### 地基层
- 通知系统：表 notifications；接口 `/api/notifications/*`；全站红点和消息中心。
- 提醒与告警：`/api/remind`(催交)、`/api/smart-alert`(智能预警)——多为手动或外部触发，不在 Vercel cron 里，不能当自动在跑。
- 定时任务（Vercel cron 仅两个）：`/api/supabase-keepalive` 每天保活数据库、`/api/notifications/cleanup` 每天清过期通知。

### 缓存现状（重要）
缓存分两种，别一句"没缓存"带过：
- 页面/接口级缓存：没用 Next 的 `unstable_cache`/`revalidate`/react-query；只有 4 个接口做了"进程内 60 秒内存缓存"（`/api/admin/content/list`、`/api/admin/dashboard-alerts`、`/api/admin/sidebar-badges`、`/api/admin/cockpit/queue-overview`）。这一层用得少，是主要优化空间。
- 请求级去重：已用 React `cache()` 对权限查询做了单次请求内去重（`src/lib/permissions.ts`、`current-permission-context.ts`），同一次请求里权限不会重复查。所以"仓库基本没缓存"的说法不准，权限链路已有去重。

## 五、接口存活判断（⚠️ 不要凭本文件删接口）

早期版本这里列过一份"疑似闲置接口"，是用"前端 grep 搜不到调用"这一种方法推断的，已证实不可靠——其中 `/api/ai/insight/period`（后台分析页在调）、`/api/admin/first-screen-monitor`、`/api/admin/cockpit/summary`、`/api/publish-drafts/approved`、`/api/publish-drafts/queue`、`/api/fulfillment/appeal/submit` 都是真实在用或有正式 route 和测试的，被误判成了闲置。该清单已作废。

结论与规矩：
- 本文件不提供"可删接口清单"。判断一个接口能否删，必须逐个查：是否有 route 定义、是否有前端调用、是否被其他接口内部调用、是否有测试、是否外部/手动触发。少查一项都不能判死。
- 静态 grep 搜不到前端调用，只能得出"前端没直接调"，得不出"闲置"。很多接口是被其他接口内部调用、cron 手动触发、或外部系统调的。
- 真要清理接口，走"代码治理"领域专项任务，逐个验证并留证据，禁止批量删。

## 六、维护规则

- 新增/删除页面或接口、改模块归属 → 当场更新第一节总览表和对应模块明细。
- 判定某功能废弃 → 移入第四节，并在代码里做重定向或删除，不留半死不活状态。
- 与代码冲突一律以代码为准并立即修本文件。
- 本文件管"有哪些模块、活还是死"；性能红线和排查流程在 `docs/架构与性能总纲.md`，两者配合使用。
- 顶部"最后核对代码日期"每次核对后更新。
