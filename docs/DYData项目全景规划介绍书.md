## 一、项目概览

### 1.1 项目定位

**DYData** 是一个面向抖音内容团队的**全链路数据管理与 AI 辅助平台**。核心解决内容生产团队的三大痛点：

| 痛点 | 解决方案 |
|------|---------|
| 数据上报不及时、口径不统一 | 标准化日报填报 + 截图 OCR 自动识别 |
| 内容质量缺乏系统性复盘 | 管理员批改台 + AI 辅助诊断 + 反馈闭环 |
| 优秀素材无法有效沉淀 | 视频资产分级入库 + 转化话术案例库 |

### 1.2 核心数据

| 维度 | 详情 |
|------|------|
| 域名 | dydata.cc / dydata.vercel.app |
| Supabase 实例 | gcrhhxaopomtposmahsw.supabase.co（新加坡） |
| 代码仓库 | git@github.com:1305085564-hue/dydata.git |
| 部署平台 | Vercel（push main 自动部署） |
| AI 模型 | Claude Sonnet 4（通过 aiapikey.net 代理） |

## 五、核心业务链路

### 5.1 数据填报 → 诊断 → 复盘全链路

```
员工端                          管理端                    系统自动
───────                        ───────                  ───────

① T+0 提交视频基本信息
   └→ videos 表
   └→ video_metrics_snapshots     ────────────→ 标记异常视频
                                                  (爆款系数算法)

② T+1 补充 24h 数据
   └→ 截图 OCR 识别 ← /api/ocr-screenshot
   └→ AI 自动打标签 ← video_tags 表
   └→ daily_reports 表

③                              ④ 批改台 /admin/content
                                  └→ 查看异常视频详情
                                  └→ AI 辅助诊断
                                  └→ 填写反馈卡
                                     ├ 草稿 → 确认 → 下发
                                     └ content_feedback_cards

⑤ 查阅反馈
   └→ 反馈卡变为 viewed

⑥ 复盘回复
   └→ 员工填写复盘
   └→ 状态：acknowledged / disputed
   └→ feedback_card_replies

⑦                              ⑧ 标记为经验 (content_experience_marks)
                                 ⑨ 视频资产分级 (S/A/B/C)
                                    └→ 入库 (ready) / 待整理 (pending)
```

### 5.2 违规话术审核闭环

```
员工提交话术                    管理员审核                  知识沉淀
──────────                    ──────────                ────────

① 6步向导提交
  └→ 类型选择
  └→ 核心内容
  └→ 违规详情
  └→ 转化数据
  └→ 审核确认
  └→ 提交

②                            ③ 审核队列
                                └→ 通过(verified) / 驳回(rejected)
                                └→ 填写结论 + 建议动作

④                            ⑤ 优质案例 → 知识库
                                └→ 标记为知识案例
                                └→ 丰富验证
```

### 5.3 文案改写对话流

```
用户操作                        AI 响应                      数据持久化
────────                      ────────                     ────────

① 输入原文 / 选择技能          ② AI 生成首版文案
  └→ / 技能选择器                 └→ 流式输出                  └→ conversations 表
  └→ 激活技能标签                 └→ 段落级渲染                └→ documents 表

③ 多轮对话优化                  ④ AI 根据上下文重写
  └→ "开头再吸引一点"              └→ 段落 diff               └→ paragraphs 表
  └→ "结尾加CTA"                                            └→ revisions 表

⑤ 局部锁定 + 其他段落重写       ⑥ 仅修改非锁定段落
  └→ 锁定段落 2                  └→ Timeline Diff 可视化

⑦ 定稿导出
  └→ 一键复制完整文案
```

## 八、设计规范与美学体系

### 8.1 设计哲学

> 让界面同时做到"顺手"和"好看" — 清晰、精准、有秩序的舒适。

**验收标准**：页面截图缩小到 25%，仍能分清 3 层以上的视觉阶梯。

### 8.2 核心设计原则

| 原则 | 说明 |
|------|------|
| 主画布通透 | 主体内容直接铺在背景上，禁止卡片包裹，仅外来信息用轻卡片 |
| 嵌套深度上限 | 禁止卡片嵌套超过一层 |
| 色温统一 | 全站只用 Zinc 灰阶，亮度差 2%-3% |
| 毛玻璃质感 | 导航栏使用 `bg-zinc-50/80 backdrop-blur-md` |
| 微交互 | 物理弹簧动效、数字动画、hover 微反馈 |
| 双字体 | 标题衬线体 + 数据 tabular-nums |

### 8.3 设计令牌

- 圆角：`--radius-card: 16px`
- 阴影 3 级：light / medium / heavy
- 主色调：`#D97757`（暖陶土色）
- 背景：骨白 Zinc-50 基调
- 动效：spring(stiffness: 260, damping: 20) 统一入场

### 8.4 重构蓝图（三批次）

| 批次 | 范围 | 核心目标 |
|------|------|---------|
| 批次一 | App Shell + NavBar | 全局架构与导航重塑，"20% 聚光灯留白" |
| 批次二 | Dashboard 工作台 | 沉浸式"知性书桌"体验 |
| 批次三 | Analytics & Admin | 高密度数据面板，双字体系统 |
